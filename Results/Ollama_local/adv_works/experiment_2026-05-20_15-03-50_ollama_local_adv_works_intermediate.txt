This is a critical architectural shift. The provided schema is a classic **Third Normal Form (3NF)** design optimized for ACID compliance and data integrity, typical of OLTP (Online Transaction Processing) systems.

However, for a **read-heavy workload** (e.g., dashboards, self-service reporting, "Who is in Sales?", "Top 10 Products by Region"), the current schema suffers from:
1.  **Join Explosion:** Queries requiring Employee + Department + Shift + History + Payroll + Address + Person data require a massive join tree.
2.  **Index Bloat:** Every join column must be indexed, and composite indexes on many-to-many relationships (like `Product` + `Photo`) are expensive to maintain.
3.  **Query Latency:** Filtering across multiple tables (e.g., `WHERE status = 'Shipped' AND territory = 'West' AND product = 'Bike'`) requires scanning multiple large tables and merging results.

Below is the **Workload-Driven Analysis** followed by the **Denormalized Physical Schema**.

---

### 1. Workload-Driven Analysis

#### A. High-Cost Join Paths & Operations
Based on typical BI and HR reporting patterns in this dataset:

1.  **The "Employee Profile" Query:**
    *   *Current Path:* `Employee` JOIN `Person` JOIN `Address` JOIN `Department` JOIN `EmployerDepartmentHistory` (current row).
    *   *Problem:* Requires 4-5 joins. If `Department` has a foreign key to `Shift`, that's 5 joins. Filtering by `currentflag` and joining lookup tables (`DepartmentName`, `ShiftName`, `AddressCity`) creates a "spider web" of joins.
    *   *Impact:* High latency as the database must resolve multiple foreign keys and filter out historical records.

2.  **The "Sales Performance" Query:**
    *   *Current Path:* `SalesOrderHeader` JOIN `SalesOrderDetail` JOIN `Product` JOIN `ProductCategory` JOIN `SalesTerritory` JOIN `SalesPerson`.
    *   *Problem:* This is the most common analytical query. Joining `SalesOrderDetail` (high cardinality) with `Product` (high cardinality) and `SalesTerritory` creates a massive result set before aggregation.
    *   *Impact:* Sorting and grouping on these joined results is computationally expensive.

3.  **The "Inventory Status" Query:**
    *   *Current Path:* `ProductInventory` JOIN `Product` JOIN `Location` JOIN `Location` (lookup).
    *   *Problem:* Requires looking up location names and product details for every line item in the inventory.

#### B. Denormalization Candidates
To optimize reads, we will duplicate data where the join cost outweighs the write penalty:
*   **HR:** Duplicate `DepartmentName`, `ShiftName`, `AddressCity`, and `FirstName`/`LastName` directly on the `Employee` table (and `EmployerDepartmentHistory` for current state).
*   **Sales:** Duplicate `ProductName`, `ProductCategoryName`, `TerritoryName`, and `SalesPersonName` on the `SalesOrderHeader` and `SalesOrderDetail` tables.
*   **Production:** Duplicate `ProductName` and `UnitMeasureName` on `ProductInventory` and `WorkOrder`.
*   **Aggregates:** Create pre-calculated summary tables for `Sales by Region`, `Inventory by Location`, and `Employee Count by Department`.

---

### 2. Denormalization Design Strategy

#### A. Star-Schema / Wide Tables
We will create "Wide" tables that contain all columns needed for a single-row lookup, eliminating the need for joins.

#### B. Pre-Computed Aggregates (Fact Tables)
Instead of calculating `SUM(SalesOrderDetail.UnitPrice)` on the fly, we will materialize these aggregates.

#### C. Index Strategy
*   **Covering Indexes:** We will use `INCLUDE` clauses to cover the columns used in `WHERE`, `GROUP BY`, and `ORDER BY` clauses, ensuring the database reads only the index (no table lookup).
*   **Composite Indexes:** Aligned to the most frequent filter patterns (e.g., `TerritoryID`, `Status`, `OrderDate`).

#### D. Partitioning
*   **Sales Orders:** Partition by `OrderDate` (Year/Month) to manage growth and speed up range scans.
*   **Transaction History:** Partition by `TransactionDate`.

---

### 3. New Physical Schema (DDL)

This DDL creates a hybrid schema:
1.  **Core Transaction Tables:** Retained for writes (with added indexes).
2.  **Read-Optimized Views/Tables:** Denormalized tables for fast reads.
3.  **Materialized Aggregates:** Pre-calculated summaries.

SELECT 
    e.businessentityid,
    e.jobtitle,
    e.birthdate,
    e.hiredate,
    e.maritalstatus,
    e.gender,
    e.vacationhours,
    e.sickleavehours,
    e.currentflag,

    a.addressline1,
    a.city,
    a.stateprovincename
FROM humanresources.employee e
JOIN humanresources.department d ON e.departmentid = d.departmentid
JOIN humanresources.shift s ON e.shiftid = s.shiftid
JOIN person.person p ON e.businessentityid = p.businessentityid
JOIN person.address a ON e.addressid = a.addressid;

-- View: Active Employees by Department
CREATE OR REPLACE VIEW humanresources.vw_dept_headcount AS
SELECT 
    d.departmentid,
    d.departmentname,
    COUNT(DISTINCT e.businessentityid) AS employee_count,
    AVG(e.vacationhours) AS avg_vacation_hours
FROM humanresources.department d
LEFT JOIN humanresources.employee e ON d.departmentid = e.departmentid AND e.currentflag = true
GROUP BY d.departmentid, d.departmentname;
