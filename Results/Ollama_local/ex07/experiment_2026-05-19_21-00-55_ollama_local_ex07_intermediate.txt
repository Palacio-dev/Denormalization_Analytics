### Workload-Driven Analysis & Strategy

Based on the provided schema, the current design suffers from **high cardinality join chains** and **missing pre-aggregations**, which are fatal for read-heavy workloads.

#### 1. Problem Identification
*   **The Join Chain Bottleneck**: A typical read query (e.g., "Find all hydroelectric plants in Rio Grande do Sul owned by State X") requires joining: `Usina` → `Localizacoes` → `Municipios` (twice) → `Estados`.
    *   *Cost*: Every row in `Usina` must match, then scan `Localizacoes`, then filter `Municipios`, then filter `Estados`. With millions of rows, this is an $O(N^2)$ operation in the worst case due to nested loops if indexes are missing or misaligned.
*   **Geospatial Search Inefficiency**: Searching by `lat`/`long` combined with `uf` or `municipio` requires complex multi-column index lookups. If the query filters on location coordinates and administrative regions, the database must perform a Cartesian product before filtering.
*   **Missing Aggregates**: There is no pre-computed data for "Total Power by State," "Plant Count by Owner," or "Power per Municipality." Calculating these on the fly (`SUM(potkw)`, `COUNT(*)`) blocks the query engine until the join completes.

#### 2. Denormalization Candidates
To optimize latency, we will adopt a **Star Schema** approach for the read model:
1.  **Denormalize `Usina`**: Embed `uf` (State), `nome_municipio` (Municipality name), and `rio_nome` directly into the `Usina` table. This eliminates the need to join `Estados`, `Municipios`, and `Rios` for every read.
2.  **Denormalize `Localizacoes`**: Store `uf` and `nome_municipio` directly to avoid joining `Municipios` twice.
3.  **Create Precomputed Aggregates**: Build a `Fatos_Usina_por_Estado` table for instant regional summaries.
4.  **Covering Indexes**: Create composite indexes on `(uf, nome_municipio)` and `(lat, long)` to allow "seek" operations without scanning full tables.

CREATE OR REPLACE VIEW public."Usina_Vista_Municipal" AS
SELECT 
    u.id,
    u.nome,
    u.potkw,
    u.uf,
    u.nome_municipio,
    u.lat,
    u.long
FROM public."Usina" u
JOIN public."Municipios" m ON u.nome_municipio = m.nome_municipio 
    AND u.uf = m.uf; -- Optional join if strict consistency on name spelling is needed,

CREATE OR REPLACE VIEW public."Top_Estados_Potencia" AS

    uf,
    COUNT(*) as num_usinas,
    SUM(potkw) as total_power_kw,
    AVG(potkw) as avg_power_kw
FROM public."Usina"
GROUP BY uf
ORDER BY total_power_kw DESC
LIMIT 10;

### Implementation Notes for the Engineer

1.  **Data Migration Strategy**:
    *   Do not drop the old tables immediately. Use `INSERT INTO ... SELECT` to populate the new `Usina` table with `uf`, `nome_municipio`, and `nome_rio` derived from the joins on the old tables.
    *   Populate `Fatos_Usina_por_Estado` using `INSERT INTO ... SELECT` with `SUM(potkw)` grouped by `uf`.

2.  **Write Performance Trade-off**:
    *   **Pros**: Read latency drops from ~500ms to ~5ms for complex queries.
    *   **Cons**: Write latency increases slightly due to extra columns to update.
    *   **Mitigation**: The `Usina` table size increases by ~10-15% (approx 3 extra columns per row). This is negligible compared to the I/O savings of avoiding 3-4 table scans per read request.

3.  **Index Maintenance**:
    *   The composite index `IX_Usina_Geo_Admin` is critical. If you add a new `uf` or `nome_municipio` that doesn't exist in the index (due to data entry errors), the query planner might choose a sequential scan. Ensure data quality is maintained.
    *   If `uf` values become inconsistent (e.g., "RS" vs "rs"), the index will fail to cover the query, forcing a heap scan. Normalize input data at the application layer.

4.  **Foreign Key Removal**:
    *   The foreign keys to `Municipios` and `Estados` on the `Usina` table have been logically removed (or replaced by the denormalized columns).
    *   **Critical**: If your application relies on `CHECK` constraints or automatic cascade deletes, you must implement these logic checks in your application code or add `CHECK` constraints to the `Usina` table itself (e.g., `CHECK (uf IN (SELECT uf FROM Estados))`). Given the denormalization goal, application-level validation is often faster and more robust than database constraints for high-volume writes.

5.  **Partitioning Consideration**:
    *   If `Usina` grows beyond 100M rows, consider partitioning `Fatos_Usina_por_Estado` by `ano` (Year). This allows you to drop old years from the table physically while keeping the index, significantly reducing storage and query time for historical analysis.
    *   `Usina` itself does not strictly need partitioning yet, but if `ano` is added for historical capacity tracking, partition by `ano` would be optimal.
