### 1. Workload-Driven Analysis

In a domain managing Power Plants (`Usinas`), read-heavy workloads typically power dashboards, maps, and analytical reports. Based on your normalized schema, here are the high-cost operations and join paths we must eliminate:

*   **The "Geographic Lookup" Penalty (4-Table Join):** To filter plants by State (`uf`) or Municipality (`municipio`), the database currently must traverse `Usina` → `Localizacoes` → `Municipios` → `Estados`. This is highly inefficient for a simple map filter or dashboard dropdown.
*   **The "Entity Resolution" Penalty (2-Table Joins):** Resolving the River (`Rios`) and Owner (`Proprietarios`) requires separate hash or nested-loop joins. 
*   **The "Regional Aggregation" Penalty:** Calculating total power capacity (`potkw` / `potfis`) per State or per Stage (`estagio`) requires scanning the base tables and aggregating on the fly, which spikes CPU usage and latency during concurrent dashboard loads.

**Denormalization Candidates:**
Since Rivers, States, and Municipalities are highly static (they rarely change names), they are perfect candidates for **flattening** directly into the `Usina` table. Owners change infrequently enough that the read-performance gain of duplicating their names far outweighs the cost of an occasional update anomaly.

### 2. Denormalization Design

To achieve sub-millisecond latency on reads, we will implement the following physical structures:

1.  **The "Wide" Read-Model Table (`usina_read_model`):** 
    We will adopt your "One Big Table" approach but optimize the data types and add a robust indexing strategy. This table acts as a read-optimized snapshot.
2.  **Precomputed Aggregate Materialized View (`mv_capacidade_uf_estagio`):**
    Dashboards frequently ask: *"What is the total capacity (potkw) currently in operation per State?"* We will create a materialized view with a grain of `(uf1, estagio)` to serve these metrics instantly without touching the detailed wide table.
3.  **Composite & Covering Indexes:**
    *   **Geographic Filtering:** A composite index on `(uf1, munic1)` to support hierarchical location drill-downs.
    *   **Stage & Capacity Sorting:** An index on `(estagio, potkw DESC)` to quickly retrieve the top-producing plants for a given operational stage.
    *   **Spatial Lookups:** A composite index on `(lat, long)` to speed up bounding-box queries for map rendering.

### 3. New Physical Schema (DDL)

Below is the PostgreSQL-compliant DDL for the new denormalized read schema, including the wide table, materialized view, and index strategy.

```sql
BEGIN;

CREATE SCHEMA IF NOT EXISTS read_optimized AUTHORIZATION ava_um_usr;
SET search_path TO read_optimized, public;

-- ==============================================================================
-- 1. THE WIDE TABLE (Flattened Read Model)
-- Eliminates 6 joins (Localizacoes, Municipios x2, Estados, Rios, Proprietarios)
-- ==============================================================================
DROP TABLE IF EXISTS read_optimized.usina_read_model CASCADE;

CREATE TABLE read_optimized.usina_read_model (
    id_usina INTEGER PRIMARY KEY,
    nome VARCHAR NOT NULL,
    estagio VARCHAR,
    
    -- Flattened Owner
    proprietario VARCHAR,
    
    -- Flattened Geography (Primary Location)
    munic1 VARCHAR NOT NULL,
    uf1 CHAR(2) NOT NULL,
    
    -- Flattened Geography (Secondary Location, if spans borders)
    munic2 VARCHAR,
    uf2 CHAR(2),
    
    -- Flattened River
    rio VARCHAR NOT NULL,
    
    -- Metrics
    potkw FLOAT,
    potfisc FLOAT,
    
    -- Coordinates
    lat DOUBLE PRECISION,
    long DOUBLE PRECISION,
    
    -- SCD / Audit dates
    iniciop DATE,
    atualizacao DATE
);

-- ==============================================================================
-- 2. INDEXING STRATEGY FOR WIDE TABLE
-- Tied directly to expected read query patterns
-- ==============================================================================

-- Pattern A: Dashboard filtering by State and Municipality
CREATE INDEX idx_usina_geo 
    ON read_optimized.usina_read_model (uf1, munic1);

-- Pattern B: Filtering by Stage, sorting by Capacity (Top N plants per stage)
-- Uses INCLUDE to create a covering index for the plant name, avoiding heap fetches
CREATE INDEX idx_usina_estagio_potkw 
    ON read_optimized.usina_read_model (estagio, potkw DESC)
    INCLUDE (nome, uf1);

-- Pattern C: Owner portfolio lookups
CREATE INDEX idx_usina_proprietario 
    ON read_optimized.usina_read_model (proprietario);

-- Pattern D: Map rendering (Bounding box queries)
-- Standard B-Tree on lat/long. (Note: If PostGIS is enabled, replace with a GiST index on a Geometry column)
CREATE INDEX idx_usina_coordinates 
    ON read_optimized.usina_read_model (lat, long);


-- ==============================================================================
-- 3. PRECOMPUTED AGGREGATE (Materialized View)
-- Grain: State (UF) + Stage (Estagio)
-- Purpose: Instant load times for high-level analytical dashboard charts
-- ==============================================================================
DROP MATERIALIZED VIEW IF EXISTS read_optimized.mv_capacidade_uf_estagio;

CREATE MATERIALIZED VIEW read_optimized.mv_capacidade_uf_estagio AS
SELECT 
    uf1 AS uf,
    estagio,
    COUNT(id_usina) AS total_usinas,
    SUM(potkw) AS capacidade_total_kw,
    SUM(potfisc) AS capacidade_total_fisc,
    MAX(atualizacao) AS ultima_atualizacao
FROM 
    read_optimized.usina_read_model
GROUP BY 
    uf1, 
    estagio;

-- Index the Materialized View to allow fast exact-match lookups by State/Stage
CREATE UNIQUE INDEX idx_mv_cap_uf_estagio 
    ON read_optimized.mv_capacidade_uf_estagio (uf, estagio);

COMMIT;
```

### Why this design wins in production:
1. **Zero-Join Point Queries:** A query like `SELECT nome, potkw FROM usina_read_model WHERE uf1 = 'SP' AND estagio = 'Operação'` requires exactly zero joins and is fully covered by the `idx_usina_estagio_potkw` or `idx_usina_geo` indexes.
2. **Heap-Fetch Avoidance:** By using the `INCLUDE (nome, uf1)` clause on the stage/capacity index, Postgres can return the top power plants directly from the index memory pages without ever touching the actual table data on disk.
3. **Pre-calculated Dashboards:** Instead of summing thousands of rows of `potkw` every time a user opens the national dashboard, the application simply queries `mv_capacidade_uf_estagio`, which returns the exact answer in `< 1ms`. (You will just need to set up a trigger or cron job to `REFRESH MATERIALIZED VIEW CONCURRENTLY` when the base normalized tables are updated).
