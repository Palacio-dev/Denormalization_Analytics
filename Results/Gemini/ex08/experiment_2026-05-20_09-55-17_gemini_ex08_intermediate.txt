As a senior database performance engineer, looking at your schema, it is clear that this is an analytical/reporting database (likely educational intelligence combining IFB student data, ENEM school performance, and RAIS employment data). 

Your current schema is highly normalized (3NF). While great for OLTP (inserts/updates), it is hostile to read-heavy analytical workloads. Every dashboard or report currently requires 4 to 6 joins, causing high CPU usage, memory spilling on large hash joins, and high latency.

Here is the plan to denormalize and optimize your physical model.

---

### 1. Workload-Driven Analysis

In this domain, read workloads typically fall into three high-cost join paths:

*   **Path A: The "Student Outcome" Query (6 tables)**
    *   *Pattern:* `aluno_ifb` → `matricula_ifb` → `curso_ifb` → `vinculo_empregaticio` → `remuneracao` → `localizacao`.
    *   *Business Intent:* "What is the average salary and employment rate of students of a specific race/gender who graduated from Course X?"
    *   *Bottleneck:* Joining massive fact tables (`matricula` and `vinculo`) through a student dimension, then branching out to remuneration and location.
*   **Path B: The "School Context & Performance" Query (5 tables)**
    *   *Pattern:* `escola` → `localizacao` → `desempenho_enem` → `indicadores_escolares` → `participacao_enem`.
    *   *Business Intent:* "Show me the historical ENEM scores, dropout rates, and participation rates for schools in a specific municipality."
    *   *Bottleneck:* `desempenho`, `indicadores`, and `participacao` all share the exact same grain (School + Year) but are split into three tables. This forces the database to join tables that should already be a single row.

### 2. Denormalization Design

To achieve sub-second latency, we will pivot from a 3NF model to a **Wide-Table / Dimensional (Star) Model**. 

**A. Pre-joined "Wide" Fact Tables:**
1.  **`fato_escola_anual` (Annual School Fact):** We will merge `desempenho_enem`, `indicadores_escolares`, and `participacao_enem` into a single table. The grain is explicitly `(co_escola_educacenso, nu_ano)`. This eliminates 2 massive joins for school historical reports.
2.  **`fato_emprego_aluno` (Student Employment Fact):** We will merge `vinculo_empregaticio` and `remuneracao`. Because `remuneracao` is purely a 1:1 metrics extension of the employment bond, keeping them separate is a performance anti-pattern. We will also duplicate the `uf` and `municipio` here to avoid joining the `localizacao` table.
3.  **`fato_matricula_aluno` (Student Enrollment Fact):** We will merge `matricula_ifb`, `aluno_ifb`, and `curso_ifb`. Enrollment is the core event. By duplicating student demographics (age, race, sex) and course details directly into the enrollment record, we create a flat table perfect for rapid filtering and aggregation.

**B. Flattened Dimensions:**
1.  **`dim_escola`:** We will embed `uf` and `municipio` directly into the school table, eliminating the `localizacao` join.

**C. Materialized View for Cross-Domain Aggregation:**
1.  **`mvw_curso_empregabilidade`:** A precomputed aggregate table linking Courses to Average Salaries and Employment rates. This answers the most expensive executive query instantly.

**D. Indexing & Partitioning Strategy:**
*   **Partitioning:** The fact tables will be partitioned by `ano` (Year). Most analytical queries filter by time (e.g., "last 3 years"). Partition pruning will allow Postgres to skip scanning irrelevant years entirely.
*   **Covering Indexes:** We will use `INCLUDE` clauses on indexes so Postgres can resolve queries entirely from the index (Index-Only Scans) without touching the table heap.

### 3. New Physical Schema (PostgreSQL DDL)

*Note: I have cast `ano` to `integer` where it was `varchar` to allow for proper Range Partitioning.*

```sql
-- ==============================================================================
-- 1. DIMENSIONS (Flattened)
-- ==============================================================================

-- Flattened School Dimension (Includes Location)
CREATE TABLE public.dim_escola (
    co_escola_educacenso varchar(50) PRIMARY KEY,
    no_escola_educacenso varchar(200) NOT NULL,
    tp_dependencia_adm_escola smallint,
    tp_localizacao_escola smallint,
    inse varchar(10),
    uf varchar(2),
    municipio varchar(100)
);
-- Index for filtering schools by location
CREATE INDEX idx_dim_escola_loc ON public.dim_escola(uf, municipio);


-- ==============================================================================
-- 2. WIDE FACT TABLES (Partitioned by Year)
-- ==============================================================================

-- A. School Annual Performance Fact (Merges ENEM, Indicators, and Participation)
-- Grain: 1 row per School per Year
CREATE TABLE public.fato_escola_anual (
    co_escola_educacenso varchar(50) NOT NULL,
    nu_ano integer NOT NULL,
    
    -- Desempenho ENEM
    nu_media_cn numeric(6, 2),
    nu_media_ch numeric(6, 2),
    nu_media_lp numeric(6, 2),
    nu_media_mt numeric(6, 2),
    nu_media_red numeric(6, 2),
    
    -- Indicadores Escolares
    nu_taxa_permanencia numeric(6, 2),
    nu_taxa_aprovacao numeric(5, 2),
    nu_taxa_reprovacao numeric(5, 2),
    nu_taxa_abandono numeric(5, 2),
    porte_escola varchar(50),
    pc_formacao_docente numeric(5, 2),
    
    -- Participacao ENEM
    nu_matriculas integer,
    nu_participantes_nec_esp integer,
    nu_participantes integer,
    nu_taxa_participacao numeric(5, 2),
    
    PRIMARY KEY (co_escola_educacenso, nu_ano)
) PARTITION BY RANGE (nu_ano);

-- Create partitions (Example for recent years)
CREATE TABLE fato_escola_anual_2021 PARTITION OF public.fato_escola_anual FOR VALUES FROM (2021) TO (2022);
CREATE TABLE fato_escola_anual_2022 PARTITION OF public.fato_escola_anual FOR VALUES FROM (2022) TO (2023);
CREATE TABLE fato_escola_anual_2023 PARTITION OF public.fato_escola_anual FOR VALUES FROM (2023) TO (2024);

-- Covering index for common dashboard queries (Scores over time)
CREATE INDEX idx_fato_escola_anual_cover 
ON public.fato_escola_anual(co_escola_educacenso, nu_ano) 
INCLUDE (nu_media_red, nu_media_mt, nu_taxa_aprovacao);


-- B. Student Enrollment Wide Fact (Merges Matricula, Aluno, and Curso)
-- Grain: 1 row per Enrollment
CREATE TABLE public.fato_matricula_aluno (
    id_matricula serial,
    ano integer NOT NULL, -- Casted to INT for partitioning
    periodo varchar(30),
    co_ciclo_matricula varchar(50),
    situacao_matricula varchar(50),
    tipo_cota varchar(50),
    atestado_baixarenda varchar(20),
    unidade_ensino varchar(100),
    
    -- Denormalized Aluno
    id_aluno integer NOT NULL,
    idade integer,
    sexo char(1),
    raca varchar(50),
    portador_deficiencia boolean,
    
    -- Denormalized Curso
    co_curso varchar(50),
    no_curso varchar(200),
    tipo_curso varchar(100),
    modalidade_ensino varchar(100),
    
    -- FK to School
    co_escola_educacenso varchar(50),
    
    PRIMARY KEY (id_matricula, ano)
) PARTITION BY RANGE (ano);

CREATE TABLE fato_matricula_aluno_2021 PARTITION OF public.fato_matricula_aluno FOR VALUES FROM (2021) TO (2022);
CREATE TABLE fato_matricula_aluno_2022 PARTITION OF public.fato_matricula_aluno FOR VALUES FROM (2022) TO (2023);
CREATE TABLE fato_matricula_aluno_2023 PARTITION OF public.fato_matricula_aluno FOR VALUES FROM (2023) TO (2024);

-- Composite index to instantly filter demographics by course and year
CREATE INDEX idx_fato_matricula_curso_demo 
ON public.fato_matricula_aluno(co_curso, ano, situacao_matricula)
INCLUDE (idade, sexo, raca);

-- Index for joining to employment data
CREATE INDEX idx_fato_matricula_aluno_id ON public.fato_matricula_aluno(id_aluno);


-- C. Employment Wide Fact (Merges Vinculo, Remuneracao, and Localizacao)
-- Grain: 1 row per Employment Bond
CREATE TABLE public.fato_emprego_aluno (
    id_vinculo serial,
    ano_rais integer NOT NULL,
    id_aluno integer NOT NULL,
    
    -- Vinculo Details
    razao_social varchar(200),
    tipo_vinculo_empregaticio varchar(200),
    data_admissao_declarada date,
    tempo_emprego integer,
    motivo_desligamento varchar(200),
    indicador_vinculo_ativo boolean,
    
    -- Denormalized Remuneracao
    vl_remun_media_nom numeric(10, 2),
    vl_remun_media_sm numeric(10, 2),
    vl_remun_dezembro_nom numeric(10, 2),
    qtd_hora_contr integer,
    vl_salario_contratual numeric(10, 2),
    
    -- Denormalized Localizacao
    uf varchar(2),
    municipio varchar(100),
    
    PRIMARY KEY (id_vinculo, ano_rais)
) PARTITION BY RANGE (ano_rais);

CREATE TABLE fato_emprego_aluno_2021 PARTITION OF public.fato_emprego_aluno FOR VALUES FROM (2021) TO (2022);
CREATE TABLE fato_emprego_aluno_2022 PARTITION OF public.fato_emprego_aluno FOR VALUES FROM (2022) TO (2023);
CREATE TABLE fato_emprego_aluno_2023 PARTITION OF public.fato_emprego_aluno FOR VALUES FROM (2023) TO (2024);

-- Index to optimize the cross-domain join (Student -> Salary)
CREATE INDEX idx_fato_emprego_aluno_salario 
ON public.fato_emprego_aluno(id_aluno, ano_rais, indicador_vinculo_ativo)
INCLUDE (vl_remun_media_nom);


-- ==============================================================================
-- 3. PRECOMPUTED AGGREGATES (Materialized Views)
-- ==============================================================================

-- Business critical query: "What is the average salary of graduates per course?"
-- This replaces a 6-table join with a single O(1) lookup table.
CREATE MATERIALIZED VIEW public.mvw_curso_empregabilidade AS
SELECT 
    m.co_curso,
    m.no_curso,
    m.ano AS ano_formatura,
    e.ano_rais AS ano_referencia_salario,
    COUNT(DISTINCT m.id_aluno) as total_alunos,
    COUNT(DISTINCT e.id_aluno) as total_empregados,
    ROUND(AVG(e.vl_remun_media_nom), 2) as media_salarial_nominal,
    ROUND(AVG(e.vl_remun_media_sm), 2) as media_salarial_salarios_minimos
FROM public.fato_matricula_aluno m
JOIN public.fato_emprego_aluno e ON m.id_aluno = e.id_aluno
WHERE m.situacao_matricula = 'Concluído' -- Assuming this string exists
  AND e.indicador_vinculo_ativo = true
GROUP BY 
    m.co_curso,
    m.no_curso,
    m.ano,
    e.ano_rais;

-- Unique index allows CONCURRENTLY refreshing the materialized view without locking reads
CREATE UNIQUE INDEX idx_mvw_curso_empregabilidade_unique 
ON public.mvw_curso_empregabilidade(co_curso, ano_formatura, ano_referencia_salario);
```

### Summary of Performance Gains:
1. **Eliminated Joins:** Querying a student's demographics alongside their course details now requires **0 joins** (previously 2). Querying school performance over time requires **0 joins** (previously 2).
2. **Reduced Heap Fetches:** The `INCLUDE` clauses on the indexes mean that if a dashboard just needs the average ENEM math score or dropout rate for a specific school, Postgres reads the B-Tree index and returns the data directly, bypassing the table entirely.
3. **Partition Pruning:** By partitioning the massive `matricula` and `vinculo` tables by year, queries filtering by `ano = 2023` will instantly drop all other partitions from the execution plan, reducing I/O by orders of magnitude.
