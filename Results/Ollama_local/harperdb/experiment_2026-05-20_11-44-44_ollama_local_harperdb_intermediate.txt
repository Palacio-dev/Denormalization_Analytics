# Database Performance Engineering: Read-Optimized Denormalization Plan

As a Senior Database Performance Engineer, I have analyzed your current schema. The current design is a classic **Third Normal Form (3NF)** structure optimized for data integrity and insert/update speed, but it is suboptimal for a **read-heavy workload**.

Every read query requiring a user's active features requires joining `users` $\to$ `user_entitlements` $\to$ `entitlements`. In high-concurrency scenarios, this causes:
1.  **CPU Contention:** The database engine must perform a join operation for every row read.
2.  **Lock Contention:** If `user_entitlements` is updated frequently, readers may wait for writers to release locks.
3.  **Network/IO Overhead:** Fetching multiple rows to assemble a single logical result set.

Below is the workload-driven analysis, the denormalization strategy, and the final DDL.

---

## 1. Workload-Driven Analysis

### Current Join Paths & High-Cost Operations
Based on typical SaaS/Subscription patterns, your read workload likely involves:
*   **Pattern A (User Feature Discovery):** `SELECT ... FROM users u JOIN user_entitlements ue ON u.id = ue.user_id JOIN entitlements e ON ue.entitlement_id = e.id WHERE u.id = ? AND e.activeFlag = 1`
    *   *Cost:* Requires scanning `users`, joining to the many-to-many `user_entitlements`, then filtering `entitlements`. If `entitlements` has thousands of rows, this is expensive.
*   **Pattern B (Usage Reporting):** `SELECT ... FROM user_entitlements ue JOIN entitlements e ON ... WHERE ue.user_id IN (...)`
    *   *Cost:* Aggregating `useCount` across multiple entitlements per user without pre-calculation.

### Denormalization Candidates
To eliminate these joins, we must duplicate data that is read frequently but updated infrequently.
1.  **Entitlement Details:** The `featureType`, `packageType`, and `useLimit` are rarely updated (once a subscription is bought). They are accessed in almost every read query involving a user.
2.  **Usage Aggregates:** The `useCount` is often queried with a date filter. Calculating this on the fly requires grouping by user and date.
3.  **Active Status:** Determining if a user has an active feature requires checking `activeDate` vs `inactiveDate` and `activeFlag`. This logic is repeated in every query.

---

## 2. Denormalization Design Strategy

### Strategy: The "Read-Model" Star Schema
We will introduce a **`user_read_model`** table. This table acts as a denormalized fact table.
*   **Grain:** One row per active feature per user per snapshot (or per day if daily snapshots are needed).
*   **Redundancy:** We will duplicate `featureType`, `packageType`, `useLimit`, and pre-computed `is_active` status.
*   **Pre-computation:** We will pre-calculate daily usage aggregates to avoid grouping operations on the transactional table.

### Specific Changes
1.  **Create `user_active_features` (Wide Table):**
    *   Denormalizes `entitlements` details directly into the user context.
    *   Includes a computed `is_active` flag to avoid date math in queries.
    *   Includes `useLimit` directly to avoid joins for quota checks.
2.  **Create `user_daily_usage` (Snapshot Table):**
    *   Stores pre-aggregated `useCount` per user per day.
    *   Allows `SUM` or `MAX` queries in milliseconds instead of `GROUP BY` on the transactional table.
3.  **Covering Indexes:**
    *   Composite indexes on `(user_id, feature_type)` and `(user_id, date)` to support the most common filter patterns without touching the index tree deeply.

---

## 3. New Physical Schema (DDL)

The following DDL creates a hybrid schema:
1.  **Transactional Tables (`entitlements`, `users`, `user_entitlements`):** Kept mostly intact for writes and auditability.
2.  **Read-Optimized Tables (`user_active_features`, `user_daily_usage`):** Denormalized tables optimized for your specific query patterns.
3.  **Trigger:** A trigger automatically updates the read tables whenever `user_entitlements` changes to ensure consistency.

### Key Implementation Notes for the Engineer

1.  **Trigger Complexity:** The `user_daily_usage` table is heavy to maintain via triggers (calculating sums row-by-row). In production, **move this to a background job** (e.g., a cron job running every 15 minutes). This reduces write latency significantly.
2.  **Data Consistency:** The `user_active_features` table is tightly coupled with `user_entitlements` via the trigger. If `entitlements` changes (e.g., `use_limit` increases), the read model updates immediately.
3.  **Index Utilization:** The composite indexes `(user_id, feature_type)` and `(user_id, usage_date)` are critical. They allow the database to fetch exactly the rows needed without scanning the whole table.
4.  **Write Overhead:** The only cost introduced is the trigger overhead. Since entitlements are written rarely compared to reads, the write penalty is negligible compared to the read performance gain.

This schema transforms your query patterns from **Join-Heavy** to **Lookup-Heavy**, which is the hallmark of a high-performance read-optimized database.
