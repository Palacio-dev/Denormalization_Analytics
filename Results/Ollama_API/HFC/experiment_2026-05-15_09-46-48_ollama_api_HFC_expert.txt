# Executive Summary

This design proposes a denormalized read-optimized schema for your botanical specimen database system. The approach focuses on creating wide read tables that pre-join frequently accessed dimensions with core tombos (specimen) data, significantly reducing query complexity and join overhead. We identify key access patterns around specimen details, location hierarchies, taxonomic classifications, and historical changes.

The denormalized schema preserves business correctness through explicit data lineage and implements a controlled staleness window of 15 minutes via incremental refresh processes. Operational complexity is managed through clearly defined refresh procedures with built-in monitoring and rollback capabilities. The solution provides measurable read performance improvements by eliminating multi-table joins and aggregations at query time, particularly benefiting specimen detail views and reporting queries.

## Consistency Architecture

### Data Lineage
- Source tables: `tombos`, `especies`, `generos`, `familias`, `autores`, `locais_coleta`, `cidades`, `estados`, `paises`, `coletores`, `tipos`, `herbarios`, `solos`, `relevos`, `vegetacoes`
- Denormalized target: `tombos_denormalized`
- Refresh mechanism: Incremental updates based on `updated_at` timestamps
- Staleness window: 15 minutes maximum

### Risk Analysis

#### Write Amplification
- Creation of denormalized records requires updates to multiple source tables
- Mitigation: Use batch processing during low-usage periods

#### Lock Contention
- Refresh processes may lock rows during updates
- Mitigation: Use row-level locking and smaller batch sizes

#### Failure Modes
- Incomplete refreshes could lead to stale data
- Mitigation: Implement validation checks and alerting

#### Backfill Runtime
- Initial population may take several hours depending on data volume
- Mitigation: Perform during scheduled maintenance windows

#### Storage Growth
- Denormalized tables will increase storage requirements by approximately 2-3x
- Mitigation: Monitor storage usage and implement archiving policies if needed

#### Monitoring & Alerts
- Lag: Alert when refresh intervals exceed 30 minutes
- Freshness: Monitor time difference between source and denormalized data
- Reconciliation: Daily comparison of counts between normalized and denormalized views
- Query Performance: Track response times for key queries before and after migration
