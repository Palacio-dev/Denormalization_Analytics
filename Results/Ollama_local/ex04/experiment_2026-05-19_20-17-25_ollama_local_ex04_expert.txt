# Executive Summary

This design addresses the read-optimized denormalization of a multi-dimensional economic and demographic dataset. The primary bottleneck in the current normalized schema is the repeated joining of `country`, `demography`, `employment`, `education`, `economy`, and `health` tables for every query, resulting in high I/O latency and complex query plans.

The proposed solution introduces **Pre-Aggregated Fact Tables** and **Wide Read Dimension Tables**. Instead of joining 6-8 tables per query, the system will materialize "Country-Year" snapshots containing pre-calculated aggregates (e.g., total population, GDP, average life expectancy) and pre-joined dimension data. This reduces query complexity from $O(N \times M)$ joins to $O(1)$ table lookups.

**Correctness Strategy:** We will not use transactional materialized views (which are unsupported in most SQL engines). Instead, we will use **Batch Refresh Jobs** that run outside business hours. A defined "Staleness Window" of 2–4 hours will be explicitly documented; data in the denormalized tables will be at most 2 hours behind the source of truth. This trade-off is necessary to achieve sub-second read latency for dashboarding and reporting workloads.

# Consistency Architecture

Since triggers are forbidden and we are denormalizing, we rely on a **Batch Refresh Architecture**.

1.  **Source of Truth (SoT):** The original normalized tables (`economy`, `demography`, etc.).
2.  **Refresh Mechanism:**
    *   **Frequency:** Hourly (every 2 hours).
    *   **Method:** A Python/SQL script runs as a background job.
    *   **Logic:**
        1.  Identify `country_code` and `survey_year` combinations where the SoT has changed (using a `last_modified` flag or CDC).
        2.  Execute a `SELECT ... INTO` query to pull the necessary columns from SoT.
        3.  Execute aggregation functions (`SUM`, `AVG`, `COUNT`) in the refresh script itself (or in a temporary table) to compute the denormalized values.
        4.  `UPSERT` (INSERT ... ON CONFLICT DO UPDATE) into `fact_country_year_snapshot`.
3.  **Staleness Handling:**
    *   The `last_refresh_ts` column is exposed to monitoring.
    *   Queries are allowed to read stale data; there is no mechanism to force "freshness" at read time (which would defeat the purpose of denormalization).
    *   **Explicit Staleness Window:** 2 hours.

# Risk Analysis & Mitigation

| Risk Category | Description | Impact | Mitigation Strategy |
| :--- | :--- | :--- | :--- |
| **Write Amplification** | Upserting millions of rows hourly can saturate WAL logs and I/O. | High latency on writes, potential blocking. | **Batch Size:** Limit refresh to 100k rows per batch. **Async Writes:** Use `COPY` command or async queues (e.g., Dead Letter Queue) for large inserts. |
| **Data Drift** | SoT data changes (e.g., new country added) while refresh is running. | Inconsistency between SoT and Denormalized table. | **Transactional Locking:** Use a `refresh_in_progress` flag in SoT. Only refresh if no other jobs are running. **Reconciliation:** Run a nightly diff job comparing `fact_country_year_snapshot` against a calculated view on SoT. |
| **Storage Growth** | Storing repeated data grows storage over time. | Cost increase, slower backups. | **TTL:** Archive old years (>10 years) to cold storage or drop partitions. **Compression:** Enable `pg_compression` or row-level compression. |
| **Query Latency Regression** | If the denormalized table is not indexed correctly. | Slower than normalized joins. | **Index Strategy:** Ensure `country_code` and `survey_year` are clustered/partitioned keys. Monitor `index_scan` vs `seq_scan`. |
| **Operational Complexity** | Managing the refresh job adds ops burden. | Risk of job failure going unnoticed. | **Alerting:** Alert on `last_refresh_ts` lag > 4 hours. Alert on refresh job failure. |

# Monitoring & Observability Plan

To ensure the denormalization strategy remains safe and effective, the following metrics must be monitored:

1.  **Freshness Lag:**
    *   *Metric:* `MAX(CURRENT_TIMESTAMP - last_refresh_ts)` per table.
    *   *Alert:* Threshold > 4 hours.
2.  **Reconciliation Diff:**
    *   *Metric:* Count of rows where `fact_country_year_snapshot` differs from a `CALCULATED_VIEW` on SoT.
    *   *Alert:* Threshold > 0 (Critical).
3.  **Query Performance:**
    *   *Metric:* Average execution time for "Country Profile" queries.
    *   *Alert:* Increase > 20% compared to baseline.
4.  **Write Throughput:**
    *   *Metric:* Rows inserted per refresh cycle.
    *   *Alert:* Sudden drop (indicates data quality issue in SoT) or spike (indicates job stuck).
