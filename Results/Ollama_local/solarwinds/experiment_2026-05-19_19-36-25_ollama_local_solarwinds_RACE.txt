### 1. Analysis of the Provided Schema

### 2. Evaluation of Denormalization Techniques

*   **Adding Redundant Attributes**: Highly effective here. Storing `product_name` directly in `order_details` eliminates the need to join `products` for every order line item.
*   **Adding Derived Attributes**: Useful for `total_price` (quantity * price) if we are pre-calculating per order, but usually, it's better to materialize the join result itself.
*   **Collapsing Relations**: The most critical technique for this model. Merging `orders` and `order_details` into a single "Fact Order" table, and merging `products` attributes into the `order_details` table, drastically reduces join complexity.
*   **Vertical Partitioning**: Not applicable here as the tables are small enough and the attributes are tightly coupled in the analytical view. We are doing the opposite (horizontal flattening).

### 3. Proposed Denormalized Schema Strategy

I will design a schema that collapses the transactional structure into **Fact Tables** populated with redundant dimensions.

**Strategy A: The "Super Fact Order" Table**
Instead of joining `orders`, `order_details`, and `products`, we create a single table `fact_orders` that contains:
*   Customer info (Redundant: `customer_name`, `customer_email`).
*   Order metadata (Redundant: `order_date`).
*   Product info (Redundant: `product_name`, `price`).
*   Line item data (Non-redundant: `quantity`).
*   **Calculated Field**: `total_line_value` (quantity * price) pre-calculated at ingestion.

This allows a query like "Total Sales per Customer" to run on a single table scan with a simple `GROUP BY`, eliminating all JOINs.

### 5. Final Selection and Rationale

The chosen schema (**Fact_Order_Details** + **Fact_Customers**) utilizes the **Collapsing Relations** and **Adding Redundant Attributes** techniques most effectively.

**Why this is the best choice:**
1.  **Elimination of Joins**: Queries previously requiring `JOIN fact_order_details products ON ... JOIN fact_orders customers ON ...` now execute as simple `SELECT ... FROM fact_order_details WHERE ...`.
2.  **Index Efficiency**: By storing `customer_name` and `product_name` in the fact table, we can create B-Tree indexes on these text columns directly. In the normalized model, these columns would not exist in the fact table, forcing the engine to look up values from dimension tables during the join process.
3.  **Pre-calculated Metrics**: The `total_line_value` column allows for immediate aggregation (e.g., `SUM(total_line_value)`) without recalculating `quantity * price` on the fly, saving CPU cycles.
4.  **Read-Optimized**: The schema sacrifices referential integrity (no Foreign Keys) to gain massive read performance. For an analytical system, data consistency is managed via batch loading (ETL), not real-time transactional constraints.

*Complexity*: High. The database must fetch rows from 4 tables, match them via IDs, and perform arithmetic for every row.

*Complexity*: Low. The database scans one table, uses the pre-calculated `total_line_value`, and groups immediately.
