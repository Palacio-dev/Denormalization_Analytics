# Executive Summary

This document outlines a denormalized, read-optimized physical schema derived from the normalized schema provided for the `bookings` database. The goal is to significantly improve read performance by reducing query complexity through pre-joined tables and materialized rollups, while preserving data correctness and ensuring operational safety.

We identify key access patterns such as frequent joins between flights, airports, aircrafts, tickets, and boarding passes, which are hotspots for performance degradation in normalized designs. To address this, we propose:
- Wide read tables that pre-join dimensions like flights with airports and aircrafts.
- Pre-aggregated rollups for booking summaries by time buckets.
- Snapshot tables for current flight statuses.
- Materialized views or equivalent engine features where appropriate.

All denormalized artifacts are tied directly to specific query patterns or analytical workloads. Operational constraints are respected, including avoiding triggers and managing refresh windows. The implementation plan includes safe rollout steps, observability hooks, and rollback strategies.

# Proposed Denormalized Schema (DDL)

### 1. Wide Read Table: `flight_details`

**Purpose**: Pre-joined view of flights with departure/arrival airports and aircraft info for fast lookups.

```sql
CREATE TABLE IF NOT EXISTS bookings.flight_details (
    flight_id integer NOT NULL PRIMARY KEY,
    flight_no character(6) NOT NULL,
    scheduled_departure timestamptz NOT NULL,
    scheduled_arrival timestamptz NOT NULL,
    actual_departure timestamptz,
    actual_arrival timestamptz,
    status varchar(20) NOT NULL,

    departure_airport_code char(3) NOT NULL,
    departure_airport_name jsonb NOT NULL,
    departure_city jsonb NOT NULL,

    arrival_airport_code char(3) NOT NULL,
    arrival_airport_name jsonb NOT NULL,
    arrival_city jsonb NOT NULL,

    aircraft_code char(3) NOT NULL,
    aircraft_model jsonb NOT NULL,
    aircraft_range integer NOT NULL,

    INDEX idx_flight_details_scheduled_departure (scheduled_departure),
    INDEX idx_flight_details_departure_airport (departure_airport_code),
    INDEX idx_flight_details_arrival_airport (arrival_airport_code),
    INDEX idx_flight_details_status (status)
);
```

**Justification**: This table avoids multiple joins on `flights`, `airports_data`, and `aircrafts_data`. It's optimized for queries about flight schedules and related metadata.

### 4. Wide Read Table: `passenger_itineraries`

**Purpose**: Denormalized itinerary view combining ticket, flight, and boarding pass information.

```sql
CREATE TABLE IF NOT EXISTS bookings.passenger_itineraries (
    ticket_no char(13) NOT NULL,
    flight_id integer NOT NULL,
    passenger_name text NOT NULL,
    flight_no char(6) NOT NULL,
    scheduled_departure timestamptz NOT NULL,
    scheduled_arrival timestamptz NOT NULL,
    departure_airport char(3) NOT NULL,
    arrival_airport char(3) NOT NULL,
    boarding_no integer,
    seat_no varchar(4),
    fare_conditions varchar(10),
    amount numeric(10,2),

    PRIMARY KEY (ticket_no, flight_id),
    INDEX idx_passenger_itineraries_flight_id (flight_id),

```

**Justification**: Eliminates costly joins between `tickets`, `ticket_flights`, `boarding_passes`, and `flights` for passenger itinerary lookups.

### 5. Materialized View (Optional Engine Equivalent): `seat_availability_per_flight`

**Purpose**: Real-time or near real-time view of seat availability per flight.

> Note: If using PostgreSQL, consider using a **materialized view** with periodic refresh. Alternatively, implement as a regular table refreshed via batch job.

```sql
-- For PostgreSQL materialized view:
CREATE MATERIALIZED VIEW bookings.seat_availability_per_flight AS
SELECT
    tf.flight_id,
    COUNT(s.seat_no) AS total_seats,
    COUNT(tf.ticket_no) AS booked_seats,
    (COUNT(s.seat_no) - COUNT(tf.ticket_no)) AS available_seats
FROM bookings.seats s
LEFT JOIN bookings.ticket_flights tf ON s.aircraft_code = (
    SELECT aircraft_code FROM bookings.flights f WHERE f.flight_id = tf.flight_id LIMIT 1
)
GROUP BY tf.flight_id;

CREATE UNIQUE INDEX idx_seat_availability_flight_id ON bookings.seat_availability_per_flight(flight_id);
```

**Justification**: Reduces complexity of computing seat availability at query time by caching it periodically.

# Risk Analysis

## Write Amplification

- Denormalized tables increase write operations during ETL/batch updates.
- Mitigation: Batch refreshes outside peak hours.

## Lock Contention

- Refresh jobs may lock tables during updates.
- Mitigation: Use staging tables + atomic swap (`RENAME`) or logical replication slots.

## Failure Modes

- Inconsistent state if refresh fails mid-process.
- Mitigation: Transactional refreshes, idempotent jobs, alerting on staleness.

## Backfill Runtime

- Initial backfill may take hours depending on data volume.
- Mitigation: Parallelize across partitions or use CDC tools.

## Storage Growth

- Denormalization increases storage footprint (~2x–5x).
- Mitigation: Monitor growth, archive old snapshots.

## Monitoring & Alerting

| Metric | Alert Threshold | Tool |
|--------|------------------|------|
| Lag in refresh | >1 hour | Prometheus / Grafana |
| Freshness of summary tables | >2 hours stale | Datadog |
| Query latency regression | +50% vs baseline | APM (e.g., New Relic) |
| Reconciliation diff | >0.1% mismatch | Custom reconciliation scripts |
