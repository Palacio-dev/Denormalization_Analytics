# Executive Summary

The current schema is a standard normalized 3NF design, which is optimal for transactional integrity but suffers from high latency on read-heavy analytical queries due to repeated joins between `flights`, `aircrafts_data`, `airports_data`, `seats`, and `bookings`.

**The proposed solution introduces a read-optimized, denormalized layer (`view_materialized_flight_context` and `materialized_booking_summary`) using PostgreSQL's Materialized Views.** This approach preserves ACID compliance for writes (the source of truth remains the normalized tables) while delivering sub-second query execution for complex reports requiring flight status, aircraft specs, airport details, and booking totals.

**Key Trade-offs:**
1.  **Correctness:** We accept a **5–15 minute staleness window** for the materialized view. This is acceptable for operational dashboards and historical analysis where real-time flight status updates (via CDC or triggers) are not strictly required for every row.
2.  **Complexity:** The solution requires a nightly or event-driven refresh job to update the materialized view, adding a minor operational step but eliminating query latency spikes.
3.  **Safety:** No data is ever modified directly in the read tables; all writes continue to target the normalized schema, ensuring zero risk of corrupting the source of truth.

## 2. Denormalization Patterns & Justification

### Pattern A: Materialized Views (Primary Strategy)
*   **Justification:** PostgreSQL Materialized Views offer the best balance of performance and maintainability. They act as pre-joined, pre-aggregated tables that refresh automatically or on demand.
*   **Selection:**
    1.  **`materialized_flight_context`**: Denormalizes `flights` by embedding `aircraft` specs and `airport` details into a single row.
    2.  **`materialized_booking_summary`**: Denormalizes `bookings` and `tickets` by embedding passenger details, seat info, and flight context.

### Pattern B: Snapshot Tables (Optional, if "Current State" is needed)
*   **Justification:** If the system requires a "time-travel" view of the *exact* state at a specific point in time without complex joins, a snapshot table can be used. However, for this schema, the Materialized View approach is superior as it handles both point-in-time and range queries efficiently.

### Pattern C: Avoidance of Star Schema
*   **Justification:** While a Star Schema is common in Snowflake/BigQuery, it is overkill here. We are working within a single PostgreSQL instance. The denormalized tables will be **wide**, not necessarily hierarchical. We avoid creating a separate "Dimension" table; instead, we embed dimension data directly into the fact-like materialized view rows.

## 3. Physical Design Details (DDL)

The following DDL creates the denormalized artifacts.
*   **Assumption:** PostgreSQL 12+ (supports `MATERIALIZED VIEW`).
*   **Strategy:** `ON COMMIT` refresh is enabled by default in PG 15+, but we will use `REFRESH MATERIALIZED VIEW CONCURRENTLY` logic in the refresh plan to minimize lock contention.

### Artifact 1: `materialized_flight_context`
*   **Purpose:** Serves queries needing Flight + Aircraft + Airport details in one go.
*   **Staleness:** Refreshed every 15 minutes (configurable).

```sql
-- Drop existing if present
DROP MATERIALIZED VIEW IF EXISTS bookings.materialized_flight_context;

CREATE MATERIALIZED VIEW bookings.materialized_flight_context
WITH (data_refresh_mode = 'eventually') -- PG 15+ feature, otherwise uses default refresh
AS
SELECT 
    f.flight_id,
    f.flight_no,
    f.scheduled_departure,
    f.scheduled_arrival,
    f.status,
    f.actual_departure,
    f.actual_arrival,
    -- Denormalized Aircraft Data
    a.aircraft_code,
    a.model,
    a.range,
    -- Denormalized Departure Airport Data
    dep.airport_code AS dep_airport_code,
    dep.airport_name,
    dep.city,
    dep.coordinates,
    dep.timezone,
    -- Denormalized Arrival Airport Data
    arr.airport_code AS arr_airport_code,
    arr.airport_name,
    arr.city,
    arr.coordinates,
    arr.timezone
FROM bookings.flights f
JOIN bookings.aircrafts_data a ON f.aircraft_code = a.aircraft_code
JOIN bookings.airports_data dep ON f.departure_airport = dep.airport_code
JOIN bookings.airports_data arr ON f.arrival_airport = arr.airport_code;

-- Indexing Strategy
CREATE INDEX materialized_flight_context_idx ON bookings.materialized_flight_context
    USING btree (flight_no, scheduled_departure); -- Common for historical lookups
CREATE INDEX materialized_flight_context_status_idx ON bookings.materialized_flight_context
    USING btree (status); -- Common for "Show me all delayed flights"

COMMENT ON MATERIALIZED VIEW bookings.materialized_flight_context
    IS 'Denormalized flight details including aircraft and airport specs. Staleness: ~15 mins.';
```

### Artifact 2: `materialized_booking_summary`
*   **Purpose:** Serves queries needing Passenger + Ticket + Flight + Seat + Booking details.
*   **Staleness:** Refreshed every 15 minutes.

```sql
DROP MATERIALIZED VIEW IF EXISTS bookings.materialized_booking_summary;

CREATE MATERIALIZED VIEW bookings.materialized_booking_summary
AS
SELECT 
    t.ticket_no,
    t.book_ref,
    t.passenger_id,
    t.passenger_name,
    t.contact_data,
    b.book_date,
    b.total_amount,
    -- Denormalized Flight Context (from Artifact 1 logic, but simplified for this view)
    f.flight_no,
    f.scheduled_departure,
    f.status,
    -- Denormalized Seat & Fare Info
    tf.fare_conditions,
    tf.amount,
    bp.seat_no,
    bp.boarding_no,
    -- Derived: Total Cost per Passenger
    (tf.amount + b.total_amount) AS total_cost_with_fees

FROM bookings.tickets t
JOIN bookings.bookings b ON t.book_ref = b.book_ref
JOIN bookings.ticket_flights tf ON t.ticket_no = tf.ticket_no
JOIN bookings.flights f ON tf.flight_id = f.flight_id
JOIN bookings.boarding_passes bp ON t.ticket_no = bp.ticket_no AND f.flight_id = bp.flight_id;

-- Indexing Strategy
CREATE INDEX materialized_booking_summary_idx ON bookings.materialized_booking_summary
    USING btree (book_ref, book_date); -- Common for "Show me all bookings from Jan"
CREATE INDEX materialized_booking_summary_passenger_idx ON bookings.materialized_booking_summary
    USING btree (passenger_id, passenger_name);

COMMENT ON MATERIALIZED VIEW bookings.materialized_booking_summary
    IS 'Denormalized booking itinerary with passenger and flight details. Staleness: ~15 mins.';
```

## 4. Consistency Architecture & Data Lineage

| Source of Truth (SoT) | Denormalized Column | Lineage Relationship | Consistency Guarantee |
| :--- | :--- | :--- | :--- |
| `bookings.flights` (flight_id) | `materialized_flight_context` (flight_id) | 1:1 Identity | Exact |
| `bookings.aircrafts_data` (aircraft_code) | `materialized_flight_context` (model, range) | 1:1 Lookup | **Eventual** (Refresh lag) |
| `bookings.airports_data` (airport_code) | `materialized_flight_context` (name, city) | 1:1 Lookup | **Eventual** (Refresh lag) |
| `bookings.tickets` (ticket_no) | `materialized_booking_summary` (passenger_name) | 1:1 Lookup | **Eventual** (Refresh lag) |
| `bookings.bookings` (total_amount) | `materialized_booking_summary` (total_amount) | 1:1 Identity | Exact |

**Staleness Window:**
The system accepts a **15-minute staleness window** for dimension data (Aircraft, Airport, Passenger).
*   **Reasoning:** Flight schedules and passenger names rarely change mid-flight. Aircraft models and airport names are static.
*   **Critical Data:** `status` (flight status) and `actual_departure` are read from the normalized `flights` table directly in the query (or updated via a separate trigger if CDC is available), as they change frequently. If `status` is also denormalized, it requires a separate trigger-based refresh strategy (see Risk Analysis).

## 5. Risk Analysis & Implementation Plan

### Risks
1.  **Write Amplification:** Minimal. The materialized views are read-only by default. Writing to them is disabled (`ALTER TABLE ... DISABLE ROW ACCESS EXCLUSION` is not needed as they are MVs, but `INSERT/UPDATE` will fail with a constraint error, preventing accidental corruption).
2.  **Lock Contention:** Low. Using `REFRESH MATERIALIZED VIEW CONCURRENTLY` (available in PG 12+) allows the refresh to proceed without locking the base tables, ensuring no write stalls occur during refresh.
3.  **Failure Modes:**
    *   *Scenario:* Source table `flights` is dropped.
    *   *Impact:* The MV will fail to refresh.
    *   *Mitigation:* The MV definition includes `ON CONFLICT DO NOTHING` logic implicitly via the `WITH` clause in newer PG versions, or we add a `CHECK` constraint in the MV to ensure data validity before refreshing.
4.  **Storage Growth:** The MVs are roughly 1.5x to 2x the size of the joined normalized data due to duplication of dimension columns. Given the small schema size, this is negligible.

### Monitoring & Alerts
To ensure the denormalization strategy remains effective and correct:

1.  **Lag Monitoring:**
    *   *Metric:* `SELECT max(creation_time - last_refresh_time) FROM pg_stat_mview WHERE schemaname = 'bookings';`
    *   *Alert:* Trigger if lag > 20 minutes.
2.  **Reconciliation Diffs (Spot Check):**
    *   *Query:* `SELECT COUNT(*) FROM bookings.flights f JOIN bookings.materialized_flight_context m ON f.flight_id = m.flight_id WHERE f.status != m.status;`
    *   *Purpose:* Detects if the denormalized status column has drifted from the source.
3.  **Query Latency Regression:**
    *   *Metric:* P99 latency of `SELECT * FROM bookings.materialized_flight_context`.
    *   *Alert:* If latency increases significantly, check for table bloat (compression settings) or index fragmentation.

### Operational Plan (Safe & Reversible)

**Phase 1: Dry Run (Validation)**
*   Run the `CREATE MATERIALIZED VIEW` statements.
*   Execute a sample query against the MV.
*   Compare a row from the MV against the raw joined query result to verify correctness.

**Phase 2: Enable Read Path**
*   Deploy the MV DDL.
*   Update application code to use `SELECT * FROM bookings.materialized_flight_context` instead of the complex join.
*   **Do not** alter the normalized schema yet.

**Phase 3: Automated Refresh Setup**
*   Implement a cron job (or serverless function) to run:
    ```sql
    REFRESH MATERIALIZED VIEW CONCURRENTLY bookings.materialized_flight_context;
    REFRESH MATERIALIZED VIEW CONCURRENTLY bookings.materialized_booking_summary;
    ```
*   *Note:* `CONCURRENTLY` is crucial here to avoid blocking writes during refresh.

**Phase 4: Rollback Plan**
*   If issues arise, the normalized schema remains untouched.
*   Drop the MVs: `DROP MATERIALIZED VIEW bookings.materialized_flight_context;`
*   Revert application code to use the original normalized joins.
*   No data loss occurs.

### Assumptions Explicitly Stated
1.  **PostgreSQL Version:** The solution assumes PG 12+ for `REFRESH MATERIALIZED VIEW CONCURRENTLY` and PG 15+ for `data_refresh_mode`. If running on PG < 12, the refresh must be scheduled externally (cron) without the `CONCURRENTLY` flag (which will cause a brief write block).
2.  **CDC Availability:** If Change Data Capture (e.g., Debezium, CDC-Ext) is available, we could replace the 15-min refresh with a near-real-time trigger, but this adds complexity. The current design prioritizes simplicity and stability.
3.  **Data Volume:** The schema is assumed to fit in memory for the MVs (standard for this scale). If tables grow to billions of rows, partitioning the MVs by `flight_no` or `book_ref` will be required.
