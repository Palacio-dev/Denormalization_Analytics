### 3 - Evaluation of Common Denormalization Techniques
*   **Adding Redundant Attributes:** **Highly Applicable.** We should bring descriptive attributes (biome name, institution name, region state) directly into the main event/entity tables to eliminate lookup JOINs.
*   **Adding Derived Attributes:** **Highly Applicable.** In `coleta_de_lixo`, there are multiple columns for different waste types. Analytical queries will frequently ask for the *total* waste collected. We should precalculate a `total_residuos` column.
*   **Collapsing Relations:** **Highly Applicable.** The 1:1 tables (`coleta_de_lixo`, `local_de_coleta`) must be collapsed back into the main `mutirao` table. The N:1 dimension tables should also be collapsed into the main tables to form flat structures.
*   **Vertical Partitioning:** **Not Applicable.** We actually want to *reverse* the existing vertical partitioning. Modern analytical databases (columnar formats) handle wide tables exceptionally well, making vertical partitioning unnecessary and detrimental to OLAP performance.

### 4 - Design Multiple Possible Schemas
*   **Design A (Star Schema):** Collapse the 1:1 relationships (`coleta_de_lixo`, `local_de_coleta`) into `mutirao_de_limpeza`. Keep `regiao`, `instituicao`, `bioma`, etc., as separate dimension tables. 
*   **Design B (Single Mega-Table):** Attempt to join both domains into a single table using `regiao` as the bridge. 
*   **Design C (Two "One-Big-Tables" / OBT):** Create two fully denormalized, independent wide tables—one for Clean-up Events and one for Conservation Units. All dimensions are collapsed into their respective fact tables.

### 5 - Evaluate and Choose the Most Consistent Schema
*   *Design A* still requires JOINs at query time, which we want to eliminate for maximum read performance.
*   *Design B* is mathematically dangerous. Because a single municipality can have multiple clean-up events and multiple conservation units, joining them directly will create a Cartesian product (fan-out), duplicating waste metrics and area metrics, leading to wildly inaccurate aggregations.
*   **Winner: Design C.** Creating two distinct "One-Big-Tables" (OBT) is the industry standard for modern data warehousing. It eliminates JOINs entirely, prevents metric duplication, and allows columnar storage engines to compress and scan the data with maximum efficiency.
