### 1 - Deep Analysis of the Provided SQL Script
The provided normalized schema represents a classic HR/Employee database (highly similar to the well-known MySQL `employees` sample database). 
* **Core Entities**: `employee`, `department`.
* **Transactional/History Tables**: `salary`, `title`.
* **Mapping/Junction Tables**: `department_employee`, `department_manager`.
* **Characteristics**: The schema is highly normalized (3NF). Every change in an employee's title, salary, or department generates a new row in the respective tables with `from_date` and `to_date`. 
* **Complexity**: To get a complete picture of an employee at any given time, one must join up to 6 tables (`employee` -> `salary` -> `title` -> `department_employee` -> `department` -> `department_manager`) while carefully matching the overlapping `from_date` and `to_date` intervals across all of them.

### 2 - Frequent Queries / Workload Context
Since we are optimizing for a **read-heavy analytical workload (OLAP)**, the database will likely face queries such as:
* *Point-in-time reporting*: "What was the average salary by department and title in 2020?"
* *Current state snapshots*: "Show me a list of all current employees, their current department, title, and salary."
* *Aggregations*: "What is the gender distribution and average tenure per department?"
* **Bottlenecks**: In the normalized model, answering these questions requires massive multi-table `JOIN` operations with complex `BETWEEN` or `>= / <=` date logic to align the timelines of salaries, titles, and departments. This will cause severe CPU and memory overhead during analytical reads.

### 3 - Evaluation of Common Denormalization Techniques
* **Adding Redundant Attributes**: Highly applicable. We can bring `dept_name` directly into the employee records to avoid joining the `department` table.
* **Adding Derived Attributes**: Highly applicable. Analytical queries frequently ask for current status. Adding an `is_current` boolean flag, `tenure_years`, and an `is_manager` flag will save expensive window functions and subqueries.
* **Collapsing Relations**: Highly applicable. We can merge `employee`, `salary`, `title`, `department_employee`, and `department` into a single "One Big Table" (OBT) to completely eliminate joins.
* **Vertical Partitioning**: Applicable if we want to separate the *current* snapshot of employees from their *historical* changes, as business users query the "current state" 80% of the time.

### 4 - Design Multiple Possible Denormalized Schemas
**Option A: The "One Big Table" (OBT) - Historical Fact Table**
* Combines all tables into a single `employee_analytics_history` table. 
* *Pros*: Zero joins required. Perfect for BI tools (Tableau, PowerBI).
* *Cons*: Requires ETL logic to slice overlapping date ranges (salary dates vs. title dates vs. department dates) into unified `effective_from_date` and `effective_to_date` periods.

**Option B: Current Snapshot + Aggregated Fact Tables**
* Table 1: `employee_current_snapshot` (Only holds the active title, salary, and department for each employee).
* Table 2: `department_yearly_metrics` (Pre-aggregated table with avg salary, headcount, etc., by year).
* *Pros*: Extremely fast for the most common queries.
* *Cons*: Loses granular row-level history for complex temporal analysis.

**Option C: Hybrid Approach (Unified History + Derived Aggregations)**
* Table 1: `employee_record_history` (OBT containing all historical intervals).
* Table 2: `department_summary_fact` (Pre-aggregated metrics per department and year).

### 5 - Evaluate and Choose the Most Consistent Schema
**Option C** is the most robust choice for a Senior Data Engineer building an analytical database. 
By creating a unified **`employee_record_history`** table, we collapse 6 tables into 1. We resolve the complex date-overlapping joins during the ETL phase, presenting analysts with a clean, flat table where every row represents a specific period where an employee's department, title, and salary remained unchanged. We will also add derived attributes (`is_current`, `is_manager`) to speed up filtering. 

Additionally, creating a **`department_yearly_summary`** table utilizes the *Adding Derived Attributes* technique to instantly serve high-level dashboard metrics without scanning millions of employee rows.

-- 1. ONE BIG TABLE (OBT): Unified Employee History
-- Collapses: employee, salary, title, department, department_employee, department_manager
DROP TABLE IF EXISTS employees.employee_record_history CASCADE;

CREATE TABLE IF NOT EXISTS employees.employee_record_history
(
    -- Employee Core Attributes
    employee_id bigint NOT NULL,
    first_name character varying(14) COLLATE pg_catalog."default" NOT NULL,
    last_name character varying(16) COLLATE pg_catalog."default" NOT NULL,
    gender employees.employee_gender NOT NULL,
    birth_date date NOT NULL,
    hire_date date NOT NULL,
    
    -- Collapsed Department Attributes
    department_id character(4) COLLATE pg_catalog."default",
    department_name character varying(40) COLLATE pg_catalog."default",
    
    -- Collapsed Title & Salary Attributes
    title character varying(50) COLLATE pg_catalog."default",
    salary_amount bigint,
    
    -- Derived Attributes
    is_manager boolean DEFAULT FALSE,     -- True if the employee was a manager during this period
    is_current boolean DEFAULT FALSE,     -- True if this is the employee's current active record
    age_at_period integer,                -- Derived: Age of employee during this specific period
    tenure_days integer,                  -- Derived: Days since hire_date at the start of this period
    
    -- Unified Slowly Changing Dimension (SCD) Dates
    -- Represents a continuous period where NOTHING (dept, title, salary) changed.
    effective_from_date date NOT NULL,
    effective_to_date date NOT NULL,
    
    CONSTRAINT pk_employee_record_history PRIMARY KEY (employee_id, effective_from_date)
);

-- Indexing for heavy analytical filtering
CREATE INDEX IF NOT EXISTS idx_emp_hist_dept ON employees.employee_record_history(department_name);
CREATE INDEX IF NOT EXISTS idx_emp_hist_title ON employees.employee_record_history(title);
CREATE INDEX IF NOT EXISTS idx_emp_hist_current ON employees.employee_record_history(is_current) WHERE is_current = TRUE;


-- 2. PRE-AGGREGATED FACT TABLE: Department Yearly Summary
-- Utilizes "Adding Derived Attributes" to instantly serve dashboard metrics
DROP TABLE IF EXISTS employees.department_yearly_summary CASCADE;

    department_name character varying(40) COLLATE pg_catalog."default" NOT NULL,
    
    -- Derived Aggregated Metrics
    total_employees integer NOT NULL,
    total_managers integer NOT NULL,
    total_female_employees integer NOT NULL,
    total_male_employees integer NOT NULL,
    avg_salary numeric(15, 2) NOT NULL,
    min_salary bigint NOT NULL,
    max_salary bigint NOT NULL,
    total_salary_expense bigint NOT NULL,
    
    CONSTRAINT pk_dept_yearly_summary PRIMARY KEY (report_year, department_id)
);

END;
