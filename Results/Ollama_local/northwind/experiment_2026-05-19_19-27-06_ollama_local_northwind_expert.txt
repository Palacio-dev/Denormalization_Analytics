# Executive Summary

The provided schema is a classic normalized "Northwind" sample database. While structurally sound for transactional integrity (ACID), it suffers from significant join overhead for analytical reporting. A single query like "Top 10 customers by sales in Region 1" currently requires joining `customers` → `orders` → `order_details` → `products` → `categories` → `regions`, resulting in a 6-table join tree.

This design proposes a **Read-Optimized Denormalized Schema** that introduces two primary artifacts:
1.  **`view_customer_sales_summary`**: A pre-aggregated wide table denormalizing `customer_id`, `region_id`, `order_date`, and `total_sales`. This eliminates the need to join `orders`, `order_details`, and `products` for standard reporting.
2.  **`materialized_view_order_metrics`**: A star-schema-style table denormalizing `order_id`, `product_id`, `quantity`, `unit_price`, and `discount` into a single fact table, eliminating the need to join `order_details` and `products` for line-item analysis.

**Correctness Strategy**: We introduce a **15-minute staleness window**. This is acceptable for reporting workloads where real-time transactional consistency is not the primary bottleneck. We enforce this via a `last_refresh_ts` column and a background refresh job (no triggers), ensuring data lineage is explicit and observable.

We will implement **Wide Read Tables** and **Pre-Aggregated Rollups**.

1.  **`fact_customer_sales` (Wide Read + Aggregation)**
    *   *Pattern*: Pre-joined dimensions (`customer`, `region`) + Aggregated metrics (`total_sales`, `avg_discount`).

    *   *Staleness*: Real-time (on write) or near-real-time (batch).

3.  **`fact_employees_territories` (Snapshot)**
    *   *Pattern*: Pre-joined `employee` and `territory` data.
    *   *Justification*: Simplifies hierarchy queries and region-based employee reporting.

## 5. Risk Analysis & Operational Plan

### Risk Assessment

| Risk | Impact | Mitigation Strategy |
| :--- | :--- | :--- |
| **Write Amplification** | Increased I/O on `INSERT` if denormalized on write. | **Strategy**: Denormalize on **Batch** (e.g., every 15 mins) rather than row-by-row. This keeps write latency low while maintaining consistency for reads. |
| **Data Staleness** | Reports show outdated figures. | **Strategy**: Explicit `last_refresh_ts`. Alerts if lag > 20 mins. Acceptable for reporting, not for transactional updates. |
| **Reconciliation Failures** | Denormalized values differ from source. | **Strategy**: Run a nightly `RECONCILIATION` query comparing `fact_customer_sales.total_sales` vs `SUM(order_details.quantity * unit_price)` from source. Log discrepancies to an `audit_log` table. |
| **Storage Growth** | Duplicate data consumes space. | **Strategy**: Enable **Row-Level Security** or **Compression** (`bloom filter` indexes) if using Postgres. The overhead is negligible compared to the I/O savings of 3-4 joins. |
| **Lock Contention** | Batch writes lock tables during peak hours. | **Strategy**: Schedule refresh jobs during off-peak (e.g., 02:00 UTC). Use `INSERT ... ON CONFLICT DO NOTHING` to prevent failures if data is already up to date. |
