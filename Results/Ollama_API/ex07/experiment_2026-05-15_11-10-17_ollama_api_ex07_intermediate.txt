# Denormalized Schema Design for Read Performance Optimization

## 1. Workload-Driven Analysis

### High-Cost Operations in Read Workload
Based on the schema, typical read queries likely involve:
- Retrieving usina details with owner names (joins Usina → Proprietarios)
- Getting location information including municipalities and states (joins Usina → Localizacoes → Municipios → Estados)
- Filtering by river information (Usina → Rios)
- Geographic queries based on coordinates
- Time-based filtering on update dates

These require multiple joins across 5+ tables, creating expensive operations especially with large datasets.

### Denormalization Candidates
1. **Proprietario name** - Frequently accessed with usina details
2. **Municipio/UF information** - Needed for regional analysis
3. **Rio name** - Often used for filtering and grouping
4. **Geographic coordinates** - Used in spatial queries
5. **Date fields** - Used for time-series analysis

## 2. Denormalization Design

### Proposed Denormalizations
1. **Wide "Usina_Dashboard" Table** - Pre-joined table with all frequently accessed attributes
2. **Usina_Summary_By_Rio** - Aggregate table grouped by river with count and average power
3. **Usina_Summary_By_State** - Aggregate table grouped by state with capacity metrics
4. **Covering indexes** for common query patterns
5. **Partitioning** by atualizacao date for time-series queries

### Index Strategy
- Composite indexes on (uf1, munic1) for regional filtering
- Index on rio for river-based queries
- Index on atualizacao for time-based queries
- Spatial index on (lat, long) for geographic queries

## 3. New Physical Schema

```sql
BEGIN;

-- Drop existing schema
DROP TABLE IF EXISTS public."Usina_Dashboard" CASCADE;
DROP TABLE IF EXISTS public."Usina_Summary_By_Rio" CASCADE;
DROP TABLE IF EXISTS public."Usina_Summary_By_State" CASCADE;

-- Main denormalized dashboard table (replaces all join operations)
CREATE TABLE public."Usina_Dashboard" (
    id INTEGER PRIMARY KEY,
    nome VARCHAR NOT NULL,
    -- Proprietario information
    proprietario VARCHAR,
    id_proprietario INTEGER,
    -- Power and operational data
    potkw FLOAT,
    potfis FLOAT,
    iniciop DATE,
    atualizacao DATE,
    estagio VARCHAR,
    -- Geographic coordinates
    lat DOUBLE PRECISION,
    long DOUBLE PRECISION,
    -- Location information (denormalized from 3 tables)
    id_localizacao INTEGER,
    longitude_loc DOUBLE PRECISION,
    latitude_loc DOUBLE PRECISION,
    -- Municipio 1 information
    munic1 VARCHAR NOT NULL,
    id_municipio1 INTEGER NOT NULL,
    uf1 CHAR(2) NOT NULL,
    -- Municipio 2 information (nullable)
    munic2 VARCHAR,
    id_municipio2 INTEGER,
    uf2 CHAR(2),
    -- River information
    rio VARCHAR NOT NULL,
    id_rio INTEGER NOT NULL
);

-- Summary table by river (precomputed aggregates)
CREATE TABLE public."Usina_Summary_By_Rio" (
    id_rio INTEGER NOT NULL,
    rio VARCHAR NOT NULL,
    usina_count INTEGER NOT NULL,
    total_potkw FLOAT NOT NULL,
    avg_potkw FLOAT NOT NULL,
    max_potkw FLOAT NOT NULL,
    min_potkw FLOAT NOT NULL,
    last_atualizacao DATE,
    PRIMARY KEY (id_rio)
);

-- Summary table by state (precomputed aggregates)
CREATE TABLE public."Usina_Summary_By_State" (
    uf CHAR(2) NOT NULL,
    usina_count INTEGER NOT NULL,
    total_potkw FLOAT NOT NULL,
    avg_potkw FLOAT NOT NULL,
    max_potkw FLOAT NOT NULL,
    operacional_count INTEGER NOT NULL,
    em_construcao_count INTEGER NOT NULL,
    PRIMARY KEY (uf)
);

-- Indexes for optimized read performance
-- Index for regional queries
CREATE INDEX idx_usina_dashboard_uf1_munic1 ON public."Usina_Dashboard" (uf1, munic1);

-- Index for river-based queries
CREATE INDEX idx_usina_dashboard_rio ON public."Usina_Dashboard" (rio);

-- Index for time-based queries
CREATE INDEX idx_usina_dashboard_atualizacao ON public."Usina_Dashboard" (atualizacao);

-- Composite index for geographic queries
CREATE INDEX idx_usina_dashboard_coordinates ON public."Usina_Dashboard" (lat, long);

-- Covering index for common dashboard queries
CREATE INDEX idx_usina_dashboard_cover ON public."Usina_Dashboard" (uf1, rio, atualizacao) 
INCLUDE (nome, proprietario, potkw, estagio);

-- Partitioning by atualizacao for time-series queries (PostgreSQL 10+)
-- This assumes the table will be recreated with partitioning in production
-- ALTER TABLE public."Usina_Dashboard" PARTITION BY RANGE (atualizacao);

-- Index for summary tables
CREATE INDEX idx_summary_rio_potkw ON public."Usina_Summary_By_Rio" (total_potkw DESC);
CREATE INDEX idx_summary_state_potkw ON public."Usina_Summary_By_State" (total_potkw DESC);

-- Materialized view for operational dashboard (refreshed periodically)
CREATE MATERIALIZED VIEW public."Usina_Operational_Dashboard" AS
SELECT 
    id,
    nome,
    proprietario,
    rio,
    uf1 AS estado,
    munic1 AS municipio,
    potkw,
    estagio,
    atualizacao
FROM public."Usina_Dashboard"
WHERE atualizacao >= CURRENT_DATE - INTERVAL '1 year'
ORDER BY atualizacao DESC;

-- Index for operational dashboard
CREATE INDEX idx_operational_dashboard ON public."Usina_Operational_Dashboard" (estado, estagio);

-- Comments for documentation
COMMENT ON TABLE public."Usina_Dashboard" IS 'Denormalized table for fast usina queries with all related information pre-joined';
COMMENT ON TABLE public."Usina_Summary_By_Rio" IS 'Precomputed aggregates by river for dashboard summaries';
COMMENT ON TABLE public."Usina_Summary_By_State" IS 'Precomputed aggregates by state for regional analysis';

COMMIT;
```

## Benefits of This Denormalization Approach

1. **Eliminates Joins**: The main dashboard table removes all join operations for typical queries
2. **Precomputed Aggregates**: Summary tables provide instant access to common metrics
3. **Optimized Indexing**: Covering indexes support common query patterns without table access
4. **Partitioning Ready**: Structure supports time-based partitioning for large datasets
5. **Materialized Views**: Provide optimized structures for specific dashboard needs
6. **Preserved Correctness**: All business-critical fields maintained with clear provenance

This design reduces typical query execution time from multiple joins across 6 tables to single table access with index support, dramatically improving read performance for dashboard and analytical workloads.
