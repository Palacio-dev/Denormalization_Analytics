# Senior Database Performance Engineer: Read-Optimized Schema Design

## 1. Workload-Driven Analysis

Based on the provided schema (Northwind sample), I have analyzed the typical read-heavy workloads for this domain. The current normalized design forces the database engine to perform expensive `JOIN` operations for almost every query, leading to high latency and complex execution plans.

### High-Cost Operations & Join Paths
1.  **Customer Profile Lookup (The "Customer Star" Pattern):**
    *   *Pattern:* `SELECT * FROM customers JOIN customers_demo ON ... JOIN categories ON ...`
    *   *Cost:* Requires scanning the `customers` table, joining it with `customers_demo` (lookup), and then `categories`. If filtering by region, the engine must scan all customers, join, filter, then join again.
    *   *Optimization Target:* Denormalize customer attributes directly into a single "Customer" table or a pre-joined view.

2.  **Sales Analysis (Order Aggregates):**
    *   *Pattern:* `SELECT SUM(quantity), AVG(unit_price) FROM orders JOIN order_details ... GROUP BY customer_id`
    *   *Cost:* Requires a full scan of `orders`, a nested loop or hash join to `order_details`, and then a `GROUP BY` operation. This is extremely slow on large datasets.
    *   *Optimization Target:* Create a **Materialized Aggregate Table** (`order_summarized`) that pre-calculates totals per customer and product.

3.  **Employee Hierarchy & Territory Lookup:**
    *   *Pattern:* `SELECT ... FROM employees JOIN employee_territories ... JOIN territories ...`
    *   *Cost:* Triple join depth. The `reports_to` relationship creates a recursive join potential if not handled carefully, but even single-level joins are costly without covering indexes.
    *   *Optimization Target:* Pre-join employee data with territory info in a "Wide Employee" table.

4.  **Product Availability & Supplier Chain:**
    *   *Pattern:* `SELECT ... FROM products JOIN order_details ... JOIN orders`
    *   *Cost:* Joining `products` with `order_details` is critical for stock vs. sold analysis.
    *   *Optimization Target:* Denormalize stock levels into the `products` table itself (read-only snapshot) to avoid joins.

### Denormalization Candidates
*   **`customers` table:** Add `customer_type_id` directly.
*   **`customers_demo` table:** Add `customer_id` to match the main table.
*   **`employees` table:** Add `territory_id` and `region_id` (derived from `employee_territories`).
*   **`products` table:** Add `category_name` and `supplier_name` (or ID) to avoid joins.
*   **`orders` table:** Add `shipper_name` to avoid joins.
*   **New Table `order_summarized`:** Pre-aggregated sales data (Grain: Customer/Product/Date).

## 2. Denormalization Design Strategy

### A. Wide Tables (Read-Optimized)
Instead of joining small tables, we will create "Wide" tables that contain the necessary foreign keys and lookup data inline. This allows queries to use simple `WHERE` clauses and `ORDER BY` without `JOIN`s.

1.  **`customers_wide`**: Combines `customers` and `customers_demo`. Includes `category_name` and `customer_desc`.
2.  **`employees_wide`**: Combines `employees` and `employee_territories`. Includes `territory_description`, `region_description`.
3.  **`products_wide`**: Combines `products` and `categories`/`suppliers`. Includes `category_name`, `supplier_name`, `unit_price`.
4.  **`orders_wide`**: Combines `orders` and `shippers`. Includes `shipper_name`.

### B. Materialized Aggregate Tables (Fact Tables)
For analytical queries (OLAP), we pre-compute aggregates to avoid `GROUP BY` operations.

1.  **`order_summarized`**:
    *   *Grain:* One row per `(customer_id, product_id, order_date)`.
    *   *Content:* `total_quantity`, `total_revenue`, `avg_discount`.
    *   *Update Strategy:* Triggers on `order_details` or `orders` to maintain freshness (or periodic ETL if write latency is low).

### C. Index Strategy
We will implement **Covering Indexes** (Index-Only Scans) where possible. This means the index contains all columns needed for the query, so the database never touches the heap (the actual table data).

*   **Composite Indexes:** `(region, customer_type_id, customer_id)` for customer lookups.
*   **Covering Indexes:** `(customer_id, category_name, customer_desc)` for `customers_wide`.
*   **Aggregate Index:** `(customer_id, product_id, order_date)` for `order_summarized` to support range scans and grouping.

### D. Partitioning
Given the date nature of orders, `orders_wide` and `order_summarized` should be partitioned by `order_date` (e.g., monthly or yearly) to allow efficient pruning of partitions during queries.

### Key Performance Improvements Explained

1.  **Elimination of Joins in Queries:**
    *   *Before:* `SELECT c.*, cd.customer_desc, cat.category_name FROM customers c JOIN customers_demo cd ON ... JOIN categories cat ON ...`
    *   *After:* `SELECT * FROM customers_wide WHERE region = 'AN'`
    *   *Impact:* The query planner can use the **covering index** `idx_customers_wide_region_type` to fetch all data without touching the heap, reducing I/O significantly.

2.  **Instant Aggregates:**
    *   *Before:* `SELECT SUM(quantity), AVG(price) FROM orders JOIN order_details ... GROUP BY customer_id` (Requires scanning millions of rows and computing aggregates at runtime).
    *   *After:* `SELECT * FROM order_summarized WHERE customer_id = 'ALAN'` (Returns pre-calculated sums).
    *   *Impact:* Latency drops from seconds to milliseconds.

3.  **Simplified Data Model:**
    *   By denormalizing `shipper_name` into `orders_wide`, queries regarding shipping speed no longer need to join `orders` -> `ship_via` -> `shippers`.
    *   By adding `category_name` to `products_wide`, product search queries become trivial `LIKE` operations without joins.

### Implementation Notes
*   **Write Performance:** The triggers (`trg_update_order_summarized`) add a small overhead to `INSERT`/`UPDATE` operations on `order_details`. This is acceptable for read-heavy workloads where the cost of recomputing aggregates on every query outweighs the trigger cost.
*   **Data Consistency:** The `customers_wide` and `employees_wide` tables are treated as read-only snapshots in this design. If the underlying `customers` table changes (e.g., a new customer is added), the wide table must be refreshed via an ETL job or a specific refresh trigger. For simplicity in this schema, I focused on the `order_summarized` trigger as it has a direct parent-child relationship.
*   **Scalability:** The partitioning suggestion (by date) is omitted from DDL for brevity but should be applied to `orders_wide` and `order_summarized` if the dataset exceeds 100 million rows.
