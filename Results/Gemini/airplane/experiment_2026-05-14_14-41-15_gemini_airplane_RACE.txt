### Step 1: Deep Analysis of the Provided SQL Script
The provided schema is a highly normalized relational model for an airline booking system. 
* **Core Entities**: `bookings`, `tickets`, `flights`, `aircrafts_data`, `airports_data`, `seats`.
* **Mapping/Transaction Entities**: `ticket_flights` (maps tickets to flights), `boarding_passes` (maps passengers to physical seats on a flight).
* **Relationships**: A booking contains multiple tickets. A ticket spans multiple flight segments (`ticket_flights`). A flight uses one aircraft and connects two airports.
* **Current Bottlenecks**: The schema is heavily fragmented. To answer a simple analytical question like *"What is the total revenue and passenger count for flights from New York to London on Boeing 737s?"*, the database must execute a 6-table `JOIN` (`ticket_flights` → `flights` → `airports_data` (dep) → `airports_data` (arr) → `aircrafts_data` → `tickets`).

### Step 2: Frequent Queries / Workload Context Inference
In an airline analytical environment, the most resource-intensive read queries typically revolve around two main "grains" of data:
1. **Flight-Level Analytics**: 
   * *Queries*: On-time performance (delays), route profitability, average seat occupancy rates, and aircraft utilization.
   * *Bottleneck*: Requires joining `flights` with `airports`, `aircrafts`, and aggregating `ticket_flights` and `seats`.
2. **Passenger/Ticket-Level Analytics**: 
   * *Queries*: Revenue analysis by fare class, booking curves (days between booking and flight), and passenger travel patterns.
   * *Bottleneck*: Requires joining `bookings`, `tickets`, `ticket_flights`, `boarding_passes`, and `flights`.

### Step 3: Evaluation of Common Denormalization Techniques
* **1 - Adding Redundant Attributes**: *Highly Applicable.* We should embed aircraft details (model) and airport details (city, name) directly into the flight records. We should also embed booking dates and flight dates into the ticket records to avoid temporal joins.
* **2 - Adding Derived Attributes**: *Highly Applicable.* We should precalculate flight delays (actual vs. scheduled times), total revenue per flight, total passengers boarded, and aircraft capacity (total seats).
* **3 - Collapsing Relations**: *Highly Applicable.* `boarding_passes` has a 1:1 relationship with `ticket_flights` and should be merged. `bookings` and `tickets` can be collapsed into the ticket segments. `airports_data` and `aircrafts_data` can be collapsed into the flights table.
* **4 - Vertical Partitioning**: *Applicable.* The `contact_data` (JSONB) in the `tickets` table is PII (Personally Identifiable Information) and is rarely used in aggregate analytics. We should partition this out to keep the analytical tables narrow and fast.

### Step 4: Design Multiple Possible Denormalized Schemas
* **Option A: Traditional Star Schema**
  * *Fact Table*: `fact_ticket_sales`
  * *Dimensions*: `dim_flight`, `dim_passenger`, `dim_airport`, `dim_aircraft`, `dim_date`.
  * *Pros*: Standard BI tool compatibility.
  * *Cons*: Still requires multiple joins (e.g., Fact to Flight to Airport) which can degrade performance on massive datasets.
* **Option B: Single "One Big Table" (OBT)**
  * A single massive table containing every ticket segment, with all flight, airport, and booking data duplicated on every row.
  * *Pros*: Zero joins required. Maximum read speed for columnar databases.
  * *Cons*: Extreme data redundancy. Analyzing flight-level metrics (like flight delays) becomes complex because flight data is duplicated across hundreds of ticket rows.
* **Option C: Hybrid Dual-Grain OBT (Recommended)**
  * Create two wide, denormalized tables representing the two primary analytical grains:
    1. `analytical_flights`: One row per flight. Contains all aircraft, route, and pre-aggregated revenue/passenger metrics.
    2. `analytical_ticket_segments`: One row per passenger flight segment. Contains all booking, ticket, boarding, and high-level flight context.

### Step 5: Evaluate and Choose the Most Consistent Schema
**Option C (Hybrid Dual-Grain OBT)** is the superior choice. 
By creating two distinct denormalized tables, we cater perfectly to the two main types of analytical queries without the severe redundancy of a single OBT or the join-heavy overhead of a Star Schema. 
* If the business wants to analyze flight performance, delays, or occupancy, they query `analytical_flights` (0 joins).
* If the business wants to analyze passenger revenue, fare classes, or booking behavior, they query `analytical_ticket_segments` (0 joins).

### Step 6: Design the New Denormalized Schema

Here is the physical schema for the denormalized analytical database.

```sql
BEGIN;

-- ==============================================================================
-- TABLE 1: analytical_flights (Flight-Level Grain)
-- Applies Techniques: Collapsing Relations, Redundant Attributes, Derived Attributes
-- ==============================================================================
DROP TABLE IF EXISTS analytics.analytical_flights CASCADE;

CREATE TABLE analytics.analytical_flights
(
    flight_id integer NOT NULL,
    flight_no character(6) NOT NULL,
    
    -- Temporal Data
    scheduled_departure timestamp with time zone NOT NULL,
    scheduled_arrival timestamp with time zone NOT NULL,
    actual_departure timestamp with time zone,
    actual_arrival timestamp with time zone,
    
    -- Derived Attributes (Precalculated for fast OLAP)
    departure_delay_minutes integer, -- (actual_departure - scheduled_departure)
    arrival_delay_minutes integer,   -- (actual_arrival - scheduled_arrival)
    is_cancelled boolean,            -- Derived from status
    
    -- Redundant Attributes: Aircraft Data (Collapsed)
    aircraft_code character(3) NOT NULL,
    aircraft_model jsonb NOT NULL,
    aircraft_range integer NOT NULL,
    aircraft_total_capacity integer, -- Derived: COUNT(seat_no) from seats table
    
    -- Redundant Attributes: Departure Airport (Collapsed)
    dep_airport_code character(3) NOT NULL,
    dep_airport_name jsonb NOT NULL,
    dep_city jsonb NOT NULL,
    dep_timezone text NOT NULL,
    
    -- Redundant Attributes: Arrival Airport (Collapsed)
    arr_airport_code character(3) NOT NULL,
    arr_airport_name jsonb NOT NULL,
    arr_city jsonb NOT NULL,
    arr_timezone text NOT NULL,
    
    -- Derived Attributes: Aggregated Ticket/Revenue Data
    total_tickets_sold integer,      -- Derived: COUNT() from ticket_flights
    total_revenue numeric(12, 2),    -- Derived: SUM(amount) from ticket_flights
    occupancy_rate numeric(5, 4),    -- Derived: total_tickets_sold / aircraft_total_capacity
    
    CONSTRAINT analytical_flights_pkey PRIMARY KEY (flight_id)
);

COMMENT ON TABLE analytics.analytical_flights IS 'Denormalized OBT for flight-level analytics (delays, occupancy, route performance).';

-- ==============================================================================
-- TABLE 2: analytical_ticket_segments (Passenger/Ticket-Level Grain)
-- Applies Techniques: Collapsing Relations, Redundant Attributes, Vertical Partitioning
-- ==============================================================================
DROP TABLE IF EXISTS analytics.analytical_ticket_segments CASCADE;

CREATE TABLE analytics.analytical_ticket_segments
(
    ticket_no character(13) NOT NULL,
    flight_id integer NOT NULL,
    
    -- Redundant Attributes: Booking Data (Collapsed)
    book_ref character(6) NOT NULL,
    book_date timestamp with time zone NOT NULL,
    days_booked_before_flight integer, -- Derived: (scheduled_departure - book_date)
    
    -- Redundant Attributes: Passenger Data (Collapsed)
    -- Note: contact_data (PII) is Vertically Partitioned OUT of this table for security and performance.
    passenger_id character varying(20) NOT NULL,
    passenger_name text NOT NULL,
    
    -- Core Ticket Flight Data
    fare_conditions character varying(10) NOT NULL,
    amount numeric(10, 2) NOT NULL,
    
    -- Redundant Attributes: Boarding Pass Data (Collapsed 1:1 relation)
    has_boarded boolean NOT NULL,      -- Derived: TRUE if boarding pass exists
    boarding_no integer,
    seat_no character varying(4),
    
    -- Redundant Attributes: Flight Context (To prevent joining back to flights table)
    flight_no character(6) NOT NULL,
    scheduled_departure timestamp with time zone NOT NULL,
    dep_airport_code character(3) NOT NULL,
    arr_airport_code character(3) NOT NULL,
    
    CONSTRAINT analytical_ticket_segments_pkey PRIMARY KEY (ticket_no, flight_id)
);

COMMENT ON TABLE analytics.analytical_ticket_segments IS 'Denormalized OBT for passenger-level analytics (revenue, fare classes, booking behavior).';

-- ==============================================================================
-- INDEXES FOR ANALYTICAL WORKLOADS
-- ==============================================================================
-- In OLAP, we index heavily on dimensions used for filtering and grouping (e.g., dates, routes, models)

CREATE INDEX idx_ana_flights_dates ON analytics.analytical_flights (scheduled_departure);
CREATE INDEX idx_ana_flights_route ON analytics.analytical_flights (dep_airport_code, arr_airport_code);
CREATE INDEX idx_ana_flights_aircraft ON analytics.analytical_flights (aircraft_code);

CREATE INDEX idx_ana_tickets_book_date ON analytics.analytical_ticket_segments (book_date);
CREATE INDEX idx_ana_tickets_flight_date ON analytics.analytical_ticket_segments (scheduled_departure);
CREATE INDEX idx_ana_tickets_fare ON analytics.analytical_ticket_segments (fare_conditions);

END;
```

### Summary of Improvements:
1. **Eliminated Joins**: Complex 6-table joins are reduced to single-table scans. 
2. **Pre-computed Aggregations**: Heavy mathematical operations (e.g., calculating delays, occupancy rates, and days booked in advance) are done once during the ETL process rather than at query time.
3. **Optimized I/O**: By vertically partitioning the heavy JSONB `contact_data` out of the analytical model, we significantly reduce the amount of disk I/O required to scan the tables.
