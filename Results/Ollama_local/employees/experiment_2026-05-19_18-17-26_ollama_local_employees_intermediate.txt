### Workload-Driven Analysis & Strategy

Based on the provided schema, the current design suffers from the classic "Entity-Relationship" model overhead, which is detrimental to read-heavy workloads. Here is the breakdown of the bottlenecks and the denormalization strategy:

#### 1. Join Paths & High-Cost Operations
In a read-heavy scenario (e.g., reporting on employee compensation, tenure, or organizational structure), the following patterns dominate:

*   **The "Employee Profile" Query:**
    *   *Current Path:* `SELECT ... FROM employees.employee e JOIN employees.department_employee de ON e.id = de.employee_id JOIN employees.department d ON de.department_id = d.id`
    *   *Cost:* Requires a hash join or nested loop join between three tables. If `department_id` is not highly selective, the optimizer might perform a full table scan on `department_employee` to find the link.
    *   *Impact:* Latency spikes as row count increases.

*   **The "Compensation History" Query:**
    *   *Current Path:* `SELECT ... FROM employees.salary s JOIN employees.employee e ON s.employee_id = e.id WHERE s.from_date < '2023-01-01'`
    *   *Cost:* The primary key is `(employee_id, from_date)`. While good for exact lookups, filtering by date ranges often requires scanning the entire index or table if the `employee_id` filter isn't tight enough to utilize the index efficiently.
    *   *Impact:* High I/O for historical reports.

*   **The "Title Evolution" Query:**
    *   *Current Path:* `SELECT ... FROM employees.title t JOIN employees.employee e ON t.employee_id = e.id`
    *   *Cost:* Similar to salary, this requires joining a sparse table (many employees have no title history) to a dense table.

#### 2. Denormalization Candidates
To optimize reads without sacrificing correctness (via foreign keys and audit trails), we will implement a **Star Schema** approach with specific denormalized views:

1.  **Denormalize `department_employee` & `department_manager`:**
    *   Instead of joining back to `department` for every row, we will include `dept_name` directly in the fact tables.
    *   *Reasoning:* Department names rarely change (low write frequency) but are accessed frequently in reports.

2.  **Create a "Wide" Employee Fact Table (`employees.employee_facts`):**
    *   Instead of joining `salary`, `title`, and `department_employee` at query time, we will denormalize these attributes into a single table.
    *   *Reasoning:* Eliminates the need for complex joins in the application layer.
    *   *Grain:* One row per employee per effective date (or a snapshot of the current state). For simplicity and performance, we will create a **Current State Snapshot** table for the most common read pattern, plus a **History** table if temporal queries are critical. Given the schema has `from_date`/`to_date`, we will create a denormalized table that includes the *latest* effective attributes for each employee to serve "Who is who?" queries instantly.

3.  **Precomputed Aggregates:**
    *   Create a summary table for "Employee Stats" (e.g., `avg_salary`, `current_dept`) if the workload involves grouping by employee.

4.  **Covering Indexes:**
    *   Replace standard B-Tree indexes with **Covering Indexes** (Index Only Scans) where possible to avoid touching the heap.

### New Physical Schema Design

The new schema introduces three key tables:
1.  **`employees.employee_facts`**: A denormalized "wide" table containing the current state of an employee (Name, Gender, Current Dept Name, Current Title, Current Salary).
2.  **`employees.employee_history`**: A denormalized table containing the *effective history* of attributes (e.g., "John Doe was in Marketing from Jan 2020 to Dec 2021").
3.  **`employees.department_stats`**: A pre-aggregated table for fast department-level reporting.

**Correctness Strategy:**
*   **Foreign Keys:** Retained on the original tables to ensure referential integrity for writes.
*   **Triggers:** PostgreSQL triggers will automatically update the denormalized tables whenever `employees.employee`, `department`, `salary`, or `title` change.
*   **Audit:** The original tables remain untouched for debugging and audit purposes.

### Implementation Notes & Best Practices

1.  **Write Overhead:** The triggers (`trg_update_*`) introduce overhead on every `UPDATE` operation.
    *   *Mitigation:* If the write load is high, consider disabling these triggers during heavy batch loads and refreshing the denormalized tables manually, or using `CREATE MATERIALIZED VIEW` with `REFRESH` commands instead of triggers.
2.  **History Maintenance:** The `employee_history` table can grow indefinitely.
    *   *Strategy:* Implement a retention policy (e.g., drop records older than 5 years) via a scheduled job (cron/PG_Cron) to prevent table bloat.
3.  **Query Optimization:**
    *   The new `employee_facts` table allows queries like `SELECT * FROM employee_facts WHERE current_dept_name = 'Sales'` to run in milliseconds (Index Only Scan) without touching the original normalized tables.
    *   The `department_stats` table eliminates the need for `GROUP BY` operations in reports, reducing CPU usage significantly.
4.  **Correctness:**
    *   The foreign keys in the original tables (`department_employee`, `salary`, etc.) remain intact. This ensures that if data is corrupted in the denormalized tables, the source of truth (the normalized tables) remains valid for debugging.
    *   The triggers ensure the denormalized tables stay in sync with the source of truth.
