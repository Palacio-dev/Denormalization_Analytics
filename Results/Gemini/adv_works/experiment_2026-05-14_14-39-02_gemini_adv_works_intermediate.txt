As a senior database performance engineer, I have analyzed your highly normalized (3NF) AdventureWorks schema. While excellent for strict OLTP write-consistency, it forces the database to perform expensive, multi-table joins for almost every read query. 

To optimize for a **read-heavy, low-latency workload**, we must shift from a purely relational model to a **Read-Optimized / Dimensional Model**.

Here is the step-by-step workload analysis, denormalization design, and the resulting PostgreSQL physical schema.

---

### 1. Workload-Driven Analysis

In a business like AdventureWorks, read-heavy workloads typically fall into three patterns. Here are the current bottlenecks:

*   **Pattern A: E-Commerce Product Catalog Browsing**
    *   *The Query:* "Show me all active products, their prices, category, subcategory, and model descriptions."
    *   *The Bottleneck:* Requires joining `Product` → `ProductSubcategory` → `ProductCategory` → `ProductModel`. This 4-table join happens on every page load.
*   **Pattern B: Customer Profile & Account Dashboards**
    *   *The Query:* "Get the customer's name, primary email, phone number, and shipping address."
    *   *The Bottleneck:* Requires joining `Customer` → `Person` → `EmailAddress` → `PersonPhone` → `BusinessEntityAddress` → `Address` → `StateProvince` → `CountryRegion`. An 8-table join just to load a user profile.
*   **Pattern C: Sales Reporting & Order History**
    *   *The Query:* "Show me order details including customer name, product name, quantity, and total price, filtered by date."
    *   *The Bottleneck:* Requires joining `SalesOrderHeader` → `SalesOrderDetail` → `Product` → `Customer` → `Person`. Aggregating this over millions of rows for dashboards causes severe CPU and memory spikes.

---

### 2. Denormalization Design

To solve these bottlenecks while preserving correctness, I propose the following denormalizations:

1.  **`production.mat_product_catalog` (Materialized View)**
    *   *Action:* Pre-join Product, Category, Subcategory, and Model.
    *   *Grain:* 1 row per `ProductID`.
    *   *Why:* Eliminates 4 joins for the website frontend. Materialized views are perfect here because product catalogs change infrequently (e.g., nightly or weekly).
2.  **`person.mat_customer_profile` (Materialized View)**
    *   *Action:* Flatten the 8-table customer hierarchy into a single wide table containing the primary email, phone, and address.
    *   *Grain:* 1 row per `CustomerID`.
    *   *Why:* O(1) lookups for user profiles.
3.  **`sales.denorm_sales_order_line` (Partitioned Fact Table)**
    *   *Action:* Create a wide "Fact" table combining Order Header, Order Detail, and snapshot attributes (Product Name, Customer Name at the time of sale).
    *   *Grain:* 1 row per `SalesOrderDetailID`.
    *   *Why:* Eliminates joins for order history. We will **Partition by Range (`orderdate`)** because sales queries almost always filter by time (e.g., "Last 30 days").
4.  **`sales.mat_daily_sales_summary` (Precomputed Aggregate)**
    *   *Action:* Roll up sales by Date, Territory, and Category.
    *   *Grain:* 1 row per Day per Territory per Category.
    *   *Why:* Dashboards will query this table in milliseconds instead of scanning millions of order lines.
5.  **Covering Indexes (`INCLUDE`)**
    *   *Action:* Create B-Tree indexes on filter columns (e.g., `orderdate`) that `INCLUDE` payload columns (e.g., `totaldue`).
    *   *Why:* Enables **Index-Only Scans**, allowing Postgres to return data directly from memory without touching the table heap.

---

### 3. New Physical Schema (DDL)

Run this script in your PostgreSQL database to generate the read-optimized schema.

```sql
BEGIN;

-- ============================================================================
-- 1. PRODUCT CATALOG (Pre-joined Wide Table / Materialized View)
-- Solves Pattern A: Eliminates 4 joins for product listing pages.
-- ============================================================================

CREATE MATERIALIZED VIEW production.mat_product_catalog AS
SELECT 
    p.productid,
    p.productnumber,
    p.name AS product_name,
    p.color,
    p.standardcost,
    p.listprice,
    p.size,
    p.weight,
    pc.name AS category_name,
    ps.name AS subcategory_name,
    pm.name AS model_name,
    p.sellstartdate,
    p.sellenddate
FROM production.product p
LEFT JOIN production.productsubcategory ps ON p.productsubcategoryid = ps.productsubcategoryid
LEFT JOIN production.productcategory pc ON ps.productcategoryid = pc.productcategoryid
LEFT JOIN production.productmodel pm ON p.productmodelid = pm.productmodelid
WHERE p.finishedgoodsflag = true;

-- Indexes for fast frontend filtering and sorting
CREATE UNIQUE INDEX idx_mat_product_catalog_id ON production.mat_product_catalog(productid);
CREATE INDEX idx_mat_product_catalog_category ON production.mat_product_catalog(category_name, subcategory_name);
CREATE INDEX idx_mat_product_catalog_price ON production.mat_product_catalog(listprice);


-- ============================================================================
-- 2. CUSTOMER PROFILE (Pre-joined Wide Table / Materialized View)
-- Solves Pattern B: Eliminates 8 joins for user profile lookups.
-- Note: Uses LATERAL/LIMIT 1 to ensure strict 1:1 grain per customer.
-- ============================================================================

CREATE MATERIALIZED VIEW person.mat_customer_profile AS
SELECT 
    c.customerid,
    p.firstname,
    p.lastname,
    CONCAT(p.firstname, ' ', p.lastname) AS full_name,
    e.emailaddress,
    ph.phonenumber,
    a.addressline1,
    a.city,
    sp.name AS state_province,
    cr.name AS country_region,
    a.postalcode
FROM sales.customer c
JOIN person.person p ON c.personid = p.businessentityid
-- Get primary email
LEFT JOIN LATERAL (
    SELECT emailaddress FROM person.emailaddress 
    WHERE businessentityid = p.businessentityid LIMIT 1
) e ON true
-- Get primary phone
LEFT JOIN LATERAL (
    SELECT phonenumber FROM person.personphone 
    WHERE businessentityid = p.businessentityid LIMIT 1
) ph ON true
-- Get primary shipping address
LEFT JOIN LATERAL (
    SELECT addressid FROM person.businessentityaddress 
    WHERE businessentityid = p.businessentityid AND addresstypeid = 5 -- Assuming 5 is Shipping
    LIMIT 1
) bea ON true
LEFT JOIN person.address a ON bea.addressid = a.addressid
LEFT JOIN person.stateprovince sp ON a.stateprovinceid = sp.stateprovinceid
LEFT JOIN person.countryregion cr ON sp.countryregioncode = cr.countryregioncode;

-- Index for instant profile lookups
CREATE UNIQUE INDEX idx_mat_customer_profile_id ON person.mat_customer_profile(customerid);
CREATE INDEX idx_mat_customer_profile_email ON person.mat_customer_profile(emailaddress);


-- ============================================================================
-- 3. DENORMALIZED SALES ORDER LINE (Partitioned Fact Table)
-- Solves Pattern C: Eliminates joins for order history and analytics.
-- We use a physical table here (populated via ETL/Triggers) partitioned by date.
-- ============================================================================

CREATE TABLE sales.denorm_sales_order_line (
    salesorderid integer NOT NULL,
    salesorderdetailid integer NOT NULL,
    orderdate timestamp without time zone NOT NULL,
    duedate timestamp without time zone NOT NULL,
    shipdate timestamp without time zone,
    status smallint NOT NULL,
    onlineorderflag boolean NOT NULL,
    customerid integer NOT NULL,
    customer_full_name character varying(150), -- Snapshot attribute
    territory_name character varying(50),      -- Snapshot attribute
    productid integer NOT NULL,
    product_name character varying(50),        -- Snapshot attribute
    category_name character varying(50),       -- Snapshot attribute
    orderqty smallint NOT NULL,
    unitprice numeric NOT NULL,
    unitpricediscount numeric NOT NULL,
    linetotal numeric GENERATED ALWAYS AS (orderqty * (unitprice - unitpricediscount)) STORED,
    CONSTRAINT pk_denorm_sales_order_line PRIMARY KEY (orderdate, salesorderid, salesorderdetailid)
) PARTITION BY RANGE (orderdate);

-- Create partitions (Example: Yearly. In production, you'd script this or use pg_partman)
CREATE TABLE sales.denorm_sales_order_line_2011 PARTITION OF sales.denorm_sales_order_line FOR VALUES FROM ('2011-01-01') TO ('2012-01-01');
CREATE TABLE sales.denorm_sales_order_line_2012 PARTITION OF sales.denorm_sales_order_line FOR VALUES FROM ('2012-01-01') TO ('2013-01-01');
CREATE TABLE sales.denorm_sales_order_line_2013 PARTITION OF sales.denorm_sales_order_line FOR VALUES FROM ('2013-01-01') TO ('2014-01-01');
CREATE TABLE sales.denorm_sales_order_line_2014 PARTITION OF sales.denorm_sales_order_line FOR VALUES FROM ('2014-01-01') TO ('2015-01-01');

-- Index Strategy for the Fact Table
-- 1. For Customer Order History UI (Covering index allows Index-Only Scan)
CREATE INDEX idx_denorm_sales_cust_date 
    ON sales.denorm_sales_order_line (customerid, orderdate DESC) 
    INCLUDE (salesorderid, product_name, orderqty, linetotal, status);

-- 2. For Product Sales Analytics
CREATE INDEX idx_denorm_sales_prod_date 
    ON sales.denorm_sales_order_line (productid, orderdate DESC)
    INCLUDE (orderqty, linetotal);


-- ============================================================================
-- 4. DAILY SALES SUMMARY (Precomputed Aggregate)
-- Solves Pattern C (Dashboards): Millisecond response times for charts.
-- ============================================================================

CREATE MATERIALIZED VIEW sales.mat_daily_sales_summary AS
SELECT 
    DATE(orderdate) AS sales_date,
    territory_name,
    category_name,
    COUNT(DISTINCT salesorderid) AS total_orders,
    SUM(orderqty) AS total_items_sold,
    SUM(linetotal) AS total_revenue
FROM sales.denorm_sales_order_line
GROUP BY 
    DATE(orderdate),
    territory_name,
    category_name;

-- Indexes for Dashboard Date-Range Queries
CREATE UNIQUE INDEX idx_mat_daily_sales_summary_pk ON sales.mat_daily_sales_summary(sales_date, territory_name, category_name);
CREATE INDEX idx_mat_daily_sales_summary_date ON sales.mat_daily_sales_summary(sales_date DESC) INCLUDE (total_revenue, total_orders);

COMMIT;
```

### How to maintain this in production:
1. **Materialized Views (`mat_product_catalog`, `mat_customer_profile`, `mat_daily_sales_summary`):** Set up a pg_cron job or Airflow DAG to run `REFRESH MATERIALIZED VIEW CONCURRENTLY <view_name>;`. This updates the read-models in the background with zero downtime or locking to the read queries.
2. **Fact Table (`denorm_sales_order_line`):** Because order data is transactional, you should populate this table using your application code (Dual-write), a Database Trigger on `SalesOrderDetail`, or a Change Data Capture (CDC) tool like Debezium streaming into this table.
