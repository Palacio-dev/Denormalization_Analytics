**B. Frequent Queries / Workload Context**
*   **Primary Workload:** Reporting on specimen attributes (location, date, taxonomy) grouped by higher-level categories (Family, Order, Collection).
*   **Bottlenecks:**
    1.  **Taxonomy Depth:** Queries often need to join `tombos` -> `especie` -> `genero` -> `familia` -> `sub_familia`.
    2.  **Location Context:** Queries need `tombos` -> `local_coleta` -> `cidade` -> `estado` -> `pais`.
    3.  **Temporal Analysis:** Grouping by `data_coleta_ano` and `fase_sucessional`.
    4.  **Collection Stats:** Aggregating `colecoes_anexas` linked to `tombos`.

**C. Evaluation of Denormalization Techniques**
1.  **Adding Redundant Attributes:** *High Value.* We will duplicate taxonomic paths (Family, Genus, Sub-family) and location paths (City, State, Country) directly in the `tombos` table to avoid joins.
2.  **Adding Derived Attributes:** *Medium Value.* We can precompute counts of specimens per family per year, but this might conflict with the "Single Source of Truth" if data changes frequently. For a read-heavy analytical model, redundancy is preferred over derived columns to ensure consistency.
3.  **Collapsing Relations:** *High Value.* We will merge the hierarchy tables (`generos`, `familias`, `sub_familias`, `sub_especies`) into the main `tombos` table. Similarly, we will merge `solos`, `relevos`, `vegetacoes`, and `locais_coleta` details.
4.  **Vertical Partitioning:** *Low Value.* The `tombos` table is large, but splitting it vertically (e.g., separating taxonomic data from geographic data) would increase the number of tables and potentially complicate the "flat" analytical view. A wide table is better for analytics here.

### **Explanation of the New Schema**

1.  **Single Table of Truth for Analytics (`tombos_analiticos`):**
    Instead of joining `tombos` with `familias`, `generos`, `paises`, `estados`, `cidades`, `solos`, `relevos`, `vegetacoes`, `fase_sucessional`, `locais_coleta`, `colecoes_anexas`, `tipos`, `autores`, `remessas`, and `coletores`, we have flattened all these relationships into a single wide table.

2.  **Redundant Attributes (Technique 1):**
    Columns like `familia_nome`, `genero_nome`, `cidade_nome`, `estado_nome`, `pais_nome`, etc., are duplicated. This eliminates the need for `JOIN` operations in queries such as:
    *   *Old:* `SELECT t.nome, f.nome FROM tombos t JOIN familias f ON t.familia_id = f.id ...`
    *   *New:* `SELECT t.nome, t.familia_nome FROM tombos_analiticos t ...`

3.  **Collapsing Relations (Technique 3):**
    The complex hierarchy of Species -> Genus -> Family -> Sub-family has been collapsed. The `tombos_analiticos` table now holds the leaf node information directly, allowing for immediate filtering by sub-species name without traversing multiple levels.

4.  **Performance Impact:**
    *   **Reads:** Queries that previously required 5-6 `JOIN`s now require 0 `JOIN`s. Indexes on the redundant text columns (e.g., `idx_analiticos_familia`) allow for extremely fast filtering by taxonomic name.
    *   **Writes:** Writing to `tombos_analiticos` requires updating the redundant columns alongside the primary `tombos` table. In a production environment, this is typically handled by triggers or a separate data loading process (ETL) that keeps the two tables synchronized, as the `tombos_analiticos` table is likely a snapshot or read-only view of the live data for reporting purposes.
