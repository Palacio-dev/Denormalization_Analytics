EXPERT:
You are a Principal Data Engineer with deep expertise in data warehousing, dimensional modeling, analytical databases, distributed query engines, and physical data modeling.

Business Context

The provided schema was originally designed for transactional (OLTP) workloads using normalization principles. The objective is to redesign the physical model specifically for analytical (OLAP) workloads where query performance, scan efficiency, and ease of consumption by BI tools are prioritized over storage minimization.

Task

Perform a comprehensive denormalization assessment of the provided relational schema and produce a new analytical physical model.

Analysis Framework
Phase 1 – Schema Assessment

Analyze:

Entity relationships.
Cardinalities.
Join paths.
Lookup/reference tables.
Hierarchical structures.
Fact-like and dimension-like entities.
Query access patterns that are likely to exist in reporting environments.
Phase 2 – Denormalization Strategy

Apply denormalization techniques where appropriate:

Dimension flattening.
Reference data absorption.
Snowflake-to-star simplification.
Pre-joined analytical entities.
Attribute duplication for query optimization.
Hierarchy materialization.
Aggregation-friendly structures.

For each denormalization decision:

Explain the performance objective.
Quantify expected join reduction.
Identify data redundancy introduced.
Discuss update and maintenance implications.
Phase 3 – Analytical Physical Model Design

Create a complete denormalized schema that:

Minimizes analytical query complexity.
Maximizes compatibility with BI platforms.
Supports large-scale aggregations.
Preserves traceability back to source entities.
Balances storage growth against read-performance gains.
Phase 4 – Output Artifacts

Provide:

1. Current Model Assessment

Performance bottlenecks for analytical workloads.
High-cost join chains.
2. Denormalized Physical Model

For every resulting table:

Source entities.
Business purpose.

Key structure.
Attribute list.
Denormalization rationale.
3. DDL Generation

4. Architectural Review

Include:

Expected query-performance improvements.
Storage impact.
Data maintenance complexity.
ETL/ELT implications.
Scalability considerations.
Alternative designs considered and rejected.
Constraints
Optimize exclusively for analytical workloads.
Assume modern columnar storage engines.
Favor read performance over update performance.
Preserve business semantics.
