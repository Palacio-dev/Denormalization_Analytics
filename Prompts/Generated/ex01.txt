BEGGINER:
You are a database expert.

I have a physical model of a relational database. I want to create a denormalized version of this database 
to improve query performance for analytics and reporting.

Please:

- Analyze the relational schema.
- Identify tables that can be joined together.
- Create a denormalized version of the schema.
- Reduce the number of joins needed for analytical queries.
- Show the new physical schema with tables, columns, primary keys, and foreign keys.
- Explain the changes you made and why they improve performance.

The goal is to optimize the database for read operations and business intelligence workloads.

Return the result in a structured format.

INTERMEDIATE:
You are an experienced Data Architect specialized in analytical databases and dimensional modeling.

Given a relational database physical model, redesign it into a denormalized analytical schema optimized for read-heavy workloads.

Requirements:

- Analyze the current normalized structure.
- Identify transactional entities, dimensions, and measures.
- Reduce join complexity while preserving analytical usability.
- Consolidate highly related entities when appropriate.
- Consider the use of fact and dimension tables when beneficial.
- Avoid unnecessary normalization that would negatively impact query performance.
- Preserve business meaning and data integrity wherever possible.

For the output:

Describe the denormalization strategy used.
List all assumptions made.
Present the new physical schema.
For each table include:
Table name
Columns
Data types
Primary key
Foreign keys
Explain which original tables were merged, duplicated, or transformed.
Describe expected performance improvements and trade-offs.

The resulting model should be suitable for BI dashboards, reporting, aggregations, and analytical queries.

EXPERT:
You are a Principal Data Architect designing a data warehouse layer for analytical workloads.

Your objective is to transform the provided normalized relational physical model into a denormalized analytical data model
 optimized for large-scale read performance, self-service analytics, and BI consumption.

Business Objective

The resulting schema will serve as the analytical layer of the platform and should prioritize:

- Fast analytical queries
- Reduced join depth
- Simplified data consumption
- Aggregation efficiency
- Scalability for historical data growth
- Compatibility with modern OLAP engines and columnar storage systems
- Analysis Tasks

Perform the following steps:

- Analyze the current physical model and identify:
- Transactional entities
- Master/reference data
- High-cardinality relationships
- Frequently joined tables
- Potential performance bottlenecks
- Determine the optimal analytical modeling strategy:
- Star schema
- Snowflake schema
- Wide-table approach
- Hybrid approach
- Justify the chosen strategy based on:
- Query patterns
- Cardinality
- Data volume
- Expected analytical use cases
- Denormalization Rules
- Eliminate joins that provide little analytical value.
- Embed low-volatility lookup attributes into analytical tables when appropriate.
- Preserve historical analysis capabilities.
- Minimize duplication that would create unacceptable maintenance costs.
- Clearly identify surrogate keys when recommended.
- Consider Slowly Changing Dimension (SCD) implications where relevant.
- Maintain traceability back to source entities.


Deliverables

1. Executive Summary

A concise explanation of the proposed analytical architecture.

3. Denormalized Physical Schema

For each table provide:

Table name
Purpose
Columns
Data types
Primary key
Foreign keys (if any)
Partitioning recommendations
Clustering/indexing recommendations
4. Analytical Considerations

Describe:

Fact tables
Dimension tables
Grain definition
Aggregation opportunities
Data freshness considerations
5. Trade-off Analysis

Discuss:

Storage increase
Update complexity
Data redundancy
Query performance gains
Maintainability implications
6. Final Recommendation

Present the complete denormalized schema and explain why it is the most appropriate design for an analytics-focused environment.
