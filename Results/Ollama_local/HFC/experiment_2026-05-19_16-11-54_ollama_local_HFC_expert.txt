# Executive Summary

This design transforms your normalized PostgreSQL schema into a **Read-Optimized Physical Schema** tailored for a biocatalogue system (Herbarium management). The primary goal is to eliminate join latency for "Tombo" (Specimen) detail views and taxonomy hierarchies, which currently suffer from deep join chains across `tombos`, `especies`, `familias`, `generos`, `reinos`, and `tipos`.

**Key Strategy:**
1.  **Snapshot Tables:** Create `tombos_snapshot` and `tombos_snapshot_fresh` to serve "Current State" queries without joins. This is the fastest path for user-facing dashboards.
2.  **Pre-Aggregated Taxonomy Rollup:** A `taxonomy_rollup` table aggregates counts of specimens per Family/Genus/Species to eliminate `COUNT(*)` joins on read paths.
3.  **Wide Read Dimension Tables:** Flatten `herbarios`, `autores`, and `locais_coleta` into wide tables to avoid joining metadata on every specimen detail request.
4.  **Staleness Contract:** Explicitly define that Snapshot tables are **T+5 minutes** (or T+1 hour depending on CDC frequency) stale. This is acceptable for read-only reporting but requires reconciliation for audit trails.

**Constraints & Assumptions:**
*   **Engine:** PostgreSQL 14+.
*   **CDC:** We assume **Postgres Logical Decoding** is available for streaming updates.
*   **Triggers:** Allowed for the refresh mechanism (or CDC listeners).
*   **Storage:** Standard PG storage; no specific columnar engine (e.g., ClickHouse) requested, so we use standard B-Tree indexes and partitioning.
*   **Correctness:** The normalized tables remain the **Source of Truth (SoT)**. The denormalized tables are derived views/snapshots.

# 1. Model-to-Workload Mapping

Based on the DDL provided, the access patterns are derived as follows:

| Query Class | Typical SQL Pattern | Join Hotspots | Aggregation Hotspots | Latency Impact |
| :--- | :--- | :--- | :--- :--- |
| **Specimen Detail** | `SELECT * FROM tombos WHERE hcf = ?` | `tombos` $\to$ `especies` $\to$ `familias` $\to$ `generos` $\to$ `reinos` $\to$ `tipos` | None (Row fetch) | **High**: Deep join tree (5-6 levels) causes CPU overhead and memory bloat. |
| **Taxonomy Stats** | `SELECT COUNT(*) FROM tombos WHERE familia_id = ?` | `tombos` $\to$ `familias` | `COUNT(*)` per family/genus | **Medium**: Scans large `tombos` table; aggregates are expensive. |
| **Herbarium List** | `SELECT * FROM herbarios` | `herbarios` $\to$ `enderecos` | None | **Medium**: Small table, but repeated joins to address/phone if not pre-joined. |
| **Collector Profile** | `SELECT * FROM autores WHERE id = ?` | `autores` $\to$ `alteracoes` (optional) | None | **Low**: Small table, but joins to `alteracoes` can be slow if `usuario_id` is missing. |
| **Geospatial** | `SELECT ... FROM tombos WHERE latitude ...` | `tombos` $\to$ `locais_coleta` | None | **Medium**: Requires spatial indexing (PostGIS) on top of the normalized structure. |

**Identified Bottlenecks:**
1.  **Taxonomy Chain:** The `tombos` table holds foreign keys to `especies`, `familias`, `generos`, `reinos`, and `tipos`. Resolving this requires a recursive or multi-table join, which is slow for list views.
2.  **Snapshot Aggregations:** Queries asking "How many specimens per family?" require a `GROUP BY` on a large fact table.
3.  **User/Herbario Context:** Every specimen query often implicitly needs the Herbario name and the Collector name, requiring joins to `herbarios` and `autores`.

# 2. Denormalization Patterns & Justification

We will implement three specific patterns:

### A. Wide Read "Current State" Snapshot (`tombos_snapshot`)
*   **Justification:** Eliminates all joins for the most common "View Specimen" query.
*   **Pattern:** Pre-joins `tombos` with `especies`, `familias`, `generos`, `reinos`, `tipos`, `herbarios`, `autores`, and `locais_coleta` into a single row.
*   **Staleness:** Updates are propagated via CDC (Postgres Logical Decoding) or a refresh job. Data is stale by ~5 minutes.
*   **Use Case:** User-facing specimen detail pages, mobile app feeds.

### B. Pre-Aggregated Taxonomy Rollup (`taxonomy_rollup`)
*   **Justification:** Eliminates `COUNT(*)` and `GROUP BY` operations on the `tombos` table for analytics.
*   **Pattern:** Stores counts of specimens grouped by `familia_id`, `genero_id`, `especie_id`.
*   **Staleness:** Updated on every `tombos` insert/update (or batched). Stale by ~1 hour (batch window).
*   **Use Case:** "Statistics by Family", "Top Collectors", "Specimen Distribution".

### C. Wide Dimension Tables (`herbarios_wide`, `autores_wide`)
*   **Justification:** Reduces join count when fetching lists of herbaria or authors.
*   **Pattern:** Pre-joins `herbarios` with `enderecos` and `telefones`; `autores` with `alteracoes`.
*   **Staleness:** Near real-time (CDC driven).
*   **Use Case:** Herbarium directory, Author profiles.

# 4. Consistency Architecture & Data Lineage

### Lineage Map
| Source Table | Target Column (Snapshot) | Target Table | Lineage Method | Staleness Window |
| :--- | :--- | :--- | :--- | :--- |
| `tombos` (All cols) | All cols | `tombos_snapshot` | CDC (Logical Decoding) | **T+5 min** |
| `tombos` (COUNT) | `count_tombos` | `taxonomy_rollup` | Batch Job (Hourly) | **T+1 hr** |
| `herbarios` + `enderecos` | `endereco_logradouro` | `herbarios_wide` | CDC | **T+5 min** |
| `autores` + `alteracoes` | `nro_alteracoes` | `autores_wide` | CDC | **T+5 min** |

### Refresh Strategy
1.  **CDC Pipeline:** Use a background worker (e.g., `pg_cron` or a dedicated Python/Node script) listening to `pg_logical`.
2.  **Trigger Mechanism:**
    *   On `INSERT`/`UPDATE` to `tombos`: Emit a "snapshot update" event.
    *   On `INSERT`/`UPDATE` to `herbarios`/`autores`: Emit a "dimension update" event.
3.  **Reconciliation:**
    *   **Snapshot:** If a row is missing in `tombos_snapshot` but exists in `tombos`, the system flags it for investigation (data loss risk).
    *   **Rollup:** A nightly job (`taxonomy_reconcile`) compares `SELECT COUNT(*) FROM tombos GROUP BY familia_id` against `taxonomy_rollup`. If diff > 0, re-run aggregation.

# 5. Risk Analysis & Operational Plan

### Risk Matrix

| Risk Category | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Write Amplification** | CDC triggers + CDC listeners + Batch jobs increase load on writers. | **Limit:** CDC listeners only consume events; they do not write to disk. Batch jobs run during low-traffic windows (e.g., 02:00-04:00 UTC). |
| **Lock Contention** | `tombos_snapshot` is read-heavy; `tombos` is write-heavy. | **Isolation:** `tombos_snapshot` is a separate table. No locks on `tombos` when reading `tombos_snapshot`. |
| **Data Freshness** | Users see 5-min or 1-hour old data. | **Documentation:** UI must display "Last updated: [Time]" next to specimen cards. Alert on lag > 15 min. |
| **Storage Growth** | Snapshot tables duplicate data. | **Retention:** `tombos_snapshot` does not grow indefinitely. Implement `TRUNCATE` strategy or archive old snapshots (e.g., keep only current day + 7 days of history). |
| **Failure Mode** | CDC pipeline breaks. | **Fallback:** If CDC fails, the `taxonomy_rollup` batch job (which uses standard SQL) will still run, ensuring some correctness. |

### Implementation Plan (Safe & Reversible)

**Phase 1: Infrastructure (Non-Disruptive)**
1.  Create `tombos_snapshot`, `taxonomy_rollup`, `herbarios_wide`, `autores_wide` tables.
2.  Create CDC listeners (Postgres Logical Decoding).
3.  Create `pg_cron` jobs for hourly rollup refresh.
4.  **Verification:** Run `EXPLAIN ANALYZE` on a production query. Compare execution time before and after.

**Phase 2: Traffic Shifting (Gradual)**
1.  **Canary Deployment:** Route 5% of "Specimen Detail" queries to `tombos_snapshot`.
2.  **Monitoring:** Watch for `tombos_snapshot` query latency (should be < 5ms) vs `tombos` join latency (baseline).
3.  **Alerting:** Set up alert if `tombos_snapshot` lag exceeds 10 minutes.

**Phase 3: Full Rollout**
1.  Route 100% of read queries to denormalized tables.
2.  Deprecate direct joins in application code (use Views or Application Layer abstraction).
3.  **Reversibility:** To revert, simply disable the CDC listeners and remove the application views; the normalized `tombos` table remains untouched and functional.

### Monitoring & Alerts
*   **Lag Metric:** `SELECT MAX(NOW() - last_refresh_ts) FROM tombos_snapshot;` (Target: < 5m)
*   **Reconciliation Diff:** `SELECT COUNT(*) FROM (SELECT familia_id, COUNT(*) as exp FROM tombos GROUP BY familia_id) t JOIN taxonomy_rollup r ON t.familia_id = r.familia_id WHERE t.count != r.count;`
*   **Query Latency:** P99 latency of `tombos_snapshot` queries (Target: < 10ms).

This design provides a measurable improvement in read performance (eliminating complex joins) while maintaining a clear path to correctness and operational stability.
