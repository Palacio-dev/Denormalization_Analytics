# Executive Summary

This design proposes a denormalized read-optimized schema to replace the current normalized model with the goal of significantly improving read performance for analytical queries that require joining multiple tables. The new `infos_anuais_paises` table consolidates data from six source tables into a single wide entity, collapsing relationships and precomputing derived metrics like per-capita emissions. This approach eliminates costly joins and aggregations at query time, enabling faster access patterns aligned with common reporting and dashboarding needs.

To maintain correctness, we implement a **refresh pipeline** that rebuilds the denormalized table nightly using batch ETL logic. Staleness is acceptable within a 24-hour window for this analytical workload. Operational complexity is mitigated through clear lineage tracking, atomic refresh via swap-table techniques, and comprehensive monitoring on freshness lag and data reconciliation diffs. The design assumes no triggers are allowed and limited batch windows exist (~2 hours), but CDC is available for future enhancements.

# Proposed Denormalized Schema (DDL)

```sql
CREATE TABLE public.infos_anuais_paises (
    pais_nome VARCHAR(50) NOT NULL,
    pais_codigo CHAR(3), -- Optional ISO code if exists in source
    continente_nome VARCHAR(50),
    ano INTEGER NOT NULL,
    populacao BIGINT,
    quantidade_emitida BIGINT,
    gee_per_capita DOUBLE PRECISION GENERATED ALWAYS AS (
        CASE WHEN populacao > 0 THEN quantidade_emitida::DOUBLE PRECISION / populacao ELSE NULL END
    ) STORED,
    porcentagem_carvao INTEGER,
    porcentagem_nuclear INTEGER,
    porcentagem_gas INTEGER,
    porcentagem_petroleo INTEGER,
    porcentagem_solar INTEGER,
    porcentagem_hidreletrica INTEGER,
    PRIMARY KEY (pais_nome, ano)
);

-- Optional indexes based on top query filters
CREATE INDEX idx_infos_anuais_pais_ano ON public.infos_anuais_paises (pais_nome, ano);
CREATE INDEX idx_infos_anuais_continente ON public.infos_anuais_paises (continente_nome);
CREATE INDEX idx_infos_anuais_emitida ON public.infos_anuais_paises (quantidade_emitida DESC);
```

### Key Design Choices:
- **PK Strategy**: `(pais_nome, ano)` supports efficient point lookups and range scans by country/year.
- **Derived Attribute**: `gee_per_capita` is computed at write time (via `GENERATED ALWAYS AS`) to avoid runtime division in queries.
- **Wide Table**: Combines all relevant fields into one relation to eliminate joins.
- **No Partitioning**: Given the relatively small size (~thousands of rows/year/country), partitioning adds unnecessary complexity unless volume grows beyond ~10M rows.

# Implementation Plan

### Phase 1: Build Initial Snapshot
1. Create a temporary table `infos_anuais_paises_temp`.
2. Populate it using a nightly ETL job that:
   - Joins `paises`, `populacao`, `emissao_g_efeito_estufa`, and explodes `consumo_anual` by energy type.
   - Aggregates energy types into separate columns (`porcentagem_carvao`, etc.).
   - Computes `gee_per_capita`.

### Phase 2: Atomic Swap
3. Once populated, rename `infos_anuais_paises_temp` → `infos_anuais_paises`.
4. Apply grants/indexes as needed.

### Phase 3: Refresh Automation
5. Schedule daily refresh during low-traffic window (~2 AM):
   - Truncate or drop/create `infos_anuais_paises_temp`.
   - Rebuild from source tables.
   - Rename atomically.

### Phase 4: Monitoring & Observability
6. Add:
   - Freshness check: `MAX(ano)` should be yesterday’s date.
   - Row count diff vs. expected combinations of countries × years.
   - Alert on failed refresh jobs or missing data.
