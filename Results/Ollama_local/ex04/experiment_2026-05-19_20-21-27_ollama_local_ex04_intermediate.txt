### Workload-Driven Analysis & Strategy

Based on the provided schema, the current physical model suffers from **high join cardinality** and **redundant I/O**. In a read-heavy analytical context (e.g., generating country reports, dashboarding, or historical trend analysis), the database engine must:
1.  **Join multiple tables per row**: A single "report row" often requires joining `country`, `survey`, `demography`, `employment`, `education`, `economy`, `population`, `life_expectancy`, and `health`.
2.  **Resolve Foreign Keys**: The engine must look up `country_code` to get region names, or `id_demography` to get age/gender breakdowns.
3.  **Scan Large Tables**: Filtering by `year` and `country_code` forces a scan across wide, sparse tables like `demography` or `employment` where many rows are missing data for specific years/countries.

**Key Bottlenecks Identified:**
*   **Join Explosion**: Queries returning a full profile of a country for a specific year likely involve 6+ table joins.
*   **Filter Pushdown Issues**: Filtering on `year` and `country_code` in wide tables (like `demography`) often results in large result sets before aggregation.
*   **Missing Aggregates**: Calculating total population or average Gini coefficient requires grouping and summing across potentially massive tables.

**Denormalization Strategy:**
1.  **Star Schema Transformation**: Convert the core analytical tables (`demography`, `employment`, `education`, `economy`, `health`) into a **Fact Table** structure with a **Dimension Table** structure. This allows efficient grouping by `year` and `country_code` without joining back to `demography` or `country` for basic attributes.
2.  **Snapshot Fact Tables**: Create denormalized "snapshot" tables that contain pre-calculated aggregates (Total Pop, Avg Gini, Total GDP) indexed by `(year, country_code)`.
3.  **Wide Dimension Tables**: Flatten `demography` and `population` into a single wide table containing all age/gender breakdowns to avoid joining `population` and `life_expectancy` for every country/year.
4.  **Covering Indexes**: Implement composite indexes on `(year, country_code)` to ensure the database can fetch and sort the required data in a single table access.

### New Physical Schema Design

#### 1. Dimension Table: `dim_country_region`
*Optimization*: Merges `country` and `region` into a single lookup table.
*Benefit*: Eliminates the join between `country` and `region`.

#### 2. Snapshot Fact Table: `fact_economy_socio`
*Optimization*: Denormalizes `economy`, `demography`, `employment`, `education`, `health`, and `survey` into a single wide table.
*Grain*: One row per `(year, country_code)`.
*Content*: Contains all metrics (GDP, Gini, Total Pop, Child Emp, etc.) pre-calculated.
*Benefit*: Reduces a 6-table join to a single table scan + filter.

#### 3. Snapshot Fact Table: `fact_population_detail`
*Optimization*: Denormalizes `demography` and `population` and `life_expectancy`.
*Grain*: One row per `(year, country_code, age_group, gender)`.
*Content*: Contains age/gender breakdowns, life expectancy, and child stats.
*Benefit*: Eliminates joins between `demography`, `population`, and `life_expectancy`.

#### 4. Snapshot Fact Table: `fact_survey_summary`
*Optimization*: Pre-aggregates survey data.
*Grain*: One row per `(survey_year, country_code, welfare_type)`.
*Content*: Pre-calculated poverty lines, headcount, Gini, etc.

#### 5. Supporting Tables (Minimal Denormalization)
*   `dim_survey_types`: Stores survey metadata (acronym, comparability).
*   `dim_health_metrics`: Stores health data (beds, physicians).

### How to Use This Schema

1.  **Simple Country Profile Query**:
    ```sql
    -- Instead of joining 6 tables, we now scan 1 table.
    SELECT * 
    FROM public.fact_economy_socio 
    WHERE year = 2023 AND country_code = 'USA';
    ```
    *Performance Gain*: 1 Table Scan + 1 Index Lookup vs. 6 Table Scans + 5 Joins.

2.  **Detailed Demographic Analysis**:
    ```sql
    -- Instead of joining demography, population, and life_expectancy
    SELECT age_group, gender, pop_ages, life_expectancy_value
    FROM public.fact_population_detail
    WHERE year = 2023 AND country_code = 'USA' AND age_group = '65+';
    ```
    *Performance Gain*: Single index scan on `(year, country_code, age_group, gender)`.

3.  **Survey Reporting**:
    ```sql
    SELECT welfare_type, gini, poverty_line
    FROM public.fact_survey_summary
    WHERE year = 2023 AND country_code = 'USA';
    ```

### Correctness & Maintenance Notes
*   **Data Consistency**: The denormalized tables (`fact_*`) are **read-only** snapshots. They must be updated periodically (e.g., monthly) via an ETL process. If you need to update a country's GDP, you update `fact_economy_socio`, not `economy`.
*   **Error Handling**: If the source data for `demography` or `economy` is missing for a specific `(year, country_code)`, the `fact_economy_socio` row will contain `NULL`s for missing columns. This is acceptable for read performance; the application can handle `NULL` values if necessary.
*   **Foreign Keys**: The original foreign key constraints are removed from the fact tables because they are aggregates. The `dim_*` tables retain their relationships if needed for historical auditing.
