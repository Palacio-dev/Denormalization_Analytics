### 1. Workload-Driven Analysis

Based on the provided schema, I infer a **Read-Heavy Analytical Workload** typical of educational data reporting (e.g., Ministry of Education dashboards).

#### High-Cost Join Paths & Operations
The current schema forces a "deep join" pattern for almost every report:
1.  **The "Mega-Join"**: A query to get "Top 10 Universities by IDEB" requires joining:
    `MUNICIPIO` → `IES` (via `id_municipio`) → `CURSO` (via `id_IES`) → `ANO` (to filter year) → `ANADE` / `SAEB` / `IDEB` (to get metrics).
    *   *Cost*: High CPU and I/O due to joining multiple large fact-like tables on small keys.
2.  **Multi-Table Aggregates**: Queries asking for "Average IDEB per UF for 2023" require aggregating across `MUNICIPIO`, `IES`, `CURSO`, and `ANADE`/`IDEB` tables simultaneously.
3.  **Filtering Granularity**: Filtering by `sigla_uf` requires a join to `UF` to get the name, then a join to `MUNICIPIO` to get city stats, etc.

#### Denormalization Candidates
To optimize latency, we must break the dependency chain into a **Star Schema** centered on the entity being reported (usually `IES` or `UF`), duplicating the "Fact" data (`ANADE`, `SAEB`, `IDEB`) into the dimension tables or creating dedicated Fact tables that are pre-joined.

**Key Candidates:**
*   **Duplicate `UF` columns**: Move `nome` and `sigla_uf` into `MUNICIPIO` to avoid the join.
*   **Pre-compute Aggregates**: `MUNICIPIO` already has aggregates, but `IES` and `CURSO` do not. We need tables that hold the *latest* or *aggregated* metrics per entity to avoid grouping in the query.
*   **Snapshot Fact Tables**: Create wide tables that combine `IES` attributes with `ANADE`, `SAEB`, and `IDEB` data for specific years, eliminating the need to join across multiple metric tables.

### 2. Denormalization Design Strategy

#### A. Dimension Tables (Read-Optimized)
*   **`UF`**: Keep as is (small, rarely joined).
*   **`MUNICIPIO`**: **Denormalize** `sigla_uf` and `nome`.
    *   *Why*: Eliminates the join `MUNICIPIO` → `UF`.
*   **`IES`**: **Denormalize** `id_municipio` (keep reference) but add a `nome_uf` column (extracted from the joined UF data if available, or derived). *Better approach*: Since `IES` doesn't have UF data, we will create a **Wide Fact Table** for `IES` that includes the UF context.

#### B. Fact Tables (Wide & Precomputed)
We will create three "Wide Fact" tables to act as the primary read targets. These tables store the pre-joined data.

1.  **`FATO_IES_METRICAS`**:
    *   *Grain*: One row per `IES` per `ANO`.
    *   *Content*: Contains `IES` attributes (`nome`, `rede`, `sigla_ies`) **PLUS** all `ANADE`, `SAEB`, and `IDEB` columns for that year.
    *   *Benefit*: A query for "IES performance in 2023" is now a single table scan with a filter on `id_ies` and `id_ano`. No joins to `ENADE`, `SAEB`, or `IDEB` tables.

2.  **`FATO_UF_METRICAS`**:
    *   *Grain*: One row per `UF` per `ANO`.
    *   *Content*: Contains `UF` attributes (`nome`, `sigla_uf`) **PLUS** aggregated metrics from all `MUNICIPIO`s, `IES`s, and metric tables for that year.
    *   *Benefit*: Eliminates the need to aggregate over millions of `IES` rows in the application logic.

#### C. Index Strategy
*   **Covering Indexes**: Critical for read performance.
    *   On `FATO_IES_METRICAS`: `(id_ies, id_ano, nome_uf)` to support `SELECT * FROM FATO_IES_METRICAS WHERE id_ies = ? AND id_ano = ?`.
    *   On `FATO_UF_METRICAS`: `(id_uf, id_ano, nome)` for UF reports.
*   **Filter Indexes**: On the wide tables, include columns frequently used in `WHERE` clauses (e.g., `sigla_uf`, `rede`, `ano`).

#### D. Partitioning
*   **Strategy**: Range partition by `id_ano` on the Wide Fact tables.
*   **Reason**: Historical data grows rapidly. Partitioning allows the DB to ignore data from 2020 when querying 2024, reducing I/O significantly.

### Key Changes Summary & Rationale

1.  **Elimination of Multi-Table Joins**:
    *   **Old**: `SELECT ... FROM IES i JOIN ANADE a ON ... JOIN SAEB s ON ...`
    *   **New**: `SELECT * FROM FATO_IES_METRICAS WHERE id_ies = ? AND id_ano = ?`
    *   **Impact**: Reduces query complexity from 4+ table joins to a single table scan (plus an index lookup).

2.  **Data Redundancy Trade-off**:
    *   We duplicated `sigla_uf` in `MUNICIPIO` and `nome_uf` in `FATO_IES_METRICAS`.
    *   **Cost**: Slight increase in storage and write latency (ETL process must update these columns).
    *   **Benefit**: Massive reduction in read latency and CPU usage during analytics.

3.  **Partitioning**:
    *   Added `PARTITION BY RANGE (id_ano)` to the Wide Fact tables.
    *   **Benefit**: If you query data for 2024, the database engine ignores partitions for 2015-2023, drastically reducing I/O for large datasets.

4.  **Covering Indexes**:
    *   Created indexes that include the `SELECT` columns (e.g., `INCLUDE (nome_ies, sigla_uf, ...)`).
    *   **Benefit**: Enables "Index Only Scans," where the database does not need to look up data in the main table heap, further speeding up reads.

5.  **Precomputed Aggregates**:
    *   `FATO_UF_METRICAS` contains `media_ideb_media` and `qtd_ies_total`.
    *   **Benefit**: Queries asking for "Average IDEB per UF" no longer require `GROUP BY` operations over millions of `IES` rows; the answer is already sitting in the pre-aggregated row.
