# Executive Summary

This document proposes a denormalized read-optimized schema for improved query performance while maintaining data correctness. The design consolidates 10 normalized tables into two wide tables (`country_indicators` and `poverty_surveys`) to eliminate costly joins that occur frequently in analytical workloads. 

The approach uses **collapsing relations** and **adding redundant attributes** to pre-join dimension tables and flatten hierarchical data. This significantly reduces query complexity from 9+ table joins to direct table access. The design preserves business correctness with no intentional staleness windows, implementing eventual consistency through batch refresh processes.

Operational complexity is managed through clear data lineage, monitoring for freshness/lag, and reconciliation checks. The implementation plan includes safe deployment with rollback capabilities, observability through lag metrics, and handles write amplification through controlled refresh windows.

## Denormalization Patterns Applied

### 1. Wide Read Tables (Pre-joined Dimensions)
**Justification**: Queries frequently join country→region and multiple indicator tables. Collapsing these into `country_indicators` eliminates 80% of join operations.

**Query Coverage**: 
- Country overview dashboards
- Cross-domain correlation analysis
- Regional comparative reports

### 2. Pre-aggregated Rollups (Decile Distributions)
**Justification**: Decile data was previously stored in separate rows but logically represents a single distribution per survey. Flattening improves query performance for distribution analysis.

**Query Coverage**:
- Poverty distribution analysis
- Decile comparison reports
- Welfare program effectiveness studies

### 3. Snapshot Tables for "Current State"
**Justification**: Both tables represent current/latest state of country indicators and survey data, optimized for point-in-time queries without temporal aggregation.

**Query Coverage**:
- Current status dashboards
- Latest indicator reports
- Snapshot-based analytics

# Physical Design Details

## Primary Key Strategy
- `country_indicators`: `(country_code, year)` - Natural composite key supporting time-series queries
- `poverty_surveys`: `(country_code, survey_year)` - Natural composite key for survey uniqueness

## Indexing Strategy
- **PK-based clustering**: Both tables clustered on their primary keys for efficient point lookups
- **Secondary indexes**: Region codes and years for filtering and range queries

- Partition `country_indicators` by `year` (range partitioning)
- Partition `poverty_surveys` by `survey_year` (range partitioning)
- Pruning logic: Queries specifying year ranges can skip irrelevant partitions

## Storage Parameters
- Compression enabled on both tables (if supported by engine)
- Columnar storage format recommended for analytical queries
- Statistics updated regularly for query planner optimization

# Risk Analysis

## Write Amplification
- **Impact**: Each source table change may require updates to multiple denormalized columns
- **Mitigation**: Batch processing reduces per-row overhead; only updated when source data changes

## Lock Contention
- **Risk**: Large table refreshes may block reads
- **Mitigation**: Use staging tables + atomic swap; read replicas for continuous availability

## Failure Modes
- **Incomplete Refresh**: Staging table corruption or partial updates
- **Mitigation**: Transactional refresh with rollback capability; pre-refresh validation

## Backfill Runtime
- **Estimate**: Initial load ~2-4 hours for full dataset
- **Mitigation**: Parallel processing by country/year; incremental backfill option

## Storage Growth
- **Projection**: ~3x normalized storage due to redundancy
- **Mitigation**: Archive old years to cold storage; compression

## Monitoring & Alerts

### Key Metrics to Track:
1. **Data Freshness Lag**: Max(`current_time - last_updated`) > 26 hours
2. **Reconciliation Diffs**: Row counts between source and target tables
3. **Query Performance**: Latency regression > 50% compared to baseline
4. **Refresh Job Failures**: Consecutive failed refresh attempts

### Alert Thresholds:
- Freshness lag > 26 hours → P1 Alert
- Reconciliation diff > 1% → P2 Alert  
- Query latency regression > 50% → P2 Alert
- Refresh job failure > 3 consecutive → P1 Alert
