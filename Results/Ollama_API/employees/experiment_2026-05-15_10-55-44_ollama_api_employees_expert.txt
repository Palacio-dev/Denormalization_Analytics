# Proposed Denormalized Schema (DDL)

We introduce two denormalized tables optimized for frequent read patterns:

### 1. `employees.employee_read_model`

A **wide read table** combining employee details, current department, and current salary.

```sql
CREATE TABLE employees.employee_read_model (
    employee_id BIGINT PRIMARY KEY,
    first_name VARCHAR(14),
    last_name VARCHAR(16),
    birth_date DATE,
    gender employees.employee_gender,
    hire_date DATE,
    department_id CHAR(4),
    department_name VARCHAR(40),
    current_salary_amount BIGINT,
    current_title VARCHAR(50),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

**Justification:**
- Supports queries needing employee details with current department and salary.
- Eliminates need to join `employee`, `department_employee`, `department`, and `salary` at query time.
- Suitable for employee directory or profile views.

**Indexing:**
```sql
CREATE INDEX idx_employee_read_dept ON employees.employee_read_model(department_id);
CREATE INDEX idx_employee_read_name ON employees.employee_read_model(last_name, first_name);
```

# Risk Analysis

### Write Amplification
- Denormalized tables increase write load due to redundancy.
- Mitigation: Use batch jobs or asynchronous pipelines (e.g., Kafka → DB Writer).

### Lock Contention
- Large batch updates may lock rows.
- Mitigation: Use partitioned tables or partial updates with row-level locking.

### Backfill Runtime
- Initial population of read models may take time.
- Mitigation: Run during off-peak hours with monitoring.

### Storage Growth
- Denormalized tables will consume more space.
- Mitigation: Monitor size, archive old snapshots, compress cold data.

### Monitoring & Alerts

| Metric | Tooling | Alert Threshold |
|--------|---------|------------------|
| Lag in refresh | Custom job log or table `updated_at` | > staleness window |
| Reconciliation diffs | Row count / checksum diff | > 0.1% |
| Query latency regressions | APM or query log analysis | > 10% slower than baseline |
