# Executive Summary

This document outlines a **denormalized read-optimized physical schema** derived from the provided normalized relational schema. The goal is to **improve read performance** for common analytical and reporting queries that involve customer orders, product details, and aggregated metrics such as total sales or item counts per order or customer.

The key transformation involves creating **wide read tables**, **pre-aggregated rollups**, and **snapshot tables** that precompute joins and aggregations, reducing query-time complexity. This design assumes that **reads significantly outweigh writes**, and that **data freshness can tolerate a small lag** (e.g., up to 15 minutes) without violating business correctness.

All artifacts are tied directly to specific query patterns (see Access-Pattern Matrix). The proposed changes are **reversible**, **observable**, and maintain **data lineage** through explicit refresh mechanisms.

# Proposed Denormalized Schema (DDL)

## 1. Wide Read Table: `order_summary`

**Purpose**: Pre-join `orders`, `customers`, and `order_details` to support fast lookups of order-level summaries.

```sql
CREATE TABLE order_summary (
    order_id INT PRIMARY KEY,
    customer_id INT NOT NULL,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    order_date DATE NOT NULL,
    total_items INT NOT NULL,          -- Precomputed: SUM(quantity)
    total_value DECIMAL(12,2) NOT NULL -- Precomputed: SUM(quantity * price)
);

-- Indexes for fast filtering/sorting
CREATE INDEX idx_order_summary_customer_id ON order_summary(customer_id);
CREATE INDEX idx_order_summary_order_date ON order_summary(order_date);
```

> **Refresh Logic**: Populated via batch job or CDC-based pipeline that recomputes per-order summary when any `order_details` record changes.

## 4. Materialized View Alternative (if supported by engine)

For engines that support materialized views (e.g., PostgreSQL, Snowflake), consider:

```sql
CREATE MATERIALIZED VIEW mv_order_summary AS
SELECT
    o.order_id,
    c.customer_id,
    c.customer_name,
    c.customer_email,
    o.order_date,
    SUM(od.quantity) AS total_items,
    SUM(od.quantity * p.price) AS total_value
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN order_details od ON o.order_id = od.order_id
JOIN products p ON od.product_id = p.product_id
GROUP BY o.order_id, c.customer_id, c.customer_name, c.customer_email, o.order_date;
```

> **Note**: Requires periodic refresh or incremental maintenance logic depending on engine capabilities.

# Risk Analysis

## Write Amplification

- Each insert/update/delete in normalized tables may trigger multiple writes in denormalized tables.
- Mitigation: Use batch processing or change data capture (CDC) pipelines to batch updates.

## Lock Contention

- High write frequency on denormalized tables could cause lock waits.
- Mitigation: Use append-only or partition-swapping strategies where possible.

## Failure Modes

- Incomplete refreshes can lead to stale data.
- Mitigation: Implement reconciliation checks and alerting on lag/freshness.

## Backfill Runtime

- Full backfills of `order_summary` or `daily_sales_summary` may take hours for large datasets.
- Mitigation: Parallelize backfills using partitioning or chunked processing.

## Storage Growth

- Denormalization increases storage footprint (typically 2x–5x).
- Mitigation: Monitor growth and archive old data if needed.
