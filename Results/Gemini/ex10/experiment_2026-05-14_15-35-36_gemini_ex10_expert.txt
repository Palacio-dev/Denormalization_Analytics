### 2. Denormalization Patterns Applied

1.  **Wide Read Table (One Big Table - OBT):** We collapse `paises`, `continente`, `populacao`, and `emissao_g_efeito_estufa` into a single flat structure. Justification: Disk space is cheap; CPU time spent on joins for analytical queries is expensive.
2.  **Pre-calculated Derived Attributes:** `gee_per_capita` is computed during write-time (`emissao_g_efeito_estufa.quantidade_emitida / populacao.quantidade_hab`). Justification: Prevents redundant division operations on every read query.
3.  **Row-to-Column Pivoting:** The `consumo_anual` table stores energy types as rows. We pivot these into explicit columns (`fonte_carvao`, `fonte_solar`, etc.). Justification: BI tools and columnar engines perform significantly better when metrics are explicitly columnar rather than requiring complex `CASE WHEN` aggregations at runtime.

### 4. Consistency Architecture & Lineage

Because the grain is yearly, we will use a **Nightly Batch Window** to refresh this table. Triggers and CDC are explicitly avoided to prevent operational complexity and write-amplification on the OLTP database.

### 5. Risk Analysis & Implementation Plan

#### Risks & Mitigations
1.  **Write Amplification:** *Low Risk.* Because updates happen in a nightly batch and the data is partitioned by year, we are only rewriting the current year's partition. Historical partitions remain untouched.
2.  **Data Anomalies (Division by Zero):** *Medium Risk.* When calculating `gee_per_capita`, if `populacao` is 0 or missing, the ETL will crash. *Mitigation:* ETL must use `NULLIF(populacao.quantidade_hab, 0)` to yield a `NULL` instead of a fatal error.
3.  **Storage Growth:** *Low Risk.* Denormalization increases footprint, but environmental data aggregated by country/year is exceptionally small (approx. 200 countries * 100 years = 20,000 rows). Storage cost is negligible.
4.  **Schema Rigidity (Pivoted Columns):** *Medium Risk.* If a new energy source is added to `fonte_energia` (e.g., "Geothermal"), the denormalized table requires a schema change (`ALTER TABLE ADD COLUMN fonte_geotermica`). *Mitigation:* Document this as standard operating procedure for the data engineering team.

#### Implementation & Observability Plan
1.  **Phase 1: Dual-Write / Backfill.** Deploy the DDL. Run a backfill script to populate historical data. The application continues reading from the normalized tables.
2.  **Phase 2: Observability.** Implement a daily reconciliation query: `SELECT SUM(populacao) FROM infos_anuais_paises WHERE ano = 2023` vs `SELECT SUM(quantidade_hab) FROM populacao WHERE ano = 2023`. Alert if the diff > 0.
3.  **Phase 3: Cutover.** Switch read traffic in the BI layer/application to `infos_anuais_paises`.
4.  **Phase 4: Reversibility.** If read latency regressions occur (unlikely), simply revert the application read endpoints back to the normalized views. The normalized source-of-truth is never dropped.
