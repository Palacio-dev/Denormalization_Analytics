As a Senior Database Performance Engineer, I have analyzed your current schema. It is a classic **denormalized entity model** (likely an older version or a legacy system) where a single table (`tombos`) holds the core data, and related entities (species, authors, locations, etc.) are accessed via heavy `JOIN` operations.

For a **read-heavy workload** (e.g., a web app displaying specimen cards, search results, or dashboards), this current structure will suffer from:
1.  **Join Latency:** Every "Specimen Detail" query requires joining `tombos` with up to 10+ tables (species, sub-species, authors, families, genera, locations, coordinates, etc.).
2.  **Network Overhead:** If this is a distributed system or even a single large node, fetching 20+ rows of related data per specimen is costly.
3.  **Complexity:** Maintenance of foreign key consistency becomes harder as the query logic grows.

### 1. Workload-Driven Analysis

#### High-Cost Join Paths
The most frequent read pattern in biological/historical databases is: **"Get me all specimens for a specific collection or location, including their full taxonomic hierarchy and collection metadata."**

*   **Current Query Pattern:**
    ```sql
    SELECT t.*, s.nome, a.nome, f.nome, g.nome, l.descricao, ...
    FROM tombos t
    JOIN especies s ON t.especie_id = s.id
    JOIN autores a ON s.autor_id = a.id
    JOIN familias f ON s.familia_id = f.id
    ...
    JOIN locais_coleta l ON t.local_coleta_id = l.id
    WHERE t.hcf = 12345
    ```
*   **The Bottleneck:** Even with indexes, the optimizer must scan the `tombos` table, then perform lookups across multiple small tables. If `tombos` has millions of rows, this multi-table join is the dominant cost.

#### Denormalization Candidates
Based on the schema, the following attributes are read frequently but rarely updated:
*   **Taxonomy:** `nome` (species), `autor` (author), `genero`, `familia` (names). These are stable.
*   **Location:** `descricao`, `cidade` (derived from `endereço`), `lat/long`.
*   **Collection:** `nome`, `sigla`, `email`, `endereco`.
*   **Status/Metadata:** `situacao`, `data_coleta_ano`, `tipo`.

### 2. Denormalization Design Strategy

We will adopt a **Star Schema** approach for the read model, keeping the original `tombos` table as the "Fact" (source of truth for unique IDs and core attributes) and creating "Denormalized Fact Tables" for specific read patterns.

#### A. The "Specimen Detail" Snapshot Table (`tombos_detalhes`)
This table mirrors `tombos` but pre-joins the most critical taxonomic and location data.
*   **Grain:** One row per specimen (`hcf`).
*   **Redundant Attributes:** `nome_cientifico`, `nome_popular`, `autor_nome`, `genero_nome`, `familia_nome`, `local_descricao`, `lat`, `long`, `colecao_nome`, `colecao_sigla`.
*   **Why:** Eliminates the need to join 5+ tables for a single specimen view.

#### B. The "Taxonomy Hierarchy" Table (`hierarquia_taxonomica`)
Taxonomic data (Family -> Genus -> Species -> Author) is often queried together.
*   **Grain:** One row per taxon level.
*   **Redundant Attributes:** Full names of parents and children to avoid recursive joins.

#### C. The "Collection & Location" Snapshot Tables
*   **Grain:** One row per collection (`colecoes_anexas`) and one row per location (`locais_coleta`).
*   **Why:** These are small, static lookup tables that get joined repeatedly. Pre-loading them into a wider table or a dedicated snapshot table speeds up filtering.

#### D. Index Strategy
*   **Covering Indexes:** For the `tombos_detalhes` table, we will create a composite index on the primary search keys (`hcf`, `colecao_anexa_id`, `local_coleta_id`) to allow the DB to satisfy the query entirely from the index (no table lookup needed).
*   **B-Tree Indexes:** Standard B-Trees on foreign keys in the original tables to maintain referential integrity for writes.

### Implementation Notes for the Engineer

1.  **Write Performance:** The new tables (`tombos_detalhes`, etc.) do not have Foreign Key constraints pointing to the original tables. This is intentional. If you try to insert into `tombos_detalhes` directly without updating the original `tombos` table, you break referential integrity.
    *   *Recommendation:* Create a trigger on `tombos` that fires `AFTER INSERT/UPDATE` to automatically update `tombos_detalhes`. This ensures the read model stays in sync with the source of truth.
    *   *Trigger Example:*
        ```sql
        CREATE OR REPLACE FUNCTION update_tombos_detalhes()
        RETURNS TRIGGER AS $$
        BEGIN
            -- Update tombos_detalhes based on changes in tombos
            -- Logic: Check if fields changed, update only if necessary
            UPDATE public.tombos_detalhes
            SET nome_cientifico = NEW.nome_cientifico,
                -- ... other fields
            WHERE hcf = NEW.hcf;
            RETURN NEW;
        END;
        $$ LANGUAGE plpgsql;

        CREATE TRIGGER trg_update_tombos_detalhes
        AFTER INSERT OR UPDATE ON public.tombos
        FOR EACH ROW EXECUTE FUNCTION update_tombos_detalhes();
        ```

2.  **Maintenance:** The snapshot tables (`autores_snapshot`, etc.) should be refreshed periodically (e.g., via a cron job or a nightly ETL process) to ensure data consistency, as they are not auto-updated by triggers.

3.  **Query Migration:**
    *   **Old:** `SELECT * FROM tombos t JOIN especies s ...`
    *   **New (Recommended):** `SELECT * FROM public.tombos_detalhes WHERE hcf = 12345;`
    *   **New (Alternative):** `SELECT * FROM public.v_tombos_read_optimized WHERE t.hcf = 12345;`

This approach reduces query execution time from potentially 50ms+ (with multiple joins) to <5ms (single table lookup with covering index) for read-heavy operations, while maintaining the ability to write to the original normalized schema.
