### Executive Summary

The current 3NF normalized schema is highly fragmented, splitting the core entities—Clean-up Events (`mutirao_de_limpeza`) and Conservation Units (`unidade_de_conservacao`)—across 10 distinct tables. Answering basic analytical questions (e.g., "How much plastic was collected in São Paulo last month?" or "What is the total marine area of conservation units by biome?") requires 4-to-5-way joins, which will degrade read performance and increase CPU/Memory overhead as data grows. 

To optimize for read-heavy analytical workloads, I propose a **Wide-Table / Materialized View architecture**. We will create two denormalized wide tables (acting as facts/dimensions) and one pre-aggregated rollup table. Because no specific operational constraints (like CDC availability) were provided, this design assumes **PostgreSQL** as the target engine and utilizes `MATERIALIZED VIEW`s. This approach is **100% safe and reversible**, as it does not modify the underlying OLTP tables, isolates read workloads, and allows for controlled, concurrent refreshes with an explicitly documented staleness window (e.g., hourly or nightly).

### 2. Denormalization Patterns Chosen

1.  **Wide Read Table (Pre-joined Fact): `mv_fato_mutirao_limpeza`**
    *   *Justification:* `mutirao_de_limpeza`, `coleta_de_lixo`, and `local_de_coleta` share a 1:1 relationship. Joining them with `regiao` and `instituicao_organizadora` flattens a 5-table join into a single O(1) lookup. Targets Q1 and Q3.
2.  **Wide Read Table (Pre-joined Dimension): `mv_dim_unidade_conservacao`**
    *   *Justification:* Flattens the 5-table snowflake schema for conservation units into a single wide dimension. Targets Q2.
3.  **Pre-aggregated Rollup: `mv_agg_coleta_mensal_estado`**
    *   *Justification:* Dashboards typically look at monthly trends by state. Pre-calculating this saves scanning millions of rows of raw waste collection data. Targets Q1 dashboarding.

### 3. Proposed Denormalized Schema (DDL)

*Note: Executing these scripts will not impact your existing tables. They create new materialized views and indexes.*

```sql
-- ==============================================================================
-- 1. WIDE FACT TABLE: Clean-up Events (Targets Q1, Q3)
-- Flattens: mutirao_de_limpeza + coleta_de_lixo + local_de_coleta + regiao + instituicao
-- ==============================================================================
CREATE MATERIALIZED VIEW public.mv_fato_mutirao_limpeza AS
SELECT 
    m.id_mutirao,
    m.data_do_ocorrido,
    TO_CHAR(m.data_do_ocorrido, 'YYYY-MM') AS mes_ano, -- Pre-computed for partitioning/grouping
    m.horario_de_inicio,
    m.duracao_total,
    m.numero_aproximado_de_participantes,
    
    -- Local & Regiao
    lc.nome AS nome_local_coleta,
    r.id_municipio,
    r.nome AS municipio_nome,
    r.estado,
    
    -- Instituicao
    i.id_instituicao_organizadora,
    i.nome AS instituicao_nome,
    i.carater AS instituicao_carater,
    
    -- Coleta de Lixo (Metrics)
    COALESCE(c.plastico, 0) AS plastico,
    COALESCE(c.borracha, 0) AS borracha,
    COALESCE(c.madeira, 0) AS madeira,
    COALESCE(c.produtos_texteis, 0) AS produtos_texteis,
    COALESCE(c.papel, 0) AS papel,
    COALESCE(c.metal, 0) AS metal,
    COALESCE(c.vidro, 0) AS vidro,
    COALESCE(c.materiais_de_pesca, 0) AS materiais_de_pesca,
    COALESCE(c.outros_residuos, 0) AS outros_residuos,
    -- Pre-computed total
    (COALESCE(c.plastico, 0) + COALESCE(c.borracha, 0) + COALESCE(c.madeira, 0) + 
     COALESCE(c.produtos_texteis, 0) + COALESCE(c.papel, 0) + COALESCE(c.metal, 0) + 
     COALESCE(c.vidro, 0) + COALESCE(c.materiais_de_pesca, 0) + COALESCE(c.outros_residuos, 0)) AS total_residuos
FROM public.mutirao_de_limpeza m
LEFT JOIN public.coleta_de_lixo c ON m.id_mutirao = c.id_mutirao
LEFT JOIN public.local_de_coleta lc ON m.id_mutirao = lc.id_mutirao
LEFT JOIN public.regiao r ON m.id_municipio = r.id_municipio
LEFT JOIN public.instituicao_organizadora i ON m.id_instituicao_organizadora = i.id_instituicao_organizadora;

-- Indexes for mv_fato_mutirao_limpeza
CREATE UNIQUE INDEX idx_mv_fato_mutirao_pk ON public.mv_fato_mutirao_limpeza(id_mutirao); -- Required for CONCURRENTLY refresh
CREATE INDEX idx_mv_fato_mutirao_data ON public.mv_fato_mutirao_limpeza(data_do_ocorrido);
CREATE INDEX idx_mv_fato_mutirao_estado ON public.mv_fato_mutirao_limpeza(estado);
CREATE INDEX idx_mv_fato_mutirao_inst ON public.mv_fato_mutirao_limpeza(instituicao_carater);


-- ==============================================================================
-- 2. WIDE DIMENSION TABLE: Conservation Units (Targets Q2)
-- Flattens: unidade_de_conservacao + bioma + categoria_manejo + esfera + regiao
-- ==============================================================================
CREATE MATERIALIZED VIEW public.mv_dim_unidade_conservacao AS
SELECT 
    uc.id_uc,
    uc.nome AS uc_nome,
    uc.ano_de_criacao,
    uc.area_soma_biomas,
    uc.area_soma_biomas_continentais,
    uc.area_marinha,
    
    b.nome AS bioma_nome,
    
    cm.categoria AS manejo_categoria,
    cm.categoria_iucn AS manejo_iucn,
    cm.plano AS manejo_plano,
    
    ea.tipo AS esfera_tipo,
    
    r.nome AS municipio_nome,
    r.estado
FROM public.unidade_de_conservacao uc
LEFT JOIN public.bioma b ON uc.id_bioma = b.id_bioma
LEFT JOIN public.categoria_manejo cm ON uc.id_manejo = cm.id_manejo
LEFT JOIN public.esfera_administrativa ea ON uc.id_esfera = ea.id_esfera
LEFT JOIN public.regiao r ON uc.id_municipio = r.id_municipio;

-- Indexes for mv_dim_unidade_conservacao
CREATE UNIQUE INDEX idx_mv_dim_uc_pk ON public.mv_dim_unidade_conservacao(id_uc);
CREATE INDEX idx_mv_dim_uc_bioma ON public.mv_dim_unidade_conservacao(bioma_nome);
CREATE INDEX idx_mv_dim_uc_estado ON public.mv_dim_unidade_conservacao(estado);


-- ==============================================================================
-- 3. PRE-AGGREGATED ROLLUP: Monthly Waste by State (Targets Q1 Dashboards)
-- ==============================================================================
CREATE MATERIALIZED VIEW public.mv_agg_coleta_mensal_estado AS
SELECT 
    mes_ano,
    estado,
    COUNT(id_mutirao) AS total_mutiroes,
    SUM(numero_aproximado_de_participantes) AS total_participantes,
    SUM(plastico) AS total_plastico,
    SUM(vidro) AS total_vidro,
    SUM(metal) AS total_metal,
    SUM(total_residuos) AS total_geral_residuos
FROM public.mv_fato_mutirao_limpeza
GROUP BY mes_ano, estado;

-- Indexes for mv_agg_coleta_mensal_estado
CREATE UNIQUE INDEX idx_mv_agg_coleta_pk ON public.mv_agg_coleta_mensal_estado(mes_ano, estado);
```

### 4. Consistency Architecture & Implementation Plan

**Data Lineage:**
*   Source-of-truth remains the normalized OLTP tables.
*   Data flows asynchronously into the Materialized Views.

**Refresh Strategy:**
Because we added `UNIQUE` indexes to all materialized views, we can use PostgreSQL's `CONCURRENTLY` feature. This allows the read tables to be refreshed without locking out read queries.

*Implementation Plan (Cron/pg_cron or Airflow task):*
```sql
-- Executed during a batch window (e.g., Nightly at 2 AM, or Hourly)
REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_fato_mutirao_limpeza;
REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_dim_unidade_conservacao;
REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_agg_coleta_mensal_estado;
```

**Intentional Staleness Window:**
Read queries pointing to the `mv_*` tables will be stale by up to the refresh interval (e.g., 1 hour or 24 hours). This is standard for analytical workloads and must be communicated to business users ("Dashboards reflect data as of the last hourly refresh").

### 5. Risk Analysis & Operational Complexity

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **Zero on OLTP.** Because we use async MVs, inserts/updates to base tables are not slowed down by triggers or synchronous index updates. |
| **Lock Contention** | **Low.** `REFRESH MATERIALIZED VIEW CONCURRENTLY` uses a snapshot and avoids exclusive locks. Base tables are only read-locked briefly during the snapshot phase. |
| **Storage Growth** | **Moderate.** We are duplicating data. However, analytical queries will scan far fewer pages. If storage becomes an issue, Postgres allows `TABLESPACE` configurations to move MVs to cheaper disks. |
| **Failure Modes** | If a refresh fails (e.g., out of memory or killed process), the MV simply retains the previous state. Users see older data, but no queries break. |

**Monitoring & Alerts to Implement:**
1.  **Freshness Lag:** Monitor the `pg_stat_user_tables.last_analyze` or maintain a `refresh_log` table to alert if the MVs haven't been refreshed in > 25 hours.
2.  **Reconciliation Diffs:** Create a daily data-quality check comparing `SELECT COUNT(*) FROM mutirao_de_limpeza` against `SELECT COUNT(*) FROM mv_fato_mutirao_limpeza`. They should match exactly after a refresh.
3.  **Query Latency Regressions:** Monitor `pg_stat_statements` for queries hitting the `mv_` tables to ensure they remain under the target SLA (e.g., < 100ms).
