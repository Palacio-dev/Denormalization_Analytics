# Executive Summary

This design proposes a **denormalized read-optimized physical schema** for the provided normalized relational model. The goal is to improve query performance by pre-joining frequently accessed tables and pre-aggregating key metrics such as customer invoice totals. The approach includes:

- **Wide read tables** for common entity access patterns (e.g., Track with Artist, Album, Genre).
- **Pre-aggregated rollups** for analytical queries involving invoices and sales.
- **Snapshot tables** for current state of customers and employees.
- A **fact-dimension star schema** for analytical workloads around Invoices.

The implementation plan includes **safe migration**, **observability**, and **rollback mechanisms**. Data correctness is preserved through explicit refresh logic and reconciliation checks. Any intentional staleness is documented and bounded within acceptable SLAs.

Operational constraints assumed:
- No triggers allowed (so refresh must be scheduled or event-driven via application/CDC).
- Limited batch windows (~1 hour daily).
- No native CDC; manual ETL or job-based refresh required.

Pre-joins Track with Album, Artist, Genre for fast metadata lookups.

# Risk Analysis

## Write Amplification
- Denormalized tables increase write volume due to redundancy.
- Mitigation: Use batch processing and avoid real-time updates unless absolutely necessary.

## Lock Contention
- Long-running ETL jobs may lock large portions of tables.
- Mitigation: Use partitioning, concurrent inserts, and incremental loads.

## Failure Modes
- ETL job failure → stale data.
- Mitigation: Alerting on lag/freshness; reconciliation diffs between source and target.

## Backfill Runtime
- Initial population of denormalized tables may take several hours.
- Mitigation: Run during off-peak hours; parallelize where possible.

## Storage Growth
- Denormalization increases storage usage (factor ~2–3x).
- Mitigation: Monitor growth rate; compress historical partitions.

## Monitoring & Observability
- **Lag**: Measure time since last successful ETL job.
- **Freshness**: Compare timestamps in source vs. target.
- **Reconciliation**: Count diffs between source and target tables.
- **Query Latency Regressions**: Baseline before/after query times.
