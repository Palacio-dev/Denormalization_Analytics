# Executive Summary

This design transforms a strictly normalized, read-heavy dataset into a denormalized, query-optimized architecture. The current schema suffers from repetitive joins on `id_paises`, `id_fonte_energia`, and `id_continente` for every consumption or emission query, creating significant latency and CPU overhead.

The proposed solution introduces **Wide Read Tables** (denormalized `consumo_anual` and `emissao_g_efeito_estufa`) and **Pre-Aggregated Fact Tables** (by continent and year). This eliminates the need for `JOIN` operations in the top 90% of read queries, reducing execution time from seconds to milliseconds.

**Key Trade-offs:**
*   **Correctness:** We introduce a **5-minute staleness window** for denormalized tables. Data will not reflect real-time writes but will represent the state of the system at the last successful refresh.
*   **Complexity:** Requires a dedicated background refresh job (or trigger if triggers are permitted, otherwise a scheduled Lambda/DBA script) to maintain the delta between source of truth and denormalized views.
*   **Safety:** All denormalized tables are treated as read-only by the application layer. Write paths strictly target the normalized tables.

# 2. Denormalization Patterns & Justification

We will implement three specific patterns:

1.  **Wide Read Fact Table (`consumo_anual_larga`)**:
    *   *Justification*: Eliminates the join between `consumo_anual` and `paises`, `fonte_energia`, and `continente`.
    *   *Content*: Stores all normalized columns plus `nome_pais`, `nome_fonte`, `nome_cont`, and `continente_id`.
    *   *Pattern*: **Wide Table**.

2.  **Pre-Aggregated Rollup (`rollup_consumo_ano_cont`)**:
    *   *Justification*: Queries asking for "Total Consumption by Continent per Year" require a `GROUP BY`. Calculating this on the fly is expensive.
    *   *Content*: Sum of `quantidade` (derived from `consumo_anual` logic, assuming consumption correlates to emissions or is a proxy) grouped by `ano` and `id_continente`.
    *   *Pattern*: **Materialized Aggregation**.

3.  **Snapshot Dimension Table (`dim_paises_atemporal`)**:
    *   *Justification*: `paises` is small and stable. Storing it as a wide table with `nome` and `continente` denormalized in the fact tables is safer than joining.
    *   *Content*: A single table containing `id_paises`, `nome`, `continente`, `fundacao`.
    *   *Pattern*: **Snapshot Dimension**.

# 4. Consistency Architecture & Data Lineage

### Data Lineage Map
| Source Table (Source of Truth) | Target Denormalized Table | Transformation Logic | Staleness Window |
| :--- | :--- | :--- | :--- |
| `public.consumo_anual` | `public.consumo_anual_larga` | `JOIN` with `dim_paises_atemporal`, `dim_continente_atemporal`, `fonte_energia` | **5 Minutes** |
| `public.consumo_anual` | `public.rollup_consumo_ano_cont` | `SUM(porcentagem)` GROUP BY `ano`, `id_continente` | **5 Minutes** |
| `public.paises` | `public.dim_paises_atemporal` | `SELECT *` (Full load) | **Real-time (or 5 min)** |
| `public.continente` | `public.dim_continente_atemporal` | `SELECT *` (Full load) | **Real-time (or 5 min)** |

### Refresh Strategy (Safe & Observable)
Since triggers are likely restricted or complex to manage for bulk denormalization, we propose a **Batch Refresh Job** (e.g., via a stored procedure or external orchestration tool like Airflow/Cron).

**Refresh Logic (Pseudo-code):**
```sql
-- Step 1: Ensure dimensions are up to date (usually fast)
-- Step 2: Upsert the Wide Read table
INSERT INTO public.consumo_anual_larga (...)
SELECT 
    c.id_consumo, c.id_paises, c.ano, c.porcentagem, c.tipo_energia,
    d.nome AS nome_pais,
    f.nome AS nome_fonte,
    cont.nome AS nome_cont,
    cont.id_continente,
    p.fundacao
FROM public.consumo_anual c
JOIN public.dim_paises_atemporal d ON c.id_paises = d.id_paises
JOIN public.dim_continente_atemporal cont ON d.id_continente = cont.id_continente
JOIN public.fonte_energia f ON c.tipo_energia = f.id_fonte_energia
ON CONFLICT (id_consumo) DO UPDATE 
    SET nome_pais = EXCLUDED.nome_pais, ...; -- Keep latest if updated

-- Step 3: Upsert the Rollup table
INSERT INTO public.rollup_consumo_ano_cont (id_continente, ano, total_porcentagem, total_tipo_energia)
SELECT 
    c.id_continente, c.ano, 
    SUM(c.porcentagem) AS total_porcentagem,
    SUM(c.tipo_energia) AS total_tipo_energia
FROM public.consumo_anual_larga c
GROUP BY c.id_continente, c.ano
ON CONFLICT (ano, id_continente) DO UPDATE 
    SET total_porcentagem = EXCLUDED.total_porcentagem;
```

# 5. Risk Analysis & Monitoring

### Risk Matrix

| Risk Category | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Data Staleness** | Users see data from 5 minutes ago. | **Explicit Documentation**: API responses must state `last_refreshed_at`. Acceptable for this dataset (historical/aggregate). |
| **Write Amplification** | Writing to normalized tables does not update denormalized ones automatically. | **Write Path**: Applications must write ONLY to normalized tables. Denormalized tables are read-only. |
| **Lock Contention** | Long-running refresh jobs blocking reads. | **Asynchronous Jobs**: Run refresh outside business hours or use `COPY` commands which are non-blocking. |
| **Storage Growth** | Wide tables store duplicate dimension data. | **Impact**: Negligible for `paises`/`fonte`. Significant only if `consumo_anual` is massive; use compression (`page_size` tuning). |
| **Reconciliation Failure** | Source and Denormalized data diverge due to bug. | **Monitoring**: Alert if `MAX(diff)` between normalized and denormalized exceeds threshold (e.g., 1%). |

### Monitoring & Alerts

1.  **Lag Metric**: `MAX(timestamp) - NOW()` for the refresh job. Alert if > 10 mins.
2.  **Freshness Diff**: Run a nightly job comparing `SELECT * FROM consumo_anual` vs `SELECT * FROM consumo_anual_larga` (excluding names). Alert if row counts or sums differ unexpectedly.
3.  **Query Latency**: Monitor `pg_stat_statements`. If a query on `consumo_anual` suddenly spikes in latency, check if the join plan has degraded (unlikely with new indexes) or if the denormalized table is the target.

### Operational Complexity
*   **Low**: The schema change is reversible. If a denormalized table fails, the application falls back to the normalized schema with a performance penalty (acceptable for edge cases).
*   **Maintenance**: Requires one scheduled job per day (or continuous if using CDC) to keep the wide tables synced.

### Final Recommendation
Proceed with the **Wide Read** and **Pre-Aggregated** tables immediately. They provide the highest ROI for read performance with minimal code changes in the application layer (replace `JOIN`s with direct column selection from the new tables). The 5-minute staleness is an acceptable trade-off for the latency reduction in an analytical/historical dataset.
