To optimize **read-heavy workloads** while preserving **data correctness**, we will denormalize the schema by:

- Eliminating frequent joins through **pre-joined wide tables**
- Precomputing aggregates into **summary tables**
- Adding **redundant attributes** where they eliminate common joins
- Creating **covering indexes** aligned with query predicates and sorts
- Designing **materialized views** or **read replicas** for complex analytical queries

---

## 🔍 1. Workload-Driven Analysis

### ✅ Common Read Queries (Implied from Schema Structure)

#### Query Patterns:
1. **Customer Invoices & Details**  
   - "Show all invoices for a customer with billing info, total, and line items."
   - Joins: `Customer → Invoice → InvoiceLine`

2. **Track Sales Performance**  
   - "Top-selling tracks per genre/media type over time."
   - Joins: `Track → Genre, MediaType, InvoiceLine → Invoice`

3. **Artist Popularity**  
   - "Top-selling artists based on track sales."
   - Joins: `Artist → Album → Track → InvoiceLine → Invoice`

4. **Support Rep Performance**  
   - "Total revenue generated by each support rep."
   - Joins: `Employee → Customer → Invoice`

5. **Playlist Track Info**  
   - "List tracks in a playlist with artist/album info."
   - Joins: `PlaylistTrack → Track → Album → Artist`

---

### ⚠️ High-Cost Operations Identified

| Query Type | Expensive Join Path | Bottleneck |
|------------|---------------------|------------|
| Customer Invoice Details | `Customer → Invoice → InvoiceLine` | Multiple joins; row explosion |
| Track Sales Analysis | `Track → Album → Artist`, `Track → InvoiceLine` | Deep join tree |
| Artist Popularity | `Artist → Album → Track → InvoiceLine` | Multi-hop joins |
| Support Rep Revenue | `Employee → Customer → Invoice` | Chain of joins |

## 🧱 2. Denormalization Design

### 🧩 Denormalization Candidates

#### 1. **Wide Table: `InvoiceWithDetails`**
- Denormalized view combining:
  - `Invoice.CustomerId`, `Invoice.InvoiceDate`, `Invoice.Total`
  - `Customer.FirstName`, `Customer.LastName`, `Customer.Country`
  - `InvoiceLine.TrackId`, `InvoiceLine.UnitPrice`, `InvoiceLine.Quantity`
- Purpose: Reduce join cost for invoice detail queries.

#### 2. **Summary Table: `TrackSalesSummary`**
- Grain: Per track per month
- Columns:
  - `TrackId`, `Month`, `Year`, `TotalQuantitySold`, `TotalRevenue`
- Purpose: Fast lookup of top-selling tracks.

#### 3. **Precomputed Aggregate: `ArtistSalesSummary`**
- Grain: Per artist per year
- Columns:
  - `ArtistId`, `Year`, `TotalRevenue`, `TrackCount`
- Purpose: Eliminate deep joins when querying artist popularity.

#### 4. **Redundant Attribute: `Track.AlbumTitle`, `Track.ArtistName`**
- Copy from `Album.Title`, `Artist.Name`
- Purpose: Avoid joining `Track → Album → Artist` for basic track info.

#### 5. **Covering Indexes**
- On `Invoice(CustomerId, InvoiceDate)` for filtering by customer/date
- On `Track(AlbumId, GenreId, MediaTypeId)` for fast filtering
- On `InvoiceLine(TrackId, InvoiceId)` for joins with `Track`, `Invoice`

## 🗃️ 3. New Physical Schema (DDL)

```sql
-- Denormalized Wide Table: InvoiceWithDetails
CREATE TABLE IF NOT EXISTS public."InvoiceWithDetails" (
    "InvoiceId" integer NOT NULL,
    "CustomerId" integer NOT NULL,
    "InvoiceDate" timestamp without time zone NOT NULL,
    "Total" numeric(10,2) NOT NULL,
    "CustomerFirstName" varchar(40) NOT NULL,
    "CustomerLastName" varchar(20) NOT NULL,
    "CustomerCountry" varchar(40),
    "TrackId" integer NOT NULL,
    "TrackName" varchar(200) NOT NULL,
    "UnitPrice" numeric(10,2) NOT NULL,
    "Quantity" integer NOT NULL,
    PRIMARY KEY ("InvoiceId", "TrackId")
);

-- Index for filtering by customer and sorting by date
CREATE INDEX idx_invoice_customer_date ON public."InvoiceWithDetails" ("CustomerId", "InvoiceDate");

-- Summary Table: TrackSalesSummary
CREATE TABLE IF NOT EXISTS public."TrackSalesSummary" (
    "TrackId" integer NOT NULL,
    "Year" int NOT NULL,
    "Month" int NOT NULL,
    "TotalQuantitySold" integer NOT NULL,
    "TotalRevenue" numeric(12,2) NOT NULL,
    PRIMARY KEY ("TrackId", "Year", "Month")
);

-- Index for top-selling tracks lookup
CREATE INDEX idx_track_sales_track_year_month ON public."TrackSalesSummary" ("TrackId", "Year", "Month");

-- Summary Table: ArtistSalesSummary
CREATE TABLE IF NOT EXISTS public."ArtistSalesSummary" (
    "ArtistId" integer NOT NULL,
    "Year" int NOT NULL,

);

-- Index for fast artist lookup
```

## 🧭 Index Strategy Recap

| Table | Index | Purpose |
|-------|-------|---------|
| `InvoiceWithDetails` | `(CustomerId, InvoiceDate)` | Fast customer invoice filtering |
| `TrackSalesSummary` | `(TrackId, Year, Month)` | Monthly sales lookup |
| `ArtistSalesSummary` | `(ArtistId, Year)` | Yearly artist revenue |
| `TrackDenormalized` | `(AlbumId)`, `(GenreId)`, `(MediaTypeId)` | Fast filtering |
| `InvoiceLine` | `(TrackId, InvoiceId)` | Join performance |
| `Invoice` | `(CustomerId)` | Join to customer |

## 📈 Benefits of This Denormalized Schema

- **Reduced Joins**: Queries like customer invoices, track sales, and artist popularity no longer require multiple joins.
- **Improved Latency**: Wide and summary tables allow direct access to precomputed or prejoined data.
- **Scalability**: Summary tables and materialized views enable efficient OLAP-style queries.
- **Data Correctness**: Redundant fields are only added when they are stable or refreshed via ETL.

---

## 🔄 ETL Considerations

- `InvoiceWithDetails`: Populate via ETL from `Invoice`, `Customer`, and `InvoiceLine`.
- `TrackSalesSummary`: Aggregated monthly from `InvoiceLine → Track → Invoice`.
- `ArtistSalesSummary`: Aggregated yearly from `Track → Album → Artist` and `InvoiceLine`.
- `TrackDenormalized`: Refreshed when `Album` or `Artist` changes.
