BEGGINER:
You are a database architect helping me redesign a relational database for analytics.

I will give you the current physical schema of a transactional database. Your task is to:

Identify which parts of the schema should be denormalized to improve read performance for analytics.
Explain the changes in simple terms.
Design a new physical schema optimized for reporting and analytical queries.
Keep the solution practical for a beginner to understand.
Requirements:

Focus on read-heavy analytics use cases, not write optimization.
Preserve the core business meaning of the data.
Suggest which tables should be merged, which columns should be duplicated, and where summary tables may help.
If appropriate, propose fact tables and dimension tables in a simple way.
Include a clear explanation of why each denormalization choice helps analytics.
Mention any trade-offs, such as duplicated data or update complexity.
Provide the final output as:
A short explanation of the strategy
A mapping from the original schema to the new schema
The new denormalized physical schema
A brief list of benefits and risks
If anything is unclear in the source schema, state your assumptions explicitly before proposing the design.

INTERMEDIATE:
You are a database designer specialized in analytics-oriented physical modeling.

Given the physical schema of a normalized relational database, redesign it into a denormalized physical schema optimized 
for analytical workloads.

Your goals are:

- Improve query performance for read-heavy BI and reporting scenarios.
- Reduce join complexity for common analytical queries.
- Create a new physical schema appropriate for a data warehouse or analytics layer.

Please do the following:

- Analyze the source tables, relationships, keys, and likely query patterns.
- Identify which entities should become facts, dimensions, snapshots, aggregates, or wide tables.
- Propose denormalization strategies such as attribute replication, table consolidation, pre-aggregated tables, and 
- summary structures.
- Define the new physical schema with table names, columns, primary keys, foreign keys if still needed, and indexes.
- Explain why each design choice supports analytics
- Highlight the main trade-offs between performance, storage, and maintenance.

Output format:

- Assumptions and workload interpretation
- Denormalization strategy
- Source-to-target table mapping
- New physical schema
- Recommended indexes and partitioning
- Risks and maintenance considerations
- Optimize the design for analytical queries over transactional updates. If multiple design options exist, choose 
  the one most suitable for a base-to-analytics transformation and explain why.

EXPERT:
You are a senior data architect designing an analytics-ready physical model from a normalized relational source system.

Your task is to transform the source physical model into a denormalized analytical physical schema that supports fast reads,
scalable BI workloads, and predictable query performance.

Instructions:

- Infer the dominant analytical access patterns from the schema and redesign accordingly.
- Convert the normalized source model into a physically optimized analytics model using dimensional modeling, wide
  denormalized tables, aggregates, snapshots, or hybrid structures as appropriate.
- Explicitly justify every denormalization decision in terms of query shapes, join elimination, scan reduction, 
  cardinality control, and maintenance cost.
- Define the target physical schema in enough detail to be implementable.
- Include indexing, partitioning, clustering, and summary-table recommendations if relevant.
- Consider slowly changing attributes, historical analysis, and grain definition where relevant.
- If the source model contains multiple subject areas, decide whether to keep them separate or consolidate them for
analytics, and explain the rationale.
- Focus on designing a base-to-analytics version of the database, not a transactional redesign.

Deliverables:

- Workload assumptions and target analytical use cases
- Grain definition for the main analytical entities
- Denormalization rationale by entity and relationship
- Source physical model to target physical model mapping
- Target schema with table definitions and key fields
- Indexing, partitioning, and storage recommendations
- Optional aggregate tables or materialized views
- Validation notes: how the design improves read performance and what trade-offs it introduces
- If helpful, produce the result in a form suitable for implementation by a DBA or data engineering team.
