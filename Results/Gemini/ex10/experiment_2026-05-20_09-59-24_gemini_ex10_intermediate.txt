As a senior database performance engineer, I have analyzed your normalized schema. Your current model is highly normalized (3NF/Snowflake), which is excellent for OLTP (writes) but creates severe bottlenecks for analytical, read-heavy workloads. 

Here is the comprehensive performance analysis and the new denormalized physical schema optimized for PostgreSQL.

---

### 1. Workload-Driven Analysis

Based on the domain (countries, continents, population, emissions, energy consumption), your read workload will predominantly consist of **time-series analytical queries** and **geographical aggregations**.

**High-Cost Operations in Current Schema:**
1. **The "Chasm Trap" (Multi-Fact Joins):** If a user wants to see Population, Emissions, and Energy Consumption for a specific country in a specific year, the database must join three separate fact tables (`populacao`, `emissao_g_efeito_estufa`, `consumo_anual`) on `id_paises` and `ano`. Because `consumo_anual` has multiple rows per year (one for each energy type), joining these directly causes a Cartesian product (fan-out), leading to inaccurate SUMs for population/emissions unless expensive `GROUP BY` subqueries are used before the join.
2. **Snowflake Dimension Joins:** Querying metrics by Continent requires traversing `Fact -> paises -> continente`. This requires two hash/nested-loop joins just to filter or group by continent.
3. **On-the-fly Calculations:** Calculating "Emissions per Capita" requires joining two massive fact tables and doing row-by-row division at query time.

---

### 2. Denormalization Design

To achieve low latency and preserve mathematical correctness, we will implement a **Star Schema with Pre-joined Wide Tables and Materialized Aggregates**.

**Specific Denormalizations:**
1. **Flattened Dimension (`dim_paises`):** We will merge `paises` and `continente`. This eliminates the `paises -> continente` join entirely.
2. **Consolidated Fact Table (`fact_country_yearly_metrics`):** We will merge `populacao` and `emissao_g_efeito_estufa`. They share the exact same grain: `(id_paises, ano)`. By combining them, we eliminate a massive fact-to-fact join. We will also add a precomputed `emissoes_per_capita` column.
3. **Denormalized Energy Fact (`fact_energy_consumption`):** We will keep this separate to prevent the "Chasm Trap" (preserving correctness for additive population/emission fields). However, we will inject the `fonte_energia_nome` directly into this table to eliminate the join to `fonte_energia`.
4. **Partitioning:** Both fact tables will be partitioned by `ano` (RANGE partitioning). Analytical queries usually filter by time (e.g., "last 10 years"). Partition pruning will allow the engine to skip scanning irrelevant years entirely.
5. **Materialized View (`mv_continent_yearly_summary`):** A pre-aggregated snapshot for executive dashboards showing total emissions and population by continent and year.

---

### 3. New Physical Schema (PostgreSQL DDL)

*Note: Run this in a new schema (e.g., `analytical`) to separate it from your OLTP tables.*

```sql
CREATE SCHEMA IF NOT EXISTS analytical;

-- ==========================================
-- 1. DIMENSIONS (Flattened / Denormalized)
-- ==========================================

-- Combines 'paises' and 'continente'
CREATE TABLE analytical.dim_paises (
    id_paises integer NOT NULL,
    pais_nome character varying(50) NOT NULL,
    fundacao integer NOT NULL,
    id_continente integer NOT NULL,
    continente_nome character varying(50) NOT NULL,
    CONSTRAINT dim_paises_pkey PRIMARY KEY (id_paises)
);

-- Index to speed up filtering by continent without sorting
CREATE INDEX idx_dim_paises_continente 
ON analytical.dim_paises (continente_nome) INCLUDE (pais_nome);


-- ==========================================
-- 2. FACT TABLES (Partitioned & Consolidated)
-- ==========================================

-- Combines 'populacao' and 'emissao_g_efeito_estufa'
-- Grain: 1 row per Country per Year
CREATE TABLE analytical.fact_country_yearly_metrics (
    id_paises integer NOT NULL,
    ano integer NOT NULL,
    quantidade_hab integer,          -- from populacao
    quantidade_emitida bigint,       -- from emissao_g_efeito_estufa
    emissoes_per_capita numeric(15,4), -- PRECOMPUTED: quantidade_emitida / quantidade_hab
    CONSTRAINT fact_country_yearly_metrics_pkey PRIMARY KEY (ano, id_paises)
) PARTITION BY RANGE (ano);

-- Create partitions for the fact table (Example: decades)
CREATE TABLE analytical.fact_country_yearly_metrics_2000s PARTITION OF analytical.fact_country_yearly_metrics FOR VALUES FROM (2000) TO (2010);
CREATE TABLE analytical.fact_country_yearly_metrics_2010s PARTITION OF analytical.fact_country_yearly_metrics FOR VALUES FROM (2010) TO (2020);
CREATE TABLE analytical.fact_country_yearly_metrics_2020s PARTITION OF analytical.fact_country_yearly_metrics FOR VALUES FROM (2020) TO (2030);
CREATE TABLE analytical.fact_country_yearly_metrics_default PARTITION OF analytical.fact_country_yearly_metrics DEFAULT;

-- Covering index for time-series queries by country
CREATE INDEX idx_fact_metrics_paises_ano 
ON analytical.fact_country_yearly_metrics (id_paises, ano) 
INCLUDE (quantidade_hab, quantidade_emitida, emissoes_per_capita);


-- Denormalized Energy Consumption
-- Grain: 1 row per Country per Year per Energy Type
CREATE TABLE analytical.fact_energy_consumption (
    id_consumo integer NOT NULL,
    id_paises integer NOT NULL,
    ano integer NOT NULL,
    id_fonte_energia integer NOT NULL,
    fonte_energia_nome character varying(50) NOT NULL, -- DENORMALIZED from fonte_energia
    porcentagem integer NOT NULL,
    CONSTRAINT fact_energy_consumption_pkey PRIMARY KEY (ano, id_consumo)
) PARTITION BY RANGE (ano);

-- Create partitions
CREATE TABLE analytical.fact_energy_consumption_2000s PARTITION OF analytical.fact_energy_consumption FOR VALUES FROM (2000) TO (2010);
CREATE TABLE analytical.fact_energy_consumption_2010s PARTITION OF analytical.fact_energy_consumption FOR VALUES FROM (2010) TO (2020);
CREATE TABLE analytical.fact_energy_consumption_2020s PARTITION OF analytical.fact_energy_consumption FOR VALUES FROM (2020) TO (2030);
CREATE TABLE analytical.fact_energy_consumption_default PARTITION OF analytical.fact_energy_consumption DEFAULT;

-- Composite index aligned to standard analytical predicates
CREATE INDEX idx_fact_energy_paises_ano 
ON analytical.fact_energy_consumption (id_paises, ano) 
INCLUDE (fonte_energia_nome, porcentagem);


-- ==========================================
-- 3. MATERIALIZED VIEWS (Dashboard Read Models)
-- ==========================================

-- Pre-aggregated "Snapshot" for Continent-level dashboards
-- Eliminates joining millions of rows for high-level BI reports
CREATE MATERIALIZED VIEW analytical.mv_continent_yearly_summary AS
SELECT 
    d.continente_nome,
    f.ano,
    SUM(f.quantidade_hab) AS total_populacao,
    SUM(f.quantidade_emitida) AS total_emissoes,
    -- Safe division for aggregate per capita
    CASE 
        WHEN SUM(f.quantidade_hab) > 0 THEN SUM(f.quantidade_emitida)::numeric / SUM(f.quantidade_hab)
        ELSE 0 
    END AS emissoes_per_capita_continente
FROM analytical.fact_country_yearly_metrics f
JOIN analytical.dim_paises d ON f.id_paises = d.id_paises
GROUP BY d.continente_nome, f.ano
WITH DATA;

-- Unique index required to allow CONCURRENTLY refreshes (zero downtime)
CREATE UNIQUE INDEX uidx_mv_continent_yearly 
ON analytical.mv_continent_yearly_summary (continente_nome, ano);

```

### Summary of Performance Gains:
1. **Zero-Join Energy Queries:** To get the energy mix of a country, you now query `fact_energy_consumption` alone. The energy name is already there.
2. **Single-Join Country Metrics:** To get population and emissions by country name, you join exactly two tables (`dim_paises` + `fact_country_yearly_metrics`) instead of four.
3. **Index-Only Scans:** The `INCLUDE` clauses in the indexes mean PostgreSQL can fetch the metric values directly from the B-Tree index without ever reading the underlying table heap pages.
4. **Partition Pruning:** If a BI tool queries `WHERE ano >= 2020`, PostgreSQL will completely ignore the tables holding data from the 2000s and 2010s, reducing I/O drastically.
