# Denormalized Schema for Read-Optimized Workloads

## 1. Workload-Driven Analysis

### Common Query Patterns:
1. **Flight details with airport information** - Requires joining flights with airports twice
2. **Booking details with passenger information** - Requires joining bookings, tickets, and ticket_flights
3. **Boarding pass information with seat details** - Requires joining boarding_passes with seats
4. **Aircraft utilization reports** - Requires joining flights with aircrafts_data
5. **Revenue reports by route** - Requires aggregating ticket_flights with flight information

### High-Cost Operations:
- Multiple joins to get complete flight information (flights → airports_data x2)
- Joining bookings → tickets → ticket_flights for booking details
- Joining boarding_passes → seats for seat information
- Complex aggregations for reporting queries

### Denormalization Candidates:
1. Pre-joined "wide" tables for flights with airport details
2. Booking summary table with passenger and flight information
3. Flight-seat mapping table for boarding pass lookups
4. Revenue summary tables for reporting

## 2. Denormalization Design

### Proposed Denormalizations:
1. **Flight details view** - Pre-joined flights with departure/arrival airport information
2. **Booking summary table** - Contains booking, ticket, and flight information in one table
3. **Boarding pass details view** - Pre-joined boarding_passes with seat information
4. **Flight revenue summary table** - Aggregated revenue by flight for reporting
5. **Passenger booking history table** - Aggregated booking history per passenger

-- Materialized view for airport route popularity
CREATE MATERIALIZED VIEW IF NOT EXISTS bookings.route_popularity AS
SELECT 
    departure_airport_code,
    arrival_airport_code,
    departure_city,
    arrival_city,
    count(*) as flight_count,
    sum(ticket_count) as total_passengers,
    avg(total_revenue) as avg_revenue
FROM bookings.flight_revenue_summary frs
JOIN bookings.flights_denorm fd ON frs.flight_id = fd.flight_id
GROUP BY departure_airport_code, arrival_airport_code, departure_city, arrival_city
ORDER BY flight_count DESC;

-- Function to update flight revenue summary
CREATE OR REPLACE FUNCTION bookings.update_flight_revenue(flight_id_param integer)
RETURNS void AS $$
BEGIN
    INSERT INTO bookings.flight_revenue_summary (
        flight_id, flight_no, scheduled_departure, 
        departure_airport_code, arrival_airport_code, 
        aircraft_code, total_revenue, ticket_count
    )
    SELECT 
        f.flight_id,
        f.flight_no,
        f.scheduled_departure,
        f.departure_airport,
        f.arrival_airport,
        f.aircraft_code,
        COALESCE(SUM(tf.amount), 0) as total_revenue,
        COUNT(tf.ticket_no) as ticket_count
    FROM bookings.flights f
    LEFT JOIN bookings.ticket_flights tf ON f.flight_id = tf.flight_id
    WHERE f.flight_id = flight_id_param
    GROUP BY f.flight_id, f.flight_no, f.scheduled_departure, 
             f.departure_airport, f.arrival_airport, f.aircraft_code
    ON CONFLICT (flight_id) 
    DO UPDATE SET 
        total_revenue = EXCLUDED.total_revenue,
        ticket_count = EXCLUDED.ticket_count,
        last_updated = now();
END;
$$ LANGUAGE plpgsql;

END;

## Summary of Denormalization Benefits

1. **Reduced Joins**: Most common queries now require 0-1 table accesses instead of 3-5
2. **Improved Read Performance**: Pre-computed joins and aggregates eliminate runtime calculations
3. **Better Indexing Strategy**: Composite indexes aligned with query patterns
4. **Partitioning**: Historical data partitioned for faster access patterns
5. **Covering Indexes**: Frequently accessed columns included in indexes to avoid table lookups
6. **Materialized Views**: Complex aggregations pre-computed for reporting queries

This schema design prioritizes read performance while maintaining data correctness through careful denormalization choices and appropriate indexing strategies.
