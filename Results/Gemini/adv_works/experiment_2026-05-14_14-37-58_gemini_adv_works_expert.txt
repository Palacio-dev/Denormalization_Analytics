### Executive Summary

To alleviate the read-latency bottlenecks inherent in the highly normalized AdventureWorks OLTP schema, I am proposing a new `read_optimized` physical schema. The current 3NF/BCNF design requires 5-to-10-way joins for standard business queries (e.g., resolving a customer’s name, address, and territory requires traversing `Customer` → `Person` → `BusinessEntityAddress` → `Address` → `StateProvince` → `CountryRegion`). 

This new architecture introduces a hybrid denormalization strategy: **Materialized Views** for operational "Wide Entity" lookups (Customer 360, Product Catalog) and a **Star Schema** (Fact/Dimension tables) for analytical aggregations. This approach isolates analytical read pressure from the OLTP engine, avoids distributed lock contention by using asynchronous refreshes (or CDC), and provides a safe, reversible migration path without altering the existing application's write paths. Intentional staleness (e.g., 15 minutes for operational reads, nightly/hourly for analytics) is introduced and explicitly documented to guarantee predictable read performance.

### 2. Denormalization Patterns & Justification

1.  **Wide Read Tables (Materialized Views)**: Used for `mvw_customer_360` and `mvw_product_catalog`. 
    *   *Justification*: The application needs millisecond-latency point-lookups for UI rendering. Pre-joining 8 tables into a single wide row eliminates nested-loop joins. We use PostgreSQL's `MATERIALIZED VIEW` because it allows `CONCURRENTLY` refreshes, ensuring read locks are not blocked during background updates.
2.  **Fact/Dimension Split (Star Schema)**: Used for `fact_sales`.
    *   *Justification*: Aggregating millions of rows across `SalesOrderHeader` and `SalesOrderDetail` degrades OLTP performance. A denormalized fact table partitioned by `orderdate` allows partition pruning and fast sequential scans for time-series reporting.
3.  **Pre-aggregated Rollups**: Embedded in `mvw_product_catalog`.
    *   *Justification*: Inventory is currently split across bins/shelves in `ProductInventory`. The read workload only cares about "Total Available Quantity". We pre-aggregate this to save CPU cycles on read.

### 3. Physical Design Details (DDL)

*Assumptions: PostgreSQL 12+ is used. The `uuid-ossp` extension is available.*

```sql
BEGIN;

CREATE SCHEMA IF NOT EXISTS read_optimized;

-- ============================================================================
-- 1. WIDE READ TABLE: Customer 360 (Operational)
-- ============================================================================
-- Refresh Strategy: REFRESH MATERIALIZED VIEW CONCURRENTLY every 15 minutes.
CREATE MATERIALIZED VIEW read_optimized.mvw_customer_360 AS
SELECT 
    c.customerid,
    p.businessentityid AS personid,
    c.storeid,
    c.territoryid,
    p.firstname || ' ' || COALESCE(p.middlename || ' ', '') || p.lastname AS full_name,
    p.persontype,
    e.emailaddress,
    ph.phonenumber,
    a.addressline1,
    a.city,
    sp.name AS state_province,
    cr.name AS country_region,
    a.postalcode
FROM sales.customer c
LEFT JOIN person.person p ON c.personid = p.businessentityid
LEFT JOIN person.emailaddress e ON p.businessentityid = e.businessentityid
-- Assuming we want the primary/first phone and address to avoid fan-out
LEFT JOIN (
    SELECT businessentityid, phonenumber, 
           ROW_NUMBER() OVER(PARTITION BY businessentityid ORDER BY modifieddate DESC) as rn 
    FROM person.personphone
) ph ON p.businessentityid = ph.businessentityid AND ph.rn = 1
LEFT JOIN (
    SELECT bea.businessentityid, a.addressline1, a.city, a.stateprovinceid, a.postalcode,
           ROW_NUMBER() OVER(PARTITION BY bea.businessentityid ORDER BY bea.modifieddate DESC) as rn
    FROM person.businessentityaddress bea
    JOIN person.address a ON bea.addressid = a.addressid
) a ON p.businessentityid = a.businessentityid AND a.rn = 1
LEFT JOIN person.stateprovince sp ON a.stateprovinceid = sp.stateprovinceid
LEFT JOIN person.countryregion cr ON sp.countryregioncode = cr.countryregioncode;

-- Unique index required for CONCURRENTLY refresh
CREATE UNIQUE INDEX uk_mvw_customer_360_customerid ON read_optimized.mvw_customer_360(customerid);
-- Index for name searches
CREATE INDEX idx_mvw_customer_360_name ON read_optimized.mvw_customer_360 USING gin(to_tsvector('english', full_name));


-- ============================================================================
-- 2. WIDE READ TABLE: Product Catalog (Operational)
-- ============================================================================
-- Refresh Strategy: REFRESH MATERIALIZED VIEW CONCURRENTLY every 5 minutes.
CREATE MATERIALIZED VIEW read_optimized.mvw_product_catalog AS
SELECT 
    p.productid,
    p.productnumber,
    p.name AS product_name,
    p.color,
    p.listprice,
    p.standardcost,
    ps.name AS subcategory_name,
    pc.name AS category_name,
    pm.name AS model_name,
    COALESCE(inv.total_quantity, 0) AS current_inventory_qty,
    p.sellstartdate,
    p.sellenddate
FROM production.product p
LEFT JOIN production.productsubcategory ps ON p.productsubcategoryid = ps.productsubcategoryid
LEFT JOIN production.productcategory pc ON ps.productcategoryid = pc.productcategoryid
LEFT JOIN production.productmodel pm ON p.productmodelid = pm.productmodelid
LEFT JOIN (
    SELECT productid, SUM(quantity) AS total_quantity
    FROM production.productinventory
    GROUP BY productid
) inv ON p.productid = inv.productid
WHERE p.finishedgoodsflag = true;

CREATE UNIQUE INDEX uk_mvw_product_catalog_productid ON read_optimized.mvw_product_catalog(productid);
CREATE INDEX idx_mvw_product_catalog_category ON read_optimized.mvw_product_catalog(category_name, subcategory_name);


-- ============================================================================
-- 3. STAR SCHEMA: Fact Sales (Analytical)
-- ============================================================================
-- Refresh Strategy: Standard table populated via CDC (Debezium) or nightly batch.
-- Partitioned by order date for fast pruning on time-series queries.
CREATE TABLE read_optimized.fact_sales (
    salesorderdetailid integer NOT NULL,
    salesorderid integer NOT NULL,
    orderdate date NOT NULL, -- Cast to date for partitioning
    customerid integer NOT NULL,
    productid integer NOT NULL,
    territoryid integer NOT NULL,
    salespersonid integer,
    orderqty smallint NOT NULL,
    unitprice numeric NOT NULL,
    unitpricediscount numeric NOT NULL,
    line_total numeric GENERATED ALWAYS AS (orderqty * unitprice * (1.0 - unitpricediscount)) STORED,
    taxamt_allocated numeric, -- Pre-calculated tax per line item
    freight_allocated numeric, -- Pre-calculated freight per line item
    onlineorderflag boolean NOT NULL,
    CONSTRAINT pk_fact_sales PRIMARY KEY (orderdate, salesorderdetailid)
) PARTITION BY RANGE (orderdate);

-- Example Partitions (Automate creation via pg_partman in production)
CREATE TABLE read_optimized.fact_sales_2011 PARTITION OF read_optimized.fact_sales FOR VALUES FROM ('2011-01-01') TO ('2012-01-01');
CREATE TABLE read_optimized.fact_sales_2012 PARTITION OF read_optimized.fact_sales FOR VALUES FROM ('2012-01-01') TO ('2013-01-01');
CREATE TABLE read_optimized.fact_sales_2013 PARTITION OF read_optimized.fact_sales FOR VALUES FROM ('2013-01-01') TO ('2014-01-01');

-- Indexes for analytical slice-and-dice
CREATE INDEX idx_fact_sales_productid ON read_optimized.fact_sales(productid);
CREATE INDEX idx_fact_sales_customerid ON read_optimized.fact_sales(customerid);
CREATE INDEX idx_fact_sales_territoryid ON read_optimized.fact_sales(territoryid);

COMMIT;
```

### 4. Consistency Architecture & Data Lineage

**Data Lineage Mapping:**
*   `mvw_customer_360.full_name` ← `person.firstname` + `middlename` + `lastname`
*   `mvw_customer_360.addressline1` ← `person.address.addressline1` (filtered to primary via window function)
*   `mvw_product_catalog.current_inventory_qty` ← `SUM(production.productinventory.quantity)`
*   `fact_sales.line_total` ← `salesorderdetail.orderqty * unitprice * (1 - unitpricediscount)`

**Refresh & Consistency Strategy:**
*   **Operational MVs (`mvw_*`)**: We will use `pg_cron` to execute `REFRESH MATERIALIZED VIEW CONCURRENTLY read_optimized.mvw_...` on a schedule (e.g., every 5-15 minutes).
    *   *Correctness Handling*: The application UI will display a small disclaimer: *"Data may be up to 15 minutes old."* This is a standard, acceptable trade-off for B2B/B2C catalogs.
*   **Analytical Tables (`fact_sales`)**: Because MVs cannot be partitioned in PostgreSQL, `fact_sales` is a standard table. We will populate this using an hourly batch ETL process (e.g., Airflow executing an `INSERT INTO ... ON CONFLICT` statement) or a CDC pipeline (Debezium → Kafka → Sink) if real-time analytics are strictly required.

### 5. Risk Analysis & Implementation Plan

#### Implementation Plan (Safe & Reversible)
1.  **Deploy DDL**: Create the `read_optimized` schema and artifacts. This takes seconds and has zero impact on the source tables.
2.  **Initial Backfill**: Run the initial population of MVs and `fact_sales` during a low-traffic window (e.g., 2:00 AM Sunday).
3.  **Dual-Read Phase (Observable)**: Modify the application to read from the new schema *asynchronously* while still relying on the old schema. Compare latency and correctness (Reconciliation Diffs).
4.  **Cutover**: Switch application read-paths to the `read_optimized` schema.
5.  **Reversibility**: If the refresh jobs fail or latency spikes, a feature flag can instantly revert reads back to the normalized OLTP tables.

#### Risk Analysis & Mitigations
*   **Write Amplification & Lock Contention**: 
    *   *Risk*: Refreshing MVs can cause heavy I/O.
    *   *Mitigation*: Using `CONCURRENTLY` ensures we do not take exclusive locks on the read tables. We isolate the refresh jobs to run sequentially, preventing I/O spikes from overwhelming the primary DB.
*   **Storage Growth**:
    *   *Risk*: Denormalization duplicates data. `fact_sales` and MVs will increase disk usage by an estimated 30-40%.
    *   *Mitigation*: Storage is cheap; compute is expensive. We utilize PostgreSQL's TOAST compression automatically, and partitioning on `fact_sales` allows us to move older partitions to cheaper, slower disk tablespaces if necessary.
*   **Staleness / Lag**:
    *   *Risk*: A user updates their address, but the UI shows the old address for 15 minutes.
    *   *Mitigation*: For the specific user making the update, the application layer should implement a **Read-Through Cache** (e.g., Redis) or locally mutate the UI state to reflect the change immediately, relying on the DB for global state.
*   **Monitoring & Alerts**:
    *   *Alert 1: Refresh Failure*: Trigger PagerDuty if `pg_stat_activity` shows the MV refresh query failing or taking > 5 minutes.
    *   *Alert 2: Data Drift*: A daily reconciliation script will assert that `SUM(line_total)` in `fact_sales` equals `SUM(orderqty * unitprice * (1-discount))` in `salesorderdetail`. Alert on diff > $0.01.
