### Executive Summary

To deliver measurable read performance improvements for your educational and employability tracking workload, I have designed a **Wide-Table Materialized View (MV) architecture**. The current normalized schema requires complex 4-to-5-way joins to answer standard analytical questions (e.g., "What is the employment rate and average salary of students from a specific course?"). Furthermore, joining `matricula_ifb` (Enrollments) directly with `vinculo_empregaticio` (Employment) creates a Cartesian product risk, as a student can have multiple enrollments and multiple jobs. 

To resolve this, the proposed physical schema splits the read domains into three denormalized materialized views: **Academic Journey**, **Employability**, and **School Panorama**. This approach completely decouples analytical reads from transactional writes, guarantees no OLTP lock contention (as no triggers are used), and relies on a scheduled batch refresh strategy (T-1 staleness).

### 2. Denormalization Patterns Chosen

1.  **Wide Read Tables (Materialized Views):** We will use PostgreSQL `MATERIALIZED VIEW`s. This is native, requires zero application-side CDC logic, and avoids trigger overhead.
2.  **Pre-joined Dimensions:** Lookups like `curso_ifb`, `localizacao`, and `remuneracao` are fully flattened into the transaction grain.
3.  **Time-Bucketed Rollups:** For school metrics, we will merge `desempenho_enem`, `indicadores_escolares`, and `participacao_enem` into a single `(School, Year)` grain.

### 3. Physical Design Details (DDL)

*Assumptions: PostgreSQL engine. Nightly batch windows are available.*

#### A. Academic Journey Wide Table
*Target: Q1. Eliminates 3-way joins. Grain: 1 row per Enrollment.*

```sql
CREATE MATERIALIZED VIEW public.mv_matricula_aluno_curso AS
SELECT 
    m.id_matricula,
    m.ano AS ano_matricula,
    m.periodo,
    m.situacao_matricula,
    m.tipo_cota,
    m.unidade_ensino,
    -- Aluno Dimensions
    a.id_aluno,
    a.idade,
    a.sexo,
    a.raca,
    a.portador_deficiencia,
    -- Curso Dimensions
    c.co_curso,
    c.no_curso,
    c.tipo_curso,
    c.modalidade_ensino,
    c.ds_eixo_tecnologico
FROM public.matricula_ifb m
JOIN public.aluno_ifb a ON m.id_aluno = a.id_aluno
JOIN public.curso_ifb c ON m.co_curso = c.co_curso;

-- UNIQUE INDEX is mandatory for CONCURRENTLY refresh
CREATE UNIQUE INDEX uk_mv_matricula ON public.mv_matricula_aluno_curso(id_matricula);

-- Read-optimized indexes for filtering and grouping
CREATE INDEX idx_mv_mat_curso ON public.mv_matricula_aluno_curso(co_curso, ano_matricula);
CREATE INDEX idx_mv_mat_aluno ON public.mv_matricula_aluno_curso(id_aluno);
CREATE INDEX idx_mv_mat_demographics ON public.mv_matricula_aluno_curso(raca, sexo);
```

#### C. School Panorama (Pre-Aggregated Snapshot)
*Target: Q3. Resolves the fragmented 1:1/1:N yearly school metrics. Grain: 1 row per School per Year.*

```sql
CREATE MATERIALIZED VIEW public.mv_escola_panorama_anual AS
WITH anos AS (
    -- Derive all distinct years across the three fact tables
    SELECT nu_ano, co_escola_educacenso FROM public.desempenho_enem
    UNION
    SELECT nu_ano, co_escola_educacenso FROM public.indicadores_escolares
    UNION
    SELECT nu_ano, co_escola_educacenso FROM public.participacao_enem
)
SELECT 
    -- Surrogate composite key for the MV
    md5(a.co_escola_educacenso || a.nu_ano::text) AS id_panorama,
    a.nu_ano,
    e.co_escola_educacenso,
    e.no_escola_educacenso,
    e.tp_dependencia_adm_escola,
    l.uf,
    l.municipio,
    -- Desempenho
    d.nu_media_cn, d.nu_media_ch, d.nu_media_lp, d.nu_media_mt, d.nu_media_red,
    -- Indicadores
    i.nu_taxa_permanencia, i.nu_taxa_aprovacao, i.nu_taxa_abandono, i.porte_escola,
    -- Participacao
    p.nu_participantes, p.nu_taxa_participacao
FROM anos a
JOIN public.escola e ON a.co_escola_educacenso = e.co_escola_educacenso
JOIN public.localizacao l ON e.id_loc = l.id_loc
LEFT JOIN public.desempenho_enem d ON a.co_escola_educacenso = d.co_escola_educacenso AND a.nu_ano = d.nu_ano
LEFT JOIN public.indicadores_escolares i ON a.co_escola_educacenso = i.co_escola_educacenso AND a.nu_ano = i.nu_ano
LEFT JOIN public.participacao_enem p ON a.co_escola_educacenso = p.co_escola_educacenso AND a.nu_ano = p.nu_ano;

CREATE UNIQUE INDEX uk_mv_escola_panorama ON public.mv_escola_panorama_anual(id_panorama);

CREATE INDEX idx_mv_escola_ano_uf ON public.mv_escola_panorama_anual(nu_ano, uf);
```

### 4. Consistency Architecture & Implementation Plan

**Data Lineage:** Source-of-truth remains the normalized OLTP tables. The MVs are strictly downstream read-replicas.
**Staleness Window:** Intentional **T-1 (24 hours) staleness**. Educational and RAIS data is typically updated in large annual/monthly batches, making real-time consistency unnecessary and computationally wasteful.

**Implementation Plan (Safe & Reversible):**
1.  **Deploy DDL:** Execute the `CREATE MATERIALIZED VIEW` statements during off-peak hours. This acts as the initial backfill.
2.  **Cron/Scheduler Setup:** Implement a nightly job (e.g., via pg_cron, Airflow, or a simple bash script) executing:
    `REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_matricula_aluno_curso;`
    *(The `CONCURRENTLY` keyword is critical: it allows reads to continue uninterrupted while the view is rebuilding in the background).*
3.  **Application Routing:** Update BI tools or read-heavy API endpoints to point to the `mv_*` tables instead of the normalized tables.
4.  **Reversibility:** If data discrepancies occur, applications can be instantly repointed to the normalized views/tables while the MVs are debugged.

### 5. Risk Analysis & Operational Complexity

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **Zero** during OLTP hours. Writes only happen during the scheduled nightly refresh. |
| **Lock Contention** | **Minimal.** Using `REFRESH ... CONCURRENTLY` requires a brief exclusive lock only at the very end of the refresh process to swap pointers. |
| **Storage Growth** | **Moderate.** Denormalizing strings (e.g., `no_curso`, `municipio`) will increase disk usage. *Mitigation:* PostgreSQL uses TOAST compression automatically for large strings, but ensure you have ~2x the storage of the normalized tables available. |
| **Failure Modes** | If the nightly refresh fails, data becomes T-2 stale. *Mitigation:* Set up alerting on the scheduler. The application will not crash; it will just serve yesterday's data. |
| **Monitoring/Alerts** | 1. Monitor `pg_stat_user_tables.last_autovacuum` and `last_autoanalyze` for the MVs.<br>2. Alert if the refresh cron job duration exceeds the batch window.<br>3. Monitor query latency on the MVs to ensure indexes are being hit (via `pg_stat_statements`). |
