As a Senior Database Performance Engineer, looking at your highly normalized schema, I see a classic OLTP (Online Transaction Processing) design. While excellent for preventing update anomalies, this 3rd Normal Form (3NF) structure will severely penalize a read-heavy analytical or reporting workload. 

Every time a user wants to view a cleanup event dashboard or a conservation unit profile, the database engine is forced to perform 4-to-5-way joins, resulting in random I/O, high CPU usage for hash/merge joins, and increased latency.

Here is the step-by-step denormalization strategy to transform this into a high-performance read model.

---

### 1. Workload-Driven Analysis

Based on the domain (Environmental Cleanups and Conservation Units), your read-heavy workload likely consists of these high-cost operations:

*   **Query Pattern A (Event Dashboards):** "Show me cleanup events in a specific State/Municipality, including the institution name, location name, and the amount of plastic/glass collected."
    *   *Current Bottleneck:* Requires joining `mutirao_de_limpeza` $\rightarrow$ `coleta_de_lixo` (1:1) $\rightarrow$ `local_de_coleta` (1:1) $\rightarrow$ `regiao` $\rightarrow$ `instituicao_organizadora`. This is a 5-table join just to read a single event's full context.
*   **Query Pattern B (Conservation Unit Profiles):** "List conservation units filtered by Biome, Management Category, or State."
    *   *Current Bottleneck:* Requires joining `unidade_de_conservacao` $\rightarrow$ `bioma` $\rightarrow$ `categoria_manejo` $\rightarrow$ `esfera_administrativa` $\rightarrow$ `regiao`. Another 5-table join for catalog browsing.
*   **Query Pattern C (Analytical Aggregations):** "What is the total amount of plastic collected per month, per state?"
    *   *Current Bottleneck:* Requires scanning and joining fact tables with dimension tables, then performing CPU-heavy `SUM()` and `GROUP BY` operations on the fly.

---

### 2. Denormalization Design

To achieve sub-millisecond latency on lookups and fast dashboard rendering, we will implement a **Star-Schema inspired Wide-Table approach** with precomputed aggregates.

1.  **Flatten 1:1 Relationships (Vertical Merging):** 
    *   `coleta_de_lixo` and `local_de_coleta` share the exact same primary key (`id_mutirao`) as `mutirao_de_limpeza`. We will merge these into a single wide fact table: `fact_mutirao_limpeza`. This eliminates two joins immediately.
2.  **Embed Slow-Changing Dimensions (SCD Type 1):**
    *   Tables like `bioma`, `esfera_administrativa`, `categoria_manejo`, and `regiao` rarely change. We will inject their descriptive text columns (`nome`, `estado`, `tipo`) directly into our main tables. This completely eliminates lookup joins.
3.  **Precompute Row-Level Metrics:**
    *   Queries often need the total trash collected. Instead of calculating `plastico + borracha + madeira...` on every read, we will add a `total_residuos` column computed at write-time.
4.  **Materialized Views for Aggregates:**
    *   For Query Pattern C, we will create a Materialized View with a grain of `Year-Month-State`. Dashboards will query this view instantly instead of scanning millions of rows.
5.  **Partitioning:**
    *   Cleanup events are time-series data. We will partition `fact_mutirao_limpeza` by `data_do_ocorrido` (yearly) to enable partition pruning on date-range queries.
6.  **Covering Indexes:**
    *   We will use B-Tree indexes aligned to the most common `WHERE` clauses (State, Date, Biome) and use the `INCLUDE` clause to append frequently `SELECT`ed columns, allowing Index-Only Scans.

---

### 3. New Physical Schema (PostgreSQL DDL)

*Note: You can keep your original tables for write-operations (OLTP) and use a CDC (Change Data Capture) tool or triggers to populate this new read-optimized schema, or replace them entirely if writes are infrequent.*

```sql
-- ==============================================================================
-- 1. WIDE FACT TABLE: CLEANUP EVENTS (Partitioned by Date)
-- Eliminates 4 joins. Combines mutirao, coleta, local, regiao, and instituicao.
-- ==============================================================================

CREATE TABLE IF NOT EXISTS public.fact_mutirao_limpeza
(
    id_mutirao integer NOT NULL,
    data_do_ocorrido date NOT NULL,
    horario_de_inicio time without time zone NOT NULL,
    duracao_total time without time zone NOT NULL,
    numero_aproximado_de_participantes integer,
    
    -- Denormalized from local_de_coleta
    local_nome text COLLATE pg_catalog."default",
    
    -- Denormalized from regiao
    id_municipio integer,
    municipio_nome text COLLATE pg_catalog."default",
    estado text COLLATE pg_catalog."default",
    
    -- Denormalized from instituicao_organizadora
    id_instituicao integer,
    instituicao_nome text COLLATE pg_catalog."default",
    instituicao_carater text COLLATE pg_catalog."default",

    -- Denormalized from coleta_de_lixo
    plastico integer DEFAULT 0,
    borracha integer DEFAULT 0,
    madeira integer DEFAULT 0,
    produtos_texteis integer DEFAULT 0,
    papel integer DEFAULT 0,
    metal integer DEFAULT 0,
    vidro integer DEFAULT 0,
    materiais_de_pesca integer DEFAULT 0,
    outros_residuos integer DEFAULT 0,
    
    -- Precomputed row-level aggregate
    total_residuos integer GENERATED ALWAYS AS (
        COALESCE(plastico, 0) + COALESCE(borracha, 0) + COALESCE(madeira, 0) + 
        COALESCE(produtos_texteis, 0) + COALESCE(papel, 0) + COALESCE(metal, 0) + 
        COALESCE(vidro, 0) + COALESCE(materiais_de_pesca, 0) + COALESCE(outros_residuos, 0)
    ) STORED,

    -- PK must include partition key in Postgres
    CONSTRAINT fact_mutirao_limpeza_pkey PRIMARY KEY (id_mutirao, data_do_ocorrido)
) PARTITION BY RANGE (data_do_ocorrido);

-- Create Partitions (Example for recent years, add more as needed)
CREATE TABLE fact_mutirao_limpeza_2023 PARTITION OF public.fact_mutirao_limpeza FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');
CREATE TABLE fact_mutirao_limpeza_2024 PARTITION OF public.fact_mutirao_limpeza FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- Index Strategy for Fact Table
-- 1. Dashboard filter by State and Date (Very common)
CREATE INDEX idx_fact_mutirao_estado_data 
    ON public.fact_mutirao_limpeza (estado, data_do_ocorrido DESC)
    INCLUDE (total_residuos, numero_aproximado_de_participantes); -- Enables Index-Only scans for high-level metrics

-- 2. Filter by Institution
CREATE INDEX idx_fact_mutirao_instituicao 
    ON public.fact_mutirao_limpeza (id_instituicao);


-- ==============================================================================
-- 2. WIDE DIMENSION TABLE: CONSERVATION UNITS
-- Eliminates 4 joins. Embeds bioma, manejo, esfera, and regiao data.
-- ==============================================================================

CREATE TABLE IF NOT EXISTS public.dim_unidade_conservacao
(
    id_uc integer NOT NULL,
    nome text COLLATE pg_catalog."default" NOT NULL,
    ano_de_criacao integer,
    area_soma_biomas double precision,
    area_soma_biomas_continentais double precision,
    area_marinha double precision,
    
    -- Denormalized from bioma
    id_bioma integer,
    bioma_nome text COLLATE pg_catalog."default",
    
    -- Denormalized from regiao
    id_municipio integer,
    municipio_nome text COLLATE pg_catalog."default",
    estado text COLLATE pg_catalog."default",
    
    -- Denormalized from categoria_manejo
    id_manejo integer,
    manejo_categoria text COLLATE pg_catalog."default",
    manejo_iucn text COLLATE pg_catalog."default",
    
    -- Denormalized from esfera_administrativa
    id_esfera integer,
    esfera_tipo text COLLATE pg_catalog."default",

    CONSTRAINT dim_unidade_conservacao_pkey PRIMARY KEY (id_uc)
);

-- Index Strategy for Dimension Table
-- 1. Filter by Biome (e.g., "Show me all Amazon UCs")
CREATE INDEX idx_dim_uc_bioma ON public.dim_unidade_conservacao (bioma_nome);

-- 2. Filter by State / Municipality
CREATE INDEX idx_dim_uc_localizacao ON public.dim_unidade_conservacao (estado, municipio_nome);


-- ==============================================================================
-- 3. PRECOMPUTED AGGREGATE: MONTHLY STATE DASHBOARD
-- Grain: Year, Month, State. 
-- Drastically reduces CPU load for analytical queries.
-- ==============================================================================

CREATE MATERIALIZED VIEW public.mvw_resumo_coleta_mensal_estado AS
SELECT 
    EXTRACT(YEAR FROM data_do_ocorrido) AS ano,
    EXTRACT(MONTH FROM data_do_ocorrido) AS mes,
    estado,
    COUNT(id_mutirao) AS total_mutiroes,
    SUM(numero_aproximado_de_participantes) AS total_participantes,
    SUM(total_residuos) AS total_residuos_geral,
    SUM(plastico) AS total_plastico,
    SUM(vidro) AS total_vidro,
    SUM(metal) AS total_metal
FROM 
    public.fact_mutirao_limpeza
GROUP BY 
    1, 2, 3
WITH DATA;

-- Index the Materialized View for instant dashboard load times
CREATE UNIQUE INDEX idx_mvw_resumo_mensal_pk ON public.mvw_resumo_coleta_mensal_estado (ano, mes, estado);
CREATE INDEX idx_mvw_resumo_mensal_estado ON public.mvw_resumo_coleta_mensal_estado (estado, ano, mes);

-- Note: You will need to refresh this view periodically (e.g., via a cron job or trigger)
```

### Summary of Performance Gains:
1. **Zero-Join Reads:** Fetching a cleanup event or a conservation unit now requires `0` joins. It is a single sequential scan or index seek.
2. **Reduced I/O:** By combining the 1:1 tables (`coleta_de_lixo`, `local_de_coleta`), the database reads contiguous blocks on disk rather than chasing pointers across three different table heaps.
3. **CPU Offloading:** The `total_residuos` calculation is done once during `INSERT`/`UPDATE`. Read queries no longer spend CPU cycles doing row-by-row math.
4. **Instant Analytics:** The Materialized View allows BI tools to render state-wide, multi-year charts in milliseconds, completely bypassing the raw fact table.
