### 2. Denormalization Patterns

**Pattern Chosen: Wide Read Table (Pre-joined Dimension)**
We will flatten the many-to-many relationship into a single table: `read_user_entitlements`. 
*   *Justification:* Q1 and Q2 are user-centric. By pre-joining `entitlements` to `user_id`, we eliminate all joins at read time. 
*   *Pre-computed Rollups:* We will introduce a boolean column `is_valid_access`. Evaluating `activeFlag = 1 AND activeDate <= NOW() AND inactiveDate >= NOW() AND useCount < useLimit` at read-time burns CPU. Pre-computing this at write-time shifts the compute burden to the asynchronous CDC pipeline.

**Pattern Chosen: Engine-Level Materialized View (For Analytics)**
We will use a standard Materialized View for Q3.
*   *Justification:* Q3 does not require user-level granularity. A simple asynchronous materialized view over the source `entitlements` table is sufficient and avoids scanning the massive wide table.

### 3. Physical Design Details (DDL)

```sql
-- ==============================================================================
-- TARGET SCHEMA: PostgreSQL (Read-Optimized)
-- ==============================================================================

-- 1. Wide Read Table for Q1 (AuthZ) and Q2 (Dashboard)
CREATE TABLE read_user_entitlements (
    user_id INT NOT NULL,
    entitlement_id INT NOT NULL,
    featureType VARCHAR(4) NOT NULL,
    packageType VARCHAR(4) NOT NULL,
    
    -- Source of truth data (replicated via CDC)
    activeFlag INT NOT NULL,
    activeDate DATE,
    inactiveDate DATE,
    useLimit INT,
    useCount INT,
    
    -- PRE-COMPUTED: Shifts CPU load from Read to Write path
    -- Updated by the stream processor when dates/counts change
    is_valid_access BOOLEAN NOT NULL, 

    -- Exact PK strategy: Composite key guarantees uniqueness per user-entitlement
    PRIMARY KEY (user_id, entitlement_id)
) 
-- Optional: Table clustering if engine supports it (e.g., Postgres CLUSTER)
-- to physically group user records together on disk for Q2.
;

-- INDEXING STRATEGY
-- Index for Q1 (AuthZ Check): "Does user X have access to feature Y?"
-- Uses a partial index to keep the B-Tree tiny and ultra-fast, storing only active grants.
CREATE INDEX idx_rue_authz 
    ON read_user_entitlements (user_id, featureType) 
    WHERE is_valid_access = TRUE;

-- Index for Q2 (User Dashboard): "List all entitlements for user X"
-- Covering index prevents heap lookups for dashboard rendering.
CREATE INDEX idx_rue_dashboard 
    ON read_user_entitlements (user_id) 
    INCLUDE (entitlement_id, featureType, packageType, useCount, useLimit);

-- 2. Materialized View for Q3 (Analytics)
CREATE MATERIALIZED VIEW mv_package_usage AS
SELECT 
    packageType,
    SUM(useCount) as total_uses,
    COUNT(id) as total_entitlements
FROM entitlements
GROUP BY packageType;

-- Index for Q3 fast retrieval
CREATE UNIQUE INDEX idx_mv_package_usage ON mv_package_usage(packageType);
```

### 4. Consistency Architecture

Data flows from the Normalized Source to the Denormalized Read schema via an Eventual Consistency model.

*   **Data Lineage:**
    *   `users.id` → `read_user_entitlements.user_id`
    *   `entitlements.*` → `read_user_entitlements.*`
    *   Stream Processor Logic: `is_valid_access` = `(activeFlag = 1 AND CURRENT_DATE BETWEEN activeDate AND inactiveDate AND useCount < useLimit)`
*   **Refresh Mechanism (CDC Pipeline):**
    1.  **Capture:** Debezium monitors the WAL (Write-Ahead Log) of the source database.
    2.  **Process:** A stream processor (e.g., Flink or Kafka Streams) consumes changes.
        *   *If `user_entitlements` changes:* Emit an UPSERT/DELETE to `read_user_entitlements`.
        *   *If `entitlements` changes:* Emit an UPSERT for **all** `user_id`s associated with that `entitlement_id`.
    3.  **Apply:** Sink connector writes to the read database.
    4.  **MV Refresh:** A cron job executes `REFRESH MATERIALIZED VIEW CONCURRENTLY mv_package_usage` every 5 minutes.

### 5. Risk Analysis

| Risk Area | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **High Risk.** If `entitlements.useCount` updates frequently, and an entitlement is shared by 10,000 users, one source update triggers 10,000 updates in the wide table. *Mitigation:* If `useCount` is highly volatile, we must remove `useCount` from the wide table and do a 2-step read (AuthZ checks wide table for validity, then checks a fast in-memory cache like Redis for current `useCount`). |
| **Staleness / Correctness** | **Medium Risk.** CDC lag means a user might successfully authorize a feature slightly after hitting their `useLimit`. *Mitigation:* Document this as a "soft limit" business rule. Implement strict limit enforcement asynchronously at the application layer, or accept the 1-2 second overage window. |
| **Storage Growth** | **Low Risk.** The wide table duplicates entitlement metadata per user. Given the narrow data types (INT, VARCHAR(4), DATE), the byte footprint per row is ~40 bytes. 100 million grants = ~4GB, which is trivial for modern storage. |
| **Monitoring & Alerts** | Must implement: <br>1. **Replication Lag Alert:** Trigger if CDC pipeline falls > 5 seconds behind.<br>2. **Reconciliation Job:** A nightly batch script that runs a `MINUS` query between the joined source tables and the wide table to detect drift.<br>3. **Query Latency:** Monitor p99 latency on Q1 to ensure the partial index is being utilized. |
