# Executive Summary

This design introduces a **Read-Optimized Physical Schema** alongside your existing OLTP normalized schema. The primary objective is to eliminate join latency for analytical queries (BI, Reporting, Dashboards) by pre-joining dimensions (NATION, REGION, SUPPLIER, PART) into fact tables (ORDERS, LINEITEM). 

**Key Trade-offs:**
1.  **Correctness:** The normalized tables remain the **Single Source of Truth (SSOT)**. The denormalized tables are read-replicas refreshed via batch/CDC.
2.  **Complexity:** Introduces an ETL pipeline. Complexity is contained to the refresh layer, not the transactional layer.
3.  **Staleness:** Denormalized data is refreshed within a **15-minute window** (assuming CDC) or **4-hour window** (assuming batch). Business logic must account for this lag if querying the denormalized layer.
4.  **Reversibility:** The denormalized tables are separate physical objects. Dropping them restores the baseline performance state without data loss in the normalized tables.

# 3. Consistency Architecture

Since we are creating **physical tables** (not materialized views), we must manage consistency explicitly.

### 3.1 Refresh Strategy (ETL)
We assume a **CDC (Change Data Capture)** or **Batch** pipeline exists.
*   **Mechanism:** A background job (e.g., Airflow, Flink, or a custom Python script) extracts changes from the normalized tables.
*   **Logic:**
    1.  Extract `INSERT`/`UPDATE`/`DELETE` events from `ORDERS`, `CUSTOMER`, etc.
    2.  Transform: Join the extracted rows with the latest dimension data.
    3.  Load: `UPSERT` (MERGE) the data into the Wide Tables.
*   **Lag:** The wide tables are **not** live. Queries must be aware of the **15-minute lag** (or 4-hour lag if batch).
*   **Reconciliation:** A nightly job compares row counts and checksums of the normalized source vs. the denormalized target to detect data drift.

### 3.2 Handling Updates
*   **Source Update:** If `CUSTOMER` name changes in the normalized table, the denormalized table **does not update immediately**.
*   **Impact:** The denormalized table reflects the customer name as of the last refresh.
*   **Mitigation:** If a user queries the denormalized table and sees a "stale" name, the application layer should check the `LAST_REFRESH_TIMESTAMP` column (added to DDL) and alert if the data is older than the acceptable threshold.
