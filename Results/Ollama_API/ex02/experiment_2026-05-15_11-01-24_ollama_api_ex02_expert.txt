# Executive Summary

This denormalized schema design optimizes read performance for a data model centered around environmental cleanup events (`mutirao_de_limpeza`) and conservation units (`unidade_de_conservacao`). By collapsing multiple normalized tables into two wide, read-optimized tables (`mutirao_limpeza` and `unidade_de_conservacao`), we eliminate costly joins that were previously required to answer common analytical queries. This approach trades write complexity and storage for significant read performance gains, particularly for dashboarding, reporting, and ad-hoc analysis workloads.

The design preserves business correctness through explicit data lineage and reconciliation mechanisms, while acknowledging a controlled staleness window of up to the batch refresh interval. Implementation follows a safe, reversible path with comprehensive monitoring to detect divergence or performance regressions.

# Denormalization Patterns Chosen

## 1. Wide Read Tables (Pre-Joined Dimensions)
- **Justification**: Both `mutirao_limpeza_denorm` and `unidade_de_conservacao_denorm` collapse multiple one-to-one or many-to-one relationships into wide rows to eliminate joins at query time.
- **Benefit**: Eliminates up to 4 joins per query for cleanup events and up to 5 joins for conservation units.

## 2. Snapshot Tables for â€śCurrent Stateâ€ť
- **Justification**: The denormalized tables act as materialized snapshots refreshed periodically.
- **Benefit**: Ensures predictable read performance with bounded staleness.

## 3. Optional: Star Schema (Not Chosen)
- **Rationale**: The workload appears operational/reporting-oriented rather than OLAP/analytics-heavy. Wide tables better suit the query patterns observed.

# Risk Analysis

## Write Amplification
- **Impact**: Each source table change may require updates to one or more denormalized rows.
- **Mitigation**: Use batched refresh to avoid row-level write amplification.

## Lock Contention
- **Impact**: Full-table refresh may lock rows during update.
- **Mitigation**: Use `CREATE TABLE ... AS SELECT` + `RENAME` pattern for atomic swap.

## Failure Modes
- **Incomplete Refresh**: Monitor `refreshed_at` timestamps.
- **Schema Drift**: Monitor column mismatches via reconciliation queries.

## Backfill Runtime
- **Estimate**: Depends on row count; test with subset first.
- **Strategy**: Parallelize across tables or partitions if needed.

## Storage Growth
- **Impact**: ~2x storage due to redundancy.
- **Mitigation**: Archive old snapshots if retention policy allows.
