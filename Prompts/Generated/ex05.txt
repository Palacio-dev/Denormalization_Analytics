BEGGINER:
You are a data modeling assistant. I have a relational database designed for transaction processing, and I want to create a denormalized physical schema for analytics and reporting.

Your task:

Review the source tables, columns, primary keys, and foreign keys.
Explain which tables should be combined and why.
Design a denormalized physical schema that improves read performance for analytics.
Keep the model simple and easy to understand.
Preserve the important business meaning of the data.
Propose a new schema that is suitable for dashboards, aggregations, and reporting.
Identify the main fact-like data and the descriptive data.
Suggest useful indexes only if they clearly help analytics queries.
Provide the output in a clear step-by-step format.
Please include:

A short explanation of the denormalization approach
The proposed target tables
The relationships or joins that were removed or flattened
Any assumptions you made
A brief example of how the new schema supports analytics queries

Input available:

Source relational schema
Table definitions
Sample queries if needed
Goal:
Create a version of the database optimized for analytics, not for transaction updates.


INTERMEDIATE:
You are a database architect specializing in analytics modeling. I need to transform an existing normalized relational 
schema into a denormalized physical schema optimized for read-heavy analytical workloads.

Context:

- The source system is a normalized OLTP database.
- The target system is an analytics-oriented physical model.
- The goal is improved query performance, simpler joins, and easier reporting.

Your task:

- Analyze the current schema and identify the core business processes represented.
- Determine the best denormalization strategy for analytics.
- Design a new physical schema, preferably following dimensional modeling principles when appropriate.
- Consolidate tables where it reduces join complexity without losing analytical meaning.
- Define the grain of each target table.
- Identify fact data, dimensions, and any degenerate attributes.
- Recommend surrogate keys if useful.
- Specify indexing, partitioning, or clustering strategies that support analytical access patterns.
- Call out any data redundancies introduced by denormalization and explain why they are acceptable.
- Highlight trade-offs between read performance, storage cost, and maintenance complexity.

Deliverables:

- A short assessment of the current model
- A denormalized target schema
- Table-by-table transformation logic
- Key design decisions and assumptions
- Suggested physical optimizations for analytics
- Example analytical queries that the new schema supports efficiently

Important:
The result should be suitable for a data warehouse or analytics mart.
Prioritize query speed and usability for reporting over normalization.

EXPERT:
You are a senior data architect and physical database designer. I need you to redesign a normalized relational OLTP 
schema into a denormalized analytics-ready physical model.

Objective:
Create a new physical schema that serves as the base layer for analytics. The design must optimize read performance, 
reduce join cost, and support common analytical workloads such as aggregation, slicing, filtering, trending, and reporting.

What you must do:

- Reverse-engineer the source schema and infer business processes, entities, and relationships.
- Define the analytical grain explicitly for each target table.
- Choose an appropriate denormalization pattern:
    - star schema
    - snowflake with partial denormalization
    - wide analytical table
    - hybrid model
- Justify the chosen pattern based on workload and query shapes.
- Model conformed dimensions where relevant.
- Flatten or pre-join descriptive attributes that are repeatedly used in analytics.
- Preserve historical analysis requirements if the source data contains slowly changing attributes.
- Address measures, additive/semi-additive/non-additive behavior, and aggregation safety.
- Propose surrogate keys, natural key retention, and any bridge tables if many-to-many relationships exist.
- Recommend partitioning, indexing, clustering, materialized views, or other physical optimizations appropriate for the target DBMS.
- Identify potential data quality risks introduced by denormalization.
- Provide migration mapping from source tables to target structures.
- Explain what should be materialized now versus computed at query time.

Required output:

- Executive summary of the modeling choice
- Detailed target physical schema
- Table definitions at the conceptual/physical level
- Source-to-target transformation rules
- Grain, measures, dimensions, and key strategy
- Performance considerations
- Assumptions and risks
- Example analytics use cases and how the schema supports them

Constraints:

- Optimize for a version of the base intended for analytics, not OLTP.
- Favor scan efficiency and fewer joins over normalization.
- Make the design explicit enough to be implemented in a warehouse or lakehouse relational layer.
