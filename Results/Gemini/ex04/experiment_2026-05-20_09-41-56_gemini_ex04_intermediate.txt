As a senior database performance engineer, I have reviewed your normalized schema. While your current model is excellent for OLTP (minimizing update anomalies), it is an anti-pattern for an analytical, read-heavy workload. 

In analytical workloads (like socio-economic data analysis), CPU cycles and I/O are wasted traversing relationships that rarely change. Below is the workload analysis, the denormalization strategy, and the optimized physical DDL.

---

### 1. Workload-Driven Analysis

**High-Cost Operations in Current Schema:**
1. **The "Country-Year" Join Explosion:** `demography`, `employment`, `education`, `economy`, and `health` all share the exact same logical grain: `(country_code, year)`. To view a country's GDP alongside its unemployment and health expenditure, the database must execute a 5-way hash/merge join. This is highly expensive and unnecessary.
2. **Snowflake Dimension Traversal:** To filter any metric by `region_name`, the engine must join the fact table -> `country` -> `region`. Regions and countries are static; this join is wasted compute.
3. **EAV (Entity-Attribute-Value) Anti-patterns:** `life_expectancy` and `population` are modeled as child tables of `demography` using key-value pairs (e.g., gender, age group). This forces aggregate queries to use slow `GROUP BY` and `CASE WHEN` pivots at runtime just to compare male vs. female life expectancy.
4. **Survey-Decile Chaining:** Analyzing income distribution over time requires joining `decile` -> `survey` -> `country`. 

**Denormalization Candidates:**
- Merge all `(country_code, year)` tables into a single Wide Fact Table.
- Flatten `region` and `country` into a single Dimension Table.
- Pivot `life_expectancy` into columns.
- Push `country_code` and `year` down into child tables (`decile`, `population`) to allow them to be queried independently.

---

### 2. Denormalization Design

**1. Pre-joined "Wide" Fact Table (`fact_country_year_indicators`)**
We will combine economy, education, employment, health, and demography into one wide table. 
*Grain:* 1 row per `country_code` per `year`.
*Benefit:* Eliminates 5-way joins. BI tools can now scan a single table to correlate GDP, education, and health metrics.

**2. Pivoted Attributes (`life_expectancy`)**
Instead of a separate table for life expectancy by gender, we will pivot this into the wide fact table as `life_exp_male` and `life_exp_female`.
*Benefit:* Removes a table entirely and eliminates runtime pivoting.

**3. Flattened Dimension (`dim_country`)**
We will merge `region` into `country`.
*Benefit:* `region_name` is now immediately available for filtering without a join.

**4. Denormalized Child Facts (`fact_survey_decile` & `fact_population`)**
We will duplicate `country_code`, `year`, and `region_name` into the `decile` and `population` tables.
*Benefit:* If a user only wants to see the population pyramid for "Sub-Saharan Africa" in 2020, they query `fact_population` directly. Zero joins required.

**5. Indexing & Partitioning Strategy**
- **Partitioning:** Fact tables will be partitioned by `year` (Range partitioning). Time-series queries usually filter by date ranges; this allows the engine to skip scanning irrelevant years (Partition Pruning).
- **BRIN Indexes:** For the `year` columns on massive tables, Block Range Indexes (BRIN) are much smaller and faster for sequential scans than B-Trees.
- **Composite B-Trees:** We will use `(country_code, year)` and `(region_name, year)` to support the most common `WHERE` and `GROUP BY` clauses.

### 3. New Physical Schema (PostgreSQL DDL)

BEGIN;

-- ==============================================================================
-- 1. DIMENSION TABLES (Flattened)
-- ==============================================================================

-- Combines Country and Region. 
-- Grain: 1 row per country.
CREATE TABLE IF NOT EXISTS public.dim_country
(
    country_code character varying NOT NULL,
    country_name character varying,
    region_code character varying NOT NULL,
    region_name character varying NOT NULL,
    PRIMARY KEY (country_code)
);

-- Index for region-based filtering
CREATE INDEX idx_dim_country_region ON public.dim_country(region_name);


-- ==============================================================================
-- 2. WIDE FACT TABLE: MACRO INDICATORS
-- ==============================================================================

-- Combines Economy, Education, Employment, Health, Demography, and Life Expectancy.
-- Grain: 1 row per country per year.
-- Partitioned by year for fast time-series pruning.
CREATE TABLE IF NOT EXISTS public.fact_country_year_indicators
(
    country_code character varying NOT NULL,
    year integer NOT NULL,
    
    -- Demography
    pop_density double precision,
    urban_pop double precision,
    rural_pop double precision,
    net_migration double precision,
    death_rate double precision,
    birth_rate double precision,
    
    -- Life Expectancy (Pivoted from rows to columns)
    life_exp_male double precision,
    life_exp_female double precision,
    life_exp_total double precision,
    
    -- Economy
    gdp double precision,
    inflation double precision,
    tax_revenue double precision,
    
    -- Employment
    child_emp double precision,
    unemp double precision,
    vulnerable_emp double precision,
    part_time double precision,
    employers double precision,
    labor_force_total double precision,
    labor_force_fem double precision,
    
    -- Education
    child_out_of_school double precision,
    progression_to_sec double precision,
    edu_expenditure double precision,
    preprim_enroll double precision,
    prim_enrol double precision,
    sec_enrol double precision,
    terti_enrol double precision,
    
    -- Health
    hospital_beds double precision,
    physicians double precision,
    health_expenditure double precision,
    
    PRIMARY KEY (country_code, year)
) PARTITION BY RANGE (year);

-- Create partitions (Example for 2000-2030)
CREATE TABLE fact_indicators_2000_2010 PARTITION OF public.fact_country_year_indicators FOR VALUES FROM (2000) TO (2010);
CREATE TABLE fact_indicators_2010_2020 PARTITION OF public.fact_country_year_indicators FOR VALUES FROM (2010) TO (2020);

CREATE INDEX idx_fact_indicators_year_brin ON public.fact_country_year_indicators USING BRIN (year);


-- ==============================================================================
-- 3. FACT TABLE: POPULATION BREAKDOWN
-- ==============================================================================

-- Denormalized to include country_code and year directly (removes demography join).
-- Grain: 1 row per country, year, age group, and gender.
CREATE TABLE IF NOT EXISTS public.fact_population
(
    id_population serial NOT NULL,
    country_code character varying NOT NULL,
    year integer NOT NULL,
    pop_ages character varying NOT NULL,
    gender character varying NOT NULL,
    "number" double precision,
    PRIMARY KEY (id_population)
) PARTITION BY RANGE (year);

CREATE TABLE fact_pop_2000_2010 PARTITION OF public.fact_population FOR VALUES FROM (2000) TO (2010);
CREATE TABLE fact_pop_2010_2020 PARTITION OF public.fact_population FOR VALUES FROM (2010) TO (2020);
CREATE TABLE fact_pop_2020_2030 PARTITION OF public.fact_population FOR VALUES FROM (2020) TO (2030);

-- Composite index covering the most common slice-and-dice dimensions
CREATE INDEX idx_fact_pop_dimensions ON public.fact_population(country_code, year, gender, pop_ages);


-- ==============================================================================
-- 4. FACT TABLE: SURVEYS
-- ==============================================================================

-- Grain: 1 row per survey. 
-- Denormalized to include region_name to avoid joining dim_country for regional averages.
CREATE TABLE IF NOT EXISTS public.fact_survey
(
    id_survey serial NOT NULL,
    country_code character varying NOT NULL,
    region_name character varying NOT NULL, -- Denormalized
    survey_year integer NOT NULL,
    welfare_type character varying,
    survey_acronym character varying,
    survey_comparability integer,
    comparable_spell character varying,
    poverty_line double precision,
    headcount double precision,
    poverty_gap double precision,
    poverty_severity double precision,
    gini double precision,
    reporting_pop integer,
    reporting_pce double precision,
    distribution_type character varying,
    spl double precision,
    survey_coverage character varying,
    reporting_level character varying,
    PRIMARY KEY (id_survey)
);

CREATE INDEX idx_fact_survey_country_year ON public.fact_survey(country_code, survey_year);
CREATE INDEX idx_fact_survey_region_year ON public.fact_survey(region_name, survey_year);


-- ==============================================================================
-- 5. FACT TABLE: SURVEY DECILES
-- ==============================================================================

-- Denormalized to include survey_year and country_code.
-- Allows plotting deciles over time for a country without joining the survey table.
CREATE TABLE IF NOT EXISTS public.fact_survey_decile
(
    id_decile serial NOT NULL,
    id_survey integer NOT NULL,
    country_code character varying NOT NULL, -- Denormalized
    survey_year integer NOT NULL,            -- Denormalized
    name character varying,                  -- e.g., 'Decile 1'
    value double precision,
    PRIMARY KEY (id_decile)
);

CREATE INDEX idx_fact_decile_country_year ON public.fact_survey_decile(country_code, survey_year);


-- ==============================================================================
-- 6. MATERIALIZED VIEW: REGIONAL AGGREGATES (Read-Model)
-- ==============================================================================

-- Pre-computes the most common high-level dashboard queries.
-- Grain: 1 row per Region per Year.
CREATE MATERIALIZED VIEW public.mv_region_year_summary AS
SELECT 
    d.region_name,
    f.year,
    SUM(f.gdp) as total_gdp,
    AVG(f.inflation) as avg_inflation,
    AVG(f.life_exp_total) as avg_life_expectancy,
    AVG(f.unemp) as avg_unemployment
FROM public.fact_country_year_indicators f
JOIN public.dim_country d ON f.country_code = d.country_code
GROUP BY d.region_name, f.year;

CREATE UNIQUE INDEX idx_mv_region_year ON public.mv_region_year_summary(region_name, year);
