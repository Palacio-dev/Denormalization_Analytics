Here is a set of prompts designed for an LLM to denormalize a relational database schema, categorized by the user's level of database expertise. 

Each prompt uses bracketed placeholders like `[Insert...]` where the user needs to provide their specific context.

---

### Level 1: The Beginner
**Target Audience:** Users who understand basic tables and columns but aren't familiar with terms like "3NF," "joins," or the trade-offs of denormalization.
**Goal:** Have the LLM act as a guide, explain the changes simply, and do the heavy lifting of deciding *how* to combine the tables.

**The Prompt:**
> "I have a database setup, but my app is running slow when trying to read or display information because it has to look up data across too many different tables. I want to 'denormalize' it—meaning I want to combine some tables or duplicate some data to make reading faster.
> 
> Here are my current tables and what they contain:
> `[Insert a simple list of tables and columns, or paste your CREATE TABLE statements here]`
> 
> Here is the information my app needs to display together the most often:
> `[Describe the screen or report, e.g., "I need to show the Customer Name, their latest Order Date, and the Total Amount of the order on one dashboard"]`
> 
> Please do the following:
> 1. Create a new, simpler database structure (schema) that combines these tables to make my specific reads faster.
> 2. Write the SQL code to create these new tables.
> 3. Explain to me in simple terms what changes you made and why it will make reading data faster.
> 4. Tell me what I need to be careful about when updating or adding new data to this new structure."

---

### Level 2: The Intermediate
**Target Audience:** Developers or analysts who understand normalization (e.g., 3NF), joins, and foreign keys, but want the LLM to design the optimal denormalized structure and write the boilerplate DDL.
**Goal:** Focus on specific read bottlenecks, balance read vs. write trade-offs, and generate clean SQL DDL.

**The Prompt:**
> "I have a highly normalized relational database schema that is suffering from poor read performance due to complex, multi-table joins. I need to denormalize this physical model to optimize it for read-heavy operations.
> 
> Here is my current physical schema (DDL):
> `[Insert your CREATE TABLE statements with primary/foreign keys]`
> 
> Here are the specific read queries or access patterns that are currently causing performance bottlenecks:
> `[Insert slow queries, or describe the access pattern, e.g., "We frequently query Orders, joining Users, Products, and Categories to generate a daily sales report"]`
> 
> Please provide:
> 1. A new physical schema (SQL DDL) optimized for these read patterns. Apply techniques like flattening relationships, adding redundant columns, or pre-calculating aggregates.
> 2. Recommended indexes for the new denormalized tables based on my access patterns.
> 3. A brief analysis of the trade-offs: what write/update anomalies or storage costs should I expect with this new schema?"

---

### Level 3: The Expert / DBA
**Target Audience:** Database Administrators or Senior Engineers dealing with large-scale systems, data warehousing, or CQRS architectures. 
**Goal:** Highly specific instructions focusing on advanced denormalization techniques (Star schemas, JSON columns, materialized views) and data consistency mechanisms.

**The Prompt:**
> "Act as an expert Database Architect. I am redesigning a normalized OLTP physical schema into a denormalized read-model (OLAP/Read-Replica) to eliminate join latency and support high-throughput analytical reads. The target RDBMS is `[Insert Database, e.g., PostgreSQL 15 / MySQL 8]`.
> 
> Current Normalized 3NF Schema:
> ```sql
> [Insert Full DDL]
> ```
> 
> Primary Read Access Patterns & SLA Constraints:
> `[Insert highly specific patterns, e.g., "Aggregating sales by user_region and product_category over 30-day windows. Reads must execute in < 50ms."]`
> 
> Perform the following tasks:
> 1. **Denormalized Physical Model:** Generate the optimized DDL. Use advanced techniques where appropriate (e.g., flattening into wide tables, embedding 1:N relationships in JSON/JSONB columns, or proposing a Star Schema).
> 2. **Pre-aggregation:** Create schemas for summary/rollup tables if necessary to satisfy the aggregation requirements.
> 3. **Indexing Strategy:** Provide exact `CREATE INDEX` statements (B-Tree, Hash, GIN/GiST, or Covering Indexes) targeted at the read access patterns.
> 4. **Data Synchronization Strategy:** The source system is still normalized. Provide the DDL/logic for how to keep this denormalized schema eventually consistent. (Suggest either Database Triggers, Materialized View refresh strategies, or an application-level event-driven approach).
> 5. **Storage footprint:** Estimate the relative storage multiplier this denormalization will incur."

---

### 💡 Tips for the User to get the best results:
*   **Specify the SQL Dialect:** Always tell the LLM which database you are using (e.g., SQL Server, PostgreSQL, Oracle). The physical implementations of things like JSON data types, arrays, or materialized views differ wildly between systems.
*   **Include Data Volumes:** If table "A" has 100 rows and table "B" has 100 million rows, tell the LLM! This drastically changes how a database architect (and the LLM) will approach denormalization.
