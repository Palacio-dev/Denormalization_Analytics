Here is a collection of prompts designed to instruct a Large Language Model (LLM) to denormalize a relational database schema. The prompts are tailored for users with varying levels of database expertise, ranging from beginner to expert. 

### 🟢 Beginner Level
**Target Audience:** Users who know basic SQL (tables, columns, basic joins) but aren't familiar with database design theory, normal forms, or the specific trade-offs of denormalization.
**Goal:** Keep the terminology simple, explain *why* changes are made, and provide easy-to-understand SQL.

**Prompt Template:**
> "I have a relational database, but it is taking too long to read data because I have to join too many tables together. I want to 'denormalize' it—meaning I want to combine some of these tables or duplicate some data to make reading faster. 
>
> Here are the tables I currently have:
> `[Insert current SQL CREATE TABLE statements or describe the tables and their connections]`
>
> Please do the following:
> 1. Suggest a new, simplified database structure where I don't have to use as many joins.
> 2. Explain to me, in simple terms, exactly what you changed and why it will make reading data faster.
> 3. Provide the new SQL `CREATE TABLE` code for this simplified structure.
> 4. Briefly warn me about any downsides this new setup might have when I try to insert or update data."

***

### 🟡 Intermediate Level
**Target Audience:** Software developers or junior data analysts who understand normalization (e.g., 3rd Normal Form), know read vs. write trade-offs, and are looking for a practical schema for reporting or faster querying.
**Goal:** Focus on specific read patterns, provide solid DDL, and outline the pros and cons of the architectural shift.

**Prompt Template:**
> "I need to denormalize an existing physical database schema to heavily optimize it for read performance. The current schema is highly normalized, which is causing bottlenecks during read-heavy operations and reporting. 
>
> Here is my current physical schema:
> `[Insert current DDL]`
>
> Here are the primary queries/read operations that are currently too slow:
> `[Insert 1-3 common slow queries, or describe the data access pattern]`
>
> Please perform the following tasks:
> 1. Analyze the schema and identify the best candidates for denormalization (e.g., pre-calculating aggregates, adding redundant columns, flattening hierarchy, or creating a star schema).
> 2. Generate the exact SQL Data Definition Language (DDL) for the new denormalized physical schema.
> 3. Provide the SQL queries showing how data will be read from this new schema (demonstrating the reduction in joins).
> 4. Detail the specific trade-offs of your proposed design, particularly focusing on write-amplification and data consistency risks."

***

### 🔴 Expert/Advanced Level
**Target Audience:** Senior Backend Engineers, Database Administrators (DBAs), or Data Architects. They need production-ready code, indexing strategies, and mechanisms to keep redundant data synchronized.
**Goal:** Act as an architectural sparring partner. Generate comprehensive DDL, address specific RDBMS dialects, and solve the data-anomaly problems that denormalization introduces.

**Prompt Template:**
> "Act as a Senior Database Architect. I need to transition an OLTP-optimized normalized physical model into a read-heavy, low-latency denormalized physical model. The target database engine is `[Insert Database, e.g., PostgreSQL 15, SQL Server 2022, MySQL 8]`.
>
> **Current Context & Schema:**
> `[Insert current DDL with existing indexes]`
> 
> **Performance Constraints & Workload:**
> - Read-to-write ratio is approximately `[e.g., 95:5]`.
> - Acceptable read latency is `[e.g., < 50ms]`.
> - Data consistency requirement: `[e.g., Strong consistency / Eventual consistency acceptable within 5 minutes]`.
> 
> Please output a comprehensive architectural proposal containing:
> 1. **Denormalization Strategy:** Explain your approach (e.g., Materialized Views, wide tables, JSONB document embedding, or CQRS projection tables).
> 2. **Target Physical Schema:** Provide the fully optimized DDL, including optimal data types for the target engine, `NOT NULL` constraints, and primary/foreign keys.
> 3. **Indexing Strategy:** Include the DDL for secondary, covering, or partial indexes tailored to maximize read throughput on the new schema.
> 4. **Synchronization Mechanism:** Because denormalization introduces data anomalies, provide the exact code required to keep the denormalized read-schema in sync with incoming writes (e.g., Database Triggers, Stored Procedures, Materialized View refresh logic, or instructions for an application-level event bus).
> 5. **Storage & Maintenance Analysis:** Provide a brief estimation of storage bloat and the lock/contention impact during updates."

***

### 💡 Pro-Tips for Getting the Best Results:
*   **Specify the Dialect:** Always tell the LLM which database system you are using (PostgreSQL, Oracle, MySQL, etc.). Denormalization techniques vary wildly between them (e.g., Postgres uses Materialized Views natively, MySQL does not).
*   **Provide Data Volume:** If you have 10,000 rows vs. 10 Billion rows, the LLM will (and should) suggest completely different physical schemas. Mention your table sizes in the prompt.
*   **Define the "Read":** "Improving read performance" is broad. Tell the LLM *what* you are reading. Are you returning single customer profiles, or are you running massive aggregations for a monthly financial report?
