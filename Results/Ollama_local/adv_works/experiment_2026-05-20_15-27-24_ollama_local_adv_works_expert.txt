# Executive Summary

The current normalized schema for the Adventure Works database is well-structured for transactional integrity but suffers from high latency on read-heavy analytical queries due to multi-table joins. This design introduces a **Read-Optimized Physical Schema** (DDL) specifically targeting the most frequent access patterns: Employee Directory Lookups, Product Catalog Aggregations, and Sales Territory Performance.

The strategy utilizes **Wide Read Tables** (pre-joined dimensions) for entity lookups to eliminate join overhead, **Pre-Aggregated Rollups** for sales and inventory analytics to reduce aggregation time, and **Snapshot Tables** for consistent "current state" views. Crucially, this design explicitly documents a **5-minute staleness window** for denormalized data to ensure business correctness without triggering ACID transaction locks during reads. The implementation plan prioritizes zero-downtime backfilling and continuous monitoring of lag metrics.

## 1. Access-Pattern Matrix & Hotspot Analysis

Based on the provided DDL, the workload is dominated by "Who, Where, What, and How Much" queries.

| Query Class | Top Access Pattern | Join Complexity (Normalized) | Latency Bottleneck | Target Optimization |
| :--- | :--- | :--- | :--- :--- |
| **Employee Lookup** | `SELECT * FROM Employee JOIN Department...` | 3-4 Table Joins | Index seek + Join execution | **Wide Read Table** (`hr_employee_current`) |
| **Product Catalog** | `SELECT P.*, C.name FROM Product JOIN ProductCategory...` | 3-4 Table Joins | Join on foreign keys + filtering | **Wide Read Table** (`prod_catalog_view`) |
| **Sales Performance** | `SELECT SUM(SO.Subtotal) FROM SalesOrderHeader JOIN...` | 5+ Table Joins + Aggregation | Aggregation over large history | **Pre-Aggregated Rollup** (`sales_agg_monthly`) |
| **Inventory Status** | `SELECT P.name, SUM(PI.quantity)...` | 2-3 Table Joins | Group By + Aggregation | **Pre-Aggregated Rollup** (`prod_inventory_rollup`) |
| **Territory Sales** | `SELECT T.name, SUM(SO.Subtotal)...` | 3-4 Table Joins | Aggregation over Territory | **Pre-Aggregated Rollup** (`sales_territory_agg`) |

**Key Observations:**
1.  **Join Hotspots:** `SalesOrderHeader` is the central hub. Queries joining `SalesOrderHeader` -> `SalesOrderDetail` -> `Product` -> `ProductCategory` -> `ProductModel` are the most expensive.
2.  **Aggregation Hotspots:** Monthly sales summaries and current inventory levels are requested frequently. These require `GROUP BY` and `SUM()`, which is slow on normalized data.
3.  **Lookup Latency:** Simple "get employee by ID" or "get product by ID" queries become slow when the application needs to fetch related attributes (e.g., department name, category name) in a single request.

## 2. Denormalization Patterns & Justification

### Pattern A: Wide Read Tables (Snapshot of Current State)
*   **Justification:** Eliminates join overhead for read-only or read-mostly queries. Ideal for dashboards, web front-ends, and reporting tools where consistency is less critical than speed (acceptable 5-min lag).
*   **Target:** Employee Directory, Product Catalog, Sales Territory.

### Pattern B: Pre-Aggregated Rollups
*   **Justification:** Moves computation from the query engine to the ETL/Refresh process. Reduces CPU usage and memory footprint for aggregation queries.
*   **Target:** Monthly Sales, Current Inventory, Year-to-Date Territory Performance.

### Pattern C: Materialized Views (Logical)
*   **Justification:** While physical materialized views exist in some engines, here we use **Snapshot Tables** to enforce a specific "Current State" definition that is easier to maintain and query than a dynamic view.

## 4. Consistency Architecture & Data Lineage

| Source-of-Truth Column | Denormalized Column | Lineage Logic | Staleness Window |
| :--- | :--- | :--- | :--- |
| `Employee.BusinessEntityID` | `hr_employee_current.businessentityid` | Direct PK match | 0s (Immediate) |
| `Employee.DepartmentID` | `hr_employee_current.department_name` | Join `Employee` -> `Department` (Latest) | **5 mins** |
| `Employee.PayHistory` | `hr_employee_current.salary` | Max() by `ratechangedate` | **5 mins** |
| `Product.ProductID` | `prod_catalog_view.category_name` | Join `Product` -> `ProductCategory` (Latest) | **5 mins** |
| `SalesOrderHeader.SalesOrderID` | `sales_territory_agg_monthly.sales_ytd` | Sum(`Subtotal`) per Month | **1 hour** (Batch refresh) |
| `ProductInventory.Quantity` | `prod_inventory_rollup.total_quantity` | Sum(`Quantity`) per Product | **15 mins** |

**Correctness Handling:**
*   **No Triggers:** No triggers are used on the denormalized tables to avoid write amplification.
*   **Explicit Staleness:** The system operates on the assumption that denormalized data is "eventually consistent" with a max lag of 5 minutes. Business logic must be aware of this if strict real-time accuracy is required (e.g., financial settlement).
*   **Reconciliation:** A nightly job (`reconcile_job`) compares the denormalized tables against the source using `EXPLAIN` diffs or checksums to alert on significant drift.

## Risk Analysis

| Risk | Impact | Mitigation |
| :--- | :--- | :--- |
| **Write Amplification** | Writing to source + denormalized tables doubles I/O for updates. | **Mitigation:** Only denormalize read-heavy tables. Do not denormalize `Password` or `PayHistory` (frequent writes). Use `INSERT ... SELECT` which is optimized, not row-by-row inserts. |
| **Data Drift** | Denormalized data diverges from source due to updates. | **Mitigation:** Accept 5-min staleness. Implement nightly reconciliation alerts. Document staleness in API responses. |
| **Storage Growth** | Wide tables store duplicate data. | **Mitigation:** Calculate storage delta. For Adventure Works, the gain in query speed far outweighs the marginal storage cost (GBs vs GBs). |
| **Lock Contention** | Backfill locks source tables during the job. | **Mitigation:** Use parallel workers and non-blocking reads. Run backfill during off-peak hours. |
| **Application Errors** | Code expects fresh data but gets stale data. | **Mitigation:** Add `modifieddate` checks in application logic. Alert developers if they query denormalized tables expecting real-time data. |

This design provides a measurable performance uplift (estimated 50-80% reduction in query latency for top reads) while maintaining operational safety through incremental loading and explicit staleness documentation.
