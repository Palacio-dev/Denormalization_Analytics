This document outlines a denormalized read-optimized schema designed to improve query performance over the provided normalized educational data model. Based on inferred common analytical workloads—such as aggregating education metrics by region or institution—we propose materializing frequently joined paths into wide-read tables, snapshotting key entities for point-in-time lookups, and pre-aggregating multi-dimensional metrics. All transformations preserve correctness through traceable column lineage and controlled staleness intervals. Changes are fully reversible via logical backups and incremental refresh strategies using scheduled ETL pipelines. This approach avoids triggers and instead relies on batch-refresh mechanisms during off-peak hours.

## 🧱 **Proposed Denormalized Schema (DDL)**

### 1️⃣ Wide Read Table: Municipality with State Info

```sql
CREATE TABLE municipio_denorm (
    id_municipio INT PRIMARY KEY,
    nome_municipio VARCHAR(100),
    id_uf INT,
    sigla_uf VARCHAR(2),
    nome_uf VARCHAR(100),
    qtd_habitantes INT,
    qtd_IES INT,
    qtd_ESCOLAS INT,
    media_ideb REAL,
    media_saeb REAL,
    media_enade REAL,
    total_participantes INT
);
```

#### Justification:
- Supports Q1 directly without joins.
- Reduces fan-out when querying multiple municipalities within a state.
- Can be refreshed nightly based on updates to either table.

## 🔁 **Consistency Architecture**

Each denormalized table includes metadata columns such as `updated_at`. Lineage is maintained explicitly through:

| Denormalized Column | Source Tables | Update Logic |
|---------------------|---------------|--------------|
| `sigla_uf`, `nome_uf` | `UF` | Copied once; updated only on change |
| `avg_nota_ideb` | `IDEB`, `ANO`, `MUNICIPIO` | Recomputed per batch job |
| `tipo_inst` | Discriminator field | Static value set during load |

ETL jobs must enforce atomicity and idempotency. Example:

```sql
BEGIN;
DELETE FROM municipio_denorm WHERE id_municipio IN (...);
INSERT INTO municipio_denorm SELECT ... JOIN UF ON ...
COMMIT;
```

## ⚠️ **Risk Analysis**

| Risk Category | Description | Mitigation |
|---------------|-------------|------------|
| **Write Amplification** | Multiple tables may require duplicate writes | Use ETL pipeline with single-source truth logic |
| **Lock Contention** | Full rebuild could block reads | Implement incremental update mode or swap-table pattern |
| **Backfill Runtime** | Large initial load might take minutes | Schedule outside peak usage times |
| **Storage Growth** | Duplication increases disk footprint (~2x baseline) | Monitor growth rate, compress infrequently changing cols |
| **Failure Modes** | ETL job failure leads to stale data | Alert on last refresh timestamp drift > threshold |
| **Query Regression** | New structure may cause planner confusion | Benchmark before/after execution plans |

### Monitoring Plan

| Metric | Tool | Threshold |
|--------|------|-----------|
| Staleness lag | Custom SQL check | >1 day behind |
| Data reconciliation diff | Row checksum comparison | >0.1% mismatch |
| Query latency regression | APM tool (e.g., pg_stat_statements) | >10% slower than baseline |
| Refresh duration anomaly | Job logging | > historical mean + 2σ |
