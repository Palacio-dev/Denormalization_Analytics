# Executive Summary

This design proposes a **Read-Optimized Denormalized Schema** for the Adventure Works Cycles dataset, specifically targeting the high-frequency reporting workload common in SQL Server Reporting Services (SSRS) and BI tools. The primary bottleneck in the normalized schema is the cost of joining multiple small dimension tables (Department, Shift, Product, Territory) with large fact-like tables (Employee, Product, SalesOrder) during complex aggregations.

The proposed solution introduces **Wide-Read Dimension Tables** and **Pre-Aggregated Fact Tables**. Instead of joining 5+ tables for a standard "Employee Performance" report, the system will query a single denormalized `EmployeeCurrentState` table. Similarly, "Sales Performance by Territory" queries will utilize a pre-aggregated `SalesTerritoryRollup` table.

**Correctness Strategy:** The normalized tables remain the **Source of Truth (SoT)**. The denormalized tables are **eventually consistent**. A "stale window" of approximately **5 minutes** is acceptable for read performance, during which a report might show a department name that hasn't been updated yet. This is explicitly documented for stakeholders. The implementation plan utilizes **CDC (Change Data Capture)** to detect changes and a background worker to refresh the denormalized tables, ensuring operational safety and reversibility.

## 1. Model-to-Workload Mapping

Based on typical Adventure Works queries (SSRS reports, ad-hoc analytics), the access patterns are:

| Query Class | Typical Join Path (Normalized) | Join Hotspots | Aggregation Hotspots | Latency Impact |
| :--- | :--- | :--- | :--- :--- |
| **Employee Lookup** | `Employee` + `Department` + `Shift` + `Person` | `DepartmentID` (FK), `ShiftID` (FK) | `COUNT(*)`, `AVG(Salary)` | **High**: 3-4 table joins, small tables but high cardinality joins |
| **Sales Analysis** | `SalesOrderHeader` + `SalesOrderDetail` + `Product` + `Territory` + `SalesPerson` | `TerritoryID`, `SalesPersonID` | `SUM(TotalDue)`, `AVG(ListPrice)` | **Critical**: Large fact table joins, multiple small dimensions |
| **Inventory Check** | `ProductInventory` + `Location` + `Product` | `LocationID`, `ProductID` | `SUM(Quantity)`, `MAX(Quantity)` | **Medium**: High row count in inventory, joins on location hierarchy |
| **Product Cost** | `Product` + `ProductCategory` + `UnitMeasure` | `ProductID` | `AVG(StandardCost)` | **Medium**: Joins on complex product hierarchy |

**Key Insight:** The most frequent queries involve joining a **Fact table** (Employee, SalesOrder, Product) with **3-5 small dimension tables**. The cost is dominated by the join operations and the lack of indexes on the foreign keys in the small dimension tables.

## 2. Denormalization Patterns & Justification

We will implement three specific patterns:

### A. Wide-Read Employee Dimension (`EmployeeCurrentState`)
*   **Pattern:** Denormalized Lookup Table.
*   **Justification:** Queries often ask for "Employee Name, Department Name, Shift Name, Hire Date". Joining `Employee`, `Department`, `Shift`, and `Person` is slow. A single table containing all current attributes allows for a single-table scan.
*   **Scope:** Only includes employees where `currentflag = 1`.

### B. Pre-Aggregated Sales Rollup (`SalesTerritoryRollup`)
*   **Pattern:** Pre-aggregated Fact Table.
*   **Justification:** "What are the sales by Territory?" requires joining `SalesOrderHeader`, `SalesOrderDetail`, `Product`, `Territory`, `CurrencyRate`, and `ShipMethod`. Calculating `SUM` and `AVG` on the fly is expensive. We pre-calculate this.
*   **Scope:** Aggregated by `TerritoryID` and `Date` (Daily/Monthly).

### C. Product Inventory Snapshot (`ProductInventoryCurrent`)
*   **Pattern:** Snapshot Table.
*   **Justification:** "How much inventory do we have per location?" requires joining `ProductInventory` with `Location` and `Product`. The `Location` table has a hierarchy; joining it is complex.
*   **Scope:** Current quantity per location per product.

## 4. Consistency Architecture & Data Lineage

To ensure correctness while allowing denormalization, we define a strict lineage:

1.  **Source of Truth (SoT):** The original normalized tables (`humanresources.department`, `sales.salesorderheader`, etc.).
2.  **Change Detection:**
    *   **Option A (CDC):** Use `pg_cron` or a custom listener to monitor `updatedat` columns in the SoT. If `modifieddate` in `department` changes, trigger a refresh of `employeecurrentstate`.
    *   **Option B (Triggerless Batch):** Run a nightly job (`0 0 * * *`) that runs `MERGE` statements.
3.  **The "Stale Window" Contract:**
    *   **Employee Table:** Max 5-minute staleness. If a user queries `employeecurrentstate` at 10:00:01, they might see a department name updated at 10:00:00 that hasn't propagated to the main table yet (or vice versa). This is acceptable for reporting.
    *   **Sales Table:** Max 24-hour staleness. Reports run overnight; data is refreshed daily at 2 AM.
    *   **Inventory:** Near real-time (seconds).

### Lineage Mapping Table
| Denormalized Column | Source Table | Source Column | Staleness Limit |
| :--- | :--- | :--- | :--- |
| `departmentname` | `humanresources.department` | `name` | 5 mins |
| `shiftname` | `humanresources.shift` | `name` | 5 mins |
| `totalrevenue` | `sales.salesorderdetail` | `unitprice * orderqty` | 24 hrs |
| `quantity` | `production.productinventory` | `quantity` | < 1 min |

## 5. Risk Analysis & Operational Plan

### Risks
1.  **Write Amplification:**
    *   *Risk:* Writing to the denormalized table on every write could cause lock contention.
    *   *Mitigation:* **Do not write directly** to the denormalized tables in the application. Use a background worker (Python/Node.js/PostgreSQL Trigger) to sync changes asynchronously.
2.  **Storage Growth:**
    *   *Risk:* `employeecurrentstate` duplicates data.
    *   *Mitigation:* Accept ~30% storage increase for HR data (small table, low volume). Sales rollup is small due to aggregation.
3.  **Reconciliation Failure:**
    *   *Risk:* Denormalized data drifts from SoT.
    *   *Mitigation:* Implement a nightly reconciliation job that compares a hash of the denormalized table against the SoT. If diffs > 0, alert the DBA.
