### Executive Summary

As a Principal Database Architect, I have reviewed your normalized schema and your proposed denormalized draft. Your intuition to use **Collapsing Relations** and **Adding Redundant Attributes** is spot on for an analytical, read-heavy workload. However, the draft model missed a critical dimensional grain: **Time (`year`)**. Socio-economic indicators (demography, economy, etc.) are recorded annually. Without a `year` column in the indicators table, you cannot store historical trends, breaking business correctness. Furthermore, surveys require a unique identifier because a single country can conduct multiple surveys in the same year (e.g., different welfare types or coverages). 

The refined physical schema below provides two wide, read-optimized tables: `country_indicators_yearly` and `poverty_surveys_denorm`. This design eliminates up to 8-way joins, pivots child rows (deciles, population demographics) into columns to prevent row-duplication, and uses a nightly batch-refresh strategy. Because socio-economic data changes infrequently, we will accept a 24-hour staleness window, optimizing entirely for read latency and analytical aggregations.

### 2. Denormalization Patterns Justification

1.  **Wide Read Tables (Pre-joined Dimensions)**: 
    *   *Action*: We embed `region_name` and `country_name` directly into the fact tables. 
    *   *Justification*: Eliminates the `country` and `region` joins. Country names and regions change very rarely; the write-amplification risk is negligible.
2.  **Pre-aggregated Rollups (Pivoting)**:
    *   *Action*: Pivot `decile` (10 rows per survey) into 10 columns (`decile_1` to `decile_10`). Pivot `population` and `life_expectancy` by gender/age.
    *   *Justification*: Analytical tools (like Tableau/PowerBI) and data scientists prefer features as columns for correlation queries (Q2, Q3). It guarantees a 1:1 relationship between a country-year and its metrics.
3.  **Engine Equivalent - Materialized Snapshot Tables**:
    *   *Action*: Implement as standard tables refreshed via a batch "Swap" (Blue/Green deployment) or `MATERIALIZED VIEW`.
    *   *Justification*: CDC (Change Data Capture) is overkill here. This data is likely loaded in batches. A full overwrite ensures absolute consistency and eliminates fragmentation.

### 4. Consistency Architecture

**Data Lineage & Transformation**:
*   `country_indicators_yearly.life_expectancy_female` $\leftarrow$ `SELECT value FROM life_expectancy WHERE gender = 'female'` joined on `id_demography`.
*   `poverty_surveys_denorm.decile_1` $\leftarrow$ `SELECT value FROM decile WHERE name = 'decile1'` joined on `id_survey`.

**Refresh Strategy (Safe, Observable, Reversible)**:
Because no triggers are allowed and we want to avoid locking read queries, we will use a **Shadow Table Swap** (or Materialized View concurrent refresh) via a scheduled batch window (e.g., Airflow/Cron at 02:00 UTC).

1.  **Build**: `CREATE TABLE country_indicators_yearly_staging AS SELECT ...` (contains the complex JOINs and `FILTER/CASE` pivots).
2.  **Validate**: Run automated data quality checks on the staging table (e.g., `SELECT COUNT(*) FROM staging WHERE gdp IS NULL`).
3.  **Swap**: 
    ```sql
    BEGIN;
    ALTER TABLE country_indicators_yearly RENAME TO country_indicators_yearly_old;
    ALTER TABLE country_indicators_yearly_staging RENAME TO country_indicators_yearly;
    COMMIT;
    DROP TABLE country_indicators_yearly_old;
    ```
*Staleness Window*: Explicitly documented as up to 24 hours. Business correctness is preserved because analytical reports will always query a consistent, point-in-time snapshot of the database.

### 5. Risk Analysis & Observability

| Risk Vector | Assessment & Mitigation |
| :--- | :--- |
| **Null Sparsity** | *Risk*: Not all countries report all metrics every year, leading to wide tables with many `NULL`s.<br>*Mitigation*: Modern columnar formats or PostgreSQL's NULL bitmap handles this efficiently. No storage bloat expected. |
| **Write Amplification** | *Risk*: A change to a `region_name` requires updating thousands of rows.<br>*Mitigation*: Handled seamlessly by the daily full-refresh batch strategy. We do not do point-updates (`UPDATE`). |
| **Failure Modes** | *Risk*: The complex pivot query fails, leaving the read tables empty.<br>*Mitigation*: The "Shadow Table Swap" ensures the live table is only replaced if the staging table builds and passes validation. |
| **Monitoring/Alerts** | 1. **Freshness**: Alert if `MAX(created_at)` of the batch job is > 26 hours.<br>2. **Reconciliation**: Alert if `COUNT(*)` in `country_indicators_yearly` differs from `SELECT COUNT(*) FROM demography` by > 1%.<br>3. **Latency**: Track P90 execution time of Q1-Q4. |
