As a senior database performance engineer, looking at your normalized schema, I see a classic OLTP (Online Transaction Processing) design. While excellent for write-heavy data integrity, it will suffer from severe read latency in analytical or dashboard-heavy workloads due to the high number of joins required to assemble a single logical entity.

Your draft denormalized model is a great start. I will elevate it to a production-ready analytical schema (OLAP/Read-Heavy) by introducing **partitioning, generated columns, covering indexes, and materialized views**.

Here is the comprehensive workload analysis, design, and the final DDL.

---

### 1. Workload-Driven Analysis

In a system tracking environmental cleanups and conservation units, the read workload typically falls into these high-cost patterns:

*   **Query Pattern A (Event Dashboards):** "Show me the details of recent cleanups, including the location, organizer, and amount of trash collected."
    *   *Bottleneck:* Requires joining 5 tables (`mutirao_de_limpeza` → `coleta_de_lixo` → `local_de_coleta` → `instituicao_organizadora` → `regiao`). In Postgres, this leads to multiple Hash Joins or Nested Loops, thrashing the buffer cache and increasing CPU time.
*   **Query Pattern B (Geographic Aggregation):** "How much plastic and glass was collected per state/municipality in the last year?"
    *   *Bottleneck:* Requires scanning the fact table, joining to regions, and aggregating on the fly. Doing this repeatedly on millions of rows causes high I/O latency.
*   **Query Pattern C (Conservation Unit Lookup):** "List all marine conservation units in a specific state, including their biome and management plan."
    *   *Bottleneck:* Requires joining 5 tables (`unidade_de_conservacao` → `bioma` → `categoria_manejo` → `esfera_administrativa` → `regiao`).

**Denormalization Candidates:**
1.  `coleta_de_lixo` and `local_de_coleta` share a strict 1:1 relationship with `mutirao_de_limpeza`. They should be **collapsed immediately**.
2.  Lookup tables (`regiao`, `instituicao_organizadora`, `bioma`, `categoria_manejo`, `esfera_administrativa`) change rarely. Their descriptive attributes should be **embedded (duplicated)** into the main tables.

---

### 2. Denormalization Design

To achieve sub-millisecond to low-millisecond read latency, we will implement a **Star-Schema inspired Wide-Table design** with the following specific techniques:

1.  **Pre-joined "Wide" Tables (Fact & Dimension):**
    *   `fact_mutirao_limpeza`: Collapses 5 tables into one. We will retain the original `id_*` columns alongside the text names. This preserves **correctness**: if a region's name changes, you can still join to a master dimension table using the ID if strict normalization is temporarily needed, while serving 99% of reads from the text columns.
    *   `dim_unidade_conservacao`: Collapses 5 tables into one wide dimension.
2.  **Generated Aggregate Columns:**
    *   Queries often need the *total* trash collected. I will add a `total_residuos` column using Postgres' `GENERATED ALWAYS AS (...) STORED` feature. This precomputes the row-level sum at write-time, saving CPU cycles on read.
3.  **Precomputed Aggregates (Materialized View):**
    *   To solve Query Pattern B, I will create a `mv_residuos_por_estado_mes` materialized view. This explicitly pre-calculates trash by state and month (the grain).
4.  **Partitioning:**
    *   `fact_mutirao_limpeza` will be **partitioned by range** on `data_do_ocorrido`. This allows the database to completely skip scanning years/months of data that aren't relevant to a time-filtered query (Partition Pruning).
5.  **Covering Indexes:**
    *   We will use composite indexes aligned to the `WHERE` and `ORDER BY` clauses, utilizing the `INCLUDE` keyword to cover the select list, enabling **Index-Only Scans**.

### 3. New Physical Schema (PostgreSQL DDL)

```sql
BEGIN;

-- ==============================================================================
-- 1. WIDE FACT TABLE: fact_mutirao_limpeza (Partitioned by Date)
-- ==============================================================================
-- Collapses: mutirao_de_limpeza, coleta_de_lixo, local_de_coleta, instituicao_organizadora, regiao
CREATE TABLE IF NOT EXISTS public.fact_mutirao_limpeza
(
    id_mutirao integer NOT NULL,
    data_do_ocorrido date NOT NULL,
    horario_de_inicio time without time zone NOT NULL,
    duracao_total time without time zone NOT NULL,
    numero_aproximado_de_participantes integer,
    
    -- Denormalized Location (Regiao & Local)
    id_municipio integer,
    nome_municipio text COLLATE pg_catalog."default",
    estado text COLLATE pg_catalog."default",
    nome_local_de_coleta text COLLATE pg_catalog."default",
    
    -- Denormalized Organizer (Instituicao)
    id_instituicao_organizadora integer,
    nome_instituicao text COLLATE pg_catalog."default",
    carater_instituicao text COLLATE pg_catalog."default",
    
    -- Collapsed 1:1 Trash Collection (Coleta de Lixo)
    plastico integer DEFAULT 0,
    borracha integer DEFAULT 0,
    madeira integer DEFAULT 0,
    produtos_texteis integer DEFAULT 0,
    papel integer DEFAULT 0,
    metal integer DEFAULT 0,
    vidro integer DEFAULT 0,
    materiais_de_pesca integer DEFAULT 0,
    outros_residuos integer DEFAULT 0,
    
    -- Precomputed Row-Level Aggregate
    total_residuos integer GENERATED ALWAYS AS (
        COALESCE(plastico, 0) + COALESCE(borracha, 0) + COALESCE(madeira, 0) + 
        COALESCE(produtos_texteis, 0) + COALESCE(papel, 0) + COALESCE(metal, 0) + 
        COALESCE(vidro, 0) + COALESCE(materiais_de_pesca, 0) + COALESCE(outros_residuos, 0)
    ) STORED,

    -- Partitioned tables require the partition key in the PK
    CONSTRAINT fact_mutirao_limpeza_pkey PRIMARY KEY (id_mutirao, data_do_ocorrido)
) PARTITION BY RANGE (data_do_ocorrido);

-- Example Partitions (You would automate this creation in production)
CREATE TABLE fact_mutirao_limpeza_2023 PARTITION OF public.fact_mutirao_limpeza FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');
CREATE TABLE fact_mutirao_limpeza_2024 PARTITION OF public.fact_mutirao_limpeza FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- ==============================================================================
-- 2. WIDE DIMENSION TABLE: dim_unidade_conservacao
-- ==============================================================================
-- Collapses: unidade_de_conservacao, bioma, categoria_manejo, esfera_administrativa, regiao
CREATE TABLE IF NOT EXISTS public.dim_unidade_conservacao
(
    id_uc integer NOT NULL,
    nome text COLLATE pg_catalog."default" NOT NULL,
    ano_de_criacao integer,
    area_soma_biomas double precision,
    area_soma_biomas_continentais double precision,
    area_marinha double precision,
    
    -- Denormalized Bioma
    id_bioma integer,
    nome_bioma text COLLATE pg_catalog."default",
    
    -- Denormalized Regiao
    id_municipio integer,
    nome_municipio text COLLATE pg_catalog."default",
    estado text COLLATE pg_catalog."default",
    
    -- Denormalized Categoria Manejo
    id_manejo integer,
    categoria_manejo text COLLATE pg_catalog."default",
    categoria_iucn text COLLATE pg_catalog."default",
    plano_de_manejo text COLLATE pg_catalog."default",
    
    -- Denormalized Esfera Administrativa
    id_esfera integer,
    esfera_administrativa text COLLATE pg_catalog."default",
    
    CONSTRAINT dim_unidade_conservacao_pkey PRIMARY KEY (id_uc)
);

-- ==============================================================================
-- 3. MATERIALIZED VIEW: Monthly Trash Aggregation by State
-- ==============================================================================
-- Grain: 1 row per State per Month. Solves heavy analytical dashboard queries.
CREATE MATERIALIZED VIEW public.mv_residuos_por_estado_mes AS
SELECT 
    estado,
    DATE_TRUNC('month', data_do_ocorrido) AS mes_ocorrencia,
    COUNT(id_mutirao) AS total_mutiroes,
    SUM(numero_aproximado_de_participantes) AS total_participantes,
    SUM(plastico) AS total_plastico,
    SUM(vidro) AS total_vidro,
    SUM(materiais_de_pesca) AS total_materiais_pesca,
    SUM(total_residuos) AS volume_total_residuos
FROM public.fact_mutirao_limpeza
GROUP BY estado, DATE_TRUNC('month', data_do_ocorrido);

-- Unique index required to allow CONCURRENTLY refreshes (zero downtime reads)
CREATE UNIQUE INDEX idx_mv_residuos_estado_mes ON public.mv_residuos_por_estado_mes (estado, mes_ocorrencia);

-- ==============================================================================
-- 4. INDEXING STRATEGY
-- ==============================================================================

-- Index 1: Optimizes "Show me cleanups in State X ordered by date, including total trash"
-- Uses INCLUDE to prevent table heap lookups (Index-Only Scan)
CREATE INDEX idx_fact_mutirao_estado_data 
    ON public.fact_mutirao_limpeza (estado, data_do_ocorrido DESC)
    INCLUDE (nome_municipio, total_residuos, numero_aproximado_de_participantes);

-- Index 2: Optimizes filtering by Organizer (e.g., "All events by NGO X")
CREATE INDEX idx_fact_mutirao_instituicao 
    ON public.fact_mutirao_limpeza (nome_instituicao);

-- Index 3: Optimizes searching Conservation Units by State and Biome
CREATE INDEX idx_dim_uc_estado_bioma 
    ON public.dim_unidade_conservacao (estado, nome_bioma)
    INCLUDE (nome, area_soma_biomas);

-- Index 4: Optimizes searching Conservation Units by Management Category
CREATE INDEX idx_dim_uc_categoria 
    ON public.dim_unidade_conservacao (categoria_manejo);

END;
```

### Why this specific design guarantees performance:
1. **Elimination of Nested Loops:** By collapsing 1:1 and 1:N lookups into the fact/dimension tables, queries that previously took `O(N * M)` time due to join overhead now execute as sequential or index scans `O(log N)`.
2. **Index-Only Scans:** The composite index `idx_fact_mutirao_estado_data` includes `total_residuos`. If a dashboard queries the total trash collected in "São Paulo" over the last 30 days, Postgres reads *only the B-Tree index* and never touches the actual table disk blocks.
3. **Partition Pruning:** If a user queries data for `2024`, the database engine completely ignores the `2023` physical file. This halves the I/O instantly for time-bound queries.
4. **Pre-aggregation:** The Materialized View shifts the CPU cost of `SUM()` and `GROUP BY` from *Read-Time* to *Write-Time* (when you refresh the view). Dashboard load times for these metrics will drop from seconds to sub-10 milliseconds.
