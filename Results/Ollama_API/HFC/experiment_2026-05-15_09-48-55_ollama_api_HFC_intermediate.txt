# Denormalized Physical Schema for Herbarium Database

## 1. Workload-driven Analysis

### High-Cost Join Paths and Operations:

1. **Tombo Details Queries**:
   - Joining `tombos` with multiple taxonomy tables (`especies`, `generos`, `familias`, `sub_especies`, `variedades`)
   - Joining with location data (`locais_coleta`, `enderecos`, `cidades`, `estados`, `paises`)
   - Joining with collectors (`coletores`, `coletores_complementares`)
   - Joining with identifiers (`identificadores`, `tombos_identificadores`)
   - Joining with classification data (`tipos`, `solos`, `relevos`, `vegetacoes`, `fase_sucessional`)

2. **Herbarium Information Queries**:
   - Joining `herbarios` with `enderecos`, `cidades`, `estados`, `paises`
   - Joining with `telefones` and `usuarios`

3. **Collection Tracking Queries**:
   - Joining `tombos` with `remessas` through `retirada_exsiccata_tombos`

### Denormalization Candidates:

1. **Tombo Wide Table** - Pre-join all frequently accessed tombo details
2. **Herbarium Summary Table** - Consolidate herbarium contact information
3. **Tombo Identifiers View** - Aggregate identifiers into a single field
4. **Tombo Photos Summary** - Include photo count in tombo records
5. **Location Hierarchy Table** - Pre-join geographic hierarchy
6. **Taxonomy Hierarchy Table** - Pre-join taxonomic hierarchy

## 2. Denormalization Design

### Proposed Denormalizations:

1. **tombo_details** - Wide table containing all tombo information with pre-joined fields
2. **herbarium_summary** - Consolidated herbarium information
3. **location_hierarchy** - Pre-joined geographic information
4. **taxonomy_hierarchy** - Pre-joined taxonomic information
5. **tombo_identifiers_agg** - Aggregated identifiers per tombo
6. **tombo_photo_summary** - Photo counts per tombo

### Index Strategy:
- Composite indexes on frequently filtered fields
- Covering indexes for common query patterns
- Partitioning by date ranges for historical data

## Summary of Optimizations

### Denormalization Strategy:

1. **Wide Table (`tombo_details`)**:
   - Pre-joined all taxonomy, location, and classification information
   - Eliminates 15+ joins in typical tombo queries
   - Includes aggregated identifiers and photo counts

2. **Summary Tables**:
   - `herbarium_summary` consolidates herbarium information
   - `tombo_statistics` materialized view for aggregated reporting

3. **Index Optimization**:
   - Composite indexes aligned with common query filters
   - Covering indexes for frequently accessed fields
   - Specialized indexes for geographic and taxonomic searches

4. **Views for Special Cases**:
   - `tombo_with_photos` for photo-related queries
   - `tombo_with_history` for alteration tracking

### Performance Improvements:

1. **Query Latency**: 
   - Reduced joins from 15+ to 0 for typical tombo detail queries
   - Eliminated need to join lookup tables for names

2. **Read Scalability**:
   - Materialized views for statistical queries
   - Covering indexes reduce I/O for common queries

3. **Data Integrity**:
   - Maintained referential integrity through careful design
   - Used materialized views where real-time accuracy isn't critical

### Implementation Notes:

1. **Maintenance Procedures**:
   - Need procedures to refresh denormalized data when base tables change
   - Regular refresh of materialized views (e.g., nightly)

2. **Storage Considerations**:
   - Increased storage due to denormalization
   - Trade-off accepted for significantly improved read performance

3. **Update Strategy**:
   - Base tables remain as source of truth
   - Denormalized tables updated via triggers or batch processes

This schema design reduces typical query complexity from multiple joins to single table access, dramatically improving read performance for the most common query patterns while preserving data correctness.
