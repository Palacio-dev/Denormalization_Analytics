BEGGINER:
You are a database design assistant. I have a normalized relational database model used for transactions, and I want to create a denormalized physical schema optimized for analytics and read performance.

Your task:

Explain, in simple terms, how the current normalized model should be transformed for analytical workloads.
Identify which tables and relationships should be denormalized.
Propose a new physical schema that supports fast reporting and aggregation.
Include the main design choices, such as fact-like tables, summary tables, duplicated attributes, and useful indexing suggestions.
Keep the design practical for a user who is new to database modeling.
Output format:

Short explanation of the strategy
List of changes from the normalized model
Proposed denormalized physical schema
Example queries the new schema should optimize
Any risks or trade-offs, such as storage growth or update complexity
Assume the goal is to move from a base transactional database to an analytics-oriented version.

INTERMEDIATE:
You are a database architect. I have a normalized relational database schema built for OLTP, and I need a denormalized
physical schema optimized for OLAP and analytics.

Please do the following:

- Analyze the normalized model and identify the main analytical access patterns.
- Recommend a denormalization strategy that improves read performance for reporting.
- Design a new physical schema for the analytics version of the database.
- Include table structures, key columns, duplicated dimensions, summary tables, and indexing or partitioning recommendations.
- Explain how the design supports faster aggregations and fewer joins.
- Note the trade-offs between query performance, storage, and data maintenance.

Output format:

- Assumptions
- Denormalization strategy
- Proposed analytics physical schema
- Table-by-table explanation
- Performance impact
- Trade-offs and risks
- Optional migration considerations from the base schema to the analytics schema

Focus on producing a schema suitable for analytical workloads rather than transactional updates.

EXPERT:
You are a senior data architect specializing in analytical database design. Given a normalized relational physical model
for a transactional system, design a denormalized physical schema for analytics and read optimization.

Requirements:

- Infer the business domain and analytical workloads from the base model.
- Transform the schema into an analytics-oriented physical design.
- Minimize join depth and optimize for reporting, aggregation, and dashboard queries.
- Propose denormalized fact-like and dimension-like structures, summary tables, materialized views if appropriate, and 
  physical optimization strategies such as partitioning, clustering, indexing, and compression.
- Preserve analytical correctness while explicitly addressing duplication, slowly changing attributes, and refresh strategy if needed.
- Provide the resulting physical schema with clear table definitions and relationship changes.
- Explain why each denormalization decision was made and what query patterns it improves.
- Include a concise assessment of ETL/ELT implications and data freshness trade-offs.

Output format:

- Domain assumptions
- Analytical access patterns
- Denormalization decisions
- New physical analytics schema
- Physical design optimizations
- Query/performance implications
- Maintenance and freshness strategy
- Risks and trade-offs
