Here is a set of prompts designed for an LLM to perform database denormalization, categorized by the user's level of database expertise. 

To use these prompts, the user simply needs to copy them, fill in the bracketed information `[like this]`, and paste them into the LLM.

---

### 🟢 Beginner Level
**Profile:** The user understands basic tables and relationships but isn't deeply familiar with database architecture, normal forms, or the exact mechanics of denormalization. They need the LLM to act as a teacher and guide.

**Prompt:**
> "I have a relational database for an application that does `[describe what your app does, e.g., an e-commerce store]`. Right now, my database is organized into many separate tables, which makes reading the data very slow. 
> 
> I want to 'denormalize' this database—meaning I want to combine some tables and duplicate some data so that my app can read information much faster without having to connect so many tables together.
> 
> Here is my current database schema:
> `[Insert current table schemas or a simple list of tables and columns]`
> 
> Here is the main information I need to read quickly:
> `[Describe what you need to display, e.g., A customer's profile along with their last 5 orders and the total amount spent]`
> 
> Please do the following:
> 1. Explain in simple terms which tables we should combine or what data we should duplicate to make this faster.
> 2. Explain the downside of doing this (e.g., what happens when data changes?).
> 3. Provide the new SQL statements (CREATE TABLE) to build this new, faster database setup."

---

### 🟡 Intermediate Level
**Profile:** The user understands Normalization (1NF, 2NF, 3NF), JOINs, and indexes. They know *why* they need to denormalize (heavy read traffic, expensive JOINs) but want the LLM to handle the architectural strategy and generate the DDL.

**Prompt:**
> "I have a highly normalized relational database schema (roughly 3NF) that is experiencing read-performance bottlenecks due to complex and expensive JOINs. I need to denormalize the physical model to optimize it for read-heavy operations.
> 
> Here is my current physical schema (DDL):
> `[Insert DDL here]`
> 
> These are the specific read queries/operations that are currently too slow:
> `[Insert slow SQL queries, or describe the read access patterns]`
> 
> Our read-to-write ratio is roughly `[e.g., 90/10]`. 
> 
> Please provide:
> 1. A strategy for denormalization (e.g., pre-calculating totals, flattening 1-to-many relationships, adding redundant columns).
> 2. The new physical schema (SQL DDL) reflecting the denormalized design.
> 3. A brief analysis of the trade-offs I will face regarding data anomalies during INSERT/UPDATE/DELETE operations, and how to mitigate them."

---

### 🔴 Advanced / Expert Level
**Profile:** Data architects, senior backend engineers, or DBAs. They understand the nuances of specific RDBMS engines (PostgreSQL, MySQL, SQL Server) and want advanced techniques like Materialized Views, JSONB document embedding, or automated trigger syncs. 

**Prompt:**
> "Act as a Senior Database Architect. I need to refactor a highly normalized OLTP schema into a read-optimized, denormalized physical model. The target RDBMS is `[Specify Engine, e.g., PostgreSQL 15 / MySQL 8]`. 
> 
> The current normalized schema DDL is here:
> `[Insert DDL]`
> 
> The core performance bottlenecks are caused by these query patterns:
> `[Insert Query EXPLAIN plans, or exact SQL queries]`
> 
> Generate a comprehensive denormalization plan and physical schema implementation. I am open to techniques including column redundancy, pre-aggregation tables, collapsing hierarchies, or utilizing NoSQL-style features (like JSONB/JSON arrays) if supported by the specified engine.
> 
> Your output must include:
> 1. **The New DDL:** The exact CREATE TABLE scripts for the denormalized physical schema, including optimized data types.
> 2. **Indexing Strategy:** Recommended indexes (B-Tree, GIN, Hash, etc.) for the new schema based on the query patterns.
> 3. **Consistency Mechanism:** Since we are introducing redundancy, provide the SQL for the mechanism to keep data consistent (e.g., Database Triggers to update duplicated columns, or Materialized View refresh strategies).
> 4. **Migration Strategy:** A rough SQL script or plan to migrate the data from the old normalized tables to the new denormalized structures."

---

### 💡 Pro-Tips for the User to get the best results:
*   **Include Constraints:** If storage space is an issue, explicitly state: *"Do not use denormalization strategies that increase storage by more than X%."*
*   **Provide Data Volume Context:** Tell the LLM how many rows are in the tables. Denormalizing a table with 10,000 rows requires a very different strategy than one with 10 billion rows.
*   **Specify the DB Engine:** Always mention if you are using PostgreSQL, MySQL, Oracle, etc., as physical modeling features (like specific data types or materialized views) vary wildly between engines.
