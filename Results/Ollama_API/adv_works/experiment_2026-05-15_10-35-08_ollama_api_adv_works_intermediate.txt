## Workload-Driven Analysis

### High-Cost Join Paths:
1. **Employee Department History**: Employee → EmployeeDepartmentHistory → Department (frequent lookups for current department)
2. **Employee Pay History**: Employee → EmployeePayHistory (salary lookups)
3. **Sales Order Details**: SalesOrderHeader → SalesOrderDetail → Product (order reporting)
4. **Customer Information**: SalesOrderHeader → Customer → Person (customer analytics)
5. **Product Hierarchy**: Product → ProductSubcategory → ProductCategory (product reporting)

### Denormalization Candidates:
- Current department and salary in Employee table
- Wide Sales Order view with customer, product, and address details
- Product hierarchy flattened
- Employee-person information combined

-- Materialized view for current employee information
CREATE MATERIALIZED VIEW humanresources.current_employee_view AS
SELECT 
    e.businessentityid,
    e.jobtitle,
    e.currentflag,
    e.hiredate,
    edh.departmentid,
    d.name as department_name,
    d.groupname as department_group,
    eph.rate as current_salary,
    p.firstname,
    p.lastname,
    p.middlename,
    ea.emailaddress
FROM humanresources.employee e
JOIN humanresources.employeedepartmenthistory edh 
    ON e.businessentityid = edh.businessentityid 
    AND edh.enddate IS NULL
JOIN humanresources.department d ON edh.departmentid = d.departmentid
LEFT JOIN humanresources.employeepayhistory eph 
    ON e.businessentityid = eph.businessentityid 
    AND eph.ratechangedate = (
        SELECT MAX(ratechangedate) 
        FROM humanresources.employeepayhistory eph2 
        WHERE eph2.businessentityid = e.businessentityid
    )
JOIN person.person p ON e.businessentityid = p.businessentityid
LEFT JOIN person.emailaddress ea ON e.businessentityid = ea.businessentityid;

## Key Optimizations Implemented:

1. **Employee Denormalized Table**: Eliminates joins to Department, EmployeePayHistory, Person, EmailAddress, and PersonPhone tables for common employee queries.

2. **Product Denormalized Table**: Flattens the product hierarchy (Product → Subcategory → Category) to avoid multiple joins.

3. **Sales Order Denormalized Table**: Pre-joins SalesOrderHeader, SalesOrderDetail, Customer, Person/Store, Address, Product, and related lookup tables.

4. **Sales Summary Table**: Precomputed aggregates by month, territory, and product category for reporting.

5. **Materialized View**: For current employee information that needs to be refreshed periodically.

6. **Partitioning**: Yearly partitioning of sales data for better query performance on date-range queries.

7. **Covering Indexes**: Strategic indexes that include all needed columns to avoid table lookups.

This schema design reduces typical query join paths from 5-8 tables down to 1-2 tables, significantly improving read performance while maintaining data correctness through controlled denormalization.
