# Executive Summary

This design transforms your normalized relational schema into a **Read-Optimized, Denormalized Physical Schema** tailored for high-frequency reporting and ad-hoc analytics on Brazilian education metrics (IES, MUNICIPIO, ANO). The core strategy is the creation of a **`DADOS_IES_ANO`** wide-table and a **`AGREGACAO_IES_ANO`** materialized view, eliminating the need for multi-table joins on the `ANO` entity.

Given the constraints (no triggers allowed, likely OLAP/OLTP hybrid workload), we will rely on **engine-level materialized views** (e.g., Snowflake `CREATE MATERIALIZED VIEW`, BigQuery `CREATE MATERIALIZED VIEW`, or PostgreSQL `CREATE VIEW WITH CHECKPOINT`) to handle complex aggregations. We explicitly document a **5-minute to 1-hour staleness window** for these denormalized artifacts. The `UF` and `MUNICIPIO` tables remain normalized as the Source of Truth (SoT) for critical counts, while `IES`, `CURSO`, and `ANO` are heavily denormalized into the wide table to support queries like "Total IDEB by Municipality for 2023" without joining 5 tables.

# 2) Denormalization Patterns & Justification

We will implement three specific patterns:

1.  **Wide Read Table (`DADOS_IES_ANO`)**:
    *   **Pattern**: Pre-joins `IES`, `CURSO`, `SAEB`, `ENADE`, and `IDEB` into a single row per `(id_ies, id_ano)`.
    *   **Justification**: Eliminates the need to join 4 foreign keys (`id_curso`, `id_enade`, `id_saeb`, `id_ideb`) in every query. Ideal for point-in-time snapshots.
    *   **Staleness**: Low (near real-time if CDC is used, otherwise T+1 batch).

2.  **Materialized Aggregation View (`AGREGACAO_IES_ANO`)**:
    *   **Pattern**: Pre-aggregates `DADOS_IES_ANO` by `id_municipio` and `id_ano`.
    *   **Justification**: The `MUNICIPIO` table does not contain granular course/school data. Queries asking for "Average IDEB per Municipality in 2023" require an `INNER JOIN` between `MUNICIPIO` and the denormalized `IES` data, followed by a `GROUP BY`. This view materializes that specific aggregation.
    *   **Staleness**: Medium (Refreshed hourly/daily depending on batch size).

3.  **Snapshot Table (`ESTADO_ATUAL_IES`)**:
    *   **Pattern**: A static copy of the latest state of `IES` + `CURSO` + `SAEB` + `ENADE` + `IDEB` (without the `ANO` dimension, or with a fixed anchor year).
    *   **Justification**: Useful for "Current State" queries where time-series history is not needed, avoiding any join logic entirely.

### D. Specialized Aggregation for Participants (`AGREGACAO_PARTICIPANTES`)
*The `ANO` table holds `total_participantes`. Joining `MUNICIPIO` -> `IES` -> `CURSO` -> `ANO` is expensive. We pre-aggregate participants by Municipality and Year.*

```sql
CREATE MATERIALIZED VIEW IF NOT EXISTS AGREGACAO_PARTICIPANTES AS
SELECT 
    a.id_ano,
    m.id_municipio,
    m.nome_municipio,
    SUM(a.total_participantes) AS total_participantes_ano,
    -- Optional: Average per course if needed
    AVG(a.total_participantes) AS avg_participantes_por_curso
FROM ANO a
JOIN IES i ON a.id_ies = i.id_ies
JOIN MUNICIPIO m ON i.id_municipio = m.id_municipio
GROUP BY a.id_ano, m.id_municipio, m.nome_municipio;
```

# 4) Consistency Architecture & Data Lineage

| Artifact | Source of Truth (SoT) | Lineage Logic | Staleness Window |
| :--- | :--- | :--- | :--- |
| **UF, MUNICIPIO, IES, CURSO, SAEB, ENADE, IDEB, ANO** | Database Writes (Transactional) | None (Primary) | 0s (Immediate) |
| **DADOS_IES_ANO** | Joins: `IES` + `CURSO` + `SAEB` + `ENADE` + `IDEB` + `ANO` | `id_ies` FK to `IES`, `id_ano` FK to `ANO` | **T+5 min** (if CDC) or **T+1 hr** (if Batch) |
| **AGREGACAO_IES_ANO** | Aggregation of `DADOS_IES_ANO` | `GROUP BY` `id_municipio` + `id_ano` on Wide Table | **T+5 min** (depends on MV refresh freq) |
| **AGREGACAO_PARTICIPANTES** | Aggregation of `ANO` + `IES` + `MUNICIPIO` | `SUM` `total_participantes` | **T+1 hr** (Batch window) |

**Correctness Guarantee:**
The denormalized tables are **not** the source of truth. They are derived artifacts.
*   **Intentional Staleness**: Queries against `DADOS_IES_ANO` or `AGREGACAO_*` will return data up to the last refresh.
*   **Reconciliation**: A nightly job (`RECONCILIATE_JOB`) runs every 24 hours:
    1.  Triggers `REFRESH MATERIALIZED VIEW` on all artifacts.
    2.  Calculates a checksum (e.g., `SUM(nota_ideb)` across all schools) for `DADOS_IES_ANO` vs. `MUNICIPIO`.
    3.  If diffs > threshold, alerts are sent.

# 5) Risk Analysis & Operational Plan

### Risks
1.  **Write Amplification**:
    *   *Impact*: Low. We do not write to `DADOS_IES_ANO`. Writes go only to source tables.
    *   *Mitigation*: Ensure the ingestion pipeline (ETL) is optimized. If using CDC, ensure the refresh logic handles incremental changes efficiently.
2.  **Lock Contention**:
    *   *Impact*: Low. Materialized views in modern engines (Snowflake/BigQuery/Postgres) use **Concurrent Refresh** by default, allowing reads during refresh.
    *   *Mitigation*: Configure `REFRESH MATERIALIZED VIEW CONCURRENTLY` explicitly.
3.  **Backfill Runtime**:
    *   *Impact*: High for historical data. If the dataset is large (e.g., 10 years of data), the first refresh of `AGREGACAO_IES_ANO` could take significant time.
    *   *Mitigation*: Use **Incremental Refresh** (add only new rows/changes since last run) rather than full table rebuild.
4.  **Storage Growth**:
    *   *Impact*: Moderate. `DADOS_IES_ANO` duplicates `IES` data + 4 other tables.
    *   *Mitigation*: Use columnar storage (Parquet/ORC) or engine-specific compression (ZSTD) if available.
