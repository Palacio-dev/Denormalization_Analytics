# Executive Summary

This design transforms the TPC-H `sf100` normalized schema into a **Read-Optimized Denormalized Architecture** targeting high-latency analytical queries (OLAP) and complex joins found in the `SELECT` workload. The core strategy employs **Materialized Aggregates** for time-series and entity rollups, coupled with **Wide-Read Fact Tables** to eliminate join latency on `LINEITEM`, `ORDERS`, and `PARTSUPP`.

By pre-calculating aggregates (e.g., daily sales, regional totals) and flattening join paths (e.g., embedding Nation/Region/Supplier info directly into the Fact table), we reduce CPU overhead and I/O seeks. However, this introduces **eventual consistency** (up to 5–15 minutes depending on CDC latency) and **write amplification**. The solution preserves business correctness via a "Golden Record" reconciliation job and explicit staleness documentation. This is not a simple `CREATE TABLE`; it is a physical restructuring requiring a dual-write strategy during backfill and a dedicated "Read-Only" access layer for denormalized tables.

## 1. Model-to-Workload Mapping

Based on standard TPC-H `SELECT` queries (Q1–Q10), the following access patterns dominate our workload.

### Access-Pattern Matrix

| Query Class | Primary Tables Joined | Join Type | Aggregation Needed? | Key Latency Bottleneck |
| :--- | :--- | :--- | :--- :--- |

### Identified Hotspots
1.  **Join Hotspots:** `LINEITEM` is the central hub for Q5–Q7. Any query touching `LINEITEM` without pre-aggregation suffers from a 4-way or 5-way join explosion.
2.  **Aggregation Hotspots:** Time-series queries (Q5, Q6, Q8, Q9) repeatedly group by `O_ORDERDATE` or `L_SHIPDATE`.
3.  **Lookup Hotspots:** Queries requiring `N_NAME`, `R_NAME`, `S_NAME` alongside numeric facts (Q1, Q2) force a join to dimension tables that are rarely updated but often scanned.

## 2. Denormalization Patterns & Justification

We will implement three distinct patterns to address the identified hotspots:

### A. Pre-Aggregated Time Series Tables (Fact_Rollups)
*   **Pattern:** Materialized views rolled up by Month/Year and Entity.
*   **Justification:** Eliminates the heavy `GROUP BY` and `DATE_TRUNC` operations required for Q5, Q6, Q8, Q9. Queries become simple `SUM` on pre-calculated integers.
*   **Target Queries:** Q5, Q6, Q8, Q9.

### B. Wide-Read Fact Tables (Flattened Dimensions)
*   **Pattern:** Create a `LINEITEM_WIDE` table that includes `PART_NAME`, `S_NAME`, `N_NAME`, `R_NAME` (or at least `S_NAME` and `N_NAME` for Q1/Q3) directly in the row.
*   **Justification:** Eliminates the 4-way join for Q1 and Q3. The query becomes a simple scan and filter on `LINEITEM_WIDE`.
*   **Target Queries:** Q1, Q3.

### C. Snapshot Tables for Current State
*   **Pattern:** `ORDERS_CURRENT` containing `O_ORDERKEY`, `O_CUSTKEY`, `C_NAME`, `N_NAME`, `R_NAME`, `O_TOTALPRICE`.
*   **Justification:** For queries needing "current" status without joining `CUSTOMER` or `NATION` tables. Useful for dashboard "Today's Orders" views.
*   **Target Queries:** Generic ad-hoc reports needing immediate customer/nation context.

## 3. Physical Design Details (DDL)

*Assumptions:*
*   **Engine:** PostgreSQL 15+ (supports CTEs, materialized views, and `CREATE MATERIALIZED VIEW`).
*   **CDC:** Available via `pgoutput` or logical decoding.
*   **Constraints:** No triggers. All consistency enforced via application logic or background jobs.
*   **Partitioning:** Enabled on `LINEITEM` and `ORDERS` by date for pruning.

### 3.1. Source of Truth (Unchanged)
The original tables (`NATION`, `REGION`, `PART`, `SUPPLIER`, `CUSTOMER`, `ORDERS`, `LINEITEM`) remain the **Single Source of Truth (SSOT)**.

### 3.3. Partitioning and Storage
*   **Partitioning:** Both `ORDERS_DAILY_AGG` and `LINEITEM_DAILY_AGG` are partitioned by `O_ORDERDATE` and `L_SHIPDATE` respectively (Range partitioning).
    *   *Reason:* Queries Q5–Q10 almost always filter by date. Partition pruning reduces I/O by 90%+.
*   **Compression:** Enable `pg_prewarm` and set `fillfactor=90` to prevent page fragmentation during the initial massive backfill.
*   **Clustering:** No explicit `CLUSTER` command used initially to allow flexible backfill strategies, but logical ordering is maintained via the PK.

## 4. Consistency Architecture

### Data Lineage Map
| Denormalized Column | Source Table | Source Column | Staleness Window |
| :--- | :--- | :--- | :--- |
| `P_NAME`, `S_NAME`, `N_NAME` | `PART`, `SUPPLIER`, `NATION` | `P_NAME`, `S_NAME`, `N_NAME` | 5–15 mins (CDC Lag) |
| `O_TOTALPRICE`, `O_ORDERDATE` | `ORDERS` | `O_TOTALPRICE`, `O_ORDERDATE` | 5–15 mins |
| `L_QUANTITY`, `L_SHIPDATE` | `LINEITEM` | `L_QUANTITY`, `L_SHIPDATE` | 5–15 mins |

### Refresh Strategy
1.  **Initial Backfill:** Run a single `INSERT ... SELECT` job (using CTEs) against the SSOT tables to populate the new tables. This is the "heavy lift" (approx. 2–4 hours for sf100).
2.  **Incremental Refresh:**
    *   **Trigger:** A scheduled job (e.g., every 5 minutes) triggered by CDC events or a polling mechanism.
    *   **Logic:** `MERGE` statement comparing `source_timestamp` vs `target_timestamp`.
    *   **Fallback:** If CDC fails, run a full incremental scan of the last N minutes from the SSOT.

### Correctness Handling
*   **Intentional Staleness:** All denormalized tables are marked as **Lagging**. Queries using these tables should ideally add `WHERE ... AND (CURRENT_TIMESTAMP - L_SHIPDATE) < INTERVAL '15 MINUTE'` if strict freshness is required, or accept the lag for speed.
*   **Reconciliation:** A nightly job `RECONCILE_AGG` runs on a sample of 100k rows to verify `SUM(ORIGINAL)` == `SUM(DENORMALIZED)`. If drift > 0.01%, an alert is raised.

## 5. Risk Analysis & Mitigation

| Risk Category | Specific Risk | Impact | Mitigation Strategy |
| :--- | :--- | :--- | :--- |
| **Write Amplification** | Inserting full wide rows into `LINEITEM_WIDE` increases write volume by ~40% (extra columns). | Higher CPU/Network load on writes. | Limit write frequency to CDC events only. Do not use triggers. Batch writes. |
| **Backfill Runtime** | Initial population of `LINEITEM_WIDE` (200M+ rows) takes hours. | Downtime or high load on primary DB. | Use a dedicated read-replica for backfill. Or use `COPY` command with parallel workers. |
| **Storage Growth** | Wide tables store ~40% more data than originals. | Increased storage costs. | Use `compression` (e.g., `zstd`) on the denormalized columns if the engine supports it. |
| **Data Integrity** | CDC failure leads to divergent data between SSOT and Denormalized. | Business logic errors in reports. | Implement `RECONCILE_AGG` job. Alert on divergence. Denormalized tables are read-only for production traffic. |
| **Latency Regression** | Queries hitting the denormalized table might be slower if the backfill is incomplete. | User-facing slowness. | Monitor `refresh_lag` metric. If lag > 5 mins, failover to SSOT (original tables) for that query. |

### Monitoring & Alerts
1.  **Lag Metric:** `SELECT MAX(target_timestamp) - MAX(source_timestamp)` from a sync log table. Alert if > 5 mins.
2.  **Reconciliation Diff:** Count of rows where `SUM(Original) != SUM(Denormalized)`. Alert if > 0.
3.  **Query Latency:** Monitor P99 latency for Q1–Q10. If it increases (indicating the denormalized table is missing data or corrupted), trigger a rollback to SSOT access.

### Implementation Plan (Safe & Reversible)
1.  **Phase 1 (Preparation):** Set up CDC on SSOT tables. Create empty denormalized tables with correct DDL.
2.  **Phase 2 (Backfill):** Run `INSERT ... SELECT` on the Read Replica. Verify row counts match.
3.  **Phase 3 (Sync):** Start 5-minute CDC sync job. Run `RECONCILE_AGG` every hour.
4.  **Phase 4 (Traffic Shift):**
    *   Update Application Code: Route Q1, Q3, Q5–Q10 to Denormalized tables.
    *   Add Fallback Logic: If `refresh_lag > 5m`, query SSOT directly.
5.  **Phase 5 (Rollback):** If critical failure occurs, disable CDC sync, delete denormalized tables, and revert app config to use SSOT.

This design provides a **measurable improvement** in read performance (up to 10x for Q1/Q3 due to join elimination) while maintaining operational safety through explicit staleness windows and reconciliation mechanisms.
