# Executive Summary

The proposed schema introduces **Denormalized Read-Optimized Artifacts** to decouple the transactional truth (normalized) from the analytical workload (denormalized). This design preserves ACID consistency for writes while accepting controlled staleness for reads, which is standard for BI and reporting layers.

We are introducing two primary artifacts:
1.  **`hr_employee_current_state`**: A **Snapshot Table** that pre-joins `Employee`, `Person`, `Department`, and `Shift` data. This eliminates the most frequent join-heavy queries (e.g., "List all employees with their current department and shift").
2.  **`hr_employee_pay_current`**: A **Wide Read Table** that pre-calculates the *effective* hourly rate by joining `EmployeePayHistory` and handling the logic of "current rate vs. last rate change". This removes complex `CASE WHEN` logic from application code.

**Correctness Strategy:** The source of truth remains the normalized tables. The denormalized tables are updated via **CDC (Change Data Capture)** streams (e.g., Debezium) or a scheduled background job (`pg_cron`). We explicitly document a **5-minute staleness window** for all denormalized data. This window is acceptable because the denormalized tables are read-only for business users and do not drive critical transactional logic.

# 1. Model-to-Workload Mapping

Based on typical `AdventureWorks` reporting patterns and the provided DDL, the access patterns are:

| Query Class | Tables Accessed (Normalized) | Join Complexity | Aggregation Needs | Latency Impact |
| :--- | :--- | :--- | :--- :--- |
| **Employee Directory** | `Employee`, `Person`, `Department`, `Shift` | High (4-way join) | None (Filter/Select) | High (I/O + CPU for joins) |
| **Employee Pay Rate** | `Employee`, `EmployeePayHistory` | Medium (2-way join + Logic) | None (Select) | Medium (Complex `CASE` logic in app) |
| **Department Stats** | `Department`, `EmployeeDepartmentHistory` | Medium | Yes (Count, Avg) | Medium |
| **Product Reviews** | `ProductReview`, `Product` | Low | Yes (Avg Rating) | Low |

**Identified Hotspots:**
*   **Join Hotspot:** `Employee` $\to$ `Department` $\to$ `Shift` $\to$ `Person`. This is the single most common join chain.
*   **Aggregation Hotspot:** Calculating the *effective* pay rate requires a subquery or window function on `EmployeePayHistory` for every employee, which is expensive on large datasets.

# 2. Denormalization Patterns & Justification

### Pattern A: Snapshot Table (`hr_employee_current_state`)
*   **Justification:** Eliminates the need to join `Person`, `Department`, and `Shift` for every employee lookup. Since employee assignments rarely change (and when they do, it's a point-in-time event), a snapshot of the "current" state is highly effective.
*   **Refresh Strategy:** Full refresh nightly or incremental via CDC on `modifieddate`.

### Pattern B: Wide Read Table (`hr_employee_pay_current`)
*   **Justification:** The effective salary is a derived value dependent on the `ratechangedate`. Doing this calculation on the fly is slow. Pre-calculating the "Current Rate" column avoids complex application-side logic.
*   **Refresh Strategy:** Incremental update triggered by `EmployeePayHistory` inserts/updates.

# 3. Physical Design Details (DDL)

*Assumptions:*
*   Engine: PostgreSQL 14+.
*   CDC Source: External stream (e.g., Debezium) or a `pg_cron` job running every 5 minutes.
*   Storage: Standard `tablespace` for speed; no partitioning required as these are dimension/slow-change tables.
*   Constraints: We use `CHECK` constraints for data integrity but **no Foreign Keys** to prevent cascading delete issues during denormalization updates.

```sql
-- ==========================================
-- 1. Snapshot Table: hr_employee_current_state
-- Purpose: Denormalized join of Employee, Person, Department, Shift.
-- Staleness: ~5 minutes.
-- ==========================================

CREATE TABLE IF NOT EXISTS hr_employee_current_state (
    employeeid integer NOT NULL,
    firstname character varying(50) COLLATE pg_catalog."default" NOT NULL,
    lastname character varying(50) COLLATE pg_catalog."default" NOT NULL,
    title character varying(20) COLLATE pg_catalog."default",
    departmentname character varying(100) COLLATE pg_catalog."default",
    groupname character varying(100) COLLATE pg_catalog."default",
    shiftname character varying(50) COLLATE pg_catalog."default",
    hiredate date NOT NULL,
    maritalstatus character(1) COLLATE pg_catalog."default",
    gender character(1) COLLATE pg_catalog."default",
    jobtitle character varying(50) COLLATE pg_catalog."default",
    currentflag "Flag" NOT NULL DEFAULT true,
    -- Indexes for fast filtering and sorting
    CONSTRAINT "PK_EmployeeCurrentState_EmployeeID" PRIMARY KEY (employeeid),
    CONSTRAINT "IX_EmployeeCurrentState_DepartmentName" UNIQUE (departmentname), -- Optional: Fast lookup by dept
    CONSTRAINT "IX_EmployeeCurrentState_HireDate" UNIQUE (hiredate)
);

-- Data Lineage Note:
-- employeeid: Source = humanresources.employee.businessentityid
-- firstname/lastname: Source = person.person.firstname/lastname
-- departmentname/groupname: Source = humanresources.department.name/groupname (via current dept)
-- shiftname: Source = humanresources.shift.name (via current shift)
-- currentflag: Source = humanresources.employee.currentflag
-- *Values are captured at the moment of refresh. If an employee changes dept, 
-- the new value appears after the refresh cycle.*

-- ==========================================
-- 2. Wide Read Table: hr_employee_pay_current
-- Purpose: Pre-calculated effective hourly rate.
-- Staleness: ~5 minutes.
-- ==========================================

CREATE TABLE IF NOT EXISTS hr_employee_pay_current (
    employeeid integer NOT NULL,
    effectivehourlyrate numeric(10, 2) NOT NULL,
    ratechangedate timestamp without time zone NOT NULL,
    payfrequency smallint NOT NULL,
    salaryflag "Flag" NOT NULL DEFAULT true,
    -- Indexes
    CONSTRAINT "PK_EmployeePayCurrent_EmployeeID" PRIMARY KEY (employeeid),
    CONSTRAINT "IX_EmployeePayCurrent_RateChangeDate" UNIQUE (ratechangedate)
);

-- Data Lineage Note:
-- employeeid: Source = humanresources.employee.businessentityid
-- effectivehourlyrate: Calculated logic:
--   IF (last ratechange < today) THEN (current rate) ELSE (last rate)
-- ratechangedate: Source = humanresources.employeepayhistory.ratechangedate
-- payfrequency: Source = humanresources.employeepayhistory.payfrequency
-- salaryflag: Source = humanresources.employee.salariedflag
```

# 4. Consistency Architecture & Implementation Plan

### Data Lineage Mapping
| Denormalized Column | Source Table | Source Column | Logic Applied |
| :--- | :--- | :--- | :--- |
| `departmentname` | `humanresources.department` | `name` | Join via `employeedepartmenthistory` where `enddate` IS NULL or `enddate` > `now()` |

### Implementation Plan (Safe & Reversible)

#### Phase 1: Validation & Baseline (Day 0)

3.  **Verification:** Run a test query comparing a random sample of `hr_employee_current_state` against a normalized join query. Ensure 100% match for existing data.

#### Phase 2: CDC / Refresh Logic (Day 1+)

*   Deploy a Debezium connector on `humanresources.employee`, `person`, `department`, `shift`, `employeedepartmenthistory`, `employeepayhistory`.
*   Stream changes to a `hr_change_log` table.
*   Run a stored procedure `sync_employee_state()` every 5 minutes that applies `INSERT/UPDATE` to the denormalized tables based on the log.

# 5. Risk Analysis

| Risk Category | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Write Amplification** | Denormalized tables grow. If we store full XML/Text in `person`, storage doubles. | **Mitigation:** Only denormalize scalar fields (Name, Title, Rate). Do not copy large BLOBs/XML unless specifically requested by a query. |
| **Lock Contention** | `INSERT` into a hot denormalized table could block writers if not careful. | **Mitigation:** Use `INSERT ... ON CONFLICT DO NOTHING` or `OR REPLACE` in the refresh script to prevent deadlocks during concurrent writes. |
| **Data Staleness** | Users see outdated department names. | **Mitigation:** Explicitly document in API contracts: "HR Data has a 5-minute delay." Use the denormalized table for *reports*, not for real-time dashboards requiring instant accuracy. |
| **Backfill Runtime** | Initial population might take time on large datasets. | **Mitigation:** Schedule initial backfill during off-peak hours (e.g., 2 AM). Monitor progress; if > 1 hour, pause and optimize query. |
| **Reversibility** | Can we go back to normalized? | **Yes.** The denormalized tables are independent. If a bug is found in the denormalized logic, drop the table and re-seed from backup. The normalized source remains untouched. |

### Final Recommendation
Adopt **`hr_employee_current_state`** immediately as it solves the most visible performance bottleneck (employee directory queries). The **pay rate** table can be added in Phase 2 if the application team reports latency on salary calculations. The 5-minute staleness window is a safe trade-off for the read performance gains.
