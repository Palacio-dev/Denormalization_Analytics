Here is a comprehensive denormalization strategy and physical schema design tailored for a read-heavy workload on the classic "Employees" dataset.

### 1. Workload-Driven Analysis

In the highly normalized schema provided, historical tracking (SCD Type 2) is implemented across four separate tables (`department_employee`, `department_manager`, `salary`, `title`) using `from_date` and `to_date`. 

**High-Cost Operations & Join Paths:**
1.  **The "Current Profile" Join Explosion:** To display a simple profile for an active employee (Name, Current Department, Current Title, Current Salary), the database must execute a 5-table join (`employee` → `department_employee` → `department` → `title` → `salary`). Furthermore, it must filter on `to_date` across three of those tables to filter out historical records.
2.  **Department Roster & Hierarchy Scans:** Fetching a list of employees in a specific department requires joining `department` → `department_employee` → `employee`. Finding the manager for those employees requires an additional subquery or join to `department_manager`.
3.  **Analytical Aggregations:** Calculating average salaries by department or title requires full table scans of the massive `salary` table (which contains every pay raise in company history), joined against titles and departments, just to aggregate the current state.

### 2. Denormalization Design

To optimize for low-latency reads while preserving the correctness of the historical write-model, we will implement a **CQRS-style Read Model** using Materialized Views and covering indexes.

**A. Pre-joined "Wide" Table (Current Employee Snapshot)**
*   **What:** A materialized view `mv_employee_current_profile` flattening the current state of an employee.
*   **Why:** Reduces the 5-table join to a single O(1) primary key lookup.
*   **Grain:** 1 row per active employee.

**B. Summary/Aggregate Table (Department Analytics)**
*   **What:** A materialized view `mv_dept_title_salary_summary`.
*   **Why:** Dashboards showing headcount and average salaries currently calculate this on the fly. Precomputing this by `department` and `title` reduces millions of rows to a few hundred.
*   **Grain:** 1 row per department, per title.

**C. Redundant Attributes (Manager Snapshot)**
*   **What:** A materialized view `mv_department_current_detail` that includes the current manager's ID and Name directly alongside the department details.
*   **Why:** Eliminates the need to traverse `department_manager` and `employee` just to label a department with its leader.

**D. Indexing Strategy for Base Tables**
*   **What:** Composite covering indexes on the historical tables.
*   **Why:** Even with materialized views, refreshing them or querying recent history requires fast access. We will add indexes on `(employee_id, to_date)` that `INCLUDE` the payload columns (`amount`, `title`, `department_id`) to enable Index-Only Scans.

### 3. New Physical Schema (DDL)

Run the following DDL to apply the read-optimized denormalized schema.

```sql
BEGIN;

-- ============================================================================
-- PHASE 1: COVERING INDEXES ON BASE TABLES
-- Optimizes the historical tables for "Index-Only Scans" when filtering for 
-- current records (where to_date is typically '9999-01-01' or NULL).
-- ============================================================================

CREATE INDEX IF NOT EXISTS idx_salary_emp_date_cover 
    ON employees.salary (employee_id, to_date DESC) 
    INCLUDE (amount);

CREATE INDEX IF NOT EXISTS idx_title_emp_date_cover 
    ON employees.title (employee_id, to_date DESC) 
    INCLUDE (title);

CREATE INDEX IF NOT EXISTS idx_dept_emp_date_cover 
    ON employees.department_employee (employee_id, to_date DESC) 
    INCLUDE (department_id);

CREATE INDEX IF NOT EXISTS idx_dept_mgr_date_cover 
    ON employees.department_manager (department_id, to_date DESC) 
    INCLUDE (employee_id);


-- ============================================================================
-- PHASE 2: DENORMALIZED READ MODELS (MATERIALIZED VIEWS)
-- Note: We assume '9999-01-01' represents the current active record, 
-- which is standard for this specific dataset.
-- ============================================================================

-- 1. WIDE TABLE: Current Employee Profile
-- Replaces 5-table joins for user profile lookups.
CREATE MATERIALIZED VIEW employees.mv_employee_current_profile AS
SELECT 
    e.id AS employee_id,
    e.first_name,
    e.last_name,
    e.gender,
    e.hire_date,
    t.title AS current_title,
    s.amount AS current_salary,
    d.id AS department_id,
    d.dept_name
FROM employees.employee e
LEFT JOIN employees.title t 
    ON e.id = t.employee_id AND t.to_date = '9999-01-01'
LEFT JOIN employees.salary s 
    ON e.id = s.employee_id AND s.to_date = '9999-01-01'
LEFT JOIN employees.department_employee de 
    ON e.id = de.employee_id AND de.to_date = '9999-01-01'
LEFT JOIN employees.department d 
    ON de.department_id = d.id;

-- Create Unique Index to allow CONCURRENT refreshes (zero downtime)
CREATE UNIQUE INDEX idx_mv_emp_profile_pk ON employees.mv_employee_current_profile(employee_id);
-- Index for department roster queries
CREATE INDEX idx_mv_emp_profile_dept ON employees.mv_employee_current_profile(department_id);
-- Index for title-based searches
CREATE INDEX idx_mv_emp_profile_title ON employees.mv_employee_current_profile(current_title);


-- 2. REDUNDANT ATTRIBUTES: Department Details with Manager
-- Replaces joins to find out who runs a department.
CREATE MATERIALIZED VIEW employees.mv_department_current_detail AS
SELECT 
    d.id AS department_id,
    d.dept_name,
    m.employee_id AS current_manager_id,
    e.first_name AS manager_first_name,
    e.last_name AS manager_last_name
FROM employees.department d
LEFT JOIN employees.department_manager m 
    ON d.id = m.department_id AND m.to_date = '9999-01-01'
LEFT JOIN employees.employee e 
    ON m.employee_id = e.id;

CREATE UNIQUE INDEX idx_mv_dept_detail_pk ON employees.mv_department_current_detail(department_id);


-- 3. AGGREGATE TABLE: Department & Title Salary Analytics
-- Replaces massive table scans for analytical dashboards.
CREATE MATERIALIZED VIEW employees.mv_dept_title_salary_summary AS
SELECT 
    p.department_id,
    p.dept_name,
    p.current_title,
    COUNT(p.employee_id) AS active_headcount,
    ROUND(AVG(p.current_salary), 2) AS avg_salary,
    MIN(p.current_salary) AS min_salary,
    MAX(p.current_salary) AS max_salary,
    SUM(p.current_salary) AS total_payroll
FROM employees.mv_employee_current_profile p
WHERE p.department_id IS NOT NULL 
  AND p.current_salary IS NOT NULL
GROUP BY 
    p.department_id,
    p.dept_name,
    p.current_title;

CREATE UNIQUE INDEX idx_mv_dept_title_summary_pk 
    ON employees.mv_dept_title_salary_summary(department_id, current_title);

END;
```

### How to use this in production:
1. **Application Reads:** Point your application's read queries (GET requests, UI dashboards) to the `mv_*` materialized views.
2. **Application Writes:** Continue writing to the normalized base tables (`employee`, `salary`, etc.) to preserve ACID guarantees and strict historical tracking.
3. **Data Sync:** Set up a pg_cron job or application trigger to run `REFRESH MATERIALIZED VIEW CONCURRENTLY employees.mv_employee_current_profile;` on a schedule (e.g., nightly, or every 5 minutes depending on business staleness tolerance). Because we added `UNIQUE` indexes to the MVs, Postgres will refresh them in the background without locking read access.
