## **Consistency Architecture**

Each denormalized table includes `_source_last_updated` or `_last_refreshed` fields to track freshness. Source lineage can be traced through these columns:

| Denormalized Column         | Source Tables                     |
|----------------------------|-----------------------------------|
| All demographic fields     | aluno_ifb                         |
| Course info                | curso_ifb                         |
| School info                | escola, localizacao               |
| ENEM metrics               | desempenho_enem                   |
| Indicators                 | indicadores_escolares             |
| Employment data            | vinculo_empregaticio, remuneracao |
| Enrollment metadata        | matricula_ifb                     |

A daily ETL process updates these tables using simple SELECT...JOIN statements. For example:
```sql
INSERT INTO student_profile ...
SELECT ... FROM aluno_ifb JOIN matricula_ifb USING(id_aluno) ...
ON CONFLICT (id_aluno) DO UPDATE SET ...
```

Freshness monitoring is achieved by comparing `_source_last_updated` against known upstream timestamps.

## **Risk Analysis**

### **Write Amplification**
Each update to a base table may trigger multiple writes to denormalized copies — especially true for frequently changing facts like `matricula_ifb`. Mitigated via batching and asynchronous processing.

### **Lock Contention**
Denormalized tables should use row-level locking only during upsert operations. Use advisory locks where necessary.

### **Backfill Runtime**
Initial load could take hours depending on volume. Suggested approach:
- Load historical batches partitioned by year.
- Use parallel workers and temp staging tables.

### **Storage Growth**
Estimated increase of ~3x due to duplication. Recommend compression settings (TOAST) and archiving old snapshots.

### **Monitoring & Alerts**
Set up:
- Lag detection between source and target tables.
- Reconciliation diffs using checksums or row counts.
- Query latency baselines to detect regressions after deployment.

Use Prometheus/Grafana or similar tools for dashboards tracking:
- `_last_refreshed`
- Row count deltas
- Query duration trends
