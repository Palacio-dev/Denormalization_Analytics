### 1. Analysis of the Provided Schema

The current schema follows a strict normalized design (3NF/BCNF principles), which is excellent for data integrity but suboptimal for analytical reporting.

**Current Structure:**
*   **Core Entity:** `MUNICIPIO` acts as the central hub, linking to `UF`, `IES`, and `ESCOLA`.
*   **Metrics Tables:** `IDE`, `ENADE`, `SAEB`, and `AN` (Year/Linker) are separate.
*   **Hierarchy:** UF -> MUNICIPIO -> IES/ESCOLA.
*   **Linker Table:** `AN` serves as the bridge between schools, years, and specific metrics (`id_ideb`, `id_enade`, etc.).

**Identified Bottlenecks & Heavy Workloads:**
In an analytical context (BI dashboards, trend analysis), the following patterns are most common:
1.  **Wide Joins on `MUNICIPIO`:** Queries often need: *State Name + City Name + IES Count + School Count + IDEB Note + ENADE Avg + SAEB Score*. Currently, this requires joining `UF`, `MUNICIPIO`, `IDE`, `ENADE`, and `SAEB`.
2.  **The `AN` Table Overhead:** Every query calculating metrics per year must join `AN` to fetch the specific `id_ideb` or `id_enade` associated with that year. If a user asks for "Average IDEB across Brazil for 2020-2023", the database must join `MUNICIPIO` -> `AN` -> `IDE` and repeat this for `ENADE` and `SAEB`.
3.  **Redundant Aggregations:** Calculating `AVG(nota_bruta_ce)` across thousands of municipalities is expensive if done on the fly.

### 2. Strategy: Denormalization Techniques Applied

I will apply the following techniques from your context list:

1.  **Adding Redundant Attributes (Materialized Joins):**
    *   *Action:* Instead of joining `MUNICIPIO` with `IDE`, `ENADE`, and `SAEB`, we will create a "Fact-like" table or expand `MUNICIPIO` to include the direct metric values.
    *   *Benefit:* Eliminates the need to join three separate metric tables per query.

2.  **Collapsing Relations:**
    *   *Action:* Merge the `UF`, `MUNICIPIO`, and the "Year-Metric" relationships into a single comprehensive table per metric (e.g., `FATIA_IDEB`, `FATIA_ENADE`).
    *   *Benefit:* Reduces join complexity from $N$ joins to 0 (single table scan).

3.  **Adding Derived Attributes (Pre-computed Aggregates):**
    *   *Action:* Since the source tables (`IDE`, `ENADE`, `SAEB`) appear to be static reference data (or aggregated once per year), we will pre-calculate the averages/counts at the municipality level.
    *   *Benefit:* Avoids runtime aggregation (`GROUP BY`), which is the most expensive operation in analytical queries.

4.  **Vertical Partitioning (Implicit):**
    *   *Action:* While we are merging, we effectively treat the "Municipality" entity and the "Metric Facts" as one logical unit for reporting.

### 3. Proposed Denormalized Schema Design

I will create a schema consisting of three main tables:
1.  **`REGIAO_ESCOLAR`**: Contains the static hierarchy (UF, Municipality Name, State Name).

3.  **`CURSO`**: Kept minimal as it links to `IES`, but can be denormalized if needed. For this optimization, we will focus on the Municipality/Metric bottleneck.

3.  **Storage:** There is a slight increase in storage overhead due to redundancy (copying `UF` and `MUNICIPIO` data), but this is negligible compared to the CPU savings gained from avoiding joins and aggregations.
4.  **Maintainability:** The schema is simpler. Developers no longer need to understand the complex foreign key relationships between `AN`, `IDE`, `ENADE`, and `SAEB` to build reports; they just query `METRICAS_ESCOLA`.
