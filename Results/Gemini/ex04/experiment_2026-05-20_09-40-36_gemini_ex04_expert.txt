### Executive Summary

The existing normalized schema is highly structured (3NF), splitting socio-economic, demographic, and survey data across 11 separate tables. While excellent for data integrity and avoiding update anomalies, this design is hostile to analytical read workloads. Querying a comprehensive "Country Profile" for a given year currently requires up to an 8-way join, leading to high CPU utilization, complex query execution plans, and poor cache hit ratios. 

To resolve this, we are introducing a read-optimized, denormalized architecture using the **One Big Table (OBT)** and **Pre-Aggregated Document** patterns. By consolidating the data into three Materialized Views, we eliminate join penalties at read time and pre-calculate expensive aggregations (like survey decile rollups). Given the nature of socio-economic data (typically appended annually/quarterly rather than updated sub-second), we assume a nightly batch refresh strategy. This guarantees exact query performance improvements with a documented, acceptable staleness window of up to 24 hours, without modifying the underlying source-of-truth tables.

### 2. Denormalization Patterns Justification

1.  **Wide Read Table (One Big Table / OBT):** 
    *   *Artifact:* `mvw_country_year_wide`
    *   *Justification:* Economy, education, employment, health, and demography all share the exact same logical grain: `(country_code, year)`. By pre-joining these into a single wide table, we reduce Q1 and Q2 from 7 joins to 0 joins. 
2.  **Pre-Aggregated Rollups (Document Pattern):**
    *   *Artifact:* `mvw_survey_insights`
    *   *Justification:* The `decile` table is strictly a 1:N child of `survey`. Instead of forcing BI tools to group by survey every time they want to see distribution, we pre-aggregate the deciles into a JSONB array. This flattens the grain to exactly 1 row per survey.
3.  **Engine Equivalent (Materialized Views):**
    *   *Justification:* PostgreSQL Materialized Views are native, easily observable, and support `REFRESH MATERIALIZED VIEW CONCURRENTLY`. This allows us to rebuild the read models in the background without blocking concurrent reads.

*(Note on Partitioning: I am explicitly **omitting** table partitioning. Assuming ~200 countries and ~60 years of data, the wide table will be ~12,000 rows. Partitioning this would cause performance degradation due to partition-planning overhead. B-Tree indexes are vastly superior here.)*

### 3. Proposed Denormalized Schema (DDL)

*The following DDL creates the read-optimized layer. It assumes PostgreSQL based on the provided data types.*

```sql
-- ==============================================================================
-- ARTIFACT 1: Wide Table for Country-Year Metrics (Targeting Q1 & Q2)
-- ==============================================================================

CREATE MATERIALIZED VIEW public.mvw_country_year_wide AS
WITH country_years AS (
    -- Driver CTE: Generate all valid (country, year) combinations
    SELECT country_code, year FROM public.demography
    UNION
    SELECT country_code, year FROM public.employment
    UNION
    SELECT country_code, year FROM public.education
    UNION
    SELECT country_code, year FROM public.economy
    UNION
    SELECT country_code, year FROM public.health
)
SELECT 
    cy.country_code,
    cy.year,
    c.country_name,
    r.region_name,
    r.region_code,
    -- Economy
    ec.gdp,
    ec.inflation,
    ec.tax_revenue,
    -- Employment
    em.unemp AS unemployment_rate,
    em.labor_force_total,
    em.child_emp,
    -- Education
    ed.expenditure AS education_expenditure,
    ed.prim_enrol AS primary_enrollment,
    ed.sec_enrol AS secondary_enrollment,
    -- Health
    h.hospital_beds,
    h.physicians,
    h.expenditure AS health_expenditure,
    -- Demography
    d.pop_density,
    d.urban_pop,
    d.rural_pop,
    d.birth_rate,
    d.death_rate
FROM country_years cy
JOIN public.country c ON cy.country_code = c.country_code
JOIN public.region r ON c.region_code = r.region_code
LEFT JOIN public.economy ec ON cy.country_code = ec.country_code AND cy.year = ec.year
LEFT JOIN public.employment em ON cy.country_code = em.country_code AND cy.year = em.year
LEFT JOIN public.education ed ON cy.country_code = ed.country_code AND cy.year = ed.year
LEFT JOIN public.health h ON cy.country_code = h.country_code AND cy.year = h.year
LEFT JOIN public.demography d ON cy.country_code = d.country_code AND cy.year = d.year;

-- Indexes for Q1 (Time-series per country) and Q2 (Regional cross-sections)
CREATE UNIQUE INDEX idx_mvw_cyw_pk ON public.mvw_country_year_wide (country_code, year);
CREATE INDEX idx_mvw_cyw_region_year ON public.mvw_country_year_wide (region_code, year);


-- ==============================================================================
-- ARTIFACT 2: Survey Insights with Pre-Aggregated Deciles (Targeting Q3)
-- ==============================================================================

CREATE MATERIALIZED VIEW public.mvw_survey_insights AS
WITH decile_agg AS (
    SELECT 
        id_survey,
        jsonb_agg(
            jsonb_build_object('name', name, 'value', value) 
            ORDER BY value ASC
        ) AS decile_distribution
    FROM public.decile
    GROUP BY id_survey
)
SELECT 
    s.id_survey,
    s.country_code,
    c.country_name,
    r.region_name,
    s.survey_year,
    s.welfare_type,
    s.poverty_line,
    s.headcount,
    s.poverty_gap,
    s.gini,
    s.reporting_pop,
    COALESCE(da.decile_distribution, '[]'::jsonb) AS decile_distribution
FROM public.survey s
JOIN public.country c ON s.country_code = c.country_code
JOIN public.region r ON c.region_code = r.region_code
LEFT JOIN decile_agg da ON s.id_survey = da.id_survey;

-- Indexes for Survey lookups
CREATE UNIQUE INDEX idx_mvw_si_pk ON public.mvw_survey_insights (id_survey);
CREATE INDEX idx_mvw_si_country_year ON public.mvw_survey_insights (country_code, survey_year);
```

### 4. Consistency Architecture

**Data Lineage:**
*   `mvw_country_year_wide` maps 1:1 to the `(country_code, year)` grain. Source-of-truth columns from `economy`, `health`, etc., flow directly into this view.
*   `mvw_survey_insights` maps 1:1 to `survey.id_survey`. `decile` rows are folded into the `decile_distribution` JSONB array.

**Refresh Strategy (Implementation Plan):**
Since triggers add operational overhead and lock contention on OLTP writes, we will use a **Scheduled Batch Refresh**.
1.  **Job Scheduler:** A cron job (e.g., via pg_cron, Airflow, or dbt) runs nightly at 02:00 AM.
2.  **Command:** `REFRESH MATERIALIZED VIEW CONCURRENTLY public.mvw_country_year_wide;`
3.  **Safety:** The `CONCURRENTLY` keyword (enabled by the `UNIQUE INDEX` we created) ensures that read queries are *never blocked* during the refresh. The engine builds a new snapshot in the background and hot-swaps the pointers.
4.  **Staleness Window:** Explicitly documented as **T+24 hours**.

### 5. Risk Analysis & Operational Considerations

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **Zero** during business hours. Because we use Materialized Views refreshed in batch, the base tables suffer no write penalty when inserts/updates occur. |
| **Lock Contention** | **Minimal**. `REFRESH CONCURRENTLY` uses an exclusive lock only for a brief microsecond during the final swap. Read queries will not queue up. |
| **Storage Growth** | **Low to Moderate**. The wide table contains many `NULL` values if a country lacks data for a specific year. PostgreSQL's physical storage handles NULLs efficiently (using a null bitmap), so storage bloat is negligible. |
| **Failure Modes** | If the nightly refresh fails, the MV simply retains yesterday's data. **Mitigation:** Alerting must be placed on the ETL scheduler. |
| **Monitoring / Alerts** | 1. **Freshness:** Monitor `pg_stat_user_tables.last_autovacuum` or a custom log table to ensure the MV was refreshed within 24h.<br>2. **Reconciliation:** Run a daily validation query: `SELECT COUNT(*) FROM survey` vs `SELECT COUNT(*) FROM mvw_survey_insights`. They must match exactly. |

**Reversibility:** 
This implementation is 100% safe and reversible. It requires **no modifications** to the existing DDL, constraints, or applications writing to the database. If the denormalized layer causes issues, the materialized views can be dropped (`DROP MATERIALIZED VIEW ...`) with zero impact on the source-of-truth normalized tables.
