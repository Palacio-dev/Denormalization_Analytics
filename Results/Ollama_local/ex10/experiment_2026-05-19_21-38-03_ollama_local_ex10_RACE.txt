### Evaluation of the New Schema

1.  **Query Performance Improvement**:
    *   **Before**: A query like `SELECT nome_continente, SUM(porcentagem) FROM consumo_anual JOIN paises ON ... JOIN continente ON ... GROUP BY nome_continente` required scanning the fact table, joining to 3 small tables, and performing a group by.
    *   **After**: `SELECT nome_continente, SUM(porcentagem) FROM fatos_consumo_emissao GROUP BY nome_continente`. The query now accesses only the fact table. The joins are eliminated entirely. This typically results in a **10x to 50x speedup** for ad-hoc analytical queries.

2.  **Write Overhead**:
    *   The only trade-off is the write overhead of storing redundant strings (`nome_pais`, `nome_continente`, etc.) in the fact tables.
    *   **Mitigation**: Since the dimension tables (`dim_paises`, etc.) are small, they can be cached in memory (Redis/Memcached) or simply joined once at the start of a batch process if the dataset is massive. However, for typical analytical workloads where the fact table grows to millions/ billions of rows and dimensions stay small, the read gain vastly outweighs the write cost.

3.  **Data Integrity**:
    *   We have preserved referential integrity by keeping the `id_paises` and `id_continente` columns in the fact tables. This allows the application to still enforce relationships if needed, but the primary key and most common join keys are now redundant, speeding up lookups.
    *   We added a Unique Constraint (`uq_fato_pais_ano_uniq`) to ensure that for a specific country and year, there is only one aggregated row (assuming the original data was unique per country/year or intended to be aggregated).

4.  **Scalability**:
    *   The schema is designed to handle high read throughput. The separation of "Fact" (heavy data) and "Dimension" (lookup data) allows for efficient indexing strategies. If the dimension tables become too large to cache, you can further denormalize by moving the `nome` columns directly into the fact tables (Technique 1 extreme), eliminating the need for the dimension tables entirely.

This schema represents a **Star Schema** where the "Star" is the fact table, and the "arms" are the dimensions. In some analytical systems (like Snowflake or BigQuery), this would be the default. In PostgreSQL, this explicit denormalization removes the latency of the `JOIN` operation.
