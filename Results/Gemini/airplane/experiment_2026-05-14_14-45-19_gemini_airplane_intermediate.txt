Here is a comprehensive, workload-driven denormalization strategy for your airline booking database, optimized for PostgreSQL.

### 1. Workload-Driven Analysis

In a typical airline system, the read workload is heavily skewed toward three specific query patterns. In your highly normalized schema, these patterns suffer from deep, expensive join trees:

*   **Pattern A: Flight Search / Status Boards (High Volume)**
    *   *Current Join Path:* `flights` → `airports_data` (departure) → `airports_data` (arrival) → `aircrafts_data`.
    *   *Pain Point:* Every time a user searches for a flight schedule or checks a flight board, the database must perform a 4-table join. 
*   **Pattern B: Passenger Itinerary Retrieval (High Volume)**
    *   *Current Join Path:* `bookings` → `tickets` → `ticket_flights` → `flights` → `boarding_passes`.
    *   *Pain Point:* To show a logged-in user their upcoming trip and seat assignments, the database executes a 5-table join.
*   **Pattern C: Gate Agent Manifests (Medium Volume, Latency Sensitive)**
    *   *Current Join Path:* `flights` → `ticket_flights` → `tickets` → `boarding_passes`.
    *   *Pain Point:* Gate agents need instant access to who is on the plane, their seat, and boarding status. This requires a 4-table join.
*   **Pattern D: Route Revenue Analytics (Heavy Analytical Reads)**
    *   *Current Join Path:* Aggregating `ticket_flights` joined to `flights` to group by date and route.

### 2. Denormalization Design

To achieve low-latency reads while preserving business correctness, we will implement the following changes:

1.  **Flatten `flights` (Pre-joined Wide Table):**
    *   *Change:* Inject `departure_city`, `arrival_city`, and `aircraft_model` directly into the `flights` table.
    *   *Why:* Eliminates the need to join the `airports_data` and `aircrafts_data` dimension tables for 99% of flight searches.
2.  **Merge `boarding_passes` into `ticket_flights`:**
    *   *Change:* Drop the `boarding_passes` table entirely. Move `boarding_no` and `seat_no` into `ticket_flights`.
    *   *Why:* A boarding pass is strictly a 1:1 or 0:1 extension of a flight segment. Merging them saves a massive join on high-volume tables.
3.  **Push Down Passenger & Booking Data to `ticket_flights`:**
    *   *Change:* Duplicate `passenger_id`, `passenger_name`, `book_ref`, and `scheduled_departure` into `ticket_flights`.
    *   *Why:* This turns `ticket_flights` into a **Covering Fact Table**. You can now retrieve a full passenger itinerary or a full flight manifest by querying *a single table*.
4.  **Partitioning Strategy:**
    *   *Change:* Partition both `flights` and `ticket_flights` by `scheduled_departure` (Range partitioning by month).
    *   *Why:* Flight data is highly temporal. Users rarely query flights from 3 years ago. Partitioning ensures that active queries only scan small, memory-resident active partitions.
5.  **Precomputed Aggregates (Materialized View):**
    *   *Change:* Create `mv_route_daily_revenue` at the grain of `(date, departure_airport, arrival_airport)`.
    *   *Why:* Serves dashboard queries instantly without summing millions of rows in `ticket_flights`.

### 3. The New Physical Schema (PostgreSQL DDL)

*Note: Dimension tables (`airports_data`, `aircrafts_data`, `seats`) and the top-level `bookings` and `tickets` tables remain largely unchanged to act as the write-heavy source of truth. The heavy lifting is done in the modified transactional tables below.*

```sql
BEGIN;

-- ==============================================================================
-- 1. MODIFIED FLIGHTS TABLE (Wide Table & Partitioned)
-- ==============================================================================
-- We partition by scheduled_departure. 
-- We bring in city and aircraft model to eliminate dimension joins on reads.

DROP TABLE IF EXISTS bookings.flights CASCADE;

CREATE TABLE bookings.flights
(
    flight_id serial NOT NULL,
    flight_no character(6) NOT NULL,
    scheduled_departure timestamp with time zone NOT NULL,
    scheduled_arrival timestamp with time zone NOT NULL,
    departure_airport character(3) NOT NULL,
    arrival_airport character(3) NOT NULL,
    status character varying(20) NOT NULL,
    aircraft_code character(3) NOT NULL,
    actual_departure timestamp with time zone,
    actual_arrival timestamp with time zone,
    
    -- DENORMALIZED COLUMNS:
    departure_city jsonb NOT NULL, 
    arrival_city jsonb NOT NULL,
    aircraft_model jsonb NOT NULL,

    CONSTRAINT flights_pkey PRIMARY KEY (flight_id, scheduled_departure),
    CONSTRAINT flights_flight_no_scheduled_departure_key UNIQUE (flight_no, scheduled_departure)
) PARTITION BY RANGE (scheduled_departure);

-- Example Partitions (You would automate this in production via pg_partman)
CREATE TABLE bookings.flights_y2023m10 PARTITION OF bookings.flights FOR VALUES FROM ('2023-10-01') TO ('2023-11-01');
CREATE TABLE bookings.flights_y2023m11 PARTITION OF bookings.flights FOR VALUES FROM ('2023-11-01') TO ('2023-12-01');

-- INDEXES FOR FLIGHTS
-- Covering index for Flight Search (Pattern A)
CREATE INDEX idx_flights_search 
    ON bookings.flights (departure_airport, arrival_airport, scheduled_departure)
    INCLUDE (flight_no, status, departure_city, arrival_city, aircraft_model);

-- Index for checking flight status by flight number
CREATE INDEX idx_flights_flight_no 
    ON bookings.flights (flight_no, scheduled_departure);


-- ==============================================================================
-- 2. MODIFIED TICKET_FLIGHTS TABLE (The Ultimate Fact Table)
-- ==============================================================================
-- Merges boarding_passes, brings down passenger details and departure time.
-- Partitioned to align with flights.

DROP TABLE IF EXISTS bookings.boarding_passes CASCADE;
DROP TABLE IF EXISTS bookings.ticket_flights CASCADE;

CREATE TABLE bookings.ticket_flights
(
    ticket_no character(13) NOT NULL,
    flight_id integer NOT NULL,
    scheduled_departure timestamp with time zone NOT NULL, -- DENORMALIZED (for partitioning & filtering)
    
    -- Original ticket_flights data
    fare_conditions character varying(10) NOT NULL,
    amount numeric(10, 2) NOT NULL,
    
    -- DENORMALIZED from tickets (Pattern B)
    book_ref character(6) NOT NULL,
    passenger_id character varying(20) NOT NULL,
    passenger_name text NOT NULL,
    
    -- DENORMALIZED from boarding_passes (Pattern C)
    boarding_no integer, -- Nullable because they might not have checked in yet
    seat_no character varying(4),

    CONSTRAINT ticket_flights_pkey PRIMARY KEY (ticket_no, flight_id, scheduled_departure)
) PARTITION BY RANGE (scheduled_departure);

-- Example Partitions
CREATE TABLE bookings.ticket_flights_y2023m10 PARTITION OF bookings.ticket_flights FOR VALUES FROM ('2023-10-01') TO ('2023-11-01');
CREATE TABLE bookings.ticket_flights_y2023m11 PARTITION OF bookings.ticket_flights FOR VALUES FROM ('2023-11-01') TO ('2023-12-01');

-- INDEXES FOR TICKET_FLIGHTS
-- 1. Passenger Itinerary Lookup (Pattern B)
CREATE INDEX idx_tf_passenger_itinerary 
    ON bookings.ticket_flights (passenger_id, scheduled_departure DESC)
    INCLUDE (ticket_no, flight_id, seat_no, boarding_no, fare_conditions);

-- 2. Gate Agent Manifest Lookup (Pattern C)
CREATE INDEX idx_tf_flight_manifest 
    ON bookings.ticket_flights (flight_id)
    INCLUDE (passenger_name, seat_no, boarding_no, ticket_no);

-- 3. Booking Reference Lookup (For customer support)
CREATE INDEX idx_tf_book_ref 
    ON bookings.ticket_flights (book_ref);


-- ==============================================================================
-- 3. MATERIALIZED VIEW FOR ANALYTICS (Pattern D)
-- ==============================================================================
-- Pre-aggregates revenue and passenger counts by day and route.
-- Grain: Date + Route

CREATE MATERIALIZED VIEW bookings.mv_route_daily_revenue AS
SELECT 
    DATE(f.scheduled_departure) AS flight_date,
    f.departure_airport,
    f.arrival_airport,
    COUNT(tf.ticket_no) AS total_passengers,
    SUM(tf.amount) AS total_revenue
FROM bookings.flights f
JOIN bookings.ticket_flights tf 
    ON f.flight_id = tf.flight_id 
    AND f.scheduled_departure = tf.scheduled_departure
GROUP BY 
    DATE(f.scheduled_departure),
    f.departure_airport,
    f.arrival_airport
WITH DATA;

-- Index for fast dashboard filtering by date range or specific route
CREATE UNIQUE INDEX idx_mv_route_daily_revenue_pk 
    ON bookings.mv_route_daily_revenue (flight_date, departure_airport, arrival_airport);

CREATE INDEX idx_mv_route_daily_revenue_route 
    ON bookings.mv_route_daily_revenue (departure_airport, arrival_airport);

-- Note: You will need to set up a cron job or trigger to run:
-- REFRESH MATERIALIZED VIEW CONCURRENTLY bookings.mv_route_daily_revenue;

COMMIT;
```

### Summary of Performance Gains:
1. **Zero-Join Flight Boards:** `SELECT * FROM flights WHERE departure_airport = 'JFK' AND arrival_airport = 'LHR'` now returns everything needed for the UI (including cities and aircraft model) via an index-only/covering index scan.
2. **Zero-Join Manifests:** Gate agents querying `SELECT passenger_name, seat_no FROM ticket_flights WHERE flight_id = 123` hit a single table, instantly returning the manifest.
3. **Zero-Join Itineraries:** A user logging in triggers `SELECT * FROM ticket_flights WHERE passenger_id = '12345'`. No need to touch the `bookings` or `tickets` tables at all for read operations.
4. **Partition Pruning:** Because `scheduled_departure` was pushed down into `ticket_flights`, the database can ignore years of historical data when a user looks up their flight for next week.
