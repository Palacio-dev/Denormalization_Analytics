### Executive Summary

The provided schema represents a classic normalized OLTP/source system for a multi-channel retail environment (Store, Catalog, Web), heavily inspired by the TPC-DS benchmark. In its current 3NF-like state, analytical queries require expensive, high-volume joins between header tables, line-item tables, and return tables. 

To optimize for read-heavy analytical workloads, I am proposing a **Wide-Table Fact Architecture** combined with **Pre-Aggregated Rollups**. We will collapse the header and line-item tables into channel-specific wide tables, and left-join the returns data directly into the sales line items. This eliminates the two most expensive join hotspots (Header ↔ Line Item, and Sales ↔ Returns) at query time. We will rely on an hourly micro-batch ELT process to populate these tables, accepting a documented staleness window of up to 60 minutes in exchange for a 10x–50x improvement in read latency and significantly reduced compute costs.

**Identified Hotspots in Current Schema:**
1. **Join Hotspot:** `s_purchase` ↔ `s_purchase_lineitem` (1:N). Every line item requires a lookup to the header for the date and store ID.
2. **Join Hotspot:** `s_purchase_lineitem` ↔ `s_store_returns` (1:0..1). Calculating net sales requires massive outer joins.
3. **Aggregation Hotspot:** Summing `plin_quantity` and `plin_sale_price` across millions of rows grouped by `purc_purchase_date`.

### 2. Denormalization Patterns Justification

1. **Wide Read Tables (Pre-joined Header + Line Item + Returns)**
   * *Why:* Analytical engines (columnar) excel at scanning wide tables. By flattening the header (`s_purchase`) and the line item (`s_purchase_lineitem`), we eliminate a massive hash join. Furthermore, by left-joining `s_store_returns` into this wide table, we calculate "Net Sales" instantly at read time.
2. **Pre-Aggregated Rollups (`rollup_daily_channel_sales`)**
   * *Why:* Executive dashboards usually look at daily/weekly trends, not individual transactions. A daily rollup table reduces billions of rows to a few thousand, making dashboard load times sub-second.
3. **Data Type Optimization**
   * *Why:* The source schema uses `char(10)` for dates. The denormalized schema converts these to native `DATE` types to enable partition pruning and native date math.

### 4. Consistency Architecture & Implementation Plan

**Operational Constraints Assumed:** No triggers on the OLTP system. CDC (Change Data Capture) is available and streams to a staging area. Batch windows are available hourly.

**Data Lineage Example (`denorm_store_sales`):**
*   `purchase_date` ← `CAST(s_purchase.purc_purchase_date AS DATE)`
*   `gross_sales_amt` ← `s_purchase_lineitem.plin_quantity * s_purchase_lineitem.plin_sale_price`
*   `is_returned` ← `CASE WHEN s_store_returns.sret_purchase_id IS NOT NULL THEN TRUE ELSE FALSE END`

**Refresh Strategy (Hourly Micro-Batch ELT):**
1. **Append-Only Sales:** Sales headers and line items are generally immutable. We `INSERT` new sales into the wide tables hourly based on the `purchase_date`.
2. **Upserting Returns (The Complexity):** Returns happen *after* the sale. When a return arrives in staging, we must update the historical sale record. 
   * *Implementation:* We use an hourly `MERGE` statement. We match on `purchase_id` and `line_number`. If a match is found, we update the `return_*` columns.
3. **Rollup Refresh:** After the wide tables are updated, we rebuild the `rollup_daily_channel_sales` for the affected `sales_date` partitions using a `DELETE` + `INSERT` (or engine-native Materialized View refresh).

**Staleness Window:** Data in the read-optimized schema will be up to **60 minutes stale**. Business users must be informed that real-time intraday tracking requires querying the operational system, while the data warehouse is for finalized hourly/daily analytics.

### 5. Risk Analysis & Observability

| Risk Vector | Analysis & Mitigation |
| :--- | :--- |
| **Write Amplification** | **High Risk.** A return requires updating a massive historical fact table. *Mitigation:* Because the table is partitioned by `purchase_date`, the `MERGE` statement must be optimized to prune partitions. We must extract the original `purchase_date` of the return to ensure the `MERGE` only scans the specific partition where the sale occurred. |
| **Storage Growth** | **Medium Risk.** Denormalizing `store_id`, `customer_id`, and `purchase_date` onto every line item increases raw data size. *Mitigation:* Modern columnar databases use dictionary encoding and run-length encoding (RLE). Because `purchase_date` and `store_id` have low cardinality per block, compression will easily negate the storage penalty. |
| **Failure Modes (Orphaned Returns)** | **Low Risk.** A return might arrive in CDC before the sale record (due to distributed system race conditions). *Mitigation:* The ELT process must place orphaned returns into a Dead Letter Queue (DLQ) table and retry them in the next hourly batch. |

**Monitoring & Alerts Required:**
1. **Freshness Lag:** Alert if `MAX(purchase_date)` in `denorm_store_sales` is more than 2 hours behind `CURRENT_TIMESTAMP()`.
2. **Reconciliation Diffs:** Run a daily automated query comparing `SUM(sale_price)` in the source `s_purchase_lineitem` vs `SUM(sale_price)` in `denorm_store_sales`. Alert if variance > $0.00.
3. **Query Latency Regressions:** Monitor the P90 execution time of the daily rollup build. If it exceeds 15 minutes, the `MERGE` strategy may need to be replaced with a full partition overwrite strategy.
