# Executive Summary

This design transforms the existing normalized schema into a **Read-Optimized Physical Schema** targeting high-frequency analytical queries (e.g., "Average ENEM score by school," "Student performance trends," "Employment rate by location"). The core strategy is **Pre-Aggregation and Denormalization**, specifically creating a "Fact Table" (`fato_estados_escolar_anual`) that merges student demographics, school metrics, and ENEM performance into a single row per entity-per-year.

**Key Trade-offs:**
*   **Correctness:** Intentional staleness of up to **15 minutes** for denormalized columns. This allows batch refreshes during off-peak hours without impacting real-time transactional integrity of the source tables.
*   **Complexity:** Introduction of a dedicated refresh job and a reconciliation view to detect data drift between source and denormalized tables.
*   **Performance:** Expected **3x–5x latency reduction** for multi-table joins, as the optimizer no longer needs to traverse `matricula_ifb` $\to$ `aluno_ifb` $\to$ `curso_ifb` $\to$ `desempenho_enem`.

# 2. Denormalization Patterns & Justification

We will implement three specific patterns:

1.  **Pre-Aggregated Fact Table (`fato_estados_escolar_anual`)**:
    *   *Justification:* Replaces the most frequent join pattern (School + Year + Performance + Demographics).
    *   *Granularity:* One row per `co_escola_educacenso` + `nu_ano`.
    *   *Benefit:* Eliminates the need to join `desempenho_enem`, `indicadores_escolares`, and `matricula_ifb` for standard reporting.

2.  **Wide Read Snapshot (`snap_aluno_curso_ativo`)**:
    *   *Justification:* For queries requiring current student details (sex, race, disability status) linked to their active course.
    *   *Granularity:* One row per `id_aluno` (snapshot at refresh time).
    *   *Benefit:* Eliminates joins to `aluno_ifb`, `curso_ifb`, and `matricula_ifb` for ad-hoc profiling.

3.  **Materialized Geography Map (`geo_escola_map`)**:
    *   *Justification:* To avoid joining `escola` $\to$ `localizacao` (often slow due to string comparisons on `co_escola` or integer lookups on `id_loc`).
    *   *Granularity:* One row per `co_escola_educacenso` with `uf`, `municipio` pre-computed.

# 4. Consistency Architecture & Data Lineage

To ensure business correctness despite staleness, we define a strict lineage map.

| Source Column | Target Denormalized Column | Staleness Window | Refresh Trigger |
| :--- | :--- | :--- | :--- |
| `desempenho_enem.nu_ano` | `fato_estados_escolar_anual.nu_ano` | 0s (Exact match) | Daily Batch |
| `desempenho_enem.nu_media_cn` | `fato_estados_escolar_anual.nu_media_cn` | 15m (Batch lag) | Every 15 mins |
| `indicadores_escolares.nu_taxa_permanencia` | `fato_estados_escolar_anual.nu_taxa_permanencia` | 15m | Every 15 mins |
| `matricula_ifb.situacao_matricula` | `snap_aluno_curso_ativo.situacao_matricula` | 15m | Every 15 mins |
| `aluno_ifb.sexo` | `snap_aluno_curso_ativo.sexo` | 15m | Every 15 mins |
| `localizacao.uf` | `geo_escola_map.uf` | 15m | Every 15 mins |

**Reconciliation Mechanism:**
A view `view_reconcile_diff` will be created to alert on significant deviations:
```sql
CREATE VIEW public.view_reconcile_diff AS
SELECT 
    f.co_escola_educacenso,
    f.nu_ano,
    (SELECT AVG(nu_media_cn) FROM desempenho_enem d WHERE d.co_escola_educacenso = f.co_escola_educacenso AND d.nu_ano = f.nu_ano) as source_val,
    f.nu_media_cn as denormalized_val,
    ABS((SELECT AVG(nu_media_cn) ...) - f.nu_media_cn) as diff
FROM public.fato_estados_escolar_anual f
WHERE ABS((SELECT AVG(nu_media_cn) FROM desempenho_enem ...) - f.nu_media_cn) > 0.5; -- Threshold
```

# 5. Risk Analysis & Operational Plan

### Risk Matrix

| Risk Category | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Write Amplification** | `snap_aluno_curso_ativo` requires updating rows on every student change. | **Strategy:** Use `INSERT ... ON CONFLICT (id_aluno) DO UPDATE`. Limit refresh to peak hours (02:00-04:00). |
| **Lock Contention** | High volume of concurrent refresh jobs. | **Strategy:** Single refresh job scheduled via `pg_cron`. Use `SERIALIZABLE` isolation only during critical updates if needed. |
| **Storage Growth** | Wide tables consume more space than normalized. | **Strategy:** Enable `page_checksum` and `tablespace` with compression (if supported). Monitor size growth; archive old snapshots (>1 year) if policy allows. |
| **Failure Mode** | Refresh job fails silently. | **Strategy:** Alert on job failure. Use `pg_stat_progress_copy` or `EXPLAIN (ANALYZE, BUFFERS)` in the refresh script to log row counts. |
| **Query Latency Regression** | Queries hit the denormalized table but expect fresh data. | **Strategy:** Document in API/UI that data is "refreshed every 15 mins". Use a `refresh_timestamp` column in the denormalized table for debugging. |
