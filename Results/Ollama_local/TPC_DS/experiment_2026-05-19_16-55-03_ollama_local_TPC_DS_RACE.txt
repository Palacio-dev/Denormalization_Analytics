### Step 3: Evaluation of Common Denormalization Techniques

We will apply the following techniques to the specific model:

1.  **Adding Redundant Attributes (Materialized Joins):**
    *   *Application:* Instead of joining `s_purchase` and `s_purchase_lineitem`, we create a `s_purchase_detail` table that contains both the order ID and the line item details (price, quantity, coupon) in a single row.
    *   *Benefit:* Eliminates the `JOIN` operation entirely for aggregation queries.

2.  **Adding Derived Attributes (Pre-aggregation):**
    *   *Application:* Pre-calculate `total_sales`, `total_quantity`, and `return_rate` at the Order level if the granularity allows, or keep the row-level but ensure the columns are present for immediate use.
    *   *Benefit:* Avoids `SUM()` and `COUNT()` functions during query execution.

3.  **Collapsing Relations:**
    *   *Application:* Merge the three line-item tables (`s_purchase_lineitem`, `s_catalog_order_lineitem`, `s_web_order_lineitem`) into a unified `s_order_lineitem` table.
    *   *Benefit:* Reduces the complexity of queries that span multiple channels. A single `SELECT` can now query all channels.

4.  **Vertical Partitioning:**
    *   *Application:* While useful, in this specific TPC-H context, vertical partitioning is less critical than **horizontal denormalization** (widening the table) because the primary bottleneck is the *number of joins*, not the *width of a single table*. However, we will keep the schema relatively wide to avoid excessive memory usage on very large datasets.

**Decision:** We will primarily use **Collapsing Relations** (merging line items) and **Adding Redundant Attributes** (embedding order details into line items).

### Step 4: Designing Possible Denormalized Schemas

#### Option A: The "Unified Order" Approach
*   **Concept:** Create a single `s_order_lineitem` table that unifies Store, Catalog, and Web line items. Create a single `s_order` table that unifies Store, Catalog, and Web orders.
*   **Pros:** Extremely simple SQL.
*   **Cons:** Loses the distinction between Store and Catalog in the physical schema (though the `order_type` flag handles this). Might make it hard to audit historical changes to specific table definitions.

#### Option B: The "Wide Line Item" Approach
*   **Concept:** Keep the order structure normalized but make the `s_order_lineitem` table extremely wide, including all attributes from the parent order tables (store_id, customer_id, etc.) directly in the line item table.
*   **Pros:** Good for "Item Performance" queries.
*   **Cons:** Bad for "Order Level" queries (e.g., "Total Orders per Customer") because you have to join the line item table back to an order table to get the count.

#### Option C: The "Hybrid Analytical" Approach (Selected)
*   **Concept:**
    1.  **Collapse Line Items:** Merge `s_purchase_lineitem`, `s_catalog_order_lineitem`, and `s_web_order_lineitem` into one massive `s_order_lineitem` table.
    2.  **Embed Order Context:** Include `order_id`, `order_date`, `order_type` (Store/Catalog/Web), and `customer_id` directly in the line item table.
    3.  **Embed Product Context:** Keep product attributes in the line item table (as they are already there).
    4.  **Separate Order Header:** Keep a lightweight `s_order` table only for order-level metadata (if needed), but ideally, the line item table is wide enough to serve as the primary source for almost everything.
    5.  **Unify Returns:** Create a single `s_return` table that merges Store and Catalog returns, embedding the original order ID and line item ID.

This approach maximizes the utility of the **Read** path by minimizing the number of tables touched.

---

### Step 5: Evaluation and Selection

*   **Option A** is too aggressive; it destroys the ability to distinguish between channels if not handled carefully with flags.
*   **Option B** causes cache thrashing for order-level queries.
*   **Option C** is the most consistent. It treats the **Line Item** as the "grain" of truth for analytical queries. Most business intelligence questions are "What did we sell of Item X in Jan?" (Line Item grain) rather than "How many orders did we have?" (Order grain), and the Line Item table in Option C can hold both the grain and the context.

**Selected Strategy:** Implement **Option C**. We will create a unified schema where the Line Item table is the primary engine, and the Order table is minimized.

### Summary of Improvements

1.  **Eliminated Joins for Line Items:** Queries asking for "Total Sales by Item" or "Average Price by Customer" no longer need to join `s_order` and `s_order_lineitem`. The context is embedded.
2.  **Unified Channels:** A single `SELECT` against `s_order_lineitem` now handles Store, Catalog, and Web data, allowing easy comparison and aggregation across channels.

4.  **Scalability:** While the tables are wider, modern columnar databases (or even well-indexed row stores) handle this well for read workloads. The reduction in join complexity usually outweighs the cost of scanning slightly more columns.
