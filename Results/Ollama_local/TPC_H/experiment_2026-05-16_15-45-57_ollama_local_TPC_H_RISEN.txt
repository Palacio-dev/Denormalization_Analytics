### Quick Comparison

| Aspect | Normalized Model | Denormalized Model |
|--------|------------------|---------------------|
| **Storage** | Multiple tables, minimal redundancy | Single wide table, more data |
| **Read Performance** | Requires JOINs, can be slower | Faster queries, fewer JOINs |
| **Write Performance** | More inserts/updates needed | Simpler, faster writes |
| **Data Integrity** | Enforced via foreign keys | Manual consistency required |
| **Use Case** | Transactional, complex queries | Analytics, reporting, dashboards |

### Example 5: Student Performance Tracking

Here's a new schema following your pattern, focusing on student performance across schools and time:

#### Normalized Model
```sql
BEGIN;

CREATE TABLE IF NOT EXISTS public."Schools" (
    "School_ID" serial PRIMARY KEY,
    "School_Name" character varying(255),
    "City" character varying(100),
    "State" character varying(2)
);

CREATE TABLE IF NOT EXISTS public."Students" (
    "Student_ID" serial PRIMARY KEY,
    "Student_Name" character varying(100),
    "School_ID" integer NOT NULL,
    "Enrollment_Date" date,
    CONSTRAINT "fk_students_school" FOREIGN KEY ("School_ID") REFERENCES public."Schools" ("School_ID")
);

CREATE TABLE IF NOT EXISTS public."Subjects" (
    "Subject_ID" serial PRIMARY KEY,
    "Subject_Name" character varying(100)
);

CREATE TABLE IF NOT EXISTS public."Grades" (
    "Grade_ID" serial PRIMARY KEY,
    "Student_ID" integer NOT NULL,
    "Subject_ID" integer NOT NULL,
    "Grade_Value" numeric(4, 2),
    "Year" integer,
    "Semester" integer,
    CONSTRAINT "fk_grades_student" FOREIGN KEY ("Student_ID") REFERENCES public."Students" ("Student_ID"),
    CONSTRAINT "fk_grades_subject" FOREIGN KEY ("Subject_ID") REFERENCES public."Subjects" ("Subject_ID")
);

CREATE TABLE IF NOT EXISTS public."Attendance" (
    "Attendance_ID" serial PRIMARY KEY,
    "Student_ID" integer NOT NULL,
    "Year" integer,
    "Semester" integer,
    "Attendance_Rate" numeric(5, 2),
    CONSTRAINT "fk_attendance_student" FOREIGN KEY ("Student_ID") REFERENCES public."Students" ("Student_ID")
);

CREATE TABLE IF NOT EXISTS public."Discipline" (
    "Discipline_ID" serial PRIMARY KEY,
    "Student_ID" integer NOT NULL,
    "Year" integer,
    "Semester" integer,
    "Incidents" integer,
    CONSTRAINT "fk_discipline_student" FOREIGN KEY ("Student_ID") REFERENCES public."Students" ("Student_ID")
);

CREATE TABLE IF NOT EXISTS public."Teacher_Assignments" (
    "Assignment_ID" serial PRIMARY KEY,
    "Teacher_ID" integer,
    "School_ID" integer NOT NULL,
    "Subject_ID" integer NOT NULL,
    "Year" integer,
    CONSTRAINT "fk_teachers_school" FOREIGN KEY ("School_ID") REFERENCES public."Schools" ("School_ID"),
    CONSTRAINT "fk_teachers_subject" FOREIGN KEY ("Subject_ID") REFERENCES public."Subjects" ("Subject_ID")
);

END;
```

### Recommendations for Your Use Case

1. **Start Normalized** — Use normalized schemas when:
   - You need strong data integrity
   - You expect frequent updates/inserts
   - You're building transactional systems

2. **Denormalize Later** — Consider denormalization when:
   - You're building analytical dashboards
   - Read performance is critical
   - You need historical snapshots

3. **Hybrid Approach** — Many production systems use:
   - Normalized tables for transactional operations
   - Materialized views or denormalized tables for reporting
