# Executive Summary

The current schema is a standard 3NF design optimized for write efficiency and ACID consistency. However, for a read-heavy workload involving customer order history, product pricing, and order summaries, the existing design forces costly join operations and prevents effective index pruning on aggregate metrics. This design introduces a **Denormalized Read-Optimized Schema** that pre-joins dimensions into wide tables and pre-aggregates order details into a fact table.

The solution preserves business correctness by maintaining the original tables as the Source of Truth (SoT) and introducing a dedicated **`orders_fact_snapshot`** table. This table holds the full expanded order details (Order ID, Customer Name, Product Name, Price, Quantity, Total) and is refreshed via a batch job rather than real-time triggers. We explicitly document a **5-minute staleness window** to avoid write amplification and lock contention, which is acceptable for most analytical reporting while still providing near-real-time insights.

# 1. Model-to-Workload Mapping

Based on typical analytics queries against this domain, the access patterns are characterized by:
*   **Join Hotspots:** Queries frequently join `orders` and `order_details` on `order_id`, then join with `customers` and `products`.
*   **Aggregation Hotspots:** Queries often need `SUM(quantity * price)`, `COUNT(DISTINCT customer_id)`, and date-range aggregations.
*   **Filter Hotspots:** Filters on `order_date`, `customer_id`, and `product_name` are common.

**Access-Pattern Matrix:**

| Query Class | Primary Tables Touched | Join Complexity | Aggregation Needs | Latency Impact (Current) |
| :--- | :--- | :--- | :--- : | :--- |

| **Customer Profile** | `customers`, `orders` (grouped by customer) | 2 Way Join | `COUNT`, `AVG(price)` | **Medium** |

# 2. Denormalization Patterns & Justification

We will implement three specific patterns to address the identified hotspots:

1.  **Wide Read Table (`orders_fact_snapshot`):**
    *   *Pattern:* Pre-joined fact table containing all columns from `orders`, `order_details`, `customers`, and `products`.
    *   *Justification:* Eliminates the need for runtime joins. A single index on `order_id` (or date range) retrieves the entire row. Ideal for "Order Summary" queries.
    *   *Trade-off:* Increased storage (approx. 2x `order_details` size) and write latency during refresh.

2.  **Pre-Aggregated Rollup (`orders_daily_rollup`):**
    *   *Pattern:* Daily aggregated table (Date, Customer ID, Product ID, Total Qty, Total Revenue).
    *   *Justification:* Drastically reduces data volume for time-series analytics. Allows scanning a 100MB table instead of 10GB.
    *   *Trade-off:* Requires a dedicated aggregation job; not suitable for sub-daily queries.

3.  **Snapshot Table (`customers_active_snapshot`):**
    *   *Pattern:* A table holding only `customer_id`, `name`, `email`, `address` with a `last_seen_order_date`.
    *   *Justification:* Optimizes queries filtering for "active" customers without joining the `orders` table.

# 4. Consistency Architecture & Data Lineage

| Artifact | Source of Truth (SoT) | Staleness Window | Refresh Mechanism |
| :--- | :--- | :--- | :--- |
| `orders_fact_snapshot` | `orders` + `order_details` + `customers` + `products` | **5 Minutes** | Scheduled Job (T+5min) |
| `orders_daily_rollup` | `orders_fact_snapshot` (or raw tables) | **1 Hour** | Batch Aggregation Job (Daily or Hourly) |
| `customers_active_snapshot` | `customers` + `orders` | **5 Minutes** | Scheduled Job (T+5min) |

**Lineage Logic:**
1.  **Insert:** New rows in `orders_fact_snapshot` are inserted only if the `order_id` does not exist. If it exists, update the row (upsert logic).
2.  **Delete:** `orders_fact_snapshot` does **not** use cascading deletes. If an order is voided in the SoT, the denormalized table retains the data until the next refresh cycle (unless a specific "voided" flag is added).
3.  **Reconciliation:** A nightly job compares checksums (e.g., `COUNT(*)` or `SUM(total_amount)`) between SoT and Denormalized tables. Diffs > 0.001% trigger an alert.

# 5. Risk Analysis & Operational Plan

### Risk Matrix

| Risk Category | Impact | Mitigation Strategy |
| :--- | :--- | :--- |
| **Write Amplification** | High write latency during refresh; potential lock contention on `orders_fact_snapshot`. | Use `INSERT ... ON CONFLICT (order_id) DO UPDATE` with a lightweight upsert. Limit batch size (e.g., 10k rows). |
| **Data Staleness** | Users see slightly outdated prices or customer names. | **Explicit:** Documented 5-min window. Critical alerts if SoT changes occur outside window. |
| **Storage Growth** | ~2x storage increase for fact table due to denormalized text columns. | Enable compression (e.g., `zstd` in PostgreSQL). Monitor storage growth; archive data older than 1 year. |
| **Write Failure** | Refresh job fails, leaving data inconsistent. | Idempotent design (Upsert). Retry logic in job scheduler. Reconciliation job detects missing rows. |
| **Query Latency Regression** | New queries accidentally join SoT instead of using new tables. | **Observability:** Add query logging. Alert if `orders_fact_snapshot` usage drops below 80% of expected load. |
