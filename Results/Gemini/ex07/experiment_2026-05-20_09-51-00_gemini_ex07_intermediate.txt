As a senior database performance engineer, I have analyzed your normalized physical schema. While your current 3NF (Third Normal Form) model is excellent for ensuring data integrity during **writes** (OLTP), it creates a "Join Hell" for **read-heavy** workloads (OLAP/Dashboards/APIs). 

To fetch a single Power Plant (`Usina`) with its owner, river, and state, the database engine must execute a **6-table join** (`Usina` → `Proprietarios` + `Rios` + `Localizacoes` → `Municipios` (x2) → `Estados`). This causes high CPU usage, random disk I/O, and unacceptable latency at scale.

Here is the strategy to denormalize your schema into a high-performance **Read Model**, assuming you maintain the normalized tables for writes and synchronize them to this read model via CDC (Change Data Capture), triggers, or asynchronous jobs.

---

### 1. Workload-Driven Analysis

Based on typical power plant (Usina) domain workloads, I have identified three critical read patterns and their bottlenecks:

*   **Pattern A: Geographic & Status Filtering (List/Search View)**
    *   *Query:* "Show me all operational plants in the state of 'SP' or on the 'Paraná' river, including owner names."
    *   *Bottleneck:* Joining `Localizacoes` to `Municipios` (twice, for border rivers) and `Estados` requires heavy hash/nested-loop joins.
    *   *Solution:* Flatten the location hierarchy and lookup tables directly into a "Wide Table".
*   **Pattern B: Map Visualization (Spatial/UI View)**
    *   *Query:* "Fetch the coordinates (lat/long), name, and power output of all plants in a specific stage to plot on a map."
    *   *Bottleneck:* Scanning the whole table and performing lookups just to get 4 columns.
    *   *Solution:* Create a **Covering Index** so the database can serve the query directly from memory (Index-Only Scan) without touching the table heap.
*   **Pattern C: Executive Dashboards (Aggregations)**
    *   *Query:* "What is the total power capacity (`potkw`) and count of plants, grouped by State and Stage?"
    *   *Bottleneck:* Aggregating millions of rows on the fly across 4 joined tables.
    *   *Solution:* Precompute these metrics using a Materialized View with a specific grain.

---

### 2. Denormalization Design

To achieve sub-millisecond read latency, we will implement the following:

1.  **The "Wide" Fact Table (`usina_read_model`):** 
    *   We will duplicate `proprietario` (Owner Name), `rio` (River Name), and the names/states of the municipalities directly into the `Usina` table.
    *   *Why?* These are Slowly Changing Dimensions (SCD Type 1). Rivers and States don't change names. Owners change rarely. The cost of updating redundant data is vastly outweighed by the read performance gains of zero-join queries.
2.  **Partitioning:**
    *   We will partition the wide table by `uf_principal` (State). 
    *   *Why?* Geographic queries usually filter by state. Partition pruning will allow the engine to scan only the data for a specific state, ignoring the rest of the country.
3.  **Precomputed Aggregate (`mv_usina_stats_estado`):**
    *   A Materialized View with a grain of `State + Stage`.
    *   *Why?* Dashboard queries will hit this tiny view (max ~100 rows) instead of aggregating the base tables, dropping latency from seconds to microseconds.
4.  **Index Strategy:**
    *   **Composite Indexes:** Aligned to `WHERE` and `ORDER BY` clauses.
    *   **Covering Indexes (`INCLUDE`):** To allow Index-Only Scans for the map UI.

### 3. New Physical Schema (DDL)

*Note: This DDL is written for PostgreSQL, utilizing its native partitioning, materialized views, and `INCLUDE` index features.*

```sql
-- ==============================================================================
-- 1. THE WIDE TABLE (READ MODEL) - Partitioned by State
-- ==============================================================================
-- This table flattens Usina, Proprietarios, Rios, Localizacoes, Municipios, and Estados.

CREATE TABLE public.usina_read_model (
    id_usina INTEGER NOT NULL,
    nome VARCHAR NOT NULL,
    potkw FLOAT,
    potfis FLOAT,
    iniciop DATE,
    atualizacao DATE,
    lat DOUBLE PRECISION,
    long DOUBLE PRECISION,
    estagio VARCHAR,
    
    -- Denormalized Lookups (Zero-Join attributes)
    proprietario_nome VARCHAR,
    rio_nome VARCHAR,
    
    -- Flattened Location Hierarchy
    -- Note: Consolidated lat/long here, dropped the redundant ones from Localizacoes
    municipio_principal_nome VARCHAR,
    uf_principal CHAR(2) NOT NULL, -- Used as Partition Key
    municipio_secundario_nome VARCHAR,
    uf_secundario CHAR(2),
    
    PRIMARY KEY (id_usina, uf_principal) -- PK must include partition key in Postgres
) PARTITION BY LIST (uf_principal);

-- Create Partitions for heavily queried states (Example: SP, MG, PR, plus a default)
CREATE TABLE public.usina_read_model_sp PARTITION OF public.usina_read_model FOR VALUES IN ('SP');
CREATE TABLE public.usina_read_model_mg PARTITION OF public.usina_read_model FOR VALUES IN ('MG');
CREATE TABLE public.usina_read_model_pr PARTITION OF public.usina_read_model FOR VALUES IN ('PR');
CREATE TABLE public.usina_read_model_default PARTITION OF public.usina_read_model DEFAULT;


-- ==============================================================================
-- 2. INDEXING STRATEGY FOR THE WIDE TABLE
-- ==============================================================================

-- Index for Pattern A: Filtering by Stage and ordering by Power (Pagination/Lists)
CREATE INDEX idx_usina_read_estagio_potkw 
ON public.usina_read_model (estagio, potkw DESC);

-- Index for Pattern A: Filtering by River (Common hydrological query)
CREATE INDEX idx_usina_read_rio 
ON public.usina_read_model (rio_nome);

-- Index for Pattern B: Map Visualization (Covering Index / Index-Only Scan)
-- Query: SELECT lat, long, nome, potkw FROM usina_read_model WHERE estagio = 'Operação';
-- The INCLUDE clause puts these columns in the index leaf nodes, preventing table heap lookups.
CREATE INDEX idx_usina_read_map_covering 
ON public.usina_read_model (estagio) 
INCLUDE (lat, long, nome, potkw);


-- ==============================================================================
-- 3. PRECOMPUTED AGGREGATE (MATERIALIZED VIEW)
-- ==============================================================================
-- Grain: State + Stage
-- Purpose: Serve executive dashboards instantly.

CREATE MATERIALIZED VIEW public.mv_usina_stats_estado AS
SELECT 
    uf_principal AS uf,
    estagio,
    COUNT(id_usina) AS qtd_usinas,
    SUM(potkw) AS total_potkw,
    SUM(potfis) AS total_potfis,
    MAX(atualizacao) AS ultima_atualizacao
FROM 
    public.usina_read_model
GROUP BY 
    uf_principal, 
    estagio;

-- Unique index required to allow CONCURRENTLY refreshes (zero downtime updates)
CREATE UNIQUE INDEX idx_mv_usina_stats_unique 
ON public.mv_usina_stats_estado (uf, estagio);

-- Index to quickly filter the dashboard by state
CREATE INDEX idx_mv_usina_stats_uf 
ON public.mv_usina_stats_estado (uf);

```

### How to maintain correctness (The Implementation Strategy):
Because you care about correctness for business-critical fields, **do not drop your original schema**. 
1. Your application should continue to `INSERT`/`UPDATE` into the normalized tables (`Usina`, `Localizacoes`, etc.) to guarantee ACID compliance and avoid update anomalies.
2. You should populate the `usina_read_model` using a background process. In PostgreSQL, this is typically done via a Database Trigger on the `Usina` table that does an `UPSERT` into the read model, or via a CDC tool like Debezium if you are using an event-driven microservices architecture.
3. Refresh the materialized view periodically (e.g., via a cron job running `REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_usina_stats_estado;`) so dashboard reads never block writes.
