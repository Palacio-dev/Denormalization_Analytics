# Consistency Architecture

### Data Lineage

| Denormalized Column | Source Table(s) | Refresh Mechanism |
|---------------------|------------------|--------------------|
| `orders_denorm.*` | `orders`, `order_details`, `products`, `categories`, `customers`, `employees` | Incremental refresh on order update |
| `sales_summary_by_customer.*` | `orders`, `order_details` | Daily rollup job |
| `current_orders.*` | `orders` | Snapshot on order status change |
| `fact_sales.*` | `orders`, `order_details` | Batch ETL or CDC-based sync |

### Refresh Strategy

- **Triggers**: Not allowed (per constraints)
- **CDC**: Preferred (if available) for incremental sync
- **Batch Jobs**: Fallback (daily/hourly rollups)
- **Staleness**: Explicitly acceptable for rollups (e.g., daily summary is stale up to 24h)

# Risk Analysis

| Risk | Description | Mitigation |
|------|-------------|------------|
| Write Amplification | Denormalized tables increase write volume | Use async batch or CDC to minimize impact |
| Lock Contention | High concurrency on refresh jobs | Use table partitioning, time-based writes, or staging tables |
| Failure Modes | ETL job failure, inconsistent state | Add checksums, reconciliation checks, alerts |
| Backfill Runtime | Large historical sync | Chunked processing, parallel jobs |
| Storage Growth | Denormalized tables increase size | Monitor and archive old data |
| Query Regressions | Denormalized tables may be misused | Enforce usage guidelines, monitor query patterns |

# Monitoring & Observability

- **Freshness Lag**: Monitor time since last refresh for each table
- **Reconciliation Checks**: Compare rollups to source queries
- **Query Latency**: Compare before/after for target queries
- **Alerts**:
  - Stale data (>1d for rollups)
  - Job failure notifications
  - Query latency degradation

# Implementation Plan (High-Level)

1. **Stage 1: Schema Creation**
   - Create all new tables in a `readstore` schema
   - Do not expose to production yet

2. **Stage 2: Initial Load**
   - Backfill data using batch jobs or CDC pipeline
   - Validate data integrity via checksums

3. **Stage 3: Refresh Jobs**
   - Set up incremental refresh jobs (hourly/daily)
   - Monitor for lag/freshness

4. **Stage 4: Enable Reads**
   - Point dashboards and reports to new tables
   - Shadow reads from old schema for validation

5. **Stage 5: Reversibility**
   - Maintain old schema in parallel
   - Switch back if issues arise
