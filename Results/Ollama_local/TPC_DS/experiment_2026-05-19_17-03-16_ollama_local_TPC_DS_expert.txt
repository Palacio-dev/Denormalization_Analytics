# Executive Summary

This design proposes a **Read-Optimized Denormalized Schema** targeting the TPC-H workload patterns inherent in your current normalized `s_*` tables. The primary goal is to eliminate expensive joins between transactional tables (orders, line items, returns) and dimension/aggregation tables (inventory, catalog, promotions) for analytical queries.

We will introduce three key artifacts:
1.  **`s_order_lineitem_flat`**: A denormalized wide-table combining `s_purchase_lineitem` and `s_catalog_order_lineitem` (plus web equivalents) into a single table with `order_id` as the PK. This eliminates the most frequent join pattern in sales reporting.
2.  **`s_return_summary_rollup`**: A pre-aggregated materialized view table for returns, grouping by `(store_id, item_id, date)` to answer "what did we sell back?" queries in milliseconds instead of seconds.
3.  **`s_inventory_snapshot`**: A daily snapshot table for `s_inventory` to avoid complex time-travel queries, assuming daily reconciliation.

**Correctness Strategy**: We define a **15-minute staleness window** for the denormalized tables. Data is refreshed via a background job (CDC + Batch) rather than triggers. Reconciliation diffs will be monitored; if a discrepancy exceeds $0.01, an alert is triggered to investigate the source of truth.

# 2. Denormalization Patterns & Justification

### Pattern A: The "Flat Order" Table (Wide Read)
*   **Pattern**: Combine all order types (Store, Catalog, Web) and their line items into a single table with a unified `order_id`.
*   **Justification**: Eliminates the need to join `s_purchase`, `s_catalog_order`, and `s_web_order` plus their respective line item tables.
*   **Staleness**: Real-time (via CDC) or near real-time (5-min batch).

### Pattern B: The "Return Rollup" Table (Pre-Aggregation)
*   **Pattern**: Aggregate `s_store_returns` and `s_catalog_returns` into a summary table.
*   **Justification**: Queries asking "Total return amount per store per day" currently require a GROUP BY across millions of rows. A pre-aggregated table allows O(1) retrieval.
*   **Staleness**: 15-minute lag (refreshed every 15 mins).

### Pattern C: The "Inventory Snapshot" Table
*   **Pattern**: A denormalized table representing inventory at a specific timestamp (e.g., end of day).
*   **Justification**: Avoids calculating `qty_on_hand` at a specific historical date by joining with `s_inventory` (which is already partitioned by date but requires a join for time travel).
*   **Staleness**: Daily (refreshed at 23:59).

-- Table: s_order_lineitem_flat
-- Purpose: Denormalized wide read for all order channels.
-- Staleness: Near Real-Time (via CDC) or 5-min Batch.
create table s_order_lineitem_flat

    store_id                  char(16),              -- Nullable for Catalog/Web

# 4. Consistency Architecture & Data Lineage

To ensure correctness while accepting measured staleness:

### Data Lineage Map
| Source Table | Denormalized Artifact | Transformation Logic | Staleness Window |
| :--- | :--- | :--- | :--- |
| `s_purchase_lineitem` + `s_catalog_order_lineitem` + `s_web_order_lineitem` | `s_order_lineitem_flat` | Union operation on `order_id` and `line_number`. Handle nulls for missing channel columns. | **5 Minutes** (Batch) |
| `s_store_returns` + `s_catalog_returns` | `s_return_summary_rollup` | `GROUP BY` (store_id, item_id, return_date) on Amounts and Quantities. | **15 Minutes** |
| `s_inventory` | `s_inventory_snapshot` | `SELECT * FROM s_inventory WHERE date = 'YYYY-MM-DD'` | **24 Hours** |

### Reconciliation Strategy
Since triggers are not allowed (per your constraint context), we use **Periodic Reconciliation Jobs**:
1.  **Frequency**: Run every 15 minutes for Order/Return tables; Daily for Inventory.
2.  **Logic**:
    *   Fetch a sample of 10,000 random rows from the source tables.
    *   Join them with the denormalized tables.
    *   Compare `SUM(quantity)` and `SUM(price)`.
3.  **Failure Mode**: If `|Source_Sum - Denorm_Sum| > 0.01`, trigger `ALERT_DATA_INTEGRITY`.
4.  **Correctness**: The denormalized data is considered "Good Enough" for reporting unless an alert fires. For financial audits, revert to the normalized source tables.

# 5. Risk Analysis & Operational Plan

### Risk Matrix

| Risk Category | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Write Amplification** | Inserting rows into the flat table requires calculating `order_line_pk` and `line_total`. | Use a `GENERATED ALWAYS AS IDENTITY` column for `order_line_pk`. Calculate `line_total` in the ETL job, not in the application. |
| **Lock Contention** | High write volume during peak hours could block the denormalized table. | Use **Partitioning** (`order_date_partition`). Writes only lock the partition for the current day, not the whole table. |
| **Backfill Runtime** | Loading historical data into the flat table is expensive. | **Phased Rollout**: Start with the last 30 days. Once stable, backfill older data in parallel batches. |
| **Storage Growth** | Wide tables store more data per row than normalized tables. | **Compression**: Enable Zstd or Snappy compression. `numeric(7,2)` and `char(16)` compress well. |
| **Staleness** | Users see slightly outdated data. | **UI Indicators**: Add a "Data Freshness" badge in the application. If lag > 10 mins, show a warning. |

### Implementation Plan (Safe & Reversible)

1.  **Phase 1: Validation (Week 1)**
    *   Create the denormalized tables as **empty**.
    *   Run the ETL pipeline on a small subset (last 24 hours).
    *   Verify row counts and sample data against the normalized source.
    *   *Success Criteria*: No reconciliation diffs > $0.01.

2.  **Phase 2: Read-Only Migration (Week 2)**
    *   Update existing read queries to use the new tables.
    *   Keep old queries running in a read-replica or shadow mode to compare latency.
    *   *Success Criteria*: Query latency improved by >50% on top 10 queries.

3.  **Phase 3: Write Enablement (Week 3)**
    *   Enable CDC listeners or scheduled jobs to populate the tables.
    *   Start the reconciliation job.
    *   *Success Criteria*: Reconciliation job runs successfully with no alerts.

4.  **Phase 4: Decommissioning (Week 4+)**
    *   Once the denormalized tables are stable and verified for 2 weeks, drop the old normalized tables (or keep them for archival only).
    *   *Note*: Do not drop immediately; keep them as "audit trail" until the denormalized tables have been in production for 30 days.
