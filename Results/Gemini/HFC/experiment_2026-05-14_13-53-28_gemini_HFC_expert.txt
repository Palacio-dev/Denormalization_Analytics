### Executive Summary

To achieve measurable read performance improvements for your botanical collection (Herbarium) system, we must decouple the highly normalized OLTP write model from the read-heavy search and analytical workloads. Currently, rendering a single specimen (`tombo`) with its full taxonomic hierarchy, geographical data, and photos requires joining upwards of 15 tables. Under concurrent load, this causes high CPU utilization, buffer cache thrashing, and unacceptable query latency.

I propose a **CQRS-lite architecture** utilizing PostgreSQL **Materialized Views with Concurrent Refresh**. We will introduce a "Wide Read Table" (`mv_tombos_catalog`) that flattens the taxonomy and geography, and pre-aggregates 1:N relationships (photos, identifiers) into `JSONB` documents. This reduces the most common queries from O(15) joins to O(1) single-table lookups. We will also introduce a pre-aggregated rollup table for dashboard analytics. This approach requires **zero changes to your existing OLTP schema**, ensuring absolute business correctness, safety, and immediate reversibility.

### 2. Denormalization Patterns Chosen

1.  **Wide Read Table (`mv_tombos_catalog`)**: We will flatten the 1:1 and N:1 relationships (Taxonomy, Geography, Ecology) into a single row per `tombo`. 
    *   *Justification:* Eliminates join overhead for 95% of user-facing queries. B-Tree indexes on this flat table will allow rapid multi-column filtering.
2.  **JSONB Pre-aggregation for 1:N**: Photos and Identifiers will be aggregated into JSONB arrays within the wide table.
    *   *Justification:* PostgreSQL handles JSONB highly efficiently. This allows the API to fetch a complete specimen document in a single disk read, mimicking a NoSQL document store while staying in Postgres.
3.  **Materialized Views (Engine Equivalent)**: We will use PostgreSQL `MATERIALIZED VIEW`.
    *   *Justification:* No triggers required (avoids write-amplification and lock contention on the OLTP tables). We can use `REFRESH MATERIALIZED VIEW CONCURRENTLY`, which updates the read models in the background without blocking `SELECT` queries.

### 3. Physical Design Details (DDL)

*Assumption: The `cidades` table was missing from the provided DDL but is referenced by `cidade_id`. I have assumed its existence for the joins.*

```sql
BEGIN;

-- ============================================================================
-- 1. WIDE READ TABLE (MATERIALIZED VIEW) FOR CATALOG SEARCH & DETAIL VIEW
-- ============================================================================

CREATE MATERIALIZED VIEW public.mv_tombos_catalog AS
SELECT 
    t.hcf AS tombo_hcf,
    t.numero_coleta,
    t.data_tombo,
    t.data_coleta_ano,
    t.situacao,
    t.ativo,
    t.latitude,
    t.longitude,
    
    -- Flattened Taxonomy
    r.nome AS reino_nome,
    f.nome AS familia_nome,
    g.nome AS genero_nome,
    e.nome AS especie_nome,
    t.nome_cientifico AS nome_cientifico_original,
    
    -- Flattened Geography & Ecology
    -- (Assuming 'cidades' exists based on schema references)
    cid.nome AS cidade_nome,
    est.sigla AS estado_sigla,
    p.nome AS pais_nome,
    sol.nome AS solo_nome,
    rel.nome AS relevo_nome,
    veg.nome AS vegetacao_nome,
    
    -- Flattened Actors
    h.sigla AS herbario_sigla,
    h.nome AS herbario_nome,
    
    -- Pre-aggregated 1:N Relationships (JSONB)
    COALESCE(
        (SELECT jsonb_agg(jsonb_build_object(
            'id', tf.id,
            'caminho_foto', tf.caminho_foto,
            'em_vivo', tf.em_vivo
        )) FROM public.tombos_fotos tf WHERE tf.tombo_hcf = t.hcf), 
    '[]'::jsonb) AS fotos_json,

    COALESCE(
        (SELECT jsonb_agg(jsonb_build_object(
            'nome', iden.nome,
            'ordem', ti.ordem
        )) 
        FROM public.tombos_identificadores ti 
        JOIN public.identificadores iden ON ti.identificador_id = iden.id 
        WHERE ti.tombo_hcf = t.hcf), 
    '[]'::jsonb) AS identificadores_json

FROM public.tombos t
LEFT JOIN public.familias f ON t.familia_id = f.id
LEFT JOIN public.reinos r ON f.reino_id = r.id
LEFT JOIN public.generos g ON t.genero_id = g.id
LEFT JOIN public.especies e ON t.especie_id = e.id
LEFT JOIN public.herbarios h ON t.entidade_id = h.id
LEFT JOIN public.solos sol ON t.solo_id = sol.id
LEFT JOIN public.relevos rel ON t.relevo_id = rel.id
LEFT JOIN public.vegetacoes veg ON t.vegetacao_id = veg.id
-- Assuming standard geography joins based on your schema
LEFT JOIN public.cidades cid ON t.cidade_id = cid.id
LEFT JOIN public.estados est ON cid.estado_id = est.id
LEFT JOIN public.paises p ON est.pais_id = p.id
WHERE t.ativo = true AND t.rascunho = false;

-- ============================================================================
-- INDEXING STRATEGY FOR READ OPTIMIZATION
-- ============================================================================

-- UNIQUE index is MANDATORY to allow CONCURRENT refreshes (zero downtime)
CREATE UNIQUE INDEX idx_mv_tombos_catalog_hcf ON public.mv_tombos_catalog (tombo_hcf);

-- B-Tree indexes for standard filtering (Catalog Search)
CREATE INDEX idx_mv_tombos_tax_familia ON public.mv_tombos_catalog (familia_nome);
CREATE INDEX idx_mv_tombos_tax_especie ON public.mv_tombos_catalog (especie_nome);
CREATE INDEX idx_mv_tombos_geo_estado ON public.mv_tombos_catalog (estado_sigla);
CREATE INDEX idx_mv_tombos_ano ON public.mv_tombos_catalog (data_coleta_ano);

-- Optional: GIN Index if you want to search inside the JSONB (e.g., find by specific identifier)
CREATE INDEX idx_mv_tombos_identificadores_gin ON public.mv_tombos_catalog USING GIN (identificadores_json);


-- ============================================================================
-- 2. PRE-AGGREGATED ROLLUP FOR DASHBOARDS
-- ============================================================================

CREATE MATERIALIZED VIEW public.mv_tombos_stats_rollup AS
SELECT 
    f.nome AS familia_nome,
    est.sigla AS estado_sigla,
    t.data_coleta_ano,
    COUNT(t.hcf) AS total_tombos
FROM public.tombos t
LEFT JOIN public.familias f ON t.familia_id = f.id
LEFT JOIN public.cidades cid ON t.cidade_id = cid.id
LEFT JOIN public.estados est ON cid.estado_id = est.id
WHERE t.ativo = true
GROUP BY f.nome, est.sigla, t.data_coleta_ano;

CREATE UNIQUE INDEX idx_mv_tombos_stats_uk ON public.mv_tombos_stats_rollup (familia_nome, estado_sigla, data_coleta_ano);

COMMIT;
```

### 4. Consistency Architecture & Refresh Strategy

**Data Lineage:**
`public.tombos` (and related normalized tables) remain the absolute Source of Truth. Writes (Inserts/Updates/Deletes) continue to hit the normalized tables exactly as they do today.

**Refresh Mechanism (Intentional Staleness):**
Because we cannot use triggers (to avoid write-penalties), we will implement a batch-refresh strategy.
*   **SLA / Staleness Window:** Read data will be up to **15 minutes stale**.
*   **Implementation:** A cron job (via OS `cron`, pgAgent, or `pg_cron` extension) will execute the following command every 15 minutes:
    ```sql
    REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_tombos_catalog;
    REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_tombos_stats_rollup;
    ```
*   *Why `CONCURRENTLY`?* It ensures that while the database is calculating the new joins, users querying the application will still be served the old data without experiencing database locks.

### 5. Risk Analysis & Operational Complexity

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **None.** Because we are using async Materialized Views, OLTP writes (`INSERT INTO tombos`) are completely unaffected. The CPU cost is shifted to the background refresh job. |
| **Lock Contention** | **Low.** `REFRESH ... CONCURRENTLY` requires a brief exclusive lock only at the very end of the transaction to swap the data pointers. |
| **Storage Growth** | **Moderate.** Denormalizing strings (like `familia_nome`) instead of integers will increase disk usage. However, PostgreSQL's TOAST compression will efficiently handle the JSONB columns. Ensure you have at least 2x the size of the `tombos` table available in free disk space. |
| **Failure Modes** | If the refresh job fails (e.g., due to a timeout), the MV simply retains the last known good state. The system does not go down; data just becomes staler than the 15-minute SLA. |
| **Reversibility** | **Immediate.** The application can be routed to read from `mv_tombos_catalog`. If issues arise, simply revert the application code to use the old ORM/SQL joins. The underlying tables were never touched. |

**Monitoring & Alerts Required:**
1.  **Freshness Alert:** Monitor the `pg_stat_user_tables` for `mv_tombos_catalog`. If `last_analyze` or `last_autovacuum` (which correlate with the refresh) is older than 30 minutes, trigger an alert to the DevOps team that the cron job failed.
2.  **Query Latency:** Monitor the 95th percentile (p95) latency of the new read queries. They should drop from seconds to milliseconds. If latency spikes, check if the B-Tree indexes are being utilized via `EXPLAIN ANALYZE`.
