### Executive Summary

As a Principal Database Architect, I have analyzed your highly normalized schema detailing Brazilian power plants (`Usinas`), their owners, rivers, and geographical locations. Because you did not provide specific queries or operational constraints, **I have made explicit assumptions**: your read workload consists of heavy map-based dashboard queries (filtering by state/stage) and analytical aggregations (total power by state), while your write workload is a nightly batch ETL (e.g., from a regulatory agency like ANEEL) with no CDC available and a 24-hour staleness tolerance. 

To eliminate the severe 5-to-6 table join penalty currently required to read a single complete plant profile, I propose a **Wide Read Table** (implemented via PostgreSQL Materialized Views) and a **Pre-Aggregated Rollup**. This design trades minor storage increases and a defined 24-hour staleness window for order-of-magnitude improvements in read latency, CPU utilization, and query simplicity. The implementation uses atomic concurrent refreshes to ensure zero downtime and absolute read consistency during batch updates.

### 2. Denormalization Patterns

1.  **Wide Read Table (Materialized View): `mv_usina_denorm_read`**
    *   *Justification:* Resolves Q1 and Q3. Flattens the 6-table snowflake schema into a single wide row per `Usina`. This completely eliminates CPU-heavy hash/nested-loop joins at read time.
    *   *Mechanism:* PostgreSQL `MATERIALIZED VIEW`. Fits the nightly batch assumption perfectly.
2.  **Pre-aggregated Rollup: `mv_usina_stats_uf_estagio`**
    *   *Justification:* Resolves Q2. Pre-calculating aggregations reduces a table scan of thousands of rows down to a simple index lookup of ~150 rows (27 states * ~5 stages).

### 3. Proposed Denormalized Schema (DDL)

*Note: PostgreSQL `MATERIALIZED VIEW` is used here. If your engine does not support this, these would be standard `TABLE`s populated via a stored procedure using a truncate/insert or swap-and-rename pattern.*

```sql
-- ==============================================================================
-- 1. WIDE READ TABLE: mv_usina_denorm_read
-- Ties to: Q1 (Map/Dashboard) and Q3 (Search)
-- ==============================================================================

CREATE MATERIALIZED VIEW public.mv_usina_denorm_read AS
SELECT 
    u.id AS usina_id,
    u.nome AS usina_nome,
    u.potkw,
    u.potfis,
    u.iniciop,
    u.atualizacao,
    -- Prefer Usina coordinates, fallback to Localizacao coordinates
    COALESCE(u.lat, l.latitude) AS latitude,
    COALESCE(u.long, l.longitude) AS longitude,
    u.estagio,
    
    -- Denormalized Dimensions
    p.proprietario AS proprietario_nome,
    r.rio AS rio_nome,
    
    -- Primary Location (Mun1)
    m1.municipio AS municipio1_nome,
    m1.uf AS uf1,
    
    -- Secondary Location (Mun2 - if applicable)
    m2.municipio AS municipio2_nome,
    m2.uf AS uf2

FROM public."Usina" u
LEFT JOIN public."Proprietarios" p ON u.id_proprietario = p.id_proprietario
LEFT JOIN public."Rios" r ON u.id_rio = r.id_rio
LEFT JOIN public."Localizacoes" l ON u.id_localizacao = l.id_localização
LEFT JOIN public."Municipios" m1 ON l.id_municipio1 = m1.id_municipio
LEFT JOIN public."Municipios" m2 ON l.id_municipio2 = m2.id_municipio;

-- Indexes for Wide Table
-- UNIQUE INDEX is strictly required for REFRESH MATERIALIZED VIEW CONCURRENTLY
CREATE UNIQUE INDEX uidx_mv_usina_denorm_id ON public.mv_usina_denorm_read(usina_id);

-- Indexes to support Q1 (Map/Dashboard filtering)
CREATE INDEX idx_mv_usina_denorm_uf1_estagio ON public.mv_usina_denorm_read(uf1, estagio);

-- Indexes to support Q3 (Text Search) - using trigram indexes if pg_trgm is enabled, 
-- otherwise standard B-Trees for prefix searches.
CREATE INDEX idx_mv_usina_denorm_nomes ON public.mv_usina_denorm_read(usina_nome, proprietario_nome, rio_nome);


-- ==============================================================================
-- 2. PRE-AGGREGATED ROLLUP: mv_usina_stats_uf_estagio
-- Ties to: Q2 (Analytics)
-- ==============================================================================

CREATE MATERIALIZED VIEW public.mv_usina_stats_uf_estagio AS
SELECT 
    m1.uf,
    u.estagio,
    COUNT(u.id) AS total_usinas,
    SUM(u.potkw) AS total_potkw,
    SUM(u.potfis) AS total_potfis,
    MAX(u.atualizacao) AS ultima_atualizacao
FROM public."Usina" u
JOIN public."Localizacoes" l ON u.id_localizacao = l.id_localização
JOIN public."Municipios" m1 ON l.id_municipio1 = m1.id_municipio
GROUP BY m1.uf, u.estagio;

-- UNIQUE INDEX for concurrent refresh
CREATE UNIQUE INDEX uidx_mv_usina_stats_uf_estagio ON public.mv_usina_stats_uf_estagio(uf, estagio);
```

### 4. Consistency Architecture

**Data Lineage:**
*   `usina_id` ← `Usina.id` (Source of Truth)
*   `uf1` ← `Municipios.uf` via `Localizacoes.id_municipio1`
*   `latitude`/`longitude` ← `Usina` (Primary) fallback to `Localizacoes` (Secondary).

**Refresh Strategy & Staleness Window:**
*   **Staleness:** Intentional 24-hour staleness. Reads will reflect the state of the database at the time of the last refresh.
*   **Refresh Mechanism:** Following the nightly ETL batch window, an orchestrator (e.g., Airflow, cron) will execute:
    ```sql
    REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_usina_denorm_read;
    REFRESH MATERIALIZED VIEW CONCURRENTLY public.mv_usina_stats_uf_estagio;
    ```
*   **Correctness Guarantee:** The `CONCURRENTLY` keyword ensures that read queries are *never* blocked during the refresh, and users never see partial data. The engine builds a new snapshot in the background and hot-swaps the pointers.

### 5. Risk Analysis & Operational Complexity

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **Low.** Because we are using Materialized Views refreshed in batch, transactional writes to the base tables (`Usina`, etc.) are completely unaffected. There are no triggers slowing down your OLTP inserts. |
| **Storage Growth** | **Medium.** Storing strings (`proprietario_nome`, `municipio_nome`) repeatedly instead of integers increases storage. However, for a dataset of power plants (typically < 100,000 records globally), this is negligible (< 50MB). |
| **Lock Contention** | **Zero.** `REFRESH ... CONCURRENTLY` requires a brief exclusive lock only at the very end of the process to swap the relfilenode. Read queries are not blocked. |
| **Failure Modes** | If the nightly refresh fails, the views remain queryable but stale. **Mitigation:** Alerting on the orchestrator task failure. |

**Observability & Monitoring Plan:**
1.  **Freshness Alerting:** Monitor the `ultima_atualizacao` column in the stats view. If `MAX(ultima_atualizacao) < CURRENT_DATE - INTERVAL '2 days'`, trigger a P2 alert.
2.  **Refresh Latency:** Log the execution time of the `REFRESH` commands. If the wide-table refresh exceeds 5 minutes, investigate index bloat or base-table fragmentation.
3.  **Reversibility:** This implementation is 100% additive. The base normalized tables remain the source of truth. If the denormalized views cause issues, the application can be pointed back to the normalized joins immediately via feature flags.
