### 4. Design of Possible Denormalized Schemas

#### Schema Option A: The "Flat Fact" Approach (Aggressive Denormalization)
*Concept:* Create one massive table per indicator type (e.g., `Fact_Economy`, `Fact_Education`) where the `country_code` and `year` are columns, not foreign keys.
*Pros:* Maximum performance for point queries.
*Cons:* High storage overhead (repeating country names), potential data consistency issues if source data changes (though in analytics, we usually accept this).

#### Schema Option B: The "Country-Year Hub" Approach (Moderate Denormalization)
*Concept:* Keep `country` and `region` as reference tables (for readability and small table optimization), but denormalize the main fact tables.
*Pros:* Balance between storage and maintainability.
*Cons:* Still requires a join to get the `region_name`.

#### Schema Option C: The "Super-Entity" Approach (Extreme Denormalization)
*Concept:* Merge all indicator tables into one giant `Fact_Indicators` table with columns for every metric.
*Pros:* Single table scan for everything.
*Cons:* Extremely wide table, hard to maintain, difficult to index specific subsets of columns effectively.

**Decision:**
I will proceed with **Schema Option B**, refined into a highly optimized structure. This keeps the small dimension tables (`country`, `region`) for metadata but denormalizes the large measurement tables to remove join overhead. This is the industry standard for Data Warehouses (Star Schema) optimized for read performance.

### 5. Evaluation and Selection

*   **Option A** is too aggressive; if we lose the `region` name, it's annoying to look up.
*   **Option C** is risky; if we need to filter only by `education` columns later, the index strategy becomes complex.
*   **Option B** is the most consistent. It allows us to keep `country` and `region` small and indexed, but removes the `JOIN` cost for the heavy lifting (GDP, Education, Health).

**Selected Strategy:**
1.  Keep `country` and `region` as small lookup tables.

### Key Improvements Explained

1.  **Elimination of JOINs:**
    *   **Before:** To get `GDP` and `Education` for a country, you had to `JOIN fact_economy` and `fact_education` on `country_code`.
    *   **After:** You simply `SELECT` from both tables. The database engine can now optimize these as simple table scans with a `WHERE` clause filter, or even perform a `UNION ALL` if the query requires combining rows, which is significantly faster than a `JOIN`.

2.  **Redundant Attributes (Technique 1):**
    *   `country_code` and `year` are repeated in every fact table.
    *   **Benefit:** No need to look up the country name in the `country` table or the region in the `region` table.
    *   **Trade-off:** Increased storage (approx. 10-20% more for text columns like `country_code`), but massive gains in query latency.

3.  **Unique Constraints (Replacing Foreign Keys):**
    *   Instead of `FOREIGN KEY (country_code) REFERENCES country(country_code)`, we use `UNIQUE (country_code, year)`.
    *   **Benefit:** This enforces data integrity (you can't have two GDP records for the same country in the same year) without the overhead of checking the parent table on every insert. It also allows the database to create a composite index on these two columns.

4.  **Specific Indexes:**
    *   Added composite indexes on `(country_code, year)` for all fact tables. This is the single most important optimization for this workload, as most analytical queries filter by a specific country and a specific year.

This schema is ready for heavy read workloads.
