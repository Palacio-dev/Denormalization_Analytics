# Denormalized Read-Optimized Schema Design

## Executive Summary

This design proposes a denormalized schema to significantly improve read performance for analytical queries over student trajectories and school performance indicators. The approach collapses key relationships into two wide tables: `fato_aluno_trajetoria` for student-level analytics and `dim_escola_indicadores_anuais` for annual school metrics. This reduces join complexity from up to 7 tables to at most 2 tables per query, while maintaining data correctness through explicit refresh processes.

The denormalization strategy preserves business semantics by:
- Maintaining source-of-truth lineage with clear refresh intervals
- Supporting both real-time transactional writes (in source tables) and batch-refreshed analytical reads
- Allowing for incremental updates without blocking analytical workloads

Operational considerations include write amplification during refresh cycles and storage growth of approximately 2.5x, but these are outweighed by expected 5-10x read performance improvements for target queries.

## Risk Analysis

### Write Amplification:
- Source tables: No change in write patterns
- Denormalized tables: 2x write amplification during refresh cycles
- Mitigation: Batch processing during off-peak hours

### Storage Growth:
- Estimated 2.5x increase in storage requirements
- Mitigation: Column compression where supported, archival of historical data

### Lock Contention:
- Minimal impact on source tables
- Potential contention during refresh on denormalized tables
- Mitigation: Use table swapping or partitioning strategies

### Failure Modes:
- Partial refresh could lead to inconsistent data
- Mitigation: Transactional refresh with rollback capability

### Backfill Runtime:
- Initial population: Estimated 4-6 hours for full dataset
- Incremental refresh: Under 30 minutes for daily updates

### Monitoring & Alerts:
- Lag detection: Compare `refreshed_at` timestamps
- Freshness alerts: Trigger if data older than SLA threshold
- Reconciliation diffs: Compare row counts and key metrics
- Query latency: Monitor before/after performance

This design balances read performance optimization with operational simplicity while preserving data correctness through explicit refresh processes and clear data lineage.
