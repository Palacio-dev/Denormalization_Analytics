## ✅ 1. Workload-Driven Analysis

### 🔍 Common Query Patterns (Inferred from Schema & Business Context)

Let’s assume these are common queries based on typical use cases:

#### 🧾 A. "List all clean-up events (`mutirao_de_limpeza`) with organizer name and location"
```sql
SELECT m.id_mutirao, m.data_do_ocorrido, i.nome AS instituicao, l.nome AS local
FROM mutirao_de_limpeza m
JOIN instituicao_organizadora i ON m.id_instituicao_organizadora = i.id_instituicao_organizadora
JOIN local_de_coleta l ON m.id_mutirao = l.id_mutirao;
```

**Problem:** Two joins per row → high cost on large datasets.

---

#### 📊 B. "Show collected waste types by event along with municipality info"
```sql
SELECT c.*, r.nome AS municipio_nome, r.estado
FROM coleta_de_lixo c
JOIN mutirao_de_limpeza m ON c.id_mutirao = m.id_mutirao
JOIN regiao r ON m.id_municipio = r.id_municipio;
```

**Problem:** Join across 3 tables for every row — expensive.

---

#### 🏞️ C. "List conservation units (`unidade_de_conservacao`) with biome, management category, administrative sphere, and municipality names"
```sql
SELECT u.nome, b.nome AS bioma, cm.categoria AS manejo, ea.tipo AS esfera, r.nome AS municipio
FROM unidade_de_conservacao u
JOIN bioma b ON u.id_bioma = b.id_bioma
JOIN categoria_manejo cm ON u.id_manejo = cm.id_manejo
JOIN esfera_administrativa ea ON u.id_esfera = ea.id_esfera
JOIN regiao r ON u.id_municipio = r.id_municipio;
```

**Problem:** Five joins – very costly.

---

### 🔁 Denormalization Candidates Identified:

| Table | Reason |
|-------|--------|
| `mutirao_de_limpeza` | Needs to carry organizer name and location name |
| `coleta_de_lixo`     | Needs municipality name/state |
| `unidade_de_conservacao` | All FKs should be flattened: bioma, manejo, esfera, município |

### 🧱 Denormalized Read Model for Clean-Up Events

```sql
-- Drop existing if exists
DROP MATERIALIZED VIEW IF EXISTS vw_mutirao_read_model;

-- Create wide read model
CREATE MATERIALIZED VIEW vw_mutirao_read_model AS
SELECT 
    m.id_mutirao,
    m.data_do_ocorrido,
    m.horario_de_inicio,
    m.numero_aproximado_de_participantes,
    m.duracao_total,
    i.nome AS instituicao_nome,
    i.carater AS instituicao_carater,
    l.nome AS local_nome
FROM mutirao_de_limpeza m
LEFT JOIN instituicao_organizadora i ON m.id_instituicao_organizadora = i.id_instituicao_organizadora
LEFT JOIN local_de_coleta l ON m.id_mutirao = l.id_mutirao;

-- Add indexes for fast lookups
CREATE INDEX idx_vw_mutirao_data ON vw_mutirao_read_model(data_do_ocorrido);
CREATE INDEX idx_vw_mutirao_instituicao ON vw_mutirao_read_model(instituicao_nome);
CREATE INDEX idx_vw_mutirao_local ON vw_mutirao_read_model(local_nome);
```

### 🧹 Waste Collection Summary Aggregation (Monthly Rollup)

```sql
-- Monthly aggregate view of collected waste
DROP MATERIALIZED VIEW IF EXISTS vw_clean_up_waste_summary;

CREATE MATERIALIZED VIEW vw_clean_up_waste_summary AS
SELECT 
    DATE_TRUNC('month', m.data_do_ocorrido) AS mes_referencia,
    r.nome AS municipio,
    SUM(c.plastico) AS total_plastico,
    SUM(c.borracha) AS total_borracha,
    SUM(c.madeira) AS total_madeira,
    SUM(c.produtos_texteis) AS total_textil,
    SUM(c.papel) AS total_papel,
    SUM(c.metal) AS total_metal,
    SUM(c.vidro) AS total_vidro,
    SUM(c.materiais_de_pesca) AS total_pesca,
    SUM(c.outros_residuos) AS total_outros
FROM coleta_de_lixo c
JOIN mutirao_de_limpeza m ON c.id_mutirao = m.id_mutirao
JOIN regiao r ON m.id_municipio = r.id_municipio
GROUP BY DATE_TRUNC('month', m.data_do_ocorrido), r.nome;

-- Speed up filtering by date and region
CREATE INDEX idx_vw_waste_summary_mes ON vw_clean_up_waste_summary(mes_referencia);
CREATE INDEX idx_vw_waste_summary_municipio ON vw_clean_up_waste_summary(municipio);
```

### 🏞️ Conservation Unit Denormalized Read Model

```sql
-- Drop old one first
DROP MATERIALIZED VIEW IF EXISTS vw_conservation_unit_read_model;

-- Flatten all foreign keys into readable names
CREATE MATERIALIZED VIEW vw_conservation_unit_read_model AS
SELECT
    u.id_uc,
    u.nome AS uc_nome,
    u.ano_de_criacao,
    u.area_soma_biomas,
    u.area_soma_biomas_continentais,
    u.area_marinha,
    b.nome AS bioma_nome,
    r.nome AS municipio_nome,
    r.estado AS estado_nome,
    cm.categoria AS manejo_categoria,
    cm.categoria_iucn,
    cm.plano,
    ea.tipo AS esfera_tipo
FROM unidade_de_conservacao u
LEFT JOIN bioma b ON u.id_bioma = b.id_bioma
LEFT JOIN regiao r ON u.id_municipio = r.id_municipio
LEFT JOIN categoria_manejo cm ON u.id_manejo = cm.id_manejo
LEFT JOIN esfera_administrativa ea ON u.id_esfera = ea.id_esfera;

-- Index for search and filter
CREATE INDEX idx_vw_uc_nome ON vw_conservation_unit_read_model(uc_nome);
CREATE INDEX idx_vw_uc_bioma ON vw_conservation_unit_read_model(bioma_nome);
CREATE INDEX idx_vw_uc_estado ON vw_conservation_unit_read_model(estado_nome);
CREATE INDEX idx_vw_uc_categoria ON vw_conservation_unit_read_model(manejo_categoria);
```
