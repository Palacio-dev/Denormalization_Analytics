# Denormalization Strategy: Read-Optimized Physical Schema

## 1. Workload-Driven Analysis

### Current Bottlenecks & Join Paths
Based on the standard TPC-H style schema provided, the primary read workloads (e.g., "Find total sales by customer," "Top 10 items by category," "Sales trend analysis") suffer from:
1.  **Multi-Join Aggregations**: Queries often require joining `s_purchase`/`s_purchase_lineitem` with `s_catalog_order`/`s_catalog_order_lineitem` (or web equivalents) to get a holistic view, or joining line items back to inventory to check stock status.
2.  **Date Range Scans**: Filtering by `purchase_date` or `return_date` (stored as `char(10)`) prevents efficient index usage unless a specific format index exists, leading to full table scans on large tables.
3.  **Repetitive Line Item Data**: The `item_id`, `quantity`, `price`, and `promotion_id` are repeated for every order line, causing high I/O when joining multiple line item tables.
4.  **Lack of Pre-aggregated Facts**: No tables exist to summarize daily/monthly sales, making time-series analysis extremely slow.

### Denormalization Candidates
To optimize latency without sacrificing correctness:
*   **Duplicate Key Attributes**: Add `customer_id`, `store_id`, `order_id` directly to line item tables to eliminate joins.
*   **Precomputed Aggregates**: Create a "Daily Sales" fact table and a "Customer Lifetime Value" summary table.
*   **Snapshot Fact Tables**: Create a wide "Order Summary" table that denormalizes the header and line item data into a single row per order.
*   **Covering Indexes**: Replace simple B-Trees on dates with composite indexes ordered by date, then by ID, to allow index-only scans for range queries.

## 2. Denormalization Design

### Strategy A: The "Wide" Order Summary Table
Instead of joining `s_purchase` and `s_purchase_lineitem` (and potentially `s_catalog_order`), we create a denormalized fact table `s_purchase_summary`.
*   **Grain**: One row per order.
*   **Redundancy**: We duplicate `plin_item_id`, `plin_quantity`, `plin_sale_price`, `plin_coupon_amt`, and `plin_promotion_id` into the header row.
*   **Benefit**: A query asking for "Total Revenue per Store" can now use a simple `GROUP BY` on the summary table without any joins.

### Strategy B: Precomputed Time-Series Aggregates
We introduce `s_daily_sales_summary`.
*   **Grain**: One row per `(store_id, date)`.
*   **Redundancy**: Contains pre-calculated `total_revenue`, `total_quantity`, `total_returns`, and `profit`.
*   **Benefit**: Eliminates the need to scan millions of line items for daily reporting.

### Strategy C: Covering Indexes & Clustering
*   **Clustering**: Order tables will be clustered by `order_date` (or `purchase_date`) followed by `order_id`. This is crucial for time-range scans.
*   **Covering Indexes**: We will use composite indexes `(order_date, order_id)` and `(date, item_id)` to allow the database engine to retrieve data without touching the table heap (Index Only Scan).

### Strategy D: Inventory Snapshot
*   **Table**: `s_inventory_snapshot`.
*   **Redundancy**: We duplicate `invn_item_id` and `invn_qty_on_hand` with a wider date range or specific "current" snapshot logic to avoid joining `s_purchase_lineitem` to check stock availability.

### Key Implementation Notes for the Engineer

1.  **Data Consistency (Correctness)**:
    *   **Triggers**: The `s_purchase_summary` table cannot be created as a static DDL if you need real-time updates. In a production environment, you would implement `AFTER INSERT/UPDATE` triggers on `s_purchase` and `s_purchase_lineitem` to automatically update `s_purchase_summary`.
    *   **Validation**: Add `CHECK` constraints to ensure `psr_total_quantity` equals the sum of line items (if triggers are used) or enforce data quality via application logic during the ETL load.

2.  **Index Strategy Rationale**:
    *   The indexes `idx_ps_purchase_date_store` and `idx_cos_date_customer` are **covering** in nature (assuming no columns are selected that aren't in the index, or using `INCLUDE` clauses if using SQL Server). This allows the database to answer "What was the revenue for Store X on Date Y?" without touching the actual row data pages, drastically reducing I/O.
    *   `char(16)` columns are indexed efficiently in most modern engines, but ensure they are stored as fixed-length strings to minimize index size.

3.  **Load Strategy**:
    *   Since these tables are denormalized aggregates, they should **not** be loaded row-by-row from the original tables.
    *   Use a batch process (ETL) or a `MERGE` statement to populate these tables once the base data is loaded.
    *   *Example Load Logic for `s_purchase_summary`*:
      ```sql
      INSERT INTO s_purchase_summary (psr_purchase_id, psr_total_quantity, ...)
      SELECT plin_purchase_id, SUM(plin_quantity), SUM(plin_sale_price), ...
      FROM s_purchase_lineitem
      GROUP BY plin_purchase_id;
      ```

4.  **Read Performance Gain**:
    *   **Before**: Querying "Top 10 Stores by Revenue" requires `JOIN s_purchase_lineitem` + `GROUP BY`. On 10M rows, this is slow.
    *   **After**: Querying `s_purchase_summary` requires `GROUP BY` on 10M rows (if not pre-aggregated) OR `SELECT * FROM s_daily_sales_summary` (if the daily summary is updated). If the daily summary is updated, the query is instantaneous.
    *   **Join Elimination**: The need to join `s_purchase` and `s_purchase_lineitem` is removed for any query that only needs order-level or line-level aggregates.

This schema transforms your system from a **Transactional Optimized** model (TPC-C style) to a **Read-Optimized** model (OLAP/Star Schema hybrid), specifically targeting the high-latency joins and date-range scans identified in your original workload.
