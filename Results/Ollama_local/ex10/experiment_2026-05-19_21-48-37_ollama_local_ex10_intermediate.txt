### 1. Workload-Driven Analysis

Based on the provided schema and typical analytics workloads for this domain (Energy, Population, Emissions), here is the analysis of the current join paths and performance bottlenecks:

**A. The Join Explosion Problem**
The current schema forces a "denormalized" join pattern where every read query must traverse the graph:
`Paises` $\to$ `Continente` $\to$ `Populacao` AND `Consumo_anual` AND `Emissao_g_efeito_estufa` AND `Fonte_energia`.

*   **High-Cost Operation:** Multi-table joins on `id_paises` and `ano`.
*   **The Bottleneck:**
    1.  **Filtering on `ano`**: Queries often filter by year (e.g., "Show 2023 data"). This forces a scan of potentially millions of rows across multiple tables.
    2.  **Join Aggregation**: Business intelligence reports usually require aggregations (Total Consumption per Continent, Emissions per Country). Doing this requires joining the fact tables (`consumo`, `emissao`) with dimension tables (`paises`, `continente`, `fonte`) and grouping.
    3.  **Attribute Lookup**: To display a report, the engine must fetch the country name, continent name, energy source name, and population for *every* row in the fact tables, then aggregate.

**B. Denormalization Candidates**
To optimize read latency, we must move from a *normalized* model (where data is split to save space/updates) to a *read-optimized* model (where data is duplicated to save I/O).

1.  **Denormalize `paises` into `consumo_anual`**: Instead of joining `paises` to get the name/continent, duplicate `nome` and `continente` directly in `consumo_anual`.
2.  **Denormalize `fonte_energia` into `consumo_anual`**: Duplicate `nome` in `consumo_anual` to avoid joining the array-based source table.
3.  **Denormalize `continente` into `paises`**: Duplicate `nome` in `paises` to avoid the join to `continente`.
4.  **Create a Precomputed Aggregate Fact Table**: Instead of aggregating on the fly, create a "Snapshot" table (`resumo_anual_por_continente`) that holds the pre-calculated totals.
5.  **Covering Indexes**: Since we are duplicating data, we must ensure the indexes cover the specific columns used in the `WHERE` and `ORDER BY` clauses to avoid index scans turning into table scans.

### 2. Denormalization Design Strategy

#### A. The "Wide" Fact Table (`consumo_anual_wide`)
We will replace `consumo_anual` with a wider version that includes all dimension attributes.
*   **Grain**: One row per (Country, Year, Energy Type).
*   **Redundant Attributes**: `nome_pais`, `nome_continente`, `nome_fonte`.
*   **Rationale**: Eliminates 3 joins (Country, Continent, Energy) for every query.

#### B. The "Wide" Emissions Table (`emissao_wide`)
*   **Grain**: One row per (Country, Year).
*   **Redundant Attributes**: `nome_pais`, `nome_continente`.
*   **Rationale**: Eliminates joins for emission data.

#### C. The Snapshot Aggregate Table (`resumo_anual_por_continente`)
*   **Grain**: One row per (Continent, Year).
*   **Content**: Total Population, Total Consumption (kWh), Total Emissions (tCO2).
*   **Rationale**: This is the fastest query possible. It replaces complex GROUP BY operations with a simple SELECT from a single table. It is updated via a background job or transactional trigger (simplified here as a materialized view logic).

#### D. Index Strategy
*   **Composite Indexes**:
    *   On `consumo_anual_wide`: `(ano, id_paises, tipo_energia)` to support time-series filtering and country grouping.
    *   On `emissao_wide`: `(ano, id_paises)`.
    *   On `resumo_anual_por_continente`: `(ano, id_continente)`.
*   **Covering Indexes**: Ensure the index includes the columns needed for the `SELECT` list to avoid "index only scans" if possible, though with large fact tables, we usually need the base table scan if we are aggregating, but for simple lookups, the index must cover the filter keys.

#### E. Partitioning
*   **Strategy**: Range partitioning by `ano` on the fact tables.
*   **Reason**: Time-series data grows linearly. Partitioning allows the database to prune partitions that are outside the query's `WHERE` clause (e.g., querying only 2023 avoids scanning 2010-2022).

### Key Implementation Notes for the Engineer

1.  **Data Consistency**: The denormalized columns (`nome_pais`, `nome_continente`) in the fact tables are now **read-only** from the application logic perspective. They should be populated during the initial load (ETL) or via a trigger that updates them when the `paises` or `fonte_energia` tables change.
    *   *Recommendation*: Add triggers on `paises` and `fonte_energia` to update the corresponding columns in `consumo_anual_wide` and `emissao_wide` automatically.

2.  **Update Latency**: Writing to a denormalized table is slightly slower than a normalized one because the database must update multiple rows (the fact row + the dimension row).
    *   *Mitigation*: If write throughput is critical, keep the original normalized tables for writes and use triggers to update the wide tables, or accept the slight overhead for the massive gain in read performance.

3.  **The Aggregate Table Maintenance**: `resumo_anual_por_continente` is a materialized view equivalent. It must be refreshed.
    *   *Option A (Background Job)*: Run a `MERGE` statement nightly to sync it with the fact tables.
    *   *Option B (Trigger)*: Use a `DO $$` trigger on the fact tables to maintain the aggregate on every insert/update (expensive for high write volume).
    *   *Option C (Materialized View)*: Create a standard Postgres Materialized View on the fact tables that aggregates by continent and refresh it manually or via cron.

4.  **Query Pattern Change**:
    *   **Before**: `SELECT p.nome, c.nome_continente, ... FROM consumo c JOIN paises p ON ...` (Slow, requires sorting/grouping).
    *   **After**: `SELECT nome_continente, SUM(porcentagem) FROM consumo_anual_wide GROUP BY nome_continente` (Fast, uses the wide table's existing columns).
    *   **Fastest**: `SELECT * FROM resumo_anual_por_continente WHERE ano = 2023;` (Instant).
