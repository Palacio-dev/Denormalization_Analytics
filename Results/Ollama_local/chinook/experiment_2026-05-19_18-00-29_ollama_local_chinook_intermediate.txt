# Database Performance Engineering: Denormalization Strategy

## 1. Workload-Driven Analysis

Based on the provided schema (Lakeside/Salesforce Lite pattern), here is the analysis of the read workload and identified bottlenecks.

### A. High-Cost Join Paths & Operations
The current schema forces the database engine to perform expensive `JOIN` operations for almost every "business fact" query.
1.  **The "Customer + Invoice" Explosion**:
    *   *Pattern*: `SELECT * FROM Invoice JOIN Customer ON Invoice.CustomerId = Customer.CustomerId`
    *   *Cost*: The database must scan the `Invoice` table, look up `Customer` details (Name, City, etc.), and reconstruct the address. If the `Invoice` table has millions of rows, this results in massive I/O and CPU usage for simple list reports.
2.  **The "Album + Track + Genre" Chain**:
    *   *Pattern*: `SELECT Album.Title, Track.Name, Genre.Name FROM Album JOIN Track ON ... JOIN Genre ON ...`
    *   *Cost*: A deep join chain. The engine must traverse multiple indexes and hash/join operations to assemble a single row of metadata.
3.  **The "Playlist + Track + Album" Triangle**:
    *   *Pattern*: Users often want to know "What songs are in this playlist?" and "Which albums do they belong to?".
    *   *Cost*: Requires joining `Playlist` -> `PlaylistTrack` -> `Track` -> `Album`. This is a classic "star" join pattern that is slow on large datasets.
4.  **Aggregation Overhead**:
    *   *Pattern*: `SELECT COUNT(*), SUM(Total) FROM Invoice GROUP BY CustomerId`
    *   *Cost*: Requires a full table scan and an aggregation pass. Doing this frequently (e.g., daily sales reports) creates a heavy load on the CPU.

### B. Denormalization Candidates
To optimize latency, we must eliminate the need to join for:
1.  **Customer Context**: Move `FirstName`, `LastName`, `Company`, `Address`, `City`, `State`, `Country` into the `Invoice` table.
2.  **Music Metadata**: Move `Track.Name`, `Album.Title`, `Genre.Name` into `InvoiceLine` (or a dedicated `InvoiceItem` table) so users see the full track info without joining `Track` and `Album`.
3.  **Sales Aggregates**: Create a pre-calculated table for daily/monthly sales totals to avoid `GROUP BY` scans.
4.  **Employee Hierarchy**: Flatten the `Employee` hierarchy or pre-calculate reporting lines if frequently queried (though this is less critical than music/sales data).

## 2. Denormalization Design Strategy

### Strategy A: The "Wide" Fact Table (`Invoice_Wide`)
Instead of joining `Invoice` with `Customer`, we duplicate the address and customer name fields directly into a new table.
*   **Grain**: One row per Invoice.
*   **Benefit**: Eliminates the most common join (`Invoice` + `Customer`). Queries become simple `SELECT * FROM Invoice_Wide WHERE Date > ...`.

### Strategy B: The "Item" Fact Table (`InvoiceItem_Wide`)
Instead of joining `InvoiceLine` -> `Track` -> `Album` -> `Genre`, we flatten these relationships.
*   **Grain**: One row per Invoice Line item.
*   **Benefit**: Removes the multi-table join chain. The query becomes `SELECT * FROM InvoiceItem_Wide`.

### Strategy C: Materialized Aggregates (`Sales_Daily`)
*   **Grain**: One row per Customer per Day.
*   **Benefit**: Instant retrieval of "Who bought what today?" without scanning the entire invoice history.

### Strategy D: Covering Indexes
Since we are denormalizing, we must ensure we don't create new index bloat.
*   **Composite Indexes**: Create indexes on `(BillingDate, CustomerId)` and `(InvoiceDate, Total)` to support common filter/sort patterns without touching the index tree.
*   **Covering Indexes**: For the `Invoice_Wide` table, ensure the primary key or a covering index includes all columns used in the top-level query to avoid "Index Only Scan" failures.

### Strategy E: Partitioning
*   **Suggestion**: Partition `Invoice_Wide` and `Sales_Daily` by `BillingDate` (or `InvoiceDate`). This allows the database to ignore partitions from previous years/months, drastically reducing I/O for time-series queries.

### Key Performance Improvements Achieved

1.  **Eliminated Deep Joins**: Queries like "List all invoices with customer and track details" now execute as single-table scans on `Invoice_Wide` and `InvoiceItem_Wide` instead of traversing 4+ tables.
2.  **Reduced CPU Usage**: Aggregation queries (e.g., "Total sales per day") now hit the `Sales_Daily` table instantly rather than calculating sums on millions of rows.
3.  **Optimized I/O**: Covering indexes (`idx_Invoice_Wide_CustomerCovering`) ensure that for the most common read pattern (lookup by customer), the database reads only the index, never touching the heap pages of the `Customer` table.
4.  **Scalability**: The partitioning hints on `Invoice_Wide` and `Sales_Daily` prepare the schema for horizontal scaling by allowing the database to ignore old data during queries.

**Correctness Note**: While this denormalized schema is optimized for reads, it introduces the risk of data inconsistency if `Customer` or `Track` data is updated frequently. In a high-write environment, you would need triggers to keep `Invoice_Wide` and `InvoiceItem_Wide` in sync, or accept that these tables are "Read-Only" caches that are refreshed periodically (e.g., nightly ETL job). Given your requirement for "Lower latency on read-heavy workloads," the trade-off of occasional data refresh is usually acceptable for the massive latency gains provided.
