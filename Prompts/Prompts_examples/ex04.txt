Here are prompts designed to instruct a Large Language Model (LLM) to denormalize a relational database schema. 

Because denormalization relies entirely on *how* the data is queried, all prompts include placeholders for the access patterns (the queries you run most often). The prompts are categorized by the user's database experience level.

---

### 🟢 Level 1: Beginner (No-Code / Low-Code User)
**Use this if:** You know your database is slow when loading data, you know what your tables look like, but you don't know the technical terms for fixing it. You want the LLM to explain its changes simply and provide ready-to-use code.

**The Prompt:**
> "I have a relational database, and I need to make it faster when reading data. I was told 'denormalization' would help. 
> 
> Here is a simple list of my current tables and the columns inside them:
> `[Paste your table names and columns, or your current SQL code here]`
> 
> The main problem is that when I try to look up `[Explain what you are trying to view, e.g., 'a User and all their recent Orders and Shipping Addresses']`, the database has to connect too many tables together, making it slow.
> 
> Please act as a helpful database teacher and do the following:
> 1. Explain how we can combine or change these tables to make reading this data much faster.
> 2. Explain the downside of doing this (like what happens when I need to update information).
> 3. Give me the new SQL code to create the new, faster tables."

---

### 🟡 Level 2: Intermediate (Software Developer / Analyst)
**Use this if:** You understand primary/foreign keys, joins, and normal forms (like 3NF). You know *why* you need to denormalize, but you want the LLM to architect the best solution, weigh the trade-offs, and generate the DDL (Data Definition Language).

**The Prompt:**
> "Act as a Database Engineer. I need to denormalize my current physical database model to optimize it for read-heavy operations. 
> 
> Here is my current normalized physical schema (DDL):
> `[Paste your current SQL DDL here]`
> 
> Here are the primary read queries/access patterns that are currently causing performance bottlenecks due to excessive joins:
> `[Paste your slow SQL queries, or describe the exact read access patterns here]`
> 
> Please provide:
> 1. **Denormalization Strategy:** Suggest 2-3 specific techniques for this schema (e.g., adding redundant columns, pre-calculating aggregate totals, or flattening tables).
> 2. **Trade-off Analysis:** For your recommended approach, explain the storage impact and the risk of data anomalies during updates/inserts.
> 3. **New Physical Schema:** Provide the complete SQL DDL for the new denormalized schema. 
> 4. **Indexes:** Include the necessary `CREATE INDEX` statements to support my read queries."

---

### 🔴 Level 3: Expert (Database Architect / DBA)
**Use this if:** You know exactly what database engine you are using (PostgreSQL, MySQL, SQL Server, etc.). You want the LLM to act as a sounding board for advanced techniques (like Materialized Views, JSONB document storage, or specific indexing strategies) and to write the boilerplate for data consistency mechanisms.

**The Prompt:**
> "Act as a Senior Database Architect specializing in `[Insert RDBMS, e.g., PostgreSQL 15]`. I am migrating a highly normalized (3NF) physical schema to a denormalized model to reduce read latency for a high-throughput system.
> 
> **Current Schema:**
> `[Paste DDL]`
> 
> **Query Access Patterns & Load:**
> `[Paste queries, Read/Write ratio, and scale, e.g., 'Read-to-write ratio is 100:1. Main query joins Users, Orders, and Products to render a dashboard.']`
> 
> I need you to design the target physical schema. Consider advanced denormalization patterns suitable for `[Insert RDBMS]`, such as Materialized Views, using Document types (JSON/JSONB) to nest child rows, or flat table structures. 
> 
> Provide the following:
> 1. **Target DDL:** The optimized physical schema, utilizing optimal data types.
> 2. **Advanced Indexing:** Appropriate indexes (B-Tree, GIN, GiST, Covering Indexes) tailored to the read patterns.
> 3. **Consistency Mechanism:** Because we are introducing redundant data, write the SQL triggers, stored procedures, or materialized view refresh logic required to mitigate write-anomalies and keep the redundant data eventually or strongly consistent.
> 4. **Write-Penalty Estimation:** A brief technical analysis of the write-amplification introduced by this new model."

---

### 💡 Pro-Tips for Getting the Best Result:
*   **Always include the "Read Patterns":** Denormalizing a database makes reads faster but writes slower. If you don't tell the LLM *what* you are reading (e.g., "I frequently search by User Last Name and City simultaneously"), it will guess incorrectly.
*   **Specify your Database Dialect:** Always tell the LLM if you are using MySQL, PostgreSQL, Oracle, or SQL Server. The physical implementation of denormalization (especially things like Materialized Views or JSON columns) varies wildly between them.
