As a senior database performance engineer, I have reviewed your normalized schema and your initial denormalization draft. 

Your draft is on the right track by collapsing relationships and adding redundant attributes. However, **it contains a critical correctness flaw**: the `country_indicators` table is missing the `year` column. In your normalized model, demography, economy, education, employment, and health are all time-series data tracked by `year`. Without `year` in the denormalized table, you will overwrite historical data, losing business-critical information.

Here is a comprehensive, workload-driven denormalization strategy that fixes this flaw, optimizes for read latency, and provides the exact DDL and indexing strategy.

---

### 1. Workload-Driven Analysis

In global economic and demographic datasets (like World Bank or UN data), read-heavy workloads typically fall into three patterns:

*   **Pattern A: Country/Region Yearly Dashboards.** Users query a specific country or region for a specific year to view all indicators (GDP, population, hospital beds, etc.).
    *   *High-Cost Operation:* In the normalized model, this requires an **8-to-10 table JOIN** (`country` → `region` → `demography` → `economy` → `health` → `education` → `population`...).
*   **Pattern B: Poverty & Inequality Analysis.** Users analyze survey results, comparing Gini coefficients and decile distributions across countries.
    *   *High-Cost Operation:* Joining `survey` to `decile` (a 1-to-10 relationship) requires the database to fetch 10 separate rows per survey and often perform expensive `GROUP BY` or pivot operations on the fly to display the distribution.
*   **Pattern C: Cross-Sectional Time-Series.** Comparing a specific metric (e.g., `gdp` vs `life_expectancy`) across all countries over a 10-year span.
    *   *High-Cost Operation:* Scanning multiple narrow tables and hash-joining them on `country_code` and `year`.

**Denormalization Candidates:**
*   **Pre-join Dimensions:** `region` into `country`, and `country` into all fact tables.
*   **Align Fact Grain:** Combine all yearly indicator tables (`demography`, `economy`, `education`, `employment`, `health`) into a single wide fact table with a grain of `(country_code, year)`.
*   **Pivot EAV/Narrow Tables:** `decile`, `population`, and `life_expectancy` act like Entity-Attribute-Value (EAV) tables. We must pivot these into distinct columns (e.g., `decile_1`, `pop_0_14_female`) to completely eliminate 1:N joins.

---

### 2. Denormalization Design

We will implement a **Star-like Read Model** utilizing Wide Tables:

1.  **`dim_country` (Dimension):** A flattened dimension table for UI dropdowns and quick lookups, preventing full table scans on the massive fact tables when only metadata is needed.
2.  **`fact_country_indicator_yearly` (Wide Fact Table):** 
    *   **Grain:** 1 row per `country_code` + `year`.
    *   **Design:** Merges all economic, health, and demographic metrics. Pivots the `population` and `life_expectancy` tables into hardcoded columns.
3.  **`fact_poverty_survey` (Wide Fact Table):**
    *   **Grain:** 1 row per `id_survey`.
    *   **Design:** Pre-joins country/region data. Pivots the 10 decile rows into 10 columns (`decile_1` through `decile_10`).
4.  **Indexing Strategy:**
    *   Composite B-Tree indexes starting with `region_code` and `year` to support the most common dashboard filters.
    *   Include/Covering clauses aren't necessary when queries typically `SELECT *` from wide tables, but clustering the data physically by `region_code` or `year` will drastically reduce disk I/O.

### 3. New Physical Schema (DDL)

Here is the PostgreSQL-optimized DDL for your new read-optimized schema.

```sql
BEGIN;

-- ==========================================
-- 1. DIMENSION TABLE
-- ==========================================
-- Used for populating UI filters without scanning fact tables.
CREATE TABLE IF NOT EXISTS public.dim_country (
    country_code VARCHAR(10) NOT NULL,

CREATE TABLE IF NOT EXISTS public.fact_country_indicator_yearly (
    country_code VARCHAR(10) NOT NULL,
    year INTEGER NOT NULL,
    
    -- Redundant Dimension Attributes (Removes joins)
    country_name VARCHAR(255),

    preprim_enrol DOUBLE PRECISION,
    prim_enrol DOUBLE PRECISION,
    sec_enrol DOUBLE PRECISION,
    terti_enrol DOUBLE PRECISION,
    
    -- Employment
    child_emp DOUBLE PRECISION,
    unemp DOUBLE PRECISION,
    vulnerable_emp DOUBLE PRECISION,
    part_time DOUBLE PRECISION,
    employers DOUBLE PRECISION,
    labor_force_total DOUBLE PRECISION,
    labor_force_fem DOUBLE PRECISION,

    hospital_beds DOUBLE PRECISION,

    life_expectancy_female DOUBLE PRECISION,
    life_expectancy_male DOUBLE PRECISION,

-- Partitioning example for the fact table (Optimizes time-series queries)
CREATE TABLE public.fact_country_indicator_yearly_pre_2010 
    PARTITION OF public.fact_country_indicator_yearly FOR VALUES FROM (MINVALUE) TO (2010);
CREATE TABLE public.fact_country_indicator_yearly_2010_2020 
    PARTITION OF public.fact_country_indicator_yearly FOR VALUES FROM (2010) TO (2020);

    PARTITION OF public.fact_country_indicator_yearly FOR VALUES FROM (2020) TO (MAXVALUE);


-- ==========================================
-- 3. WIDE FACT TABLE: POVERTY SURVEYS
-- ==========================================
-- Grain: 1 row per Survey
-- Collapses: survey, decile
CREATE TABLE IF NOT EXISTS public.fact_poverty_survey (
    id_survey SERIAL NOT NULL,
    survey_year INTEGER NOT NULL,
    
    -- Redundant Dimension Attributes
    country_code VARCHAR(10) NOT NULL,
    country_name VARCHAR(255),
    region_code VARCHAR(10),
    region_name VARCHAR(255),

    poverty_line DOUBLE PRECISION,

    distribution_type VARCHAR(100),

    decile_2 DOUBLE PRECISION,
    decile_3 DOUBLE PRECISION,
    decile_4 DOUBLE PRECISION,
    decile_5 DOUBLE PRECISION,
    decile_6 DOUBLE PRECISION,
    decile_7 DOUBLE PRECISION,
    decile_8 DOUBLE PRECISION,
    decile_9 DOUBLE PRECISION,
    decile_10 DOUBLE PRECISION,

-- ==========================================
-- 4. INDEXING STRATEGY
-- ==========================================

-- Index 1: Optimize cross-sectional queries by Region and Year
-- Example: "Show me all GDP and Education metrics for Sub-Saharan Africa in 2020"
CREATE INDEX idx_fact_indicators_region_year 
    ON public.fact_country_indicator_yearly (region_code, year);

-- Index 2: Optimize time-series queries for a specific country
-- Example: "Show me the trend of hospital beds in Brazil from 2000 to 2020"
-- (Note: country_code is already the first column in the PK, so a separate index isn't strictly needed, 
-- but explicitly indexing year + country_code helps reverse sorts).
CREATE INDEX idx_fact_indicators_year_country 
    ON public.fact_country_indicator_yearly (year DESC, country_code);

-- Index 3: Optimize Survey lookups by Region and Year
CREATE INDEX idx_fact_surveys_region_year 
    ON public.fact_poverty_survey (region_code, survey_year DESC);

-- Index 4: Optimize Survey lookups by Welfare Type (common analytical filter)
CREATE INDEX idx_fact_surveys_welfare 
    ON public.fact_poverty_survey (welfare_type, survey_year DESC);

COMMIT;
```

### Key Performance Wins of this Design:
1. **Zero-Join Reads:** Your application can now populate an entire dashboard for a country's given year with a single `SELECT * FROM fact_country_indicator_yearly WHERE country_code = 'X' AND year = Y;`. Latency drops from milliseconds (doing hash joins) to microseconds (single primary key lookup).
2. **Elimination of EAV Anti-Pattern:** By pivoting `decile`, `population`, and `life_expectancy`, you avoid the database having to read multiple blocks of disk to reconstruct a single logical entity. Calculating the difference between `decile_10` and `decile_1` is now a simple row-level math operation (`decile_10 - decile_1`) rather than a complex window function or self-join.
3. **Partition Pruning:** By partitioning the indicators table by `year`, historical queries (e.g., "Show me 2022 data") will entirely skip scanning disk blocks containing older data.
