### 2. Denormalization Patterns Justification

1.  **Wide Read Tables (One-Big-Table)**: We will use your proposed two-table design (`read_mutirao_limpeza` and `read_unidade_de_conservacao`). This eliminates 4-5 joins per query, reducing CPU and memory usage during query execution.
2.  **Surrogate Key Retention**: I have added `id_municipio` back into both tables. While BI tools will read the text columns (`nome_municipio`), any cross-domain queries (Q3) will join on `id_municipio` (Integer), which is orders of magnitude faster than joining on `(nome_municipio, estado)` (Text).
3.  **Partitioning**: `read_mutirao_limpeza` is a time-series fact table. I have applied **Range Partitioning** on `data_do_ocorrido`. This allows the database to skip scanning entire years/months of data for time-bounded queries (Partition Pruning).

### 4. Consistency Architecture & Implementation Plan

To prevent locking the normalized OLTP tables during heavy BI reads, we decouple the write and read paths.

*   **Data Lineage**: 
    *   `read_mutirao_limpeza` is sourced via `LEFT JOIN`s from `mutirao_de_limpeza` → `regiao`, `local_de_coleta`, `instituicao_organizadora`, `coleta_de_lixo`.
    *   `read_unidade_de_conservacao` is sourced via `LEFT JOIN`s from `unidade_de_conservacao` → `bioma`, `regiao`, `categoria_manejo`, `esfera_administrativa`.
*   **Refresh Strategy (ELT Batch)**: Do not use database triggers (they cause severe write amplification and transaction latency). Instead, use an hourly or nightly cron job (e.g., via `dbt`, Airflow, or a simple pg_cron function) that performs a **Blue/Green table swap** or an `INSERT ... ON CONFLICT DO UPDATE`. 
*   **Staleness Window**: Business users must be informed that the analytical tables are up to *X hours stale* (depending on your batch frequency). The `last_refreshed_at` column provides explicit observability of this staleness.

### 5. Risk Analysis & Operational Complexity

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Storage Growth** | **High.** Repeating strings like `nome_municipio` and `estado` millions of times consumes disk space. *Mitigation:* PostgreSQL uses TOAST compression for large texts, but if storage becomes an issue, consider enabling LZ4 compression (PG 14+) on the text columns. |
| **Write Amplification** | **Low.** Because we are not using triggers, the transactional normalized tables experience zero write amplification. The cost is shifted entirely to the background batch ETL process. |
| **Lock Contention** | **Zero.** The analytical queries hit the `read_*` tables. If you use a Blue/Green swap pattern (create new table, populate, `ALTER TABLE RENAME`), the refresh process causes zero read locks. |
| **Data Anomalies (Nulls)** | **Medium.** Because we use `LEFT JOIN`s to build the denormalized tables, missing child records (e.g., a cleanup without logged trash) will result in `NULL`s. *Mitigation:* I added `DEFAULT 0` to the waste metric columns in the DDL to ensure BI aggregations (`SUM`) do not fail or require `COALESCE` functions. |
| **Monitoring / Alerts** | Implement a monitoring query that compares `COUNT(*)` of `public.mutirao_de_limpeza` against `public.read_mutirao_limpeza`. If the difference is > 0 after a batch run, alert the data engineering team of a pipeline failure. Monitor `MAX(last_refreshed_at)` to alert on stale data. |
