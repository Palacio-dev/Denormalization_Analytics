# Denormalized Schema Design for Read Performance Optimization

## Executive Summary

This design proposes a denormalized read-optimized schema that addresses the complex join patterns in the normalized model while maintaining data correctness. The approach focuses on three primary denormalized artifacts: a consolidated Municipio dimension table, a wide Escola fact table with pre-joined educational metrics, and a Curso table combining IES and ENADE data. These changes will reduce query complexity from 4-5 table joins to single-table lookups for common analytical queries.

The design preserves business correctness through explicit data lineage tracking and implements a controlled refresh process with acceptable staleness windows of up to 24 hours for batch processing. Operational complexity is managed through clear separation of source-of-truth tables and denormalized read replicas, with a safe implementation plan including backfill strategies and rollback procedures.

## Proposed Denormalized Schema (DDL)

```sql
-- Dimension table consolidating UF and MUNICIPIO information
CREATE TABLE dim_municipio (
    id_municipio INT PRIMARY KEY,
    sigla_uf VARCHAR(2) NOT NULL,
    nome_uf VARCHAR(100) NOT NULL,
    nome_municipio VARCHAR(100) NOT NULL,
    qtd_habitantes INT CHECK(qtd_habitantes >= 0),
    qtd_IES INT CHECK (qtd_IES >= 0),
    qtd_ESCOLAS INT CHECK (qtd_ESCOLAS >= 0),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Wide fact table for school-level analytics (addresses School Performance Analysis queries)
CREATE TABLE fact_escola (
    cod_escola INTEGER PRIMARY KEY,
    nome_escola VARCHAR(100) NOT NULL,
    rede VARCHAR(100) NOT NULL,
    id_municipio INT NOT NULL,
    ano_referencia INT NOT NULL,
    -- IDEB metrics
    rend_1 REAL,
    rend_2 REAL,
    rend_3 REAL,
    rend_4 REAL,
    ind_rend REAL,
    nota_ideb REAL,
    -- SAEB metrics
    nota_mat REAL,
    nota_port REAL,
    nota_padrao REAL,
    -- Aggregated metrics
    media_ideb REAL,
    media_saeb REAL,
    total_participantes INT CHECK (total_participantes >= 0),
    -- Metadata for freshness tracking
    source_ideb_id INT,
    source_saeb_id INT,
    last_refreshed TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (cod_escola, ano_referencia),
    FOREIGN KEY (id_municipio) REFERENCES dim_municipio(id_municipio)
);

-- Wide fact table for course-level analytics (addresses Course Performance Analysis queries)
CREATE TABLE fact_curso (
    codigo_ies INT NOT NULL,
    nome_ies VARCHAR(100) NOT NULL,
    sigla_ies VARCHAR(20),
    rede_ies VARCHAR(100) NOT NULL,
    nome_curso VARCHAR(100) NOT NULL,
    id_municipio INT NOT NULL,
    ano_referencia INT NOT NULL,
    -- ENADE metrics
    total_inscritos INT CHECK (total_inscritos >= 0),
    total_concluintes INT CHECK (total_concluintes >= 0),
    nota_bruta_ce REAL,
    nota_padronizada_ce REAL,
    nota_bruta_fg REAL,
    nota_padronizada_fg REAL,
    nota_enade_continua REAL,
    nota_enade_faixa REAL,
    -- Metadata for lineage and freshness
    source_enade_id INT,
    source_ies_id INT,
    last_refreshed TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (codigo_ies, nome_curso, ano_referencia),
    FOREIGN KEY (id_municipio) REFERENCES dim_municipio(id_municipio)
);

-- Pre-aggregated rollup for municipal-level summaries (addresses Regional Comparison Reports)
CREATE TABLE agg_municipio_ano (
    id_municipio INT NOT NULL,
    ano_referencia INT NOT NULL,
    qtd_escolas INT,
    avg_nota_ideb REAL,
    avg_nota_saeb REAL,
    avg_nota_mat REAL,
    avg_nota_port REAL,
    total_participantes INT,
    qtd_cursos INT,
    avg_nota_enade REAL,
    last_refreshed TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id_municipio, ano_referencia),
    FOREIGN KEY (id_municipio) REFERENCES dim_municipio(id_municipio)
);

-- Indexes for optimized read performance
CREATE INDEX idx_fact_escola_municipio_ano ON fact_escola(id_municipio, ano_referencia);
CREATE INDEX idx_fact_escola_rede ON fact_escola(rede);
CREATE INDEX idx_fact_curso_municipio_ano ON fact_curso(id_municipio, ano_referencia);
CREATE INDEX idx_fact_curso_ies ON fact_curso(codigo_ies);
CREATE INDEX idx_agg_municipio_uf ON agg_municipio_ano((SELECT sigla_uf FROM dim_municipio WHERE dim_municipio.id_municipio = agg_municipio_ano.id_municipio));
```

## Data Lineage and Consistency Architecture

**Source of Truth → Denormalized Mapping:**
- `UF` + `MUNICIPIO` → `dim_municipio`
- `ESCOLA` + `MUNICIPIO` + `SAEB` + `IDEB` + `ANO` → `fact_escola`
- `CURSO` + `IES` + `MUNICIPIO` + `ENADE` + `ANO` → `fact_curso`

**Consistency Guarantees:**
- Staleness window: Maximum 24 hours for batch refresh
- Reconciliation process: Daily comparison checksums between source and target
- Error handling: Failed refreshes maintain previous valid data

## Risk Analysis

**Write Amplification:** 
- 3x to 5x increase in write operations during refresh
- Mitigation: Schedule during low-traffic periods

**Storage Growth:**
- Estimated 150-200% increase in storage requirements
- Mitigation: Implement data archiving policies for historical data

**Failure Modes:**
- Refresh job failures: Previous data remains available
- Data inconsistency: Detection through reconciliation queries
- Query performance regression: Rollback capability to normalized schema

**Monitoring Requirements:**
- Data freshness alerts (>24 hours stale)
- Reconciliation diffs monitoring
- Query latency comparison dashboards
- Storage growth tracking

This design provides significant read performance improvements while maintaining data correctness and operational safety through explicit staleness handling and comprehensive monitoring.
