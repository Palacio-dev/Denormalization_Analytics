# Denormalized Read-Optimized Schema Design

## Executive Summary

This design proposes a denormalized physical schema to optimize read performance for entitlement-based access patterns. The normalized schema requires joins across three tables (`users`, `entitlements`, `user_entitlements`) for common queries, creating join hotspots that can be eliminated through strategic denormalization. We propose two main artifacts:

1. **Wide Read Table (`user_entitlement_details`)**: Pre-joins user and entitlement data to eliminate runtime joins for user entitlement lookups
2. **Snapshot Table (`current_user_entitlements`)**: Stores only currently active entitlements for fast "what can this user access right now?" queries

The design preserves business correctness through clear data lineage and implements a controlled staleness window of 15 minutes for the snapshot table. The implementation plan includes safe rollout with backfill strategies, comprehensive monitoring, and rollback procedures.

## Access-Pattern Matrix

| Query Pattern | Tables Accessed | Join Requirements | Aggregation Needs | Frequency |
|---------------|----------------|-------------------|-------------------|-----------|
| Get all entitlements for a user | users, user_entitlements, entitlements | 2 joins (users→user_entitlements→entitlements) | None | High |
| Check if user has specific entitlement | users, user_entitlements, entitlements | 2 joins | None | Very High |
| List active entitlements for user | users, user_entitlements, entitlements | 2 joins | Filter by activeFlag, date ranges | High |
| Count entitlements by feature/package type | entitlements, user_entitlements | 1 join | GROUP BY | Medium |
| Bulk user entitlement reporting | all tables | Multiple joins | Aggregation | Low |

## Join Hotspots and Aggregation Hotspots

**Join Hotspots:**
- `user_entitlements` → `entitlements` (high frequency)
- `users` → `user_entitlements` (high frequency)

**Aggregation Hotspots:**
- Count/group by operations on entitlement types
- Date range filtering for active entitlements

## Proposed Denormalized Schema (DDL)

```sql
-- Wide read table: pre-joined user and entitlement data
CREATE TABLE user_entitlement_details (
    user_id INT NOT NULL,
    entitlement_id INT NOT NULL,
    activeDate DATE,
    inactiveDate DATE,
    useLimit INT,
    useCount INT,
    featureType VARCHAR(4),
    packageType VARCHAR(4),
    activeFlag INT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    PRIMARY KEY (user_id, entitlement_id),
    INDEX idx_user_active (user_id, activeFlag),
    INDEX idx_feature_type (featureType),
    INDEX idx_package_type (packageType),
    INDEX idx_active_dates (activeDate, inactiveDate)
);

-- Snapshot table: current active entitlements only
CREATE TABLE current_user_entitlements (
    user_id INT NOT NULL,
    entitlement_id INT NOT NULL,
    featureType VARCHAR(4),
    packageType VARCHAR(4),
    useLimit INT,
    useCount INT,
    snapshot_timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    PRIMARY KEY (user_id, entitlement_id),
    INDEX idx_user_lookup (user_id),
    INDEX idx_feature_lookup (featureType),
    INDEX idx_snapshot_time (snapshot_timestamp)
);

-- Metadata table for tracking refresh status
CREATE TABLE denormalization_refresh_log (
    table_name VARCHAR(100) NOT NULL,
    last_refresh_start TIMESTAMP,
    last_refresh_end TIMESTAMP,
    record_count INT,
    status VARCHAR(20), -- 'SUCCESS', 'FAILED', 'RUNNING'
    PRIMARY KEY (table_name)
);
```

## Denormalization Patterns Applied

### 1. Wide Read Tables (`user_entitlement_details`)
**Justification**: Eliminates the most frequent join pattern (user → entitlements) by pre-joining all relevant data. Supports high-frequency queries without joins.

**Target Queries**: 
- "Get all entitlements for user X"
- "Check if user X has entitlement Y"
- "List active entitlements for user X"

### 2. Snapshot Tables (`current_user_entitlements`)
**Justification**: Optimizes the most common access pattern - checking current active entitlements. Reduces data volume by ~80% (assuming 80% of entitlements are inactive).

**Target Queries**:
- "What can user X access right now?"
- "List current entitlements for reporting"

## Physical Design Details

### Primary Key Strategy
- `user_entitlement_details`: (user_id, entitlement_id) - natural composite key
- `current_user_entitlements`: (user_id, entitlement_id) - same for consistency
- `denormalization_refresh_log`: (table_name) - tracking table

### Indexing Strategy
- Covering indexes for common query patterns
- Type-based indexes for feature/package filtering
- Date-based indexes for time-bound queries

### Partitioning (Optional)
If data volume exceeds 100M rows:
- Partition `user_entitlement_details` by `user_id` hash (mod 100)
- Partition `current_user_entitlements` by `user_id` hash (mod 10)

## Consistency Architecture

### Data Lineage
```
Source of Truth:
  users.id ← user_entitlements.user_id
  entitlements.* ← user_entitlements.entitlement_id
  
Denormalized Artifacts:
  user_entitlement_details ← (users JOIN user_entitlements JOIN entitlements)
  current_user_entitlements ← (user_entitlement_details WHERE active conditions)
```

### Staleness Windows
- **user_entitlement_details**: Real-time (refreshed every 5 minutes)
- **current_user_entitlements**: 15-minute staleness window
- **Business Impact**: Acceptable for read queries; critical updates still go to source tables

## Risk Analysis

### Write Amplification
- 2x to 3x increase in write operations during refresh cycles
- Mitigation: Staggered refresh schedules, batch processing

### Lock Contention
- Table-level locks during full refresh possible
- Mitigation: Partitioned refresh, incremental updates where possible

### Failure Modes
- Refresh job failures → stale data
- Partial refresh → data inconsistency
- Mitigation: Atomic refresh with validation, rollback procedures

### Backfill Runtime
- Initial backfill: Estimated 2-4 hours for 10M user-entitlement pairs
- Incremental refresh: < 5 minutes for typical daily changes

### Storage Growth
- Additional storage: ~200% of original table size
- `user_entitlement_details`: ~150MB per 1M records
- `current_user_entitlements`: ~80MB per 1M active records

## Implementation Plan

1. **Phase 1**: Deploy tables in staging environment with sample data
2. **Phase 2**: Implement refresh jobs with dry-run validation
3. **Phase 3**: Backfill historical data during maintenance window
4. **Phase 4**: Deploy to production with canary routing
5. **Phase 5**: Monitor and tune performance
6. **Rollback Plan**: Disable denormalized query routing, rely on original normalized schema

This design provides measurable read performance improvements while maintaining data correctness and operational safety.
