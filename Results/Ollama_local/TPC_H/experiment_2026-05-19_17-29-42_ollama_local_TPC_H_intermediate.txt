# Denormalization Strategy: Read-Optimized TPC-H Schema

## 1. Workload-Driven Analysis

Based on the standard TPC-H read patterns (OLAP/Reporting) and your request for low-latency reads, here is the analysis of the current schema's bottlenecks:

### A. The Join Explosion Problem
The current schema forces every read query to traverse a complex join graph:
`LINEITEM` $\to$ `PART` $\to$ `SUPPLIER` $\to$ `NATION` $\to$ `REGION`.
*   **Cost:** To get a single row of `LINEITEM` data enriched with context (part name, supplier city, region), the database engine must perform multiple random I/O lookups.
*   **Latency Impact:** As table sizes grow (millions of rows), the time to resolve these foreign keys increases exponentially. A simple `SELECT * FROM lineitem l JOIN part p ON l.l_partkey = p.p_partkey` becomes slow due to the need to fetch `p` from disk, then potentially `s`, then `n`, then `r`.

### B. The "Aggregate" Bottleneck
Calculations like "Total Sales per Region" require scanning `LINEITEM`, joining `PART` (for price calculation), `SUPPLIER` (to link to `NATION`), and `REGION`.
*   **Cost:** Scanning billions of `LINEITEM` rows to compute a regional sum is computationally expensive.
*   **Latency Impact:** Queries returning a list of regions with their total sales will be slow because the database must calculate the sum on the fly for every query execution.

### C. Filter Selectivity
Queries often filter by `NATIONKEY` or `REGIONKEY`.
*   **Cost:** If `NATIONKEY` is not the primary key of the denormalized view, the database must scan the whole table to find the matching rows before aggregating.

---

## 2. Denormalization Design Strategy

To optimize read performance while preserving correctness, we will adopt a **Star Schema** approach with **Materialized Aggregates**.

### A. Read Model: "Wide" Fact Table (`FCT_READ_LINEITEM`)
Instead of joining, we duplicate the necessary columns from `PART`, `SUPPLIER`, `NATION`, and `REGION` directly into a new fact table.
*   **Why:** Eliminates the join step entirely. The query becomes a simple `SELECT ... FROM FCT_READ_LINEITEM WHERE ...`.
*   **Redundancy:** We duplicate `P_NAME`, `P_TYPE`, `S_NAME`, `S_ADDRESS`, `N_NAME`, `R_NAME`.
*   **Grain:** One row per `LINEITEM`.

### B. Materialized Aggregate Table (`FCT_READ_REGIONAL_SALES`)
We pre-calculate the most common complex query: "Total Sales per Region".
*   **Why:** Avoids scanning the fact table and performing complex joins during aggregation.
*   **Grain:** One row per `REGION`.
*   **Content:** `R_REGIONKEY`, `R_NAME`, `TotalSales`, `AvgPrice`, `OrderCount`.

### C. Index Strategy (Covering Indexes)
Since we are denormalizing, we cannot rely on foreign key indexes to speed up joins. We must create **Covering Indexes** on the denormalized tables.
*   **Strategy:** Create composite indexes that include the filter columns and the columns needed for grouping/ordering, effectively allowing the database to answer queries entirely from the index (Index Only Scan).

### D. Partitioning
*   **Suggestion:** Partition `FCT_READ_LINEITEM` by `O_ORDERDATE` (or a year range). This allows the database to ignore partitions where the date doesn't match, drastically reducing I/O for time-range queries.

## 3. New Physical Schema (DDL)

Below is the complete DDL for the denormalized schema. This replaces the original `LINEITEM` with the optimized read model and adds the aggregate table.

```sql
-- ==========================================
-- 1. DENORMALIZED FACT TABLE (Read-Optimized)
-- ==========================================
-- Purpose: Replaces LINEITEM + JOINs for fast reads.
-- Grain: One row per Line Item.
-- Optimization: Duplicates Part, Supplier, Nation, Region columns to avoid joins.
-- Partitioning: By Year/Order Date range for time-series filtering.

CREATE TABLE FCT_READ_LINEITEM (
    L_ORDERKEY      INTEGER NOT NULL,
    L_LINENUMBER    INTEGER NOT NULL,
    L_QUANTITY      DECIMAL(15,2) NOT NULL,
    L_EXTENDEDPRICE DECIMAL(15,2) NOT NULL,
    L_DISCOUNT      DECIMAL(15,2) NOT NULL,
    L_TAX           DECIMAL(15,2) NOT NULL,
    L_RETURNFLAG    CHAR(1) NOT NULL,
    L_LINESTATUS    CHAR(1) NOT NULL,
    L_SHIPDATE      DATE NOT NULL,
    L_COMMITDATE    DATE NOT NULL,
    L_RECEIPTDATE   DATE NOT NULL,
    L_SHIPINSTRUCT  CHAR(25) NOT NULL,
    L_SHIPMODE      CHAR(10) NOT NULL,
    L_COMMENT       VARCHAR(44) NOT NULL,
    
    -- DENORMALIZED ATTRIBUTES (Duplicates from PART, SUPPLIER, NATION, REGION)
    -- These allow filtering and grouping without joining
    
    -- Part Attributes
    P_PARTKEY       INTEGER NOT NULL,
    P_NAME          VARCHAR(55) NOT NULL,
    P_MFGR          CHAR(25) NOT NULL,
    P_BRAND         CHAR(10) NOT NULL,
    P_TYPE          VARCHAR(25) NOT NULL,
    P_SIZE          INTEGER NOT NULL,
    P_CONTAINER     CHAR(10) NOT NULL,
    P_RETAILPRICE   DECIMAL(15,2) NOT NULL,
    P_COMMENT       VARCHAR(23) NOT NULL,
    
    -- Supplier Attributes
    S_SUPPKEY       INTEGER NOT NULL,
    S_NAME          CHAR(25) NOT NULL,
    S_ADDRESS       VARCHAR(40) NOT NULL,
    S_PHONE         CHAR(15) NOT NULL,
    S_ACCTBAL       DECIMAL(15,2) NOT NULL,
    S_COMMENT       VARCHAR(101) NOT NULL,
    
    -- Nation Attributes
    N_NATIONKEY     INTEGER NOT NULL,
    N_NAME          CHAR(25) NOT NULL,
    N_REGIONKEY     INTEGER NOT NULL,
    N_COMMENT       VARCHAR(152),
    
    -- Region Attributes
    R_REGIONKEY     INTEGER NOT NULL,
    R_NAME          CHAR(25) NOT NULL,
    R_COMMENT       VARCHAR(152)
) PARTITION BY RANGE (L_SHIPDATE) (
    PARTITION p2023 VALUES LESS THAN (DATE '2024-01-01'),
    PARTITION p2024 VALUES LESS THAN (DATE '2025-01-01'),
    PARTITION p2025 VALUES LESS THAN (MAXVALUE)
);

-- ==========================================
-- 2. MATERIALIZED AGGREGATE TABLE (Regional Sales)
-- ==========================================
-- Purpose: Pre-calculated sums to avoid scanning the fact table for "Sales by Region".
-- Grain: One row per Region.
-- Maintenance: Should be updated via a stored procedure or scheduled job, 
--              but for a physical schema, we define it as a static table.

CREATE TABLE FCT_READ_REGIONAL_SALES (
    R_REGIONKEY      INTEGER NOT NULL,
    R_NAME           CHAR(25) NOT NULL,
    
    -- Aggregated Metrics
    TOTAL_SALES      DECIMAL(15,2) NOT NULL, -- SUM(L_EXTENDEDPRICE * (1 - L_DISCOUNT) * (1 + L_TAX))
    ORDER_COUNT      INTEGER NOT NULL,        -- COUNT(*)
    AVG_ORDER_PRICE  DECIMAL(15,2) NOT NULL,  -- AVG(TOTAL_SALES / ORDER_COUNT)
    PART_COUNT       INTEGER NOT NULL,         -- COUNT(DISTINCT L_PARTKEY)
    SUPPLIER_COUNT   INTEGER NOT NULL          -- COUNT(DISTINCT S_SUPPKEY)
);

-- ==========================================
-- 3. INDEX STRATEGY (Covering Indexes)
-- ==========================================
-- These indexes are designed to allow "Index Only Scans" for common query patterns.

-- Index for: Filtering by Region + Sorting by Sales (Top N Regions)
-- Pattern: SELECT * FROM FCT_READ_REGIONAL_SALES WHERE R_REGIONKEY = X ORDER BY TOTAL_SALES DESC
CREATE INDEX IX_REGION_SALES_SORT 
ON FCT_READ_REGIONAL_SALES (R_REGIONKEY, TOTAL_SALES DESC);

-- Index for: Filtering by Nation + Sorting by Sales
-- Pattern: SELECT * FROM FCT_READ_LINEITEM WHERE N_NATIONKEY = X ... GROUP BY N_NATIONKEY
CREATE INDEX IX_NATION_SALES_GROUP 
ON FCT_READ_LINEITEM (N_NATIONKEY, L_ORDERKEY);

-- Index for: Filtering by Part Type + Supplier + Region
-- Pattern: Complex joins removed, now a simple filter on denormalized cols.
CREATE INDEX IX_PART_SUPP_REGION 
ON FCT_READ_LINEITEM (P_TYPE, S_SUPPKEY, R_REGIONKEY);

-- Index for: Time-range filtering (Partition aware)
-- Note: Partitioning handles the bulk of this, but a secondary index helps within partitions.
CREATE INDEX IX_TIME_RANGE 
ON FCT_READ_LINEITEM (L_SHIPDATE);

-- ==========================================
-- 4. VIEW FOR "ORIGINAL" LOGIC (Optional)
-- ==========================================
-- If you need to run queries on the original schema structure (e.g., for legacy apps),
-- this view projects the denormalized data back to the original table structure.
-- This allows zero-cost joins for legacy queries that don't use the new indexes.

CREATE VIEW V_ORIGINAL_LINEITEM_WITH_CONTEXT AS
SELECT 
    l.L_ORDERKEY,
    l.L_LINENUMBER,
    l.L_QUANTITY,
    l.L_EXTENDEDPRICE,
    l.L_DISCOUNT,
    l.L_TAX,
    l.L_RETURNFLAG,
    l.L_LINESTATUS,
    l.L_SHIPDATE,
    l.L_COMMITDATE,
    l.L_RECEIPTDATE,
    l.L_SHIPINSTRUCT,
    l.L_SHIPMODE,
    l.L_COMMENT,
    
    -- Context columns joined via the denormalized fact table
    p.P_PARTKEY,
    p.P_NAME,
    p.P_MFGR,
    p.P_BRAND,
    p.P_TYPE,
    p.P_SIZE,
    p.P_CONTAINER,
    p.P_RETAILPRICE,
    p.P_COMMENT,
    
    s.S_SUPPKEY,
    s.S_NAME,
    s.S_ADDRESS,
    s.S_PHONE,
    s.S_ACCTBAL,
    s.S_COMMENT,
    
    n.N_NATIONKEY,
    n.N_NAME,
    n.N_REGIONKEY,
    n.N_COMMENT,
    
    r.R_REGIONKEY,
    r.R_NAME,
    r.R_COMMENT
FROM FCT_READ_LINEITEM l
JOIN PART p ON l.P_PARTKEY = p.P_PARTKEY
JOIN SUPPLIER s ON l.S_SUPPKEY = s.S_SUPPKEY
JOIN NATION n ON l.N_NATIONKEY = n.N_NATIONKEY
JOIN REGION r ON l.R_REGIONKEY = r.R_REGIONKEY;

-- Note: While V_ORIGINAL_LINEITEM_WITH_CONTEXT looks like a join, it is actually 
-- a projection from the denormalized table. The database will still optimize this 
-- by using the covering indexes defined above if the query pattern matches.
```

### Key Performance Improvements Explained

1.  **Elimination of Join Latency**:
    *   **Old:** `Query` $\to$ Scan `LINEITEM` $\to$ Hash Join $\to$ Scan `PART` $\to$ Hash Join $\to$ Scan `SUPPLIER`...
    *   **New:** `Query` $\to$ Scan `FCT_READ_LINEITEM` (single table).
    *   **Gain:** Removes $N-1$ join operations and associated random I/Os.

2.  **Covering Indexes**:
    *   The index `IX_NATION_SALES_GROUP` allows the database to find all rows for a specific nation and sort them by order key entirely within the index structure (if the data is ordered that way) or use an Index Scan to avoid a full table scan.
    *   The index `IX_PART_SUPP_REGION` covers queries that previously required filtering by Part Type, then joining to find the Supplier, then the Region. Now, it's a single predicate check.

3.  **Precomputed Aggregates**:
    *   Queries like `SELECT R_REGIONKEY, SUM(...) FROM ... GROUP BY R_REGIONKEY` now hit `FCT_READ_REGIONAL_SALES`.
    *   **Gain:** Instead of summing millions of `LINEITEM` rows, the database sums a small number of rows in the aggregate table. This reduces CPU usage and memory pressure during aggregation.

4.  **Partitioning**:
    *   Queries filtering by `L_SHIPDATE` (e.g., "Sales in Q4 2023") will automatically skip the `p2024` and `p2025` partitions. This reduces the dataset size by orders of magnitude for time-based queries.

### Correctness Preservation
*   **Redundancy Control:** The denormalized table `FCT_READ_LINEITEM` duplicates columns but does not add any new derived logic (like `L_EXTENDEDPRICE * (1 - L_DISCOUNT)`). It is a 1:1 copy of the original grain.
*   **Aggregate Integrity:** The `FCT_READ_REGIONAL_SALES` table is the *only* source of truth for the aggregates. In a production system, you would run a maintenance job to update this table nightly to ensure it matches the latest `FCT_READ_LINEITEM`. However, for the purpose of the physical schema design, this table ensures that read queries are instant and consistent relative to each other (since they read from the same snapshot).
