Here is a set of prompts designed for an LLM (like ChatGPT, Claude, etc.) to perform the database denormalization task. They are categorized by the user's level of database expertise. 

To use these, copy the prompt that matches your experience level and fill in the information inside the **[brackets]**.

---

### 1. Beginner Level
**Use case:** The user knows their database is slow when reading data but doesn't fully understand the technical mechanics of "denormalization." They need the LLM to act as a guide, explain the changes, and write simple code.

**Prompt:**
> "I am a beginner with databases. I have a relational database, and I need to make it faster when people are reading or searching for data. I was told I need to 'denormalize' it. 
>
> Here are the tables I currently have:
> **[Paste your current CREATE TABLE statements or just list the tables and their columns here]**
>
> Here is the information my app usually needs to show the user all at once:
> **[Explain in plain English what data you usually show together, e.g., "I need to show the User's name, their total number of orders, and the date of their last order on one screen."]**
>
> Please do the following:
> 1. Explain what denormalization means in simple terms based on my tables.
> 2. Propose a new, simplified database design (schema) that combines some of these tables so the database doesn't have to work as hard to read the data.
> 3. Provide the basic SQL `CREATE TABLE` statements for this new design.
> 4. Briefly tell me what the downside of this new design might be."

---

### 2. Intermediate Level
**Use case:** The user understands primary/foreign keys, joins, and normal forms. They know *what* needs to be done but want the LLM to do the heavy lifting of drafting the new schema and identifying exactly where the denormalization should happen based on specific queries.

**Prompt:**
> "Act as a database architect. I have a normalized relational database (3NF), but we are experiencing slow read performance due to heavy JOIN operations. I want to selectively denormalize the physical model to optimize for our most common read workloads.
>
> Here is the current physical schema (DDL):
> **[Paste your current SQL DDL here]**
>
> Here are the top 3 read queries (or access patterns) that need to be optimized:
> 1. **[Paste query or describe access pattern, e.g., "Fetching a full e-commerce product catalog with category names and average review scores"]**
> 2. **[Paste query 2]**
> 3. **[Paste query 3]**
>
> Please output the following:
> 1. **Denormalization Strategy:** Identify which tables should be flattened, where redundant columns should be added, or where pre-aggregated summary columns make sense.
> 2. **New Physical Schema:** Provide the complete SQL DDL for the new denormalized tables.
> 3. **Indexing:** Include suggested `CREATE INDEX` statements for the new schema based on the queries I provided.
> 4. **Write-Penalty Analysis:** Briefly explain how these changes will impact INSERT/UPDATE operations and how I should handle data anomalies (e.g., triggers, application logic)."

---

### 3. Expert Level
**Use case:** The user is a DBA or Senior Engineer. They are dealing with high-scale systems, specific SQL dialects (e.g., PostgreSQL, Snowflake), and need advanced physical modeling techniques (Materialized Views, JSONB arrays, table partitioning) alongside strict trade-off analyses.

**Prompt:**
> "As an expert Database Engineer, I need to refactor a highly normalized transactional database into a highly denormalized physical model optimized for sub-millisecond read latency. The target database engine is **[Specify Engine, e.g., PostgreSQL 15 / MySQL 8]**.
>
> **Current state and constraints:**
> *   Current Schema DDL: **[Paste DDL]**
> *   Data Volume: **[e.g., 50 million rows in the main fact table, growing by 100k daily]**
> *   Read/Write Ratio: **[e.g., 90% Reads, 10% Writes]**
> *   Primary Read Workload: **[Paste complex SQL queries or strict SLA requirements]**
>
> Generate a comprehensive physical schema refactoring plan that includes:
> 1. **Advanced Denormalization DDL:** Provide exact, production-ready DDL. Use advanced physical storage features appropriate for the engine (e.g., Materialized Views, nested JSON/NoSQL-style columns, array types, or table partitioning).
> 2. **Pre-aggregation:** Define specific schema structures for rolling up historical data if applicable.
> 3. **Index Strategy:** Provide exact DDL for covering indexes, composite indexes, or partial indexes required to support the read workload on the new schema.
> 4. **Data Integrity & Synchronization:** Because this introduces redundancy, provide the architecture to keep the denormalized read-model in sync (e.g., write the DDL for database triggers, or suggest a CQRS/Eventual Consistency pattern for the application layer).
> 5. **Storage vs. Compute Trade-off:** Quantify the expected increase in storage footprint versus the expected reduction in CPU/Memory usage during read execution plans."

---

### Pro-Tip for getting the best results:
If the LLM generates the schema but you aren't sure if it's correct, use this **Follow-up Prompt**:
> *"Write a simple SQL script to migrate data from my old schema (the first one I provided) into the new denormalized schema you just created. This will help me test if the new tables hold the data correctly."*
