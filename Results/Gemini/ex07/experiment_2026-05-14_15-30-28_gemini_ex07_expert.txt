### 2. Denormalization Patterns

**Chosen Pattern: Wide Read Table (Materialized View / OBT)**
*   **Justification**: The normalized schema represents a classic "Snowflake" dimension model. For read-heavy workloads (dashboards, maps, aggregations), traversing 6 tables creates CPU overhead and random I/O. A wide table pre-joins all dimensions.
*   **Implementation Engine**: PostgreSQL `MATERIALIZED VIEW`. 
*   **Why not a standard table updated via Triggers?** Triggers introduce write-amplification and lock contention on the source tables (e.g., updating a Municipality name would lock all associated Usinas). A Materialized View decouples writes from reads.
*   **Why not CDC?** While Debezium/Kafka could stream this to an external OLAP store, keeping it in-database via Materialized Views is the *minimal change* required to achieve the performance goal without introducing new infrastructure complexity.

### 4. Consistency Architecture

**Data Lineage**
*   `id`, `nome`, `potkw`, `potfisc`, `iniciop`, `atualizacao`, `estagio` $\leftarrow$ Source: `Usina`
*   `proprietario` $\leftarrow$ Source: `Proprietarios.proprietario`
*   `rio` $\leftarrow$ Source: `Rios.rio`
*   `munic1`, `uf1` $\leftarrow$ Source: `Municipios` (via `Localizacoes.id_municipio1`)
*   `lat`, `long` $\leftarrow$ Source: `Usina` (with fallback to `Localizacoes`)

**Refresh Strategy & Staleness Window**
*   **Mechanism**: A scheduled background worker (e.g., `pg_cron` or an external orchestrator like Airflow) will execute:
    `REFRESH MATERIALIZED VIEW CONCURRENTLY public.usina_read_optimized;`
*   **Staleness Window**: Intentional staleness is defined by the cron schedule (e.g., every 15 minutes). Business stakeholders must be informed that the read model reflects the state of the database *as of the last refresh*.
*   **Why Concurrently?** It builds a new snapshot in the background and hot-swaps the data using the Unique Index (`id`), meaning zero downtime for `SELECT` queries.

### 5. Risk Analysis & Operational Complexity

| Risk Factor | Assessment & Mitigation |
| :--- | :--- |
| **Write Amplification** | **Low.** Because we are using a Materialized View rather than triggers, OLTP writes to the normalized tables are completely unaffected. Write cost is shifted to the batch refresh window. |
| **Lock Contention** | **Zero on Reads.** `CONCURRENTLY` ensures reads are never blocked. Write locks on source tables during the refresh are minimal because it is a pure `SELECT` operation. |
| **Storage Growth** | **Moderate.** We are duplicating strings (Owner names, River names, Municipality names). However, power plant datasets typically have low row counts (e.g., tens of thousands, not billions). Storage impact is likely negligible (< 100MB). |
| **Failure Modes** | If the refresh job fails, the read table simply becomes more stale; it does not go offline. **Mitigation**: Alert on job failure. |
| **Monitoring / Alerts** | 1. **Freshness Lag**: Monitor the `pg_stat_user_tables.last_analyze` or a custom metadata log to alert if the view hasn't refreshed in > 30 mins.<br>2. **Query Latency**: Monitor read latency on the new view to ensure indexes are being utilized. |
| **Reversibility** | **High.** The original normalized schema is untouched. If the denormalized model fails, application read queries can be immediately reverted to the original normalized `JOIN` queries. |
