EXPERT:
You are a senior data architect and physical database designer. Redesign the following normalized relational physical model 
into a denormalized analytics-oriented physical schema optimized for query performance, low-latency reads, and scalable reporting.

Primary objective:
Create a new physical schema suitable for analytics, including a dimensional or denormalized structure where appropriate, 
while preserving a clear path for loading and maintaining the data.

Inputs:
- Current physical schema with tables, columns, PKs, FKs, indexes, and constraints: [PASTE FULL MODEL]
- Business domain and subject areas: [PASTE]
- Top analytical query patterns and SLA targets: [PASTE]
- DBMS and version: [PASTE]
- Data volume, refresh frequency, and concurrency: [PASTE]
- Known pain points in the current schema: [PASTE IF ANY]

What to do:
1. Reverse-engineer the current model and identify the core business processes and entities.
2. Propose an analytics-first physical design, favoring denormalization where it materially improves read performance.
3. Define the target schema with explicit table grain, column set, surrogate keys if needed, and relationships.
4. Decide where to use:
   - wide denormalized tables
   - fact tables
   - dimension tables
   - aggregate tables
   - materialized views
   - partitioning or clustering
   - columnstore or other engine-specific features if appropriate
5. Explain which normalization rules are intentionally relaxed and why.
6. Preserve auditability and historical analysis where needed, including handling slowly changing attributes if relevant.
7. Optimize the schema for the provided analytical workloads, not transactional OLTP use.
8. Include physical implementation details that matter for performance, such as indexing strategy, distribution/partition keys,
 compression, and refresh approach.

Required output:
- Executive summary of the redesign
- Workload assumptions
- Proposed target physical schema
- Table-by-table specification with grain, keys, and purpose
- Recommended indexes, partitions, and storage optimizations
- Mapping from source model to target model
- Performance tradeoffs and risks
- Implementation notes for ETL/ELT loading
- Optional SQL DDL for the core tables

Important constraints:
- Prioritize read performance for analytics.
- Minimize joins in the most common queries.
- Avoid unnecessary theoretical discussion.
- If multiple design options exist, choose the best one and justify it.
- State assumptions explicitly when source information is incomplete.
