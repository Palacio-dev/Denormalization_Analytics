# **Executive Summary**

This project proposes a denormalized read-optimized schema designed to significantly improve query performance on analytical workloads involving conservation units (`unidade_de_conservacao`), cleanup events (`mutirao_de_limpeza`, `coleta_de_lixo`), and related reference data. The approach focuses on **prejoining frequently joined dimensions**, creating **pre-aggregated summaries** for common reporting queries, and building **wide read tables** that reduce runtime joins and aggregations.

All artifacts preserve referential correctness through clear lineage tracking and are refreshed via controlled batch pipelines within defined SLA windows. No triggers are used; consistency is maintained through application-controlled ETL jobs during off-peak hours.

The resulting schema enables faster dashboard rendering, report generation, and ad-hoc analysis by reducing join fan-out, avoiding real-time computation of totals, and enabling efficient index-based filtering.

## 🛠️ 2. Denormalization Patterns Chosen

| Artifact | Purpose | Justification |
|---------|--------|---------------|
| **Wide Read Table**: `vw_conservation_unit_enriched` | Reduce joins for UC listing/reporting | Prejoin biome, management category, administrative sphere into one table |
| **Rollup View/Table**: `agg_cleanups_by_date_and_region` | Fast lookup of daily cleanup stats | Avoid recomputing participant count and event volume |
| **Snapshot Table**: `snap_current_cleanups_summary` | Support dashboards needing latest state | Provides up-to-date totals for current month/year |
| **Materialized View**: `mv_top_institutions_by_event_count` | Rank organizers efficiently | Refreshable summary avoids full scan at runtime |
| *(Optional)* Star Schema Fact-Dim Split | For BI/analytics engines | Not implemented unless required for OLAP tools like PowerBI/Tableau |

> Assumption: Queries typically involve small result sets but high frequency of execution over recent time ranges.

## 🧱 3. Proposed Denormalized Schema (DDL)

```sql
-- Wide read table: Enriched Conservation Unit view
CREATE TABLE public.fact_conservation_unit (
    id_uc integer NOT NULL PRIMARY KEY,
    nome text NOT NULL,

    -- Dimensional enrichment
    nome_bioma text,
    nome_categoria_manejo text,
    categoria_iucn text,
    plano_gestao text,
    tipo_esfera text,
    nome_municipio text,
    estado text,

    -- Measures
    ano_de_criacao integer,
    area_soma_biomas double precision,
    area_soma_biomas_continentais double precision,
    area_marinha double precision,

    -- Lineage metadata
    updated_at timestamptz DEFAULT now()
);

-- Indexes for fast filtering
CREATE INDEX idx_fact_conservation_unit_nome ON public.fact_conservation_unit(nome);
CREATE INDEX idx_fact_conservation_unit_estado ON public.fact_conservation_unit(estado);
CREATE INDEX idx_fact_conservation_unit_bioma ON public.fact_conservation_unit(nome_bioma);

COMMENT ON COLUMN public.fact_conservation_unit.updated_at IS 'Source-of-truth timestamp for last update';
```

```sql
-- Rollup Aggregate: Cleanups Per Day By Region
CREATE TABLE public.agg_cleanups_by_day (
    event_date date NOT NULL,
    id_municipio integer NOT NULL,
    nome_municipio text,
    estado text,
    num_events integer,
    total_participants integer,
    avg_duration interval,
    
    PRIMARY KEY(event_date, id_municipio)
);

CREATE INDEX idx_agg_cleanups_by_day_date ON public.agg_cleanups_by_day(event_date DESC);
CREATE INDEX idx_agg_cleanups_by_day_municipio ON public.agg_cleanups_by_day(id_municipio);
```

```sql
-- Snapshot Summary Table: Monthly Stats for Dashboard
CREATE TABLE public.snap_monthly_cleanup_summary (
    period_month date NOT NULL PRIMARY KEY,
    total_events integer,
    total_participants bigint,
    avg_participants_per_event numeric,
    top_region_name text,
    top_institution_name text,
    updated_at timestamptz DEFAULT now()
);

COMMENT ON COLUMN public.snap_monthly_cleanup_summary.period_month IS 'First day of month being summarized';
```

```sql
-- Materialized View: Top Organizers by Event Count
CREATE MATERIALIZED VIEW public.mv_top_institutions_by_event_count AS
SELECT 
    i.nome AS instituicao_nome,
    COUNT(m.id_mutirao) AS total_events,
    SUM(m.numero_aproximado_de_participantes) AS total_participants
FROM public.mutirao_de_limpeza m
JOIN public.instituicao_organizadora i ON m.id_instituicao_organizadora = i.id_instituicao_organizadora
GROUP BY i.nome
ORDER BY total_events DESC;

CREATE UNIQUE INDEX mv_top_inst_nome_idx ON public.mv_top_institutions_by_event_count(instituicao_nome);
```

## 🔁 4. Consistency Architecture

Each artifact has a clearly defined source of truth:

| Artifact | Source Tables | Update Frequency | Method |
|----------|----------------|------------------|--------|
| `fact_conservation_unit` | `unidade_de_conservacao`, `bioma`, etc. | Batch nightly | Full replace based on FK joins |
| `agg_cleanups_by_day` | `mutirao_de_limpeza`, `regiao` | Incremental hourly | Append new dates, recompute yesterday |
| `snap_monthly_cleanup_summary` | All relevant fact tables | Daily | Full rebuild |
| `mv_top_institutions_by_event_count` | `mutirao_de_limpeza`, `instituicao_organizadora` | Weekly | REFRESH CONCURRENTLY |

> ✅ Correctness preserved via atomic updates using staging tables before swapping in final results.

## ⚠️ 5. Risk Analysis

| Aspect | Description |
|-------|-------------|
| **Write Amplification** | High – multiple copies of same data exist across de-normalized structures |
| **Lock Contention** | Low risk due to batch-only writes and staged replacement |
| **Failure Modes** | Stale data possible if job fails; mitigated via alerting |
| **Backfill Runtime** | Initial population may take minutes depending on dataset size |
| **Storage Growth** | ~2x original footprint expected due to duplication |
| **Monitoring Needs** | Lag metrics between base and derived tables; freshness checks; reconciliation diffs against raw data |

### 📈 Suggested Alerts/Metrics:
- Max lag between source and materialized object > 6h
- Row count mismatch between fact tables and their sources exceeds threshold
- Query latency regression after deployment (via A/B testing)
