## ⚠️ 5. Risk Analysis

| Risk Category         | Description                                                                 | Mitigation Plan |
|-----------------------|-----------------------------------------------------------------------------|-----------------|
| Write Amplification   | Refresh processes update denormalized rows even when only one base field changes | Batch writes via ETL pipeline; deduplicate changes if CDC used |
| Lock Contention       | Bulk refresh may block readers                                              | Use MVCC engines or staging tables before swapping live versions |
| Failure Modes         | Incomplete refresh leads to stale/inconsistent data                         | Implement atomic swaps using temp tables + rename |
| Backfill Runtime      | Full rebuild of wide table takes minutes                                    | Schedule during off-hours; stream partial updates via CDC |
| Storage Growth        | Denormalized copies increase disk usage                                     | Monitor retention policies; archive old snapshots |
| Query Regressions     | Poorly indexed denormalized table degrades performance                      | Validate execution plans; benchmark post-deploy |
| Lag/Freshness         | Stale data visible until next sync                                          | Track timestamp deltas between source and target |
| Reconciliation Diff   | Mismatch between base and derived tables                                   | Daily checksum validation routines |

### Monitoring Recommendations

- **Lag Alert**: Trigger alert if `updated_at` differs > X minutes from latest source row
- **Freshness Metric**: Expose last-refresh timestamp in metadata endpoint
- **Query Latency Dashboard**: Compare pre/post denormalization SLIs
- **Data Drift Detection**: Hash-based comparison job nightly between sources and targets
