BEGGINER:
You are a database assistant. I have a relational database with a normalized physical model, and I want to create a denormalized physical schema optimized for analytics and fast read queries.

Your task is to:

Analyze the normalized physical model.
Identify the tables, relationships, and main analytical access patterns.
Propose a denormalized version of the physical schema that improves read performance.
Explain which tables should be merged, which attributes should be duplicated, and why.
Design the new physical schema for analytics, including table names, columns, primary keys, and foreign keys if needed.
Highlight any trade-offs, such as increased storage or slower writes.
Please explain your reasoning in simple terms and organize the answer clearly. If any information is missing, state the assumptions you are making.

INTERMEDIATE:
Act as a database design specialist. I have an existing normalized relational physical model, and I need a denormalized
physical schema for a reporting and analytics workload.

Your goal is to:

- Review the source schema and infer the likely analytical queries and reporting needs.
- Design a denormalized physical model oriented toward read-heavy analytics.
- Reduce join complexity by flattening relevant dimensions into fact-like structures where appropriate.
- Preserve the business meaning of the data while improving query performance.
- Produce a new physical schema with:
- table definitions
- columns and data types
- primary keys
- indexes you recommend
- denormalized relationships or embedded attributes
- Explain the denormalization strategy used for each table.
- Identify performance benefits and maintenance costs.
- Call out any data quality or consistency risks introduced by denormalization.
- Return the result as a structured design document, and include assumptions if the source model does not provide enough context.

EXPERT:
You are a senior database architect specializing in analytics platforms and physical data modeling. I will provide a
normalized relational physical schema, and your task is to redesign it into a denormalized physical schema optimized for
analytical workloads, such as dashboards, aggregates, and BI queries.

Requirements:

- Analyze the source physical model, including entities, relationships, cardinalities, keys, and likely query paths.
- Infer the target analytical use cases and optimize for read performance, low join cost, and query simplicity.
- Produce a denormalized physical schema suitable for a reporting/analytics layer, potentially using:
    - wide tables
    - fact-style tables
    - denormalized dimensions
    - pre-joined structures
    - selective attribute duplication
- Minimize normalization where it harms analytical performance, but preserve enough structure to support maintainability and governance.
- For each target table, specify:
    - table name
    - business purpose
    - grain
    - columns
    - keys
    - indexing strategy
    - partitioning/clustering suggestions if relevant
    - source table mappings
- Explain the denormalization decisions in terms of:
    - query patterns
    - cardinality
    - update frequency
    - storage cost
    - consistency trade-offs
- If helpful, propose both:
- a base transactional schema adaptation
- an analytics-ready physical schema
- and clearly distinguish them.


Deliver the final answer as:
- a concise architecture summary
- the proposed physical schema
- a table-by-table transformation rationale
- risks, assumptions, and implementation notes

Optimize the design specifically for analytics, not OLTP. If there are multiple valid denormalization strategies, 
choose the one that best balances query performance, semantic clarity, and maintenance cost.
