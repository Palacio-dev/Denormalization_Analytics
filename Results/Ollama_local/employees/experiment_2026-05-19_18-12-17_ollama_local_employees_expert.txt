# Executive Summary

This design proposes a **Read-Optimized Denormalized Schema** for the `employees` dataset, specifically targeting analytical queries that require joining `employee`, `department`, `salary`, and `title` attributes frequently. Under a standard OLTP workload, joins on large historical datasets (e.g., `from_date` < '2020-01-01') cause significant I/O latency due to scanning multiple tables and resolving temporal relationships.

The solution introduces **Wide Read Fact Tables** (denormalized aggregates) and **Snapshot Views** to eliminate join latency. Crucially, because the source data is time-dependent (`from_date`, `to_date`), the denormalized schema explicitly handles **intentional staleness**. We will not attempt real-time consistency; instead, we will enforce a **T+5 minute freshness window** via background refresh jobs. This trade-off allows us to pre-compute complex temporal logic (e.g., "Who is the manager of Dept X as of today?") into a single table lookup, reducing query complexity from $O(N)$ joins to $O(1)$ lookups while preserving business correctness within the defined latency bounds.

## 1. Model-to-Workload Mapping

Based on the provided DDL and typical HR analytics workloads, the access patterns are characterized by:

| Query Class | Top Query Pattern | Join Hotspots | Aggregation Hotspots | Latency Bottleneck |
| :--- | :--- | :--- | :--- :--- |
| **Org Structure** | `SELECT * FROM employee, department, department_employee` | `department_employee` (Many-to-Many) | `COUNT(*)`, `MAX(to_date)` | Joining `department_employee` and `department_manager` on `(emp_id, dept_id)` for historical state |
| **Compensation** | `SELECT salary, title FROM employee, salary, title` | `salary` (Many-to-One) | `AVG(amount)`, `SUM(amount)` | Scanning `salary` history for each employee over a date range |
| **Reporting** | `SELECT dept_name, COUNT(emp), AVG(sal) FROM ... GROUP BY dept` | `department_employee` (Many-to-One) | `GROUP BY` on date ranges | Multiple table scans + sort operations |

**Key Observations:**
1.  **Temporal Complexity:** The normalized schema requires complex `CASE WHEN` logic or window functions to determine the "current" state or state at a specific date.
2.  **Fan-out:** The `department_employee` and `department_manager` tables create a fan-out that explodes when joining `employee` data with `department` data.
3.  **Write Frequency:** `salary` and `title` updates are frequent, making a live materialized view expensive to maintain without CDC.

## 2. Denormalization Patterns & Justification

We will employ three specific patterns, justified by the need to reduce join depth and simplify temporal logic:

1.  **Wide Read Fact Table (`emp_dept_snapshot`)**:
    *   *Justification:* Pre-joins `employee`, `department`, `department_employee`, and `department_manager` into a single wide table.
    *   *Benefit:* Eliminates 3+ joins per query. Stores the "current" state (as of a refresh timestamp) directly.
2.  **Pre-Aggregated Rollup (`dept_emp_stats`)**:
    *   *Justification:* Materializes counts, sums, and averages grouped by Department and Time Bucket.
    *   *Benefit:* Replaces `GROUP BY` operations on massive datasets with a single row lookup.
3.  **Temporal Snapshot Strategy (Not Live View)**:
    *   *Justification:* Given the `from_date`/`to_date` constraints, we cannot maintain a single live view of "current" state across all time.
    *   *Strategy:* We implement a **Batch Refresh** mechanism. The denormalized tables store the state valid at `REFRESH_TIME`. Queries must specify the target time or default to `REFRESH_TIME`.

### C. Source of Truth Definition
To ensure correctness, we define the lineage explicitly in comments and documentation:
*   **Source of Truth:** `employees.employee`, `employees.department`, `employees.department_employee`, `employees.department_manager`, `employees.salary`, `employees.title`.
*   **Denormalized Artifact:** `employees.emp_dept_snapshot`.
*   **Correctness Rule:** The `valid_as_date` column dictates the truth. A query asking for "Current State" implies `valid_as_date = CURRENT_DATE`. A query asking for "State on 2022-01-01" must filter `WHERE valid_as_date = '2022-01-01'`.

## 4. Consistency Architecture & Refresh Plan

Since triggers are forbidden and CDC is not assumed in the provided DDL, we rely on a **Batch Refresh Job**.

### Refresh Logic (Pseudo-Code for Scheduler)
This job runs every 5 minutes. It calculates the "current" state (where `to_date > CURRENT_DATE` or `to_date IS NULL`) and upserts into the denormalized tables.

```sql
-- Pseudo-logic for a background job (e.g., via pg_cron or external script)
-- 1. Identify Current State Employees
WITH current_employees AS (
    SELECT e.id, e.birth_date, e.first_name, e.last_name, e.gender, e.hire_date
    FROM employees.employee e
    WHERE e.id IN (
        SELECT employee_id FROM employees.department_employee 
        WHERE department_id IN (SELECT id FROM employees.department)
    )
),
current_salaries AS (
    SELECT s.employee_id, s.amount
    FROM employees.salary s
    WHERE s.employee_id = (SELECT id FROM current_employees)
    AND (s.to_date > CURRENT_DATE OR s.to_date IS NULL)
),
current_titles AS (
    SELECT t.employee_id, t.title
    FROM employees.title t
    WHERE t.employee_id = (SELECT id FROM current_employees)
    AND (t.to_date > CURRENT_DATE OR t.to_date IS NULL)
),
current_depts AS (
    SELECT de.employee_id, de.department_id, d.dept_name
    FROM employees.department_employee de
    JOIN employees.department d ON de.department_id = d.id
    WHERE de.employee_id = (SELECT id FROM current_employees)
    AND (de.to_date > CURRENT_DATE OR de.to_date IS NULL)
),
current_managers AS (
    SELECT dm.employee_id, dm.department_id, d.dept_name
    FROM employees.department_manager dm
    JOIN employees.department d ON dm.department_id = d.id
    WHERE dm.employee_id = (SELECT id FROM current_employees)
    AND (dm.to_date > CURRENT_DATE OR dm.to_date IS NULL)
)
-- Upsert to Wide Read Table
INSERT INTO employees.emp_dept_snapshot (emp_id, dept_id, dept_name, title, salary_amount, valid_as_date, refresh_timestamp)
SELECT 
    ce.id,
    cd.department_id,
    cd.dept_name,
    ct.title,
    cs.amount,
    CURRENT_DATE,
    NOW()
FROM current_employees ce
JOIN current_salaries cs ON ce.id = cs.employee_id
JOIN current_titles ct ON ce.id = ct.employee_id
JOIN current_depts cd ON ce.id = cd.employee_id
LEFT JOIN current_managers cm ON ce.id = cm.employee_id
-- Handle multi-dept logic here (simplified for DDL clarity)
-- In production, use a CTE to merge multiple dept_ids per employee
ON CONFLICT (emp_id, valid_as_date) DO UPDATE 
    SET dept_name = EXCLUDED.dept_name, -- Handle conflict resolution strategy
        title = EXCLUDED.title,
        salary_amount = EXCLUDED.salary_amount;
```

### Handling Staleness
*   **Explicit Staleness:** All queries targeting `employees.emp_dept_snapshot` must be aware that the data is valid only up to `NOW() - 5 minutes`.
*   **Reconciliation:** A nightly job runs a diff query:
    ```sql
    SELECT 
        e.id,
        e.dept_name,
        s.dept_name as snapshot_dept_name,
        CASE WHEN e.dept_name != s.dept_name THEN 'MISMATCH' ELSE 'OK' END as status
    FROM employees.employee e
    JOIN employees.emp_dept_snapshot s ON e.id = s.emp_id
    WHERE e.id IN (SELECT employee_id FROM employees.department_employee)
    AND e.id NOT IN (SELECT emp_id FROM employees.emp_dept_snapshot WHERE valid_as_date = CURRENT_DATE);
    ```
    If mismatches occur, the source data is audited.

## 5. Risk Analysis & Mitigation

| Risk Category | Specific Risk | Impact | Mitigation Strategy |
| :--- | :--- | :--- | :--- |
| **Correctness** | **Staleness:** Users see outdated manager/salary info. | Medium | Document in API/Schema that data is T+5 min. Add a `last_refreshed` column to queries if dynamic freshness is critical. |
| **Consistency** | **Race Conditions:** Two writes happen simultaneously, corrupting the snapshot. | Low | Use `ON CONFLICT DO UPDATE` in the refresh job to handle atomic updates. |
| **Performance** | **Write Amplification:** Large `INSERT`/`UPDATE` volumes on `emp_dept_snapshot` during peak hours. | Medium | Limit refresh to off-peak hours (e.g., 02:00–04:00 UTC). Batch inserts if volume > 10k rows. |
| **Failure** | **Backfill Failure:** Job fails, leaving a gap in history. | Low | Idempotent design ensures partial failures don't corrupt data. Keep a `refresh_log` table to track job status. |
| **Storage** | **Growth:** Snapshot tables grow indefinitely. | Medium | Partition by year/month. Archive data older than 1 year to cold storage or drop partitions if policy allows. |

### Monitoring & Alerts
Define the following metrics for the dashboard:
1.  **Refresh Lag:** `NOW() - refresh_timestamp` (Alert if > 10 mins).
2.  **Reconciliation Diff Count:** Number of mismatches between source and snapshot (Alert if > 0).
3.  **Query Latency Regression:** Time taken for `SELECT * FROM emp_dept_snapshot` (Alert if > 100ms).

### Operational Complexity
*   **Reversibility:** The schema is reversible. If the denormalized approach causes issues, drop `emp_dept_snapshot` and revert to the normalized DDL. The source tables (`employee`, `department`, etc.) remain untouched.
*   **Minimal Changes:** Only two new tables are created. No existing tables are modified. Foreign keys in the source remain intact.
