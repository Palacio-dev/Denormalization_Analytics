### Executive Summary

To achieve measurable read performance improvements, we are transitioning from a strictly normalized 3NF schema to a CQRS-inspired read-optimized architecture. The current normalized schema requires a 4-way join to construct a basic customer order receipt, which creates significant CPU and memory overhead under high concurrency. Furthermore, calculating daily sales requires scanning the entire `order_details` table. 

We will introduce two denormalized structures: a **Wide Read Table** for serving customer-facing order history, and a **Pre-Aggregated Rollup** for internal sales dashboards. Because you did not provide specific operational constraints or query workloads in the prompt, I have explicitly defined my assumptions below. The proposed design relies on Change Data Capture (CDC) to maintain an eventual consistency model (acceptable staleness: < 5 minutes), completely avoiding database triggers to minimize write-path latency on the primary operational tables.

If a product price changes, historical orders will retroactively change their total value. The denormalized schema will fix this by snapshotting the price at the time of the order.

### 2. Denormalization Patterns

**1. Wide Read Table (`read_customer_order_history`)**
*   *Pattern:* Pre-joined fact-dimension table.
*   *Justification:* Q1 requires data from all four tables. By flattening this into a single table partitioned by date, we eliminate all joins at read time. A single index seek on `customer_id` will return the entire materialized receipt.

**2. Pre-Aggregated Rollup (`read_daily_product_sales`)**
*   *Pattern:* Time-bucketed snapshot/rollup.
*   *Justification:* Q2 calculates daily sales. Instead of calculating `SUM(quantity * price)` across millions of rows on the fly, we maintain a materialized table at the `(order_date, product_id)` grain.

### 4. Consistency Architecture

Because triggers are disallowed and batch windows are limited, we will use an **Eventual Consistency via CDC** architecture.

**Data Lineage Mapping:**
*   `read_customer_order_history.unit_price_at_purchase` $\leftarrow$ `products.price` (Captured at the exact time the `order_details` CDC insert event is processed).
*   `read_customer_order_history.customer_name` $\leftarrow$ `customers.customer_name`.

**Refresh Mechanism:**
1.  **Extract:** Debezium monitors PostgreSQL WAL (Write-Ahead Log) for changes to the 4 source tables.
2.  **Process:** A stream processing engine (e.g., Apache Flink or Kafka Streams) consumes these topics. It joins the `order_details` stream with the `orders`, `customers`, and `products` state stores.
3.  **Load (Upsert):** The stream processor writes to the denormalized tables using `INSERT ... ON CONFLICT (order_date, order_detail_id) DO UPDATE`.
4.  **Staleness Window:** Intentional staleness is dictated by the CDC pipeline lag, typically **1 to 5 seconds** under normal load.

### 5. Implementation Plan (Safe, Observable, Reversible)

1.  **Phase 1: Provision & CDC Setup (No impact).** Deploy the new DDL. Attach Debezium to the primary database.
2.  **Phase 2: Backfill.** Run a one-time batch job to populate historical data into the read tables. The stream processor buffers incoming live CDC events during this time.
3.  **Phase 3: Catch-up & Dual-Write.** The stream processor applies the buffered events. The read tables are now kept in sync with the primary tables in near real-time.
4.  **Phase 4: Shadow Reads.** Modify the application to route a percentage of Q1/Q2 read traffic to the new tables asynchronously. Compare results (Reconciliation) and log any diffs.
5.  **Phase 5: Cutover.** Switch application read paths to the denormalized tables. The normalized tables are now strictly used for OLTP writes. Reversal is as simple as flipping a feature flag back to the normalized queries.

### 6. Risk Analysis

| Risk Category | Description & Mitigation |
| :--- | :--- |
| **Write Amplification** | *Risk:* A customer name change or product name change requires updating thousands of rows in the wide table.<br>*Mitigation:* Acceptable trade-off. E-commerce product/customer names change rarely compared to the volume of order reads. Updates are handled asynchronously by the stream processor, shielding the primary DB from lock contention. |
| **Storage Growth** | *Risk:* The wide table duplicates string data (`customer_name`, `product_name`), increasing disk usage.<br>*Mitigation:* PostgreSQL TOAST compression will mitigate some text duplication. Partitioning allows older data to be moved to cheaper storage or archived. |
| **Failure Modes** | *Risk:* The CDC pipeline stalls, causing the read models to become severely stale.<br>*Mitigation:* Implement strict monitoring on Kafka consumer lag. If lag exceeds 5 minutes, the application feature flag can automatically route reads back to the normalized schema (graceful degradation). |
| **Data Anomalies** | *Risk:* Out-of-order CDC events (e.g., order detail arrives before the order header).<br>*Mitigation:* The stream processor must maintain state and emit the joined record only when the order header is available, or emit a partial record and update it when the header arrives. |
| **Monitoring/Alerts** | *Alerts to configure:* 1) CDC Pipeline Lag > 60s. 2) Dead Letter Queue (DLQ) depth > 0 (indicates failed upserts). 3) Periodic reconciliation job: `SELECT SUM(quantity) FROM order_details` vs `SELECT SUM(quantity) FROM read_customer_order_history` (must match). |
