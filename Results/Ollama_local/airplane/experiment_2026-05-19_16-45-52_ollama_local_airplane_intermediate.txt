### 1. Workload-Driven Analysis

Based on the provided schema, the current design suffers from the classic "Anomalies of Normalization" in a read-heavy environment. Here is the breakdown of the bottlenecks:

*   **The Join Explosion (The "N+1" Problem):**
    *   **Scenario:** A user requests a list of upcoming flights with full details (name, city, airline model, distance, seat availability).
    *   **Current Cost:** To get this, the database must perform a `JOIN` between `flights`, `airports_data` (2x), `aircrafts_data`, and potentially `seats` or `ticket_flights`.
    *   **Latency Impact:** Even with indexes, scanning multiple tables to stitch together a single logical entity (a flight with its context) adds up. If the query filters on `scheduled_departure` and `status`, the planner must still join to get the descriptive fields (`airport_name`, `model`).

*   **Frequent Repeated Joins:**
    *   **Scenario:** Reporting on "Total revenue by route" or "Flight status history".
    *   **Current Cost:** Aggregating `bookings` + `tickets` + `ticket_flights` + `flights` requires a massive join tree.
    *   **Latency Impact:** Complex aggregates are slow because the engine has to fetch raw rows, join them, compute sums, and then apply filters.

*   **Lack of Pre-computed Aggregates:**
    *   **Scenario:** "How many seats are available on Flight X?"
    *   **Current Cost:** Requires joining `flights` -> `ticket_flights` -> `boardings_passes` (or `seats`) and counting distinct values.
    *   **Latency Impact:** This is a read operation that is being turned into a write-heavy aggregation process.

*   **Missing Spatial Optimization:**
    *   **Scenario:** "Find all airports within 500km of London."
    *   **Current Cost:** Using `point` without a proper GIST index or a denormalized distance column forces a full table scan or an inefficient index scan.

### 2. Denormalization Design Strategy

To optimize for low-latency reads while preserving data integrity, we will adopt a **Star Schema** approach for reporting and a **Read-Optimized Physical Model** for transactional reads.

1.  **Denormalize Reference Data (`factless` tables):**
    *   **Strategy:** Duplicate `airport_name`, `city`, `model`, and `range` directly into the `flights` table.
    *   **Rationale:** Eliminates the most common join (`flights` -> `airports`/`aircrafts`). Since these are read-only reference data, duplicating them introduces zero correctness risk.

2.  **Create a "Flight Facts" Table (Denormalized Flight):**
    *   **Strategy:** Create a new table `flights_read_model` that includes all columns from `flights` plus the denormalized reference data.
    *   **Rationale:** This serves as the single source of truth for "What is this flight?" queries. It allows for `SELECT * FROM flights_read_model WHERE ...` without any joins.

3.  **Denormalize Booking Aggregates:**
    *   **Strategy:** Create a `bookings_read_model` table. Instead of joining `tickets`, `ticket_flights`, and `boardings_passes`, we pre-calculate the `total_amount` and passenger count per booking reference.
    *   **Rationale:** Reporting queries on `book_ref` become instant.

4.  **Spatial Optimization:**
    *   **Strategy:** Add a `distance_from_origin` column to `flights` (calculated on the fly or pre-computed if static origin is known, but for now, we rely on the `point` GIST index which is better than a denormalized distance column for dynamic queries). *Correction:* For extreme speed on "nearby airports", we can add a `h3_index` or simply rely on the GIST index on `coordinates` which is standard for PostGIS. We will optimize the `flights` table to ensure the `point` column is indexed.

5.  **Index Strategy:**
    *   **Composite Indexes:** Align indexes to the most frequent filter patterns (e.g., `status`, `scheduled_departure`, `flight_no`).
    *   **Covering Indexes:** For the `flights_read_model`, create a covering index on the most queried columns to avoid tuple fetches.

-- Populate the read model from the transactional flight table

CREATE OR REPLACE FUNCTION populate_flights_read_model()
RETURNS void AS $$
BEGIN
    INSERT INTO bookings.flights_read_model (
        flight_id, flight_no, scheduled_departure, scheduled_arrival, status,
        departure_airport, arrival_airport,
        airport_name, airport_city, departure_airport_coordinates, 
        arrival_airport_coordinates, aircraft_code, aircraft_model, aircraft_range
    )
    SELECT 
        f.flight_id, f.flight_no, f.scheduled_departure, f.scheduled_arrival, f.status,
        f.departure_airport, f.arrival_airport,
        a.airport_name, a.city, a.coordinates,
        a.coordinates, -- Reuse for simplicity or keep distinct if logic differs
        f.aircraft_code,
        ac.model, ac.range
    FROM bookings.flights f
    JOIN bookings.airports_data a ON f.departure_airport = a.airport_code
    JOIN bookings.airports_data b ON f.arrival_airport = b.airport_code
    JOIN bookings.aircrafts_data ac ON f.aircraft_code = ac.aircraft_code

    RETURN;
END;
$$ LANGUAGE plpgsql;

### Implementation Notes for the Engineer

1.  **Data Migration**:
    *   Run `populate_flights_read_model()`, `populate_bookings_read_model()`, and `populate_seats_read_model()` once at startup.
    *   These functions use `ON CONFLICT ... DO NOTHING` to handle duplicates safely if the load is run multiple times.

2.  **Query Pattern Changes**:
    *   **Old**: `SELECT f.*, a.name, ac.model FROM flights f JOIN airports a ...`
    *   **New**: `SELECT * FROM bookings.flights_read_model WHERE flight_no = 'BA123'`
    *   **Performance Gain**: The new query uses the composite index `idx_flights_read_model_flight_no` and returns rows immediately without touching the `airports_data` or `aircrafts_data` tables.

3.  **Write Performance**:
    *   The transactional tables (`flights`, `tickets`, etc.) remain unchanged. Writes are not impacted by denormalization because we are not adding columns to the primary transactional tables.
    *   The read tables are populated asynchronously or on startup. If real-time denormalization is needed, consider adding triggers that update `flights_read_model` whenever `flights` changes, but this adds write overhead. For now, the initial load is recommended.

4.  **Correctness**:
    *   Business-critical fields (amounts, dates, codes) are preserved in the transactional tables.
    *   The read tables act as a "shadow" copy. If a critical update occurs in the transactional table that hasn't propagated to the read table yet, the read model will be stale. For most read-heavy workloads, this latency is acceptable. If strict consistency is required, consider materialized views or application-level caching (Redis) for the most critical read tables.
